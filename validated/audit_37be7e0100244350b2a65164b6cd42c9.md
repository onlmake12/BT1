### Title
Subscription Balance Check in `updatePriceFeeds` Only Validates Against `pythFee` Before Deduction, Allowing Keeper Transactions to Revert After Gas Consumption — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `updatePriceFeeds` function checks that `status.balanceInWei >= pythFee` before deducting the Pyth fee and executing the expensive Pyth call. Only after that deduction does `_processFeesAndPayKeeper` check whether the remaining balance covers `totalKeeperFee`. If the remaining balance is insufficient, the transaction reverts — but only after the keeper has already consumed significant gas. A subscription manager can deliberately keep the balance just above `pythFee` but below `pythFee + totalKeeperFee`, causing keeper transactions to consistently fail and waste gas without compensation.

---

### Finding Description

In `updatePriceFeeds`:

```solidity
// Line 292-297: Only checks against pythFee
uint256 pythFee = pyth.getUpdateFee(updateData);
if (status.balanceInWei < pythFee) {
    revert SchedulerErrors.InsufficientBalance();
}

// Line 305-319: pythFee deducted and Pyth call made (expensive)
status.balanceInWei -= pythFee;
status.totalSpent += pythFee;
...
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(...);

// Line 345: keeper fee check happens AFTER all the above work
_processFeesAndPayKeeper(status, startGas, params.priceIds.length);
```

Inside `_processFeesAndPayKeeper`:

```solidity
// Line 846-854: totalKeeperFee calculated dynamically, then checked
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;

if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
```

The initial guard at line 295 only validates `balance >= pythFee`. It does **not** validate `balance >= pythFee + totalKeeperFee`. If `balance` is in the range `[pythFee, pythFee + totalKeeperFee)`, the initial check passes, the Pyth call is made (consuming gas and executing all validation logic), and then the transaction reverts at the keeper fee check. The keeper's gas is not refunded on revert in the EVM.

This is structurally identical to the btc-staking report: a value is checked against a threshold, a fee is subtracted, and the post-subtraction value can violate a second constraint — except here the second constraint is checked after expensive work has already been done. ---

### Impact Explanation

- Keepers are permissionless actors who call `updatePriceFeeds` and are compensated from the subscription balance.
- When a subscription balance satisfies `pythFee <= balance < pythFee + totalKeeperFee`, the keeper's transaction passes the initial check, executes the Pyth call and all validation logic, then reverts at the keeper fee check.
- The keeper loses all gas spent on the transaction with no compensation.
- A malicious subscription manager can deliberately maintain this balance range to grief keepers, making it economically irrational to service that subscription.
- This can also occur naturally as repeated updates drain the balance below the minimum, eventually landing in the vulnerable range. [4](#0-3) 

---

### Likelihood Explanation

- **Attacker-controlled entry path:** Any subscription manager (unprivileged role) can call `addFunds` to set the balance to exactly `pythFee + 1 wei`, or simply stop topping up the balance as it drains.
- **No privileged access required:** `updatePriceFeeds` is callable by any keeper; the subscription manager controls the balance.
- **Natural occurrence:** Even without malicious intent, as updates drain the balance, it will naturally pass through the vulnerable range before hitting zero.
- **Dynamic `totalKeeperFee`:** Because `gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice` is computed at runtime, keepers cannot pre-screen whether the balance will be sufficient after the pythFee deduction. [5](#0-4) 

---

### Recommendation

Perform a combined upfront balance check before deducting `pythFee` and before calling the Pyth contract. Since `totalKeeperFee` is partially dynamic, use a conservative lower bound for the keeper fee (e.g., `GAS_OVERHEAD * tx.gasprice + singleUpdateKeeperFeeInWei * numPriceIds`) for the pre-check:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
uint256 minKeeperFee = (GAS_OVERHEAD * tx.gasprice) +
    (uint256(_state.singleUpdateKeeperFeeInWei) * params.priceIds.length);

// Check combined minimum upfront
if (status.balanceInWei < pythFee + minKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}

status.balanceInWei -= pythFee;
// ... Pyth call ...
_processFeesAndPayKeeper(status, startGas, params.priceIds.length);
```

This mirrors the fix recommended in the btc-staking report: validate the post-deduction value against the constraint before performing the deduction. [6](#0-5) 

---

### Proof of Concept

1. Admin deploys `Scheduler` with `minimumBalancePerFeed = 0.01 ETH`, `singleUpdateKeeperFeeInWei = 0.0001 ETH`.
2. Subscription manager creates a subscription with 2 feeds, deposits exactly `minimumBalance = 0.02 ETH`.
3. After several updates, the balance drains to `pythFee + 1 wei` (e.g., `pythFee = 0.001 ETH`, balance = `0.001 ETH + 1 wei`).
4. Keeper calls `updatePriceFeeds`:
   - Line 295: `0.001 ETH + 1 wei >= 0.001 ETH` → check passes.
   - Line 305: `balance = 1 wei`.
   - Pyth call executes (gas consumed).
   - `_processFeesAndPayKeeper` calculates `totalKeeperFee ≈ gasCost + 0.0002 ETH >> 1 wei`.
   - Line 852: `1 wei < totalKeeperFee` → **revert**.
5. Keeper loses all gas. Subscription is not updated. Malicious manager repeats indefinitely. [7](#0-6) [4](#0-3)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
