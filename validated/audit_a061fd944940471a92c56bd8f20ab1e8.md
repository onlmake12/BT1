Audit Report

## Title
Expired Alert Replay via Missing Tombstone in `clear_expired_alerts` - (File: `util/network-alert/src/notifier.rs`)

## Summary
`Notifier::clear_expired_alerts` removes expired alerts from `received_alerts` and `noticed_alerts` using `retain`, but never records their IDs in `cancel_filter`. After expiry, `has_received(id)` returns `false` for those IDs, allowing any P2P peer that retained the original alert bytes (bearing valid developer signatures) to replay the alert. The P2P relay path in `AlertRelayer::received` performs no `notice_until` check, so the replayed alert passes all guards, is re-accepted, re-added to state, and re-broadcast to all connected peers.

## Finding Description

**Root cause — `clear_expired_alerts` does not tombstone expired IDs:**

`clear_expired_alerts` uses `retain` with `_id` (unused) and never calls `self.cancel_filter.put(*id, ())` for any expired entry. [1](#0-0) 

**Deduplication gate — `has_received` checks only two stores:**

After expiry, both `received_alerts.contains_key(&id)` and `cancel_filter.contains(&id)` return `false`, so `has_received` returns `false` and the alert ID is "unknown" again. [2](#0-1) 

**Test suite explicitly confirms the gap:**

After `clear_expired_alerts(after_expired_time)`, the existing test asserts `has_received(1)` is `false` and `received_alerts().len()` is 0 — directly confirming the ID is no longer tracked anywhere. [3](#0-2) 

**Trigger — `clear_expired_alerts` is called on every new peer connection:**

The `connected` handler calls `self.clear_expired_alerts()` before sending current alerts to the new peer. An attacker connects, triggering the cleanup, then immediately sends the retained expired alert bytes. [4](#0-3) 

**No `notice_until` check in the P2P relay path:**

`AlertRelayer::received` checks only UTF-8 validity, `has_received`, and signature validity. There is no `notice_until > now` guard. The replayed alert passes all three checks. [5](#0-4) 

**The RPC `send_alert` path does enforce this check**, but the P2P relay path does not, leaving the gap fully exploitable. [6](#0-5) 

**Contrast with the cancel path — which does tombstone:**

`cancel` correctly calls `self.cancel_filter.put(cancel_id, ())`, permanently recording the ID. `clear_expired_alerts` should do the same. [7](#0-6) 

**`cancel_filter` capacity is bounded at 128:**

Even if expired IDs were tombstoned, LRU eviction after 128 entries would eventually allow replay of very old IDs. This is a secondary weakness. [8](#0-7) 

## Impact Explanation

The accepting node calls `nc.quick_filter_broadcast` to all connected peers, propagating the replayed alert across the entire P2P network. Each peer that had cleared the alert also re-accepts and re-broadcasts it, creating a network-wide cascade. The attacker can repeat this on every reconnect with zero cost (no PoW, no stake, no key material). This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**. A secondary impact is alert ID squatting: after replay, `has_received(id)` returns `true` on all affected nodes, causing any subsequent legitimate alert with the same ID to be silently dropped, potentially suppressing a genuine emergency communication.

## Likelihood Explanation

- **Attacker requirement**: Any peer connected during the original broadcast retains the full alert bytes including valid developer signatures. No key material needs to be stolen or forged.
- **Trigger**: The attacker simply connects to a target node (triggering `clear_expired_alerts`) and sends the retained bytes over the P2P Alert protocol.
- **Persistence**: The attacker can repeat this on every reconnect to keep the expired alert alive indefinitely, or rely on the network-wide re-broadcast to propagate it without further effort.
- **No rate limiting or cost**: There is no proof-of-work, stake, or other cost to connecting and sending an alert message.

## Recommendation

In `clear_expired_alerts`, tombstone each expired alert ID into `cancel_filter` before removing it from `received_alerts`:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until > now {
            true
        } else {
            self.cancel_filter.put(*id, ()); // tombstone expired IDs
            false
        }
    });
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

Additionally, add a `notice_until > now` check in `AlertRelayer::received` to reject expired alerts at the P2P boundary, mirroring the check already present in the RPC path. Consider increasing `CANCEL_FILTER_SIZE` beyond 128 to prevent LRU eviction from re-opening old IDs.

## Proof of Concept

1. CKB core team broadcasts alert with `id=42`, `notice_until=T+1h`. All nodes receive and store it.
2. Time advances past `T+1h`. Alert expires.
3. Attacker (any previously connected peer) retains the original alert bytes.
4. Attacker connects to target node → `AlertRelayer::connected` fires → `clear_expired_alerts` removes alert 42 from `received_alerts` and `noticed_alerts`. `cancel_filter` is untouched.
5. Attacker sends the retained alert bytes over the Alert P2P protocol.
6. `AlertRelayer::received`: `has_received(42)` → `false` (cleared). Signatures valid. Alert accepted.
7. `notifier.add(&alert)`: alert 42 re-enters `received_alerts` and `noticed_alerts`.
8. Node calls `nc.quick_filter_broadcast` to all connected peers. Each peer that had cleared the alert repeats steps 4–8.
9. All nodes now display the expired alert. If the CKB team later sends a new alert with `id=42`, it is silently dropped on all nodes because `has_received(42)` returns `true`.

A minimal unit test to reproduce: extend `test_clear_expired_alerts` in `util/network-alert/src/tests/test_notifier.rs` — after `clear_expired_alerts(after_expired_time)`, call `notifier.add(&alert1)` again and assert `has_received(1)` returns `true` and `received_alerts().len()` equals 1. Under the current code, the re-add succeeds, confirming the replay.
