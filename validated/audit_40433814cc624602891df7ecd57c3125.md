### Title
`total_tx_size`/`total_tx_cycles` Inflated When Evictions Occur During `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the running totals `total_tx_size` and `total_tx_cycles` are computed as a pre-eviction snapshot before `check_and_record_ancestors` is called. If that function evicts transactions (via `remove_entry_and_descendants`), those evictions correctly decrement `self.total_tx_size`/`self.total_tx_cycles` — but the final two lines of `add_entry` then unconditionally overwrite those fields with the stale pre-eviction snapshot. The evicted transactions' sizes and cycles are never subtracted from the running totals, causing permanent inflation. An unprivileged transaction sender can craft a sequence of transactions that repeatedly triggers this path, inflating `total_tx_size` until the pool incorrectly believes it is full and rejects all new transactions.

---

### Finding Description

In `tx-pool/src/component/pool_map.rs`, `add_entry` executes as follows: [1](#0-0) 

```
(A) let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    // Snapshot: total_tx_size = self.total_tx_size + entry.size (LOCAL variable)

(B) evicts = self.check_and_record_ancestors(&mut entry)?;
    // May call remove_entry_and_descendants -> remove_entry -> update_stat_for_remove_tx
    // which DECREMENTS self.total_tx_size and self.total_tx_cycles

    ...

(C) self.total_tx_size = total_tx_size;   // OVERWRITES with pre-eviction snapshot
    self.total_tx_cycles = total_tx_cycles; // OVERWRITES with pre-eviction snapshot
```

At step (A), `updated_stat_for_add_tx` computes `total_tx_size = self.total_tx_size + entry.size` and stores it in a **local variable**. [2](#0-1) 

At step (B), `check_and_record_ancestors` may enter the eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`: [3](#0-2) 

Inside that branch, `remove_entry_and_descendants` is called, which calls `remove_entry` for each evicted transaction: [4](#0-3) 

`remove_entry` calls `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles`: [5](#0-4) 

At step (C), `self.total_tx_size = total_tx_size` **overwrites** the correctly-decremented `self.total_tx_size` with the stale pre-eviction value. The evicted transactions' sizes are effectively added back, inflating `total_tx_size` by exactly the total size of all evicted entries.

The `total_tx_size` field is the authoritative pool-size counter used to gate new transaction admission: [6](#0-5) 

---

### Impact Explanation

`total_tx_size` is the running total used to enforce `max_tx_pool_size`. When inflated, the pool believes it is fuller than it actually is. Once `total_tx_size` exceeds `max_tx_pool_size`, every subsequent call to `updated_stat_for_add_tx` returns `Reject::Full`, and the pool stops accepting any new transactions — even though the actual pool occupancy is well below the limit. This is a complete tx-pool denial-of-service: miners cannot fill blocks with new transactions, and users' submitted transactions are rejected.

---

### Likelihood Explanation

The eviction branch in `check_and_record_ancestors` is triggered when a submitted transaction has more than `max_ancestors_count` (default 25) in-pool ancestors, but the excess is entirely due to `cell_ref_parents` (in-pool transactions referenced as cell deps). An attacker can deliberately construct this scenario:

1. Submit 3 standalone transactions `tx_A`, `tx_B`, `tx_C`.
2. Submit a chain of 22 transactions `tx_E1 → tx_E2 → … → tx_E22`.
3. Submit `tx_F` that spends `tx_E22`'s output **and** uses `tx_A`, `tx_B`, `tx_C` as cell deps.
   - `ancestors = {tx_A, tx_B, tx_C, tx_E1, …, tx_E22}` → `ancestors_count = 26`
   - `cell_ref_parents = {tx_A, tx_B, tx_C}` → `26 − 3 = 23 ≤ 25`
   - Eviction branch fires; `tx_A` (and its descendants) are evicted.
   - `total_tx_size` is inflated by the evicted entries' sizes.

This requires no privileged access, no majority hashpower, and no social engineering. Any transaction sender reachable via the P2P relay or RPC `send_transaction` endpoint can execute this. The attacker can repeat the cycle to accumulate inflation until the pool is permanently DoS'd.

---

### Recommendation

In `add_entry`, do not capture the pre-eviction totals before calling `check_and_record_ancestors`. Instead, apply the new entry's contribution to `self.total_tx_size` and `self.total_tx_cycles` **after** all evictions have completed, so that evicted entries' decrements are not overwritten:

```rust
// After check_and_record_ancestors returns (evictions already applied):
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .ok_or_else(|| Reject::Full(...))?;
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .ok_or_else(|| Reject::Full(...))?;
```

Alternatively, recompute the totals from scratch after evictions using `recompute_total_stat` and then add the new entry's contribution.

---

### Proof of Concept

**Root cause path:**

1. `PoolMap::add_entry` (line 210–211): `total_tx_size = self.total_tx_size + entry.size` stored in local variable.
2. `PoolMap::check_and_record_ancestors` (line 618): `self.remove_entry_and_descendants(next_id)` → `remove_entry` → `update_stat_for_remove_tx` decrements `self.total_tx_size`.
3. `add_entry` (line 218): `self.total_tx_size = total_tx_size` overwrites the decremented value with the stale snapshot.

**Concrete trigger sequence:**

- Submit `tx_A`, `tx_B`, `tx_C` (standalone, no inputs from pool).
- Submit chain `tx_E1 → tx_E2 → … → tx_E22` (22 chained transactions).
- Submit `tx_F`: inputs spend `tx_E22`'s output; cell deps reference `tx_A`, `tx_B`, `tx_C`.
  - `ancestors_count = 26 > 25 = max_ancestors_count`
  - `cell_ref_parents = {tx_A, tx_B, tx_C}`, `26 − 3 = 23 ≤ 25`
  - Eviction fires; `tx_A` (and any descendants) removed; `self.total_tx_size` decremented.
  - Line 218 overwrites: `self.total_tx_size` inflated by `size(tx_A) + size(descendants)`.
- Repeat. After enough iterations, `total_tx_size > max_tx_pool_size`; pool rejects all new transactions with `Reject::Full`. [1](#0-0) [3](#0-2) [5](#0-4)

        &mut self,
        mut entry: TxEntry,
        status: Status,
    ) -> Result<(bool, HashSet<TxEntry>), Reject> {
        let tx_short_id = entry.proposal_short_id();
        let mut evicts = Default::default();
        if self.entries.get_by_id(&tx_short_id).is_some() {
            return Ok((false, evicts));

        let (total_tx_size, total_tx_cycles) =
            self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

        self.record_entry_edges(&entry)?;

        self.record_entry_descendants(&entry);
        self.track_entry_statics(None, Some(status));
        self.total_tx_size = total_tx_size;
        self.total_tx_cycles = total_tx_cycles;
        Ok((true, evicts))
    }

    /// Calculate size and cycles statistics for adding a tx.
    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
