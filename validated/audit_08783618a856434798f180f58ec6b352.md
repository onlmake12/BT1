### Title
Provider and Fee Manager Accrued Fees Permanently Frozen When Caller Cannot Receive ETH - (File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol)

---

### Summary

In `Entropy.sol`, both `withdraw()` and `withdrawAsFeeManager()` unconditionally send ETH to `msg.sender` with no option to specify an alternative recipient address. If the provider or fee manager is a smart contract that cannot receive ETH (no `receive()`/`fallback()` function, or one that reverts on receipt), all accrued fees are permanently inaccessible. The same pattern exists in `Echo.sol`'s `withdrawFees()` and `withdrawAsFeeManager()`.

---

### Finding Description

**`Entropy.withdraw()`** deducts `accruedFeesInWei` and sends ETH exclusively to `msg.sender`:

```solidity
providerInfo.accruedFeesInWei -= amount;
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "withdrawal to msg.sender failed");
``` [1](#0-0) 

**`Entropy.withdrawAsFeeManager()`** has the identical pattern — the fee manager's withdrawal also goes exclusively to `msg.sender`:

```solidity
providerInfo.accruedFeesInWei -= amount;
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "withdrawal to msg.sender failed");
``` [2](#0-1) 

Neither function accepts a `recipient` parameter. There is no mechanism in the contract to transfer a provider's `accruedFeesInWei` balance to a different address.

The same issue exists in `Echo.sol`:

- `withdrawFees()` sends Pyth protocol fees to `msg.sender` (admin) only.
- `withdrawAsFeeManager()` sends provider fees to `msg.sender` (fee manager) only. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

If a provider or fee manager is a smart contract address that cannot receive ETH (e.g., a multisig that gets bricked, a contract without a `receive()` function, or a contract whose fallback reverts), every call to `withdraw()` or `withdrawAsFeeManager()` will revert at the `require(sent, ...)` check. Because the ETH transfer failure causes a full revert, the state change is also rolled back — but the provider has no alternative path to retrieve their fees. The `accruedFeesInWei` balance remains in the contract indefinitely with no recovery mechanism.

For `Entropy.withdraw()`, a partial mitigation exists: the provider can set a fee manager via `setFeeManager()` and have that EOA call `withdrawAsFeeManager()`. However, `withdrawAsFeeManager()` also hardcodes `msg.sender` as recipient, so if the fee manager is also a non-receiving contract, the same freeze applies.

For `Echo.withdrawFees()`, there is no mitigation at all — the admin address is the sole recipient and there is no admin-equivalent of `withdrawAsFeeManager()`. [5](#0-4) 

---

### Likelihood Explanation

Providers and fee managers are often operated as smart contracts (multisigs, proxy contracts, keeper bots). A contract that registers as a provider or fee manager may later become unable to receive ETH due to an upgrade, self-destruct, or simply because it was deployed without a `receive()` function. This is a low-probability but realistic operational scenario, matching the severity classification of the reference report.

---

### Recommendation

Add a `recipient` parameter to `withdraw()`, `withdrawAsFeeManager()`, and `Echo.withdrawFees()` / `Echo.withdrawAsFeeManager()`, so the caller can direct funds to any address they control:

```solidity
// Entropy.sol
function withdraw(uint128 amount, address recipient) public override {
    ...
    (bool sent, ) = recipient.call{value: amount}("");
    require(sent, "withdrawal failed");
    emit EntropyEventsV2.Withdrawal(msg.sender, recipient, amount, bytes(""));
}

function withdrawAsFeeManager(address provider, uint128 amount, address recipient) external override {
    ...
    (bool sent, ) = recipient.call{value: amount}("");
    require(sent, "withdrawal failed");
}
```

The same change should be applied to `Echo.withdrawFees(address recipient, uint128 amount)` and `Echo.withdrawAsFeeManager(address provider, uint128 amount, address recipient)`.

Note that `EntropyGovernance.withdrawFee()` already implements this pattern correctly (it accepts a `targetAddress` parameter), confirming the fix is straightforward. [6](#0-5) 

---

### Proof of Concept

1. Deploy a contract `ProviderContract` with no `receive()` function.
2. Call `Entropy.register(...)` from `ProviderContract` — it becomes a registered provider.
3. Users request randomness from `ProviderContract`, causing `providerInfo.accruedFeesInWei` to accumulate.
4. `ProviderContract` calls `Entropy.withdraw(amount)`.
5. The ETH transfer `msg.sender.call{value: amount}("")` fails because `ProviderContract` has no `receive()`.
6. The transaction reverts. `accruedFeesInWei` is unchanged.
7. There is no alternative function to redirect the fees to a different address.
8. All accrued fees are permanently frozen in the `Entropy` contract. [5](#0-4) [7](#0-6)
