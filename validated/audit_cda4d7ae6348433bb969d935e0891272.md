Audit Report

## Title
`forward_rate_limiter` Bypassed via Attacker-Controlled `from`/`to` Fields — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

The `HolePunching` protocol's `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` values taken directly from the attacker-supplied message body, with no verification that `content.from` matches the actual sending session's peer ID. An attacker can trivially bypass this limiter by cycling through distinct fabricated `(from, to)` pairs within the 30 req/s outer limit, causing the node to forward an unbounded number of hole-punching messages per second and grow the rate-limiter hashmap without bound.

## Finding Description

**Outer limiter (correct):** In `network/src/protocols/hole_punching/mod.rs`, the outer limiter is keyed by the real, non-spoofable `session_id`:

```rust
if self.rate_limiter.check_key(&(session_id, msg.item_id())).is_err() { … }
``` [1](#0-0) 

This correctly caps each session at 30 req/s.

**Inner limiter (broken):** The `forward_rate_limiter` is keyed by attacker-controlled fields in `ConnectionRequestProcess::execute()`:

```rust
self.protocol.forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
``` [2](#0-1) 

The same broken pattern appears in `ConnectionRequestDeliveredProcess::execute()`: [3](#0-2) 

And in `ConnectionSyncProcess::execute()`: [4](#0-3) 

**No `from` verification:** After parsing `content`, the code checks `listen_addrs` length, `max_hops`, `route` length, and whether the local node is already in the route — but there is no check that `content.from` equals the peer ID of the actual sending session. [5](#0-4) 

**Exploit path:**
1. Attacker connects as a normal P2P peer.
2. Attacker sends 30 `ConnectionRequest` messages/second (outer limiter allows this).
3. Each message carries a distinct fabricated `(from, to)` pair.
4. Each unique pair creates a fresh bucket in the `forward_rate_limiter` hashmap — all 30 pass the inner check.
5. For each message where `content.to` is not the local node and `max_hops > 0`, the node calls `forward_message()`, which broadcasts to `sqrt(total_peers)` connected peers. [6](#0-5) 

**Unbounded hashmap growth:** `retain_recent()` is called only in `disconnected()`, not in the periodic `notify()` handler, so the `forward_rate_limiter` hashmap grows at 30 entries/second per attacker connection indefinitely during the session. [7](#0-6) 

The comment at the rate limiter setup explicitly states the intent — "the same group of from/to should not be received by the same node more than 1 times within one second" — which is completely defeated: [8](#0-7) 

## Impact Explanation

**Impact: High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Each attacker message that passes the outer limiter causes the node to broadcast to `sqrt(N)` peers (e.g., with 100 peers: 10 forwarded messages per attacker message). At 30 messages/second per attacker connection, this yields ~300 forwarded messages/second per connection. With multiple attacker connections, the forwarding load multiplies linearly. Additionally, the `forward_rate_limiter` hashmap grows at 30 entries/second per connection without bound, consuming memory over the lifetime of the session. Sustained across multiple connections, this constitutes a realistic network congestion and resource exhaustion attack.

## Likelihood Explanation

- **Preconditions:** Only a standard P2P connection to the HolePunching protocol is required — no keys, stake, or privileged role.
- **Ease:** Crafting molecule-encoded messages with varying `from`/`to` byte sequences is straightforward with any P2P client.
- **Repeatability:** The attack is fully sustainable; the outer limiter (30 req/s) is the only real bound, and the inner limiter is completely neutralized.
- **Detection:** Since the inner limiter is never triggered, no warning-level log is emitted; only debug-level logs appear on outer-limiter hits.

## Recommendation

1. **Verify `content.from` against the actual session:** After parsing, look up the peer ID for the sending `session_id` from the peer registry and assert it equals `content.from`. Reject (and ban) messages where they do not match.
2. **Re-key `forward_rate_limiter` by session:** Use `(peer_index, msg_item_id)` — the same key as the outer limiter — so the forwarding budget is tied to the real sender, not attacker-supplied fields.
3. **Periodically prune the hashmap:** Call `forward_rate_limiter.retain_recent()` inside the `notify()` handler (every `CHECK_INTERVAL`) rather than only on disconnect, to bound memory growth regardless of session duration.

## Proof of Concept

```
1. Connect to a CKB node on the HolePunching protocol (SupportProtocols::HolePunching).
2. In a loop at ≤30 iterations/second (to stay under the outer limiter):
   a. Generate a fresh random PeerId for `from` (or increment a counter).
   b. Build a valid ConnectionRequest molecule message:
        from = <fresh_random_peer_id>
        to   = <any valid peer id, e.g. random>
        max_hops = 6
        listen_addrs = [<any valid TCP multiaddr with from's peer id appended>]
        route = []
   c. Send it to the node.
3. Observe: the node forwards each message to sqrt(N) other peers
   (no TooManyRequests is returned for the inner limiter).
4. After T seconds, the node's forward_rate_limiter hashmap contains T*30 entries,
   and T*30*sqrt(N) forwarding sends have been issued to other peers.
5. Confirm: repeat with a second attacker connection to double the forwarding load.
```
