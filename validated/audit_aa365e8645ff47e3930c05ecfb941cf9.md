### Title
Missing Per-Peer Rate Limiting on `GetBlockFilterCheckPoints` Enables High-Amplification DoS — (`sync/src/filter/get_block_filter_check_points_process.rs`)

### Summary
The Filter protocol handler (`BlockFilter`) has no rate limiter. Any peer can repeatedly send a 9-byte `GetBlockFilterCheckPoints{start_number:0}` message, causing the server to execute up to 4 000 RocksDB point-lookups and return a ~64 KB response per message, with no per-peer throttle of any kind.

### Finding Description

`GetBlockFilterCheckPointsProcess::execute` iterates up to `BATCH_SIZE = 2000` times, advancing by `CHECK_POINT_INTERVAL = 2000` blocks per step. Each iteration issues two store reads:

```
get_block_hash(block_number)          // snapshot read
get_block_filter_hash(&block_hash)    // ChainDB/RocksDB read
``` [1](#0-0) [2](#0-1) 

The only guard before the loop is `latest >= start_number`. With `start_number = 0`, this is satisfied as soon as genesis filter data exists, because `get_latest_built_filter_block_number` returns `unwrap_or_default()` (i.e., 0) and `0 >= 0` is always true. [3](#0-2) [4](#0-3) 

The `BlockFilter` struct carries only `shared: Arc<SyncShared>` — no rate limiter field exists anywhere in `sync/src/filter/`: [5](#0-4) 

`BlockFilter::try_process` dispatches directly to the handler with zero rate-check: [6](#0-5) 

Compare with `Relayer`, which has an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` and rejects any non-CompactBlock message that exceeds 30 req/s per (peer, message_type): [7](#0-6) [8](#0-7) 

Also note that `GetBlockFiltersProcess` has an explicit 1.8 MB response-size guard to prevent oversized frames, but `GetBlockFilterCheckPointsProcess` has no analogous work or size cap: [9](#0-8) 

### Impact Explanation

| Dimension | Value |
|---|---|
| Request size | ~9 bytes (1-byte union tag + 8-byte `Uint64` struct) |
| DB reads triggered | up to 4 000 (2 000 × `get_block_hash` + 2 000 × `get_block_filter_hash`) |
| Response size | up to 2 000 × 32 B = **64 KB** |
| Rate limit | **none** |

A single attacker peer can sustain a continuous stream of these messages. On a node with a long chain (≥ 4 000 000 blocks), every message saturates the full 2 000-iteration loop. Multiple attacker peers multiply the effect linearly. The result is sustained RocksDB I/O exhaustion and outbound bandwidth amplification, degrading or halting normal block/transaction sync for honest peers — matching the "high-amplification DoS / network congestion" scope.

### Likelihood Explanation

The Filter protocol (`/ckb/filter`, protocol ID 121) is a standard supported protocol. Any peer that negotiates it can send `GetBlockFilterCheckPoints` without authentication, PoW, or any prior state. The message is a fixed 9-byte struct with no variable-length fields, so it cannot be rejected by the frame-size limit. The precondition (genesis filter built) is met on every node that has the filter feature enabled. This is trivially reachable from the internet.

### Recommendation

Add a per-peer, per-message-type rate limiter to `BlockFilter`, mirroring the pattern already used in `Relayer`:

1. Add `rate_limiter: RateLimiter<(PeerIndex, u32)>` to the `BlockFilter` struct.
2. In `try_process`, check `rate_limiter.check_key(&(peer, message.item_id()))` before dispatching, returning `StatusCode::TooManyRequests` on failure.
3. A quota of 1–2 `GetBlockFilterCheckPoints` requests per second per peer is sufficient for legitimate light-client use.

### Proof of Concept

```
# On a node with filter data built (any mainnet/testnet node):
# 1. Connect as a peer on /ckb/filter (protocol 121)
# 2. In a tight loop, send:
#    BlockFilterMessage::GetBlockFilterCheckPoints { start_number: 0 }
#    (encoded: union tag 0x04 + 8 zero bytes = 9 bytes total)
# 3. Observe: each message causes ~4000 RocksDB reads and a ~64 KB response
# 4. No rate-limit error is ever returned; the node processes every message
# 5. Monitor node I/O: rocksdb.block.cache.miss / disk reads spike proportionally
# 6. Repeat from multiple peers to multiply the effect
``` [10](#0-9)

    pub async fn execute(self) -> Status {
        let active_chain = self.filter.shared.active_chain();
        let start_number: BlockNumber = self.message.to_entity().start_number().into();
        let latest: BlockNumber = active_chain.get_latest_built_filter_block_number();

        let mut block_filter_hashes = Vec::new();

        if latest >= start_number {
            let mut block_number = start_number;
            for _ in 0..BATCH_SIZE {
                if let Some(block_filter_hash) = active_chain
                    .get_block_hash(block_number)
                    .and_then(|block_hash| active_chain.get_block_filter_hash(&block_hash))
                {
                    block_filter_hashes.push(block_filter_hash);
                } else {
                    break;
                }
                let Some(next_block_number) = block_number.checked_add(CHECK_POINT_INTERVAL) else {
                    break;
                };
                block_number = next_block_number;
            }
            let content = packed::BlockFilterCheckPoints::new_builder()
                .start_number(start_number)
                .block_filter_hashes(block_filter_hashes)
                .build();

            let message = packed::BlockFilterMessage::new_builder()
                .set(content)
                .build();
            async_send_message_to(&self.nc, self.peer, &message).await
        } else {
            Status::ignored()
        }
    }
