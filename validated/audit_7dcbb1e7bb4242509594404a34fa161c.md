### Title
Keeper Can Drain Subscription Funds via Inflated `tx.gasprice` in `updatePriceFeeds` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `Scheduler.sol` contract pays keepers based on `tx.gasprice` with no upper bound. Because `updatePriceFeeds` is permissionless, any caller can set an arbitrarily high gas price and drain a subscription's entire ETH balance in a single transaction, stealing funds deposited by subscription managers.

### Finding Description
In `Scheduler.sol`, `updatePriceFeeds` is a fully permissionless external function — no access control modifier restricts who may call it. [1](#0-0) 

After parsing price feeds, it calls `_processFeesAndPayKeeper`, which computes the keeper payment as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [2](#0-1) 

There is **no cap on `tx.gasprice`**. The only guard is that `totalKeeperFee` must not exceed `status.balanceInWei`: [3](#0-2) 

The keeper fee is then sent directly to `msg.sender`: [4](#0-3) 

An attacker can therefore:
1. Observe any active subscription with a large `balanceInWei`.
2. Obtain valid `updateData` satisfying the subscription's update criteria (heartbeat or deviation — both are observable on-chain).
3. Call `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice` set to a value large enough that `gasCost` alone equals `status.balanceInWei`.
4. Receive the entire subscription balance as "keeper fee."

The subscription manager deposited ETH expecting it to be consumed gradually by legitimate price updates, but the attacker drains it entirely in one transaction.

### Impact Explanation
Subscription managers lose all deposited ETH. The `balanceInWei` field tracks user-deposited funds: [5](#0-4) 

A single malicious `updatePriceFeeds` call with inflated `tx.gasprice` can set `status.balanceInWei` to zero, transferring all funds to the attacker. This is a direct theft of user principal, not merely a fee overcharge.

### Likelihood Explanation
- `updatePriceFeeds` has no access control; any EOA or contract can call it.
- Setting `tx.gasprice` to an arbitrary value is trivial on any EVM chain.
- Valid `updateData` satisfying update criteria is publicly available from Pyth's price feed infrastructure.
- No special role, leaked key, or governance majority is required.

### Recommendation
Enforce a maximum acceptable `tx.gasprice` in `_processFeesAndPayKeeper`. For example, add a configurable `maxGasPriceInWei` state variable and cap the calculation:

```solidity
uint256 effectiveGasPrice = tx.gasprice < _state.maxGasPriceInWei
    ? tx.gasprice
    : _state.maxGasPriceInWei;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, cap `totalKeeperFee` to a reasonable multiple of the minimum balance per feed.

### Proof of Concept

```solidity
// Attacker contract
contract DrainSubscription {
    IScheduler scheduler;

    constructor(address _scheduler) {
        scheduler = IScheduler(_scheduler);
    }

    // Called with tx.gasprice set to a very large value
    function drain(uint256 subscriptionId, bytes[] calldata updateData) external {
        // updateData is valid data satisfying the subscription's update criteria
        // tx.gasprice is set by the attacker to e.g. 1e18 wei/gas
        // gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice
        // With tx.gasprice = 1e18, even 1 gas unit of overhead yields enormous fees
        scheduler.updatePriceFeeds(subscriptionId, updateData);
        // msg.sender (this contract) receives totalKeeperFee ≈ subscriptionBalance
    }

    receive() external payable {}
}
```

Steps:
1. Deploy `DrainSubscription`.
2. Identify a subscription with `balanceInWei = B`.
3. Compute required `tx.gasprice` such that `GAS_OVERHEAD * tx.gasprice >= B`.
4. Call `drain(subscriptionId, validUpdateData)` with that gas price.
5. The subscription balance is transferred to the attacker contract. [6](#0-5)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
