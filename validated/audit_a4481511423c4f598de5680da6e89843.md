### Title
PostgreSQL-only `row_lock`/`row_type` variable confusion in `script_exists_in_output` silently deletes type-script-only scripts during rollback — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

A copy-paste error at line 252 of `script_exists_in_output` causes the function to re-evaluate `row_lock` instead of `row_type` when checking whether a script is referenced as a `type_script_id` in any surviving output. On PostgreSQL this always returns `false` for the type-script check, so any script that is used exclusively as a `type_script` (never as a `lock_script`) is incorrectly classified as unreferenced and deleted from the `script` table during every `rollback_block` call. The Nervos DAO system script is the canonical example: it is always attached as a `type_script`, never as a `lock_script`, so a single natural reorg that touches any DAO-bearing block permanently removes it from the indexer's script table, making every DAO deposit cell invisible to all subsequent `get_cells` queries.

---

### Finding Description

`script_exists_in_output` performs two sequential SQL `EXISTS` queries:

1. **`row_lock`** — checks `WHERE lock_script_id = $1`
2. **`row_type`** — checks `WHERE type_script_id = $1`

The function returns early (`Ok(true)`) if `row_lock` is true. If not, it fetches `row_type` and then evaluates the result. The bug is at line 252:

```rust
// line 237-249: row_type is fetched correctly
let row_type = sqlx::query( ... WHERE type_script_id = $1 ... )
    .fetch_one(tx.as_mut()).await ...;

// line 252: BUG — row_lock is read again, not row_type
match row_lock.try_get::<bool, _>(0) {   // <-- should be row_type
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

**PostgreSQL path:** `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL `EXISTS` returns `BOOLEAN`). Because the early-return guard at line 223–227 already confirmed `row_lock` is `false`, this second evaluation of `row_lock` also returns `Ok(false)`. The freshly-fetched `row_type` result is never consulted. The function returns `false` regardless of whether the script exists as a `type_script_id` in thousands of other outputs. [2](#0-1) 

**SQLite path:** `row_lock.try_get::<bool, _>(0)` returns `Err` (SQLite returns `BIGINT`), so the `Err` arm falls through to `row_type.get::<i64, _>(0) == 1`, which correctly reads `row_type`. SQLite is unaffected.

The caller `rollback_block` deletes all outputs of the rolled-back block **before** calling `script_exists_in_output`:

```rust
remove_batch_by_blobs("output", "tx_id", &tx_id_list, tx).await?;  // line 25
// ...
for (_, lock_script_id, type_script_id) in output_lock_type_list {
    if !script_exists_in_output(lock_script_id, tx).await? { ... }
    if let Some(type_script_id) = type_script_id
        && !script_exists_in_output(type_script_id, tx).await? {
        script_id_list_to_remove.push(type_script_id);  // line 36
    }
}
remove_batch_by_blobs("script", "id", &script_id_list_to_remove, tx).await?;  // line 39
``` [3](#0-2) 

So the check is correctly scoped to surviving outputs — but the bug makes it always return `false` on PostgreSQL for the type-script branch, causing the script row to be unconditionally deleted.

---

### Impact Explanation

The DAO system script (`0x82d76d1b75fe2fd9a27dfbaa65a039221a380d76a978e6cf3c1ac4b77f4f5d43`) is attached as a `type_script` on every DAO deposit and withdrawal cell. It is never used as a `lock_script`. After any reorg on a PostgreSQL-backed rich-indexer node:

- The DAO system script's row is deleted from the `script` table.
- All `get_cells` / `get_cells_capacity` RPC calls that filter by the DAO type script hash return zero results.
- Wallets and tooling built on the indexer see no DAO deposits, cannot construct withdrawal transactions, and cannot claim matured withdrawals.
- The damage persists until the indexer is fully re-synced from genesis; there is no self-healing path.

The same class of breakage applies to any other system script or user-defined script that is exclusively used as a `type_script` (e.g., xUDT, SUDT, Spore).

---

### Likelihood Explanation

- Reorgs are a **normal, unprivileged, zero-attacker event** on any live chain. Even a 1-block natural reorg is sufficient.
- The only precondition is that the rolled-back block contains at least one output whose `type_script` is also referenced by outputs in earlier, surviving blocks — an extremely common condition for any widely-used system script.
- The bug is PostgreSQL-specific and silent: no error is returned, no log warning is emitted, the deletion succeeds, and the node continues operating with a corrupted index.

---

### Recommendation

Change line 252 from `row_lock` to `row_type`:

```rust
// BEFORE (buggy)
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}

// AFTER (correct)
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

Additionally, add a deduplication step before `script_id_list_to_remove` is passed to `remove_batch_by_blobs` to avoid double-deletion when the same script appears in multiple outputs of the same rolled-back block.

---

### Proof of Concept

1. Start a PostgreSQL-backed CKB rich-indexer node.
2. Index several blocks containing DAO deposit cells (outputs with the DAO system script as `type_script`).
3. Index one additional block containing a DAO withdrawal transaction (also uses the DAO system script as `type_script`).
4. Trigger `rollback_block` (e.g., via a natural or forced reorg that removes the withdrawal block).
5. Query `get_cells` with `script = <DAO type script hash>`, `script_type = "type"`.
6. **Expected:** all DAO deposit cells are returned.
7. **Actual (PostgreSQL):** zero results — the DAO system script row has been deleted from the `script` table, so the join in `get_cells` finds no matching script and returns nothing.
