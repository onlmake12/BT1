### Title
Permanent Locking of All Protocol Update Fees — No Fee Withdrawal Path Exists for `FEE_STORAGE_KEY` — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
All SUI fees paid by users via `update_single_price_feed` are deposited into a dynamic object field (`FEE_STORAGE_KEY`) on each `PriceInfoObject` and are permanently irrecoverable. No function in any module withdraws or transfers coins from `FEE_STORAGE_KEY`. The `fee_recipient_address` stored in `State` is explicitly acknowledged in the code as unused. Additionally, `deposit_fee_coins` is `public` with no bounds, allowing any caller to permanently lock arbitrary SUI into any `PriceInfoObject`.

### Finding Description

**1. `deposit_fee_coins` is unconditionally public with no bounds:** [1](#0-0) 

The function is `public` (not `public(friend)`), accepts any `Coin<SUI>` of any value, and either initializes or `coin::join`s into the `FEE_STORAGE_KEY` dynamic object field. There is no minimum, maximum, or caller restriction.

**2. Legitimate update fees are deposited via the same path:** [2](#0-1) 

Every call to `update_single_price_feed` deposits the fee coin directly into the `PriceInfoObject` via `price_info::deposit_fee_coins`. These fees accumulate and are never moved out.

**3. No withdrawal function exists anywhere:**

A search across all `.move` files under `target_chains/sui/contracts/sources/` for `withdraw`, `dynamic_object_field::remove`, `coin::split`, `coin::take`, or any transfer from `FEE_STORAGE_KEY` returns **zero matches**. The only operations on `FEE_STORAGE_KEY` are `add`, `borrow`, and `borrow_mut` (for `coin::join`). [3](#0-2) 

**4. `fee_recipient_address` is stored but explicitly acknowledged as unused:** [4](#0-3) 

The module comment states: *"The previous version of the contract sent the fees to a recipient address but this state is not used anymore. This module is kept for backward compatibility."* The `get_fee_recipient` getter exists in `state.move` but is never called in any fee-collection or fee-transfer logic. [5](#0-4) 

**5. `fee_recipient_address` is set but never referenced in fee flow:** [6](#0-5) 

The field exists in `State`, is settable via governance, but is never read during fee deposit or any transfer operation.

### Impact Explanation

- **All legitimately paid update fees** (paid by every integrator/dApp calling `update_single_price_feed`) are permanently locked inside `PriceInfoObject` shared objects with no recovery path. The protocol earns zero revenue despite enforcing a fee.
- **Any unprivileged attacker** can call `deposit_fee_coins` directly (it is `public`) with dust amounts, permanently locking their own SUI into any `PriceInfoObject`. While this is self-harm, it also permanently inflates the locked balance with no recourse.
- The `fee_recipient_address` governance mechanism is vestigial — it can be set but has no effect on fund recovery.

### Likelihood Explanation

This is not a conditional or theoretical path. Every single call to `update_single_price_feed` on mainnet locks fees permanently. The missing withdrawal function is a confirmed, unconditional absence across the entire codebase.

### Recommendation

1. Add a `withdraw_fees` function (gated by `LatestOnly` or a capability held by `fee_recipient_address`) that calls `dynamic_object_field::remove` on `FEE_STORAGE_KEY` and `transfer::public_transfer`s the coin to `state::get_fee_recipient(pyth_state)`.
2. Restrict `deposit_fee_coins` to `public(friend)` so only `pyth::pyth` can call it during legitimate updates.
3. Remove or repurpose the now-vestigial `set_fee_recipient` governance action if fee collection is intentionally disabled, or restore the fee-forwarding logic.

### Proof of Concept

```
// Anyone can call this directly — no signer check, no minimum
price_info::deposit_fee_coins(&mut price_info_object, coin::split(&mut my_coins, 1, ctx));
// Coins are now in FEE_STORAGE_KEY. No function exists to remove them.
// Confirmed: grep for "FEE_STORAGE_KEY" returns only price_info.move (11 matches),
// all of which are add/borrow/join — never remove or transfer.
```

The invariant that "fees paid to the protocol are claimable by `fee_recipient_address`" is broken unconditionally. All update fees paid since deployment are permanently locked.

            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);

                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
