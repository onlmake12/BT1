### Title
Ignored Return Value of `async_send_message_to` Leaves Compact Block Reconstruction in Broken State - (`sync/src/relayer/block_transactions_process.rs`)

### Summary

In `BlockTransactionsProcess::execute`, after a compact block reconstruction attempt fails due to missing transactions, the node sends a `GetBlockTransactions` follow-up request to the peer. The return value of `async_send_message_to` is explicitly discarded with `let _ignore`, but the in-memory state tracking which transactions are expected is unconditionally updated immediately afterward. If the send fails, the node believes it has requested specific missing transactions from the peer, but no request was actually delivered. The compact block reconstruction is permanently stalled for that block hash until a timeout, with no retry or error propagation.

### Finding Description

In `sync/src/relayer/block_transactions_process.rs`, the `execute` method handles incoming `BlockTransactions` relay messages. When block reconstruction is incomplete (`ReconstructionResult::Missing` or `ReconstructionResult::Collided`), the node builds a `GetBlockTransactions` message and attempts to send it back to the peer:

```rust
let _ignore = async_send_message_to(&self.nc, self.peer, &message).await;

let _ignore_prev_value =
    mem::replace(expected_transaction_indexes, missing_transactions);
let _ignore_prev_value = mem::replace(expected_uncle_indexes, missing_uncles);
``` [1](#0-0) 

The send result is discarded. Immediately after, `expected_transaction_indexes` and `expected_uncle_indexes` — the fields inside the `pending_compact_blocks` map entry for this block hash — are overwritten with the new missing set. This state update is unconditional: it happens regardless of whether the network send succeeded or failed.

If `async_send_message_to` returns an error (e.g., the peer's send buffer is full, the session was closed, or the underlying channel is at capacity), the peer never receives the `GetBlockTransactions` request. However, the node's internal state now records the new missing indexes as "pending" for this peer. No further request will be issued for those indexes because the state has already been advanced. The compact block entry remains in `pending_compact_blocks` with stale expected indexes, and the block cannot be reconstructed until the entry expires.

### Impact Explanation

An unprivileged relay peer can send a `BlockTransactions` message that triggers the missing-transaction path. If the network send fails at that moment (due to congestion, a full send buffer, or a race with session teardown), the node silently loses the ability to reconstruct that compact block from that peer. The block propagation for that block hash is stalled on this node. In a scenario where the attacker deliberately causes send failures (e.g., by flooding the relay channel), this can delay or prevent a node from accepting newly mined blocks, degrading its view of the chain tip and potentially affecting mining or fee estimation.

### Likelihood Explanation

The `async_send_message_to` utility can fail when the underlying P2P send channel is full or the session has been torn down between the time the `BlockTransactions` message was received and the time the follow-up is sent. This is a realistic race condition under normal network load, not a purely theoretical scenario. Any peer that relays compact blocks can trigger this path without any special privileges.

### Recommendation

Check the return value of `async_send_message_to` before updating `expected_transaction_indexes` and `expected_uncle_indexes`. If the send fails, either:
- Do not update the expected indexes (so the old request state is preserved and a retry can be attempted), or
- Log an error and return an appropriate `Status` so the caller can handle the failure.

```rust
if async_send_message_to(&self.nc, self.peer, &message).await.is_ok() {
    let _ignore_prev_value =
        mem::replace(expected_transaction_indexes, missing_transactions);
    let _ignore_prev_value = mem::replace(expected_uncle_indexes, missing_uncles);
} else {
    warn!("Failed to send GetBlockTransactions to peer {}", self.peer);
    return StatusCode::Network.with_context(block_hash);
}
```

A secondary instance of the same pattern exists in `network/src/peer_store/peer_store_db.rs` (WASM path), where three consecutive `db.put(...)` calls for the address manager, ban list, and anchors all discard their results:

```rust
let _ignore = db.put(addr_manager_path.into_bytes(), addr_manager).await;
let _ignore = db.put(ban_list_path.into_bytes(), ban_list).await;
let _ignore = db.put(anchors_path.into_bytes(), anchors).await;
``` [2](#0-1) 

A failed write to the ban list would silently leave banned peers unpersisted, allowing them to reconnect after a node restart.

### Proof of Concept

1. Attacker peer A connects to a CKB node and announces a new block via compact block relay.
2. The node sends `GetBlockTransactions` to peer A for missing short IDs.
3. Peer A responds with a `BlockTransactions` message that is still incomplete (e.g., intentionally omits some transactions).
4. The node enters `ReconstructionResult::Missing`, builds a new `GetBlockTransactions` message, and calls `async_send_message_to`.
5. At this moment, peer A closes the session or the send buffer is full — `async_send_message_to` returns `Err(...)`.
6. The error is discarded via `let _ignore`. The node overwrites `expected_transaction_indexes` with the new missing set.
7. No request is ever delivered to peer A (or any other peer). The compact block entry for this block hash remains in `pending_compact_blocks` with the updated-but-never-requested indexes.
8. The block cannot be reconstructed from this peer. The node must wait for another peer to announce the same block, or for the pending entry to expire — delaying block acceptance. [3](#0-2)
