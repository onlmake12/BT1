Audit Report

## Title
`migrate_transaction_info` Traverses `COLUMN_UNCLES` Instead of `COLUMN_TRANSACTION_INFO`, Corrupting Transaction Info on Migrated Nodes — (File: util/migrate/src/migrations/table_to_struct.rs)

## Summary
The `ChangeMoleculeTableToStruct` migration's `migrate_transaction_info` function calls `db.traverse(COLUMN_UNCLES, ...)` at line 93 instead of `db.traverse(COLUMN_TRANSACTION_INFO, ...)`. Any node that ran this migration over pre-v0.35.0 data never converts its `COLUMN_TRANSACTION_INFO` entries from the old 68-byte molecule Table format to the new 52-byte Struct format. Subsequent calls to `get_transaction_info` and `get_transaction` on those nodes read fields at wrong offsets, producing corrupted or missing results.

## Finding Description
`ChangeMoleculeTableToStruct` (version `20200703124523`) rewrites four column families. Three sub-migrations correctly traverse their own column; `migrate_transaction_info` does not:

```
migrate_header           → db.traverse(COLUMN_BLOCK_HEADER, ...)  ✓  (line 40)
migrate_uncles           → db.traverse(COLUMN_UNCLES, ...)        ✓  (line 66)
migrate_transaction_info → db.traverse(COLUMN_UNCLES, ...)        ✗  (line 93)
migrate_epoch_ext        → db.traverse(COLUMN_EPOCH, ...)         ✓  (line 120)
``` [1](#0-0) 

Because `migrate_uncles` runs first and converts uncle records to 240-byte Struct format, when `migrate_transaction_info` traverses `COLUMN_UNCLES` it finds 240-byte records. The guard `value.len() != TRANSACTION_INFO_SIZE` (52) is always true for 240-byte uncle records, so the closure fires for every uncle and writes 224 bytes of uncle data (`value[16..]`) into `COLUMN_TRANSACTION_INFO` under uncle hashes. The actual `COLUMN_TRANSACTION_INFO` entries (keyed by tx hash, 68 bytes each in old Table format) are never visited and remain unconverted. [2](#0-1) 

`get_transaction_info` then calls `packed::TransactionInfoReader::from_slice_should_be_ok` on the 68-byte old-format slice. The Struct reader applies fixed offsets: `block_number` at `[0..8]` reads the 4-byte `total_size` field plus the first offset entry (garbage); `block_epoch` at `[8..16]` reads remaining offset bytes (garbage); `key` at `[16..52]` reads the actual `block_number` + `block_epoch` + first 4 bytes of `key` (shifted). The resulting `tx_info.key()` is wrong, causing `get_transaction_with_info` to look up `COLUMN_BLOCK_BODY` with a corrupted key and return `None`. [3](#0-2) [4](#0-3) 

## Impact Explanation
The concrete impact is incorrect or absent responses from the `get_transaction` RPC for any transaction confirmed before the migration ran. If `from_slice_should_be_ok` enforces exact size (52 bytes) and panics on the 68-byte old-format slice, the RPC handler crashes. This fits **Note (0–500 points): Any local RPC API crash**. The bug does not affect consensus, network stability, or the CKB economy; it is local to nodes that ran the migration over pre-v0.35.0 data.

## Likelihood Explanation
Only nodes that held chain data from before v0.35.0 (released 2020) and ran `ckb migrate` or auto-migrated on startup are affected. Nodes that synced from genesis after v0.35.0 have empty columns at migration time and are unaffected. The affected population is limited to long-running nodes that have not resynced since 2020.

## Recommendation
Change line 93 of `util/migrate/src/migrations/table_to_struct.rs` from:
```rust
db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
```
to:
```rust
db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
```
Because the migration version `20200703124523` is already stamped in affected databases, a new migration with a new version string must be added to re-scan `COLUMN_TRANSACTION_INFO` and rewrite any remaining 68-byte Table-format entries to the 52-byte Struct format.

## Proof of Concept
1. Start a CKB node on any version prior to v0.35.0 and sync at least one block so `COLUMN_TRANSACTION_INFO` contains 68-byte Table-format entries.
2. Upgrade the binary to v0.35.0+ and run `ckb migrate` (or start the node, which auto-migrates).
3. Confirm migration version `20200703124523` is stamped.
4. Call `get_transaction(<tx_hash>)` for any pre-migration transaction via RPC.
5. Observe `null` or a panic, because `get_transaction_with_info` resolves a corrupted `block_hash` from the mangled `key` field and fails to find the block body in `COLUMN_BLOCK_BODY`.
