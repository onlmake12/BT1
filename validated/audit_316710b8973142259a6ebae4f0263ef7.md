Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Permanently Inflated After Cell-Dep Ancestor Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, a pre-eviction snapshot of `total_tx_size`/`total_tx_cycles` is captured before `check_and_record_ancestors` runs. When that function evicts cell-dep ancestor transactions, each eviction correctly decrements `self.total_tx_size` and `self.total_tx_cycles` via `update_stat_for_remove_tx`. However, the stale pre-eviction snapshot is then unconditionally written back, silently overwriting those decrements. The result is a permanently inflated `total_tx_size`, which causes `limit_size` to expel legitimate transactions from the pool with `Reject::Full` even when the pool is not actually full.

## Finding Description

**Step 1 — Pre-eviction snapshot (lines 210–211):**

`updated_stat_for_add_tx` reads the current `self.total_tx_size` and `self.total_tx_cycles`, adds the new entry's contribution, and returns the result as local variables. It does not mutate any field. [1](#0-0) 

**Step 2 — Evictions decrement the live counters (line 213):**

`check_and_record_ancestors` enters the cell-dep eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants` for each evict candidate. [2](#0-1) 

Each `remove_entry_and_descendants` call chains into `remove_entry`, which calls `update_stat_for_remove_tx`, correctly decrementing `self.total_tx_size` and `self.total_tx_cycles`. [3](#0-2) 

**Step 3 — Stale snapshot overwrites the decrements (lines 218–219):**

After all evictions have already decremented the live counters, the pre-eviction local variables are unconditionally assigned back: [4](#0-3) 

Net result: `self.total_tx_size` ends up as `(original_total + entry.size)` instead of the correct `(original_total − Σevicted_sizes + entry.size)`. The inflation equals exactly the aggregate serialized size of all evicted transactions and persists for the lifetime of the pool.

**Downstream — `limit_size` uses `total_tx_size` as its sole eviction guard:** [5](#0-4) 

An inflated counter causes the `while` loop to fire and expel legitimate pending/proposed transactions with `Reject::Full` even though the pool is not actually full.

## Impact Explanation

Every time the cell-dep ancestor eviction path fires, `total_tx_size` is over-counted by the aggregate size of the evicted entries. Subsequent `limit_size` calls (invoked after every successful `submit_entry`) then eject honest user transactions. The error accumulates across submissions, progressively shrinking effective pool capacity. The inflated value is also surfaced directly through the `tx_pool_info` RPC, giving callers incorrect telemetry. This maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — an unprivileged attacker can repeatedly trigger the path to degrade or effectively disable the tx-pool of any targeted node, preventing honest transactions from being relayed or mined.

## Likelihood Explanation

No privileged access, leaked keys, or majority hashpower is required. The attacker only needs the ability to call `send_transaction` via the standard RPC. The setup requires building an ancestor chain of length approaching `max_ancestors_count` (default 1000) and then submitting a transaction that references an intermediate output as a cell-dep, pushing the total ancestor count over the limit. This is non-trivial but entirely feasible for a motivated attacker. The inflation accumulates with each successful trigger, so repeated submissions amplify the effect progressively.

## Recommendation

Move the `updated_stat_for_add_tx` call to **after** `check_and_record_ancestors` completes, so that any eviction-driven decrements are already reflected in `self.total_tx_size` before the new entry's contribution is added:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Snapshot taken after all evictions:
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, use the existing `recompute_total_stat()` after all mutations for a correctness-first approach, accepting its O(n) cost. [6](#0-5) 

## Proof of Concept

1. Configure a node with `max_ancestors_count = N` (default 1000).
2. Submit `tx_root` spending a live cell.
3. Submit a linear chain `tx_1 → tx_2 → … → tx_N`, each spending the previous output.
4. Submit `tx_dep` that uses `tx_root`'s output as a **cell-dep**.
5. Submit `tx_trigger` that spends `tx_N`'s output AND uses `tx_root`'s output as a cell-dep.
   - Ancestor count = N (input chain) + 1 (`tx_root` cell-dep) + 1 (self) = N+2 > N.
   - `cell_ref_parents = {tx_root}`, so `N+2 − 1 = N+1 ≤ N+1` — within limit after removing `tx_root`.
   - `check_and_record_ancestors` evicts `tx_root` and `tx_dep` via `remove_entry_and_descendants`, calling `update_stat_for_remove_tx` twice.
   - Lines 218–219 then restore the pre-eviction snapshot, inflating `total_tx_size` by `tx_root.size + tx_dep.size`.
6. Query `tx_pool_info` via RPC: `total_tx_size` exceeds the true sum of sizes of entries in the pool.
7. Submit additional transactions filling the real remaining capacity: `limit_size` evicts them with `Reject::Full` despite the pool not being full.
8. Repeat steps 2–7 to accumulate further inflation. [7](#0-6)

    /// Calculate size and cycles statistics for adding a tx.
    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
