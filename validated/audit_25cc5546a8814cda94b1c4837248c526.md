### Title
Provider Can Front-Run `requestV2` by Calling `setProviderFee` to Drain User's Full `msg.value` — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract allows any registered provider to update their fee instantly via `setProviderFee` with no timelock or delay. Because `requestV2` computes the required fee at execution time and does not accept a caller-specified maximum fee, a malicious provider can observe a pending `requestV2` transaction in the mempool, front-run it with a `setProviderFee` call that raises the fee to exactly the user's `msg.value` minus the Pyth protocol fee, and cause the user to pay the provider's maximum extractable amount rather than the fee they observed when constructing the transaction.

---

### Finding Description

The Entropy contract exposes two interacting functions with no slippage protection between them:

**`setProviderFee`** — callable by any registered provider at any time, with immediate effect and no delay:

```solidity
// Entropy.sol:810-827
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei, bytes(""));
}
```

**`requestHelper`** — computes the required fee at execution time and accepts any `msg.value >= requiredFee` without refunding the excess:

```solidity
// Entropy.sol:234-239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

The interface documentation explicitly warns that fees change over time and instructs callers to send a buffer:

> *"Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2()` prior to each invocation."*
> *"Note that excess value is **not** refunded to the caller."*

These two properties together create the attack surface. A user who sends `msg.value = fee + buffer` to guard against reverts is exposed to having that entire buffer extracted by the provider.

**Attack flow:**

1. User calls `getFeeV2()` off-chain and observes fee `F`.
2. User submits `requestV2{value: F + buffer}()` to avoid a revert if the fee ticks up slightly.
3. Provider sees the pending transaction in the mempool and reads `msg.value = F + buffer`.
4. Provider front-runs with `setProviderFee(F + buffer - pythFeeInWei)`.
5. User's `requestV2` executes: `requiredFee = (F + buffer - pythFeeInWei) + pythFeeInWei = F + buffer`. Since `msg.value == requiredFee`, the check passes.
6. Provider accrues `F + buffer - pythFeeInWei` instead of the original `F - pythFeeInWei`. The user's entire buffer is extracted.

Even without a buffer, the provider can simply front-run to raise the fee above `msg.value`, causing the user's transaction to revert with `InsufficientFee` and wasting their gas — a pure DoS variant of the same root cause.

---

### Impact Explanation

- **Financial loss**: A user who sends any `msg.value` above the exact fee (a common defensive pattern given the "no refund" behavior) can have the entire overage extracted by the provider. The provider profits by the difference between the original fee and the user's `msg.value` minus the fixed Pyth protocol fee.
- **DoS**: A provider can repeatedly front-run user requests to cause `InsufficientFee` reverts, preventing any user from successfully requesting randomness from that provider and wasting user gas on every attempt.
- **Trust violation**: Users select a provider based on an advertised fee. The front-running attack allows the provider to silently charge more than the fee the user agreed to pay.

---

### Likelihood Explanation

- Any registered provider can execute this attack — provider registration is permissionless.
- `setProviderFee` has no timelock, cooldown, or rate limit, so the fee change takes effect in the same block.
- Transactions are visible in the public mempool on all EVM chains where Entropy is deployed, making the pending `requestV2` observable before inclusion.
- The Fortuna keeper service (`adjust_fee_if_necessary` in `apps/fortuna/src/keeper/fee.rs`) already performs automated fee updates, demonstrating that fee changes concurrent with user requests are a normal operational condition — a malicious provider simply weaponizes the same mechanism.
- The interface documentation's own warning that "excess value is not refunded" incentivizes users to send a buffer, which is exactly what the attacker needs to extract value.

---

### Recommendation

Add a `maxFee` parameter to the `requestV2` family of functions so callers can specify the maximum fee they are willing to pay. Revert if the computed fee at execution time exceeds this bound:

```solidity
function requestV2(uint32 gasLimit, uint128 maxFee)
    external payable returns (uint64 assignedSequenceNumber);
```

Inside `requestHelper`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (requiredFee > maxFee) revert EntropyErrors.FeeTooHigh();
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
```

This mirrors the standard slippage-protection pattern used in AMMs and gives callers a guarantee that their transaction will not execute under terms worse than they observed.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

import "./IEntropyV2.sol";
import "./IEntropyConsumer.sol";

contract FrontRunPoC is IEntropyConsumer {
    IEntropyV2 public entropy;
    address public provider;

    constructor(address _entropy, address _provider) {
        entropy = IEntropyV2(_entropy);
        provider = _provider;
    }

    // Step 1: User calls this, sending fee + 0.001 ether buffer
    function requestWithBuffer() external payable {
        // User observed fee = X off-chain, sends X + buffer
        // Provider front-runs setProviderFee(msg.value - pythFee)
        // Provider now accrues msg.value - pythFee instead of X - pythFee
        entropy.requestV2{value: msg.value}(provider, 0);
    }

    function entropyCallback(uint64, address, bytes32) internal override {}
    function getEntropy() internal view override returns (address) {
        return address(entropy);
    }
}
```

**Sequence:**
1. Deploy `FrontRunPoC` pointing at the live Entropy contract and a provider the attacker controls.
2. Call `requestWithBuffer{value: 0.01 ether}()` — transaction enters the mempool.
3. Provider calls `setProviderFee(0.01 ether - pythFeeInWei)` with higher gas to front-run.
4. `requestWithBuffer` executes: `requiredFee = 0.01 ether`, `msg.value = 0.01 ether` — passes.
5. Provider's `accruedFeesInWei` increases by `0.01 ether - pythFeeInWei` instead of the original smaller fee.
6. Provider calls `withdraw(...)` to collect the extracted funds.
