### Title
Stale `total_tx_size`/`total_tx_cycles` Overwrite After Ancestor-Eviction in `add_entry` Inflates Pool Accounting — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new aggregate pool statistics (`total_tx_size`, `total_tx_cycles`) are computed **before** ancestor-eviction occurs, then unconditionally written back **after** eviction. Because each eviction correctly decrements the running totals via `update_stat_for_remove_tx`, the final assignment overwrites those correct decrements with a stale, inflated value. The result is that `total_tx_size` and `total_tx_cycles` are permanently over-counted by the sum of all evicted transactions' sizes and cycles. Any unprivileged RPC caller can trigger this path by submitting a transaction whose cell-dep ancestor count exceeds `max_ancestors_count`.

---

### Finding Description

In `tx-pool/src/component/pool_map.rs`, `add_entry` executes the following sequence:

```
// Step 1 – snapshot new totals BEFORE any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211

// Step 2 – may evict N transactions; each calls update_stat_for_remove_tx
//          which correctly decrements self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

// Step 3 – OVERWRITES the correctly-decremented values with the stale snapshot
self.total_tx_size  = total_tx_size;                            // line 218
self.total_tx_cycles = total_tx_cycles;                         // line 219
``` [1](#0-0) 

`updated_stat_for_add_tx` computes `new_total = self.total_tx_size + new_tx.size` and stores it in a local variable. [2](#0-1) 

`check_and_record_ancestors` calls `remove_entry_and_descendants` for each evicted transaction, which in turn calls `remove_entry`, which calls `update_stat_for_remove_tx` — correctly decrementing `self.total_tx_size` and `self.total_tx_cycles` for every evicted entry. [3](#0-2) [4](#0-3) 

After all evictions complete, lines 218–219 unconditionally assign the **pre-eviction** snapshot back to `self.total_tx_size` / `self.total_tx_cycles`, erasing every decrement performed during eviction. The final value is:

```
self.total_tx_size = original_total + new_tx.size
```

instead of the correct:

```
self.total_tx_size = original_total + new_tx.size − Σ(evicted_tx.size)
```

The eviction path is reached inside `check_and_record_ancestors` when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`: [5](#0-4) 

---

### Impact Explanation

`total_tx_size` is the sole variable governing the `limit_size` eviction loop:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size { … }
``` [6](#0-5) 

With an inflated `total_tx_size`, `limit_size` will evict additional legitimate transactions that would otherwise remain in the pool. The inflation persists until the pool is cleared (e.g., on reorg or restart), because every subsequent `updated_stat_for_add_tx` call builds on the already-inflated baseline: [7](#0-6) 

Concrete consequences:
- Legitimate pending/proposed transactions are silently evicted beyond the configured `max_tx_pool_size` limit.
- The `updated_stat_for_add_tx` overflow guard fires prematurely, causing valid transactions to be rejected with `Reject::Full` even when the pool has real capacity.
- The `total_tx_size` reported via `get_tx_pool_info` RPC is incorrect, misleading operators and monitoring tools. [8](#0-7) 

---

### Likelihood Explanation

The trigger condition — a transaction whose cell-dep ancestor count exceeds `max_ancestors_count` (default 25) but whose non-cell-ref ancestor count is within the limit — is reachable by any unprivileged `send_transaction` RPC caller. No special privilege, key, or majority hash power is required. The test `TxPoolLimitAncestorCount` in the integration suite demonstrates that 2000+ cell-ref transactions can be submitted and that the eviction path fires normally, confirming the path is exercised in practice. [9](#0-8) 

---

### Recommendation

Move the `total_tx_size`/`total_tx_cycles` assignment to **after** `check_and_record_ancestors` completes, and account for the sizes of evicted entries. The simplest correct fix is to compute the final totals after eviction:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Recompute after evictions have already updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, subtract the evicted totals from the pre-computed snapshot before assigning, or rely on `recompute_total_stat` after the full operation. [10](#0-9) 

---

### Proof of Concept

1. Fill the pool with `N` transactions that each reference the same in-pool cell as a `cell_dep` (as in `TxPoolLimitAncestorCount`), where `N` is close to `max_ancestors_count`.
2. Submit a new transaction that **consumes** that cell as an input. Its ancestor count = N+1 > `max_ancestors_count`, but `N+1 - N = 1 <= max_ancestors_count`, so the eviction branch fires and removes `K` cell-ref transactions.
3. After the submission, query `get_tx_pool_info`. Observe that `total_tx_size` equals `(original_total + new_tx.size)` rather than `(original_total + new_tx.size - K * evicted_tx.size)`.
4. Repeat step 2 several times. `total_tx_size` grows unboundedly relative to the actual pool contents.
5. Observe that `limit_size` begins evicting transactions even though the real pool occupancy is well below `max_tx_pool_size`, or that new valid transactions are rejected with `Reject::Full`. [11](#0-10) [12](#0-11)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
