### Title
Unvalidated `providerToCredit` in `Echo.sol::executeCallback` Enables Fee Theft from Legitimate Providers — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol::executeCallback` accepts a fully user-controlled `providerToCredit` address that is only validated during the exclusivity window. After that window, any caller can pass their own address as `providerToCredit`, causing the contract to credit the entire request fee to the attacker before the callback fires. Because `registerProvider` is permissionless and `setFeeManager` is self-service, the attacker can immediately drain those credited fees via `withdrawAsFeeManager`.

---

### Finding Description

`executeCallback` is an open, permissionless function. Its first substantive action after parsing the request is to credit fees to the caller-supplied `providerToCredit`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

clearRequest(sequenceNumber);
// ...
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
```

The only guard on `providerToCredit` is the exclusivity-period check:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

Once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, the check is skipped entirely and `providerToCredit` is unconstrained. There is no requirement that `providerToCredit` equals `msg.sender`, equals `req.provider`, or even be a registered provider.

The fee-withdrawal path (`withdrawAsFeeManager`) does not check provider registration status — it only checks that `msg.sender == _state.providers[provider].feeManager`. Because `registerProvider` is permissionless and `setFeeManager` is self-service for any registered provider, an attacker can set up the full withdrawal path before the attack.

---

### Impact Explanation

A malicious actor can steal 100 % of the fee a requester paid for any pending request once the exclusivity period lapses. The legitimate provider who was supposed to fulfill the request receives nothing. Because `updateData` is publicly available from Pyth's Hermes price service, the attacker incurs no additional cost beyond gas and the Pyth parse fee (which is already accounted for in `req.fee`).

---

### Likelihood Explanation

- `executeCallback` has no access control.
- `registerProvider` and `setFeeManager` are permissionless.
- Valid `updateData` is freely available from Hermes.
- The attack requires only two setup transactions and one exploit transaction.
- Any pending request whose exclusivity period has elapsed is a target.

---

### Recommendation

1. **Bind `providerToCredit` to `msg.sender`**: require `providerToCredit == msg.sender` so only the actual caller can receive credit.
2. **Alternatively, validate registration**: require `_state.providers[providerToCredit].isRegistered` before crediting fees.
3. **Consider requiring the caller to be a registered provider** regardless of `providerToCredit`.
