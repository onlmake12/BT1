### Title
Unbounded O(n) Hash Loop in `constructProviderCommitment` Enables DOS on Entropy Reveal — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any unprivileged user can make an arbitrary number of randomness requests against a provider that has not set `maxNumHashes`. Each request increments the provider's `sequenceNumber`, which inflates the `numHashes` field stored in subsequent requests. When the provider later calls `revealWithCallback()` or `reveal()`, the internal `constructProviderCommitment()` function iterates `numHashes` times in a `while` loop. If `numHashes` is large enough, the reveal transaction exceeds the block gas limit and permanently cannot be executed, permanently locking the affected request.

---

### Finding Description

`requestHelper()` stores `numHashes` in each new request as the distance between the current sequence number and the provider's last-known commitment anchor:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
``` [1](#0-0) 

A guard exists, but only fires when `maxNumHashes != 0`:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [2](#0-1) 

`maxNumHashes` defaults to `0` in `ProviderInfo` (zero-initialized storage) and is never set during `register()`. A provider must separately call `setMaxNumHashes()` to enable the guard. [3](#0-2) 

When `revealWithCallback()` or `reveal()` is called, `revealHelper()` invokes:

```solidity
bytes32 providerCommitment = constructProviderCommitment(
    req.numHashes,
    providerContribution
);
``` [4](#0-3) 

`constructProviderCommitment` performs exactly `numHashes` iterations of `keccak256`:

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
``` [5](#0-4) 

Each `keccak256` costs ~30 gas. At Ethereum's 30 M gas block limit, approximately 1 million iterations exhaust the block. An attacker making ~1 million requests against a provider with `maxNumHashes == 0` causes the last request's reveal to be permanently unexecutable.

---

### Impact Explanation

- The affected request is permanently stuck: `revealWithCallback` and `reveal` both revert out-of-gas for that sequence number.
- Users who submitted the high-`numHashes` request never receive their random number and cannot recover their fee.
- The provider's reputation and service availability are degraded.
- The provider cannot use `advanceProviderCommitment` to rescue the stuck request because that function also calls `constructProviderCommitment` with the same large `numHashes`, hitting the same gas wall. [6](#0-5) 

---

### Likelihood Explanation

- `maxNumHashes` is **not set by default** during `register()`. Any provider that does not explicitly call `setMaxNumHashes()` is vulnerable.
- The attack requires paying the provider fee per request. At low fee settings (e.g., 1 wei), the cost to make 1 million requests is negligible. Even at higher fees the attack is economically feasible for a motivated adversary.
- The entry point (`request()` / `requestV2()`) is fully permissionless — no role or privilege is required. [7](#0-6) 

---

### Recommendation

1. **Enforce a non-zero `maxNumHashes` at registration time.** Require providers to supply a `maxNumHashes` value in `register()`, or set a protocol-level default (e.g., 1000) that applies when the provider has not configured one.
2. **Remove the `!= 0` bypass.** Change the guard to always apply:
   ```solidity
   if (req.numHashes > providerInfo.maxNumHashes) {
       revert EntropyErrors.LastRevealedTooOld();
   }
   ```
3. **Cap `numHashes` at a safe constant** (e.g., `MAX_NUM_HASHES = 10_000`) to bound worst-case gas regardless of provider configuration.

---

### Proof of Concept

```
Setup:
  - Provider P registers via register(...) without calling setMaxNumHashes().
    → providerInfo.maxNumHashes == 0 (default)
    → providerInfo.currentCommitmentSequenceNumber == 0

Attack:
  for i in range(1_000_000):
      entropy.request{value: fee}(P, userCommitment, false)
      // Each call increments providerInfo.sequenceNumber by 1.
      // Request i gets numHashes = i - 0 = i.

Result:
  - Request #1_000_000 has numHashes = 1_000_000.
  - Provider calls revealWithCallback(P, 1_000_000, ...).
  - revealHelper → constructProviderCommitment(1_000_000, ...) loops 1M times.
  - Gas consumed ≈ 1,000,000 × 30 = 30,000,000 gas → exceeds block gas limit.
  - Transaction reverts. Request is permanently unresolvable.
``` [5](#0-4) [8](#0-7)
