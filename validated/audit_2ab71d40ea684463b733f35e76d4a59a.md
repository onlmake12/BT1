### Title
Admin Can Drain Accumulated Pyth Protocol Fees to Arbitrary Address via Dual-Role `_authoriseAdminAction` — (`EntropyGovernance.sol`)

---

### Summary

`EntropyUpgradable` deploys a two-role model (owner + admin) where the contract comment explicitly states *"The admin can set some config parameters only."* However, `_authoriseAdminAction()` grants the admin identical access to `withdrawFee(address targetAddress, uint128 amount)` as the owner, allowing the admin to drain the entire `accruedPythFeesInWei` balance to any arbitrary address — including one they control. Symmetrically, the owner can override the admin's fee-management decisions. Neither role is constrained to a designated treasury address.

---

### Finding Description

`EntropyUpgradable._authoriseAdminAction()` passes for both `owner()` and `_state.admin`:

```solidity
// EntropyUpgradable.sol L63-66
function _authoriseAdminAction() internal view override {
    if (msg.sender != owner() && msg.sender != _state.admin)
        revert EntropyErrors.Unauthorized();
}
``` [1](#0-0) 

Every function in `EntropyGovernance` that calls `_authoriseAdminAction()` is therefore callable by either role:

| Function | Intended role per comment | Actual access |
|---|---|---|
| `setPythFee` | admin | owner **or** admin |
| `setDefaultProvider` | admin | owner **or** admin |
| `withdrawFee` | admin | owner **or** admin |
| `proposeAdmin` | admin | owner **or** admin | [2](#0-1) 

The critical function is `withdrawFee`:

```solidity
// EntropyGovernance.sol L103-116
function withdrawFee(address targetAddress, uint128 amount) external {
    require(targetAddress != address(0), "targetAddress is zero address");
    _authoriseAdminAction();

    if (amount > _state.accruedPythFeesInWei)
        revert EntropyErrors.InsufficientFee();

    _state.accruedPythFeesInWei -= amount;

    (bool success, ) = targetAddress.call{value: amount}("");
    require(success, "Failed to withdraw fees");

    emit FeeWithdrawn(targetAddress, amount);
}
``` [3](#0-2) 

`targetAddress` is completely unconstrained (only non-zero). There is no designated treasury address enforced on-chain. The admin can call `withdrawFee(adminControlledAddress, accruedPythFeesInWei)` and drain the entire Pyth protocol fee balance to an address they control, bypassing the owner's intended fee management. The same applies in reverse.

The same dual-role pattern is replicated in `SchedulerGovernance` / `SchedulerUpgradeable` for the Pulse Scheduler contract:

```solidity
// SchedulerUpgradeable.sol L57-60
function _authorizeAdminAction() internal view override {
    if (msg.sender != owner() && msg.sender != _state.admin)
        revert SchedulerErrors.Unauthorized();
}
``` [4](#0-3) 

allowing either role to override `setSingleUpdateKeeperFeeInWei` and `setMinimumBalancePerFeed`. [5](#0-4) 

A secondary griefing vector exists in `proposeAdmin`: since both owner and admin can call it, either party can silently overwrite the other's pending admin transfer by proposing a different address, blocking the intended admin rotation. [6](#0-5) 

---

### Impact Explanation

- **Fee theft**: A compromised or malicious admin can call `withdrawFee(attackerAddress, totalAccruedFees)` and drain all accumulated Pyth protocol fees (ETH) to an arbitrary address. The owner has no on-chain mechanism to prevent this once the admin key is used.
- **Fee parameter manipulation**: Either role can call `setPythFee(0)` to zero out the protocol fee, eliminating Pyth's revenue stream from Entropy requests.
- **Default provider hijack**: Either role can call `setDefaultProvider` to redirect users to a provider they control, affecting randomness quality or enabling front-running.
- **Admin transfer griefing**: Either role can overwrite the other's pending `proposedAdmin`, blocking legitimate admin key rotation.

The `accruedPythFeesInWei` balance is funded by every `request()` call from all users across all chains where `EntropyUpgradable` is deployed. [7](#0-6) 

---

### Likelihood Explanation

The admin key is a separate operational key from the owner. If the admin key is compromised (e.g., via a hot-wallet breach in the Fortuna keeper infrastructure), an attacker can immediately drain all accrued Pyth fees in a single transaction. The attack requires no user interaction, no oracle manipulation, and no complex setup — just a single call to `withdrawFee`. The Fortuna keeper service actively monitors and interacts with the Entropy contract, making the admin key an active operational key with elevated exposure. [8](#0-7) 

---

### Recommendation

1. **Restrict `withdrawFee` to owner-only**, or require a pre-configured on-chain treasury address rather than an arbitrary `targetAddress` parameter. The admin should not have the ability to redirect funds.
2. **Separate admin and owner capabilities explicitly**: functions that move funds (`withdrawFee`) should be `onlyOwner`; functions that set operational parameters (`setPythFee`, `setDefaultProvider`) can remain `_authoriseAdminAction`.
3. **For `proposeAdmin`**: restrict to `onlyOwner` to prevent the current admin from self-perpetuating their role or blocking owner-initiated admin rotation.
4. Apply the same separation to `SchedulerGovernance`.

---

### Proof of Concept

```solidity
// Attacker controls `adminKey`
// Step 1: Admin drains all accrued Pyth fees to attacker address
entropyUpgradable.withdrawFee(attackerAddress, entropyUpgradable.getAccruedPythFees());
// Result: accruedPythFeesInWei = 0, attackerAddress receives all ETH

// Step 2 (separate vector): Admin zeroes out the protocol fee
entropyUpgradable.setPythFee(0);
// Result: all future requests pay 0 Pyth protocol fee

// Step 3 (separate vector): Admin blocks owner's pending admin rotation
// Owner called proposeAdmin(newAdmin) — admin overwrites it:
entropyUpgradable.proposeAdmin(adminControlledAddress);
// Result: newAdmin can no longer acceptAdmin()
``` [3](#0-2) [9](#0-8) [10](#0-9)
