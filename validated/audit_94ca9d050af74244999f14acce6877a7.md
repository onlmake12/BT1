## Guard Analysis: Lines 201–205

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT  // = 1000
```

With `last_n_blocks = u64::MAX` on a 64-bit system:

- `u64::MAX as usize = usize::MAX = 2^64 - 1`
- `usize::MAX * 2 = 2^65 - 2`, which **wraps** to `usize::MAX - 1` in Rust release mode (no panic, two's-complement wrap)
- `difficulties.len() + (usize::MAX - 1)`:
  - `len = 0`: result = `usize::MAX - 1` → huge → check fires → **rejected**
  - `len = 1`: result = `usize::MAX` → huge → check fires → **rejected**
  - `len = 2`: result = `usize::MAX + 1` → wraps to **`0`** → `0 > 1000` is **false** → **check bypassed**
  - `len = 3`: wraps to `1` → **bypassed**

So with `difficulties.len() >= 2` (two U256 values = 64 bytes, trivially within any message size limit), the guard silently wraps to a small number in a release build and is bypassed entirely. [1](#0-0) [2](#0-1) 

## Reorg Path Analysis: Lines 245–246

Once the guard is bypassed, the reorg branch executes:

```rust
let min_block_number = start_block_number - min(start_block_number, last_n_blocks);
(min_block_number..start_block_number).collect()
```

With `last_n_blocks = u64::MAX`:
- `min(start_block_number, u64::MAX) = start_block_number` (real block numbers are far below `u64::MAX`)
- `min_block_number = start_block_number - start_block_number = 0`
- `(0..start_block_number).collect::<Vec<_>>()` → allocates **`start_block_number`** entries of `u64` [3](#0-2) 

The only bound on `start_block_number` is `start_block_number <= last_block_number` (line 231–235). On mainnet, `last_block_number` is in the millions. A Vec of 10 million `u64` entries = ~80 MB per request. Multiple concurrent requests compound this. [4](#0-3) 

## Verdict

### Title
Integer Overflow in Guard Allows Unbounded `reorg_last_n_numbers` Allocation via `last_n_blocks=u64::MAX` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
An unprivileged light-client peer can send a crafted `GetLastStateProof` P2P message with `last_n_blocks = u64::MAX` and `difficulties.len() >= 2`. The size guard at line 201 overflows (wraps to 0 in release mode), bypassing the `GET_LAST_STATE_PROOF_LIMIT = 1000` check. The reorg branch then allocates a `Vec` of `start_block_number` entries with no upper bound, enabling OOM or excessive CPU on any node serving the light-client protocol.

### Finding Description
The guard expression `difficulties.len() + (last_n_blocks as usize) * 2` uses unchecked arithmetic. In Rust release builds, `(u64::MAX as usize) * 2` wraps to `usize::MAX - 1`, and adding `difficulties.len() >= 2` wraps the total to a small value (0, 1, …) that passes the `> 1000` check. The reorg allocation `(0..start_block_number).collect()` is then unbounded by any limit.

### Impact Explanation
- Each malicious request allocates up to `last_block_number * 8` bytes (e.g., ~80 MB at block 10,000,000).
- Multiple concurrent requests from different peers can exhaust node memory.
- The `complete_headers` loop at line 358 then iterates over all entries, performing a DB lookup per entry, causing excessive CPU and I/O. [5](#0-4) 

### Likelihood Explanation
The attack requires only a valid P2P connection to a node running the light-client protocol server. No PoW, no keys, no privileged role. The crafted message is 64+ bytes. The reorg condition (start hash not on main chain) is trivially satisfied by sending any non-canonical hash.

### Recommendation
Replace the unchecked arithmetic with a saturating or checked operation, and add a direct bound on `last_n_blocks` before any computation:

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("last_n_blocks too large");
}
if self.message.difficulties().len().saturating_add((last_n_blocks as usize).saturating_mul(2))
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, the reorg allocation at line 246 should be independently capped:

```rust
let reorg_count = (start_block_number - min_block_number) as usize;
if reorg_count > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("reorg window too large");
}
```
