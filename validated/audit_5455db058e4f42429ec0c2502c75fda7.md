### Title
`ScriptHashTypeVerifier` Omits Type Script `hash_type` Consensus Check, Allowing Disallowed VM Versions in UTXO Set — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify` enforces `ENABLED_SCRIPT_HASH_TYPE` only on `output.lock().hash_type()`. It never inspects `output.type_()`. A transaction output whose lock uses an allowed hash type (e.g., `Data=0`) but whose type script uses a disallowed one (e.g., `Data3=6`) passes this verifier unconditionally.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is defined as `{0, 1, 2, 4}` — only `Data`, `Type`, `Data1`, `Data2` are consensus-permitted. [1](#0-0) 

`ScriptHashTypeVerifier::verify` iterates outputs and checks only the lock script: [2](#0-1) 

The type script field is never read. A `CellOutput` with `lock.hash_type = 0` (Data, allowed) and `type_.hash_type = 6` (Data3, not in `ENABLED_SCRIPT_HASH_TYPE`) returns `Ok(())` from this verifier.

The lower-level `check_data` in `CellOutputReader` does check both lock and type script, but only for structural validity via `verify_value` (even number or 1) — not against the consensus-permitted set. `Data3=6` is even, so it passes `check_data` too. [3](#0-2) 

No other location in the verification pipeline applies `ENABLED_SCRIPT_HASH_TYPE` to type scripts on non-cellbase transaction outputs. The grep across all `verification/**/*.rs` confirms `ENABLED_SCRIPT_HASH_TYPE` is only referenced in `block_verifier.rs` (cellbase lock check) and `transaction_verifier.rs` (output lock check).

The existing test suite covers the lock-script path (`test_not_enabled_hash_type_output_lock`) but has no corresponding test for the type script path. [4](#0-3) 

---

### Impact Explanation

An unprivileged submitter can craft a transaction whose output has a valid lock `hash_type` but a type script with `hash_type = Data3 (6)`. `ScriptHashTypeVerifier` returns `Ok`. Whether the transaction ultimately lands in the UTXO set depends on the script execution phase:

- If the type script body is trivially passing (e.g., always-success code), or if the node's VM already supports Data3, the transaction is fully accepted and the UTXO enters the set with a type script using a consensus-disallowed hash type.
- This violates the invariant that all script hash types in transaction outputs must be within the consensus-permitted range.
- Nodes running different VM versions may disagree on acceptance, creating a potential consensus split.

The stated scoped impact — *suboptimal state storage where outputs with disallowed type script hash types are accepted into the UTXO set* — is reachable.

---

### Likelihood Explanation

The path is straightforward: submit a transaction via P2P or RPC with a crafted `CellOutput`. No privilege, key, or majority hashpower is required. The gap is a simple missing branch in a 19-line function. The proof of concept is a unit test that can be run locally without any network.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify` to also check the type script's `hash_type` when present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        check_hash_type(output.lock().hash_type())?;
        // missing type script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the existing `TryInto::<ScriptHashType>` + `ENABLED_SCRIPT_HASH_TYPE.contains` logic. [2](#0-1)

// Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}
