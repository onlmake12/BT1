Audit Report

## Title
Stale Tx-Pool Snapshot Allows Dead-Cell Transactions to Bypass Admission Validation — (`tx-pool/src/pool.rs`, `tx-pool/src/process.rs`)

## Summary
The `TxPool` maintains its own `Arc<Snapshot>` that is updated asynchronously via a bounded `try_send` channel. During the window between block commitment and snapshot update, a transaction spending an already-consumed (dead) cell resolves as live against the stale snapshot and is unconditionally admitted to the mempool, because the re-validation guard in `submit_entry` only fires when the snapshot tip changed between `pre_check` and `submit_entry`. If the snapshot remains stale throughout the entire admission pipeline, no re-validation occurs and the invalid transaction enters the pool and is relayed to peers.

## Finding Description

**Root cause — stale snapshot:**
`TxPool.snapshot()` is explicitly documented as potentially inconsistent with the chain: [1](#0-0) 

The snapshot is updated only when `_update_tx_pool_for_reorg` processes a reorg notification. That notification is sent via `try_send` on a bounded channel: [2](#0-1) 

`try_send` returns an error (only logged) if the channel is full, leaving the snapshot stale across an arbitrary number of blocks. The channel capacity is `DEFAULT_CHANNEL_SIZE = 512`. [3](#0-2) 

**Exploit path through `_process_tx`:**

1. `pre_check` acquires a read lock and clones the (potentially stale) snapshot. `resolve_tx_from_pool` builds an `OverlayCellProvider` over the stale snapshot; a cell spent in a recently committed block appears `Live`: [4](#0-3) 

2. `verify_rtx` runs with the stale snapshot's data loader and stale `tip_header` as `tx_env`: [5](#0-4) 

3. `submit_entry` contains the only mitigation — a re-check of cell liveness — but it is **conditional on the snapshot tip having changed**: [6](#0-5) 

If the snapshot has **not** been updated by the time `submit_entry` runs (i.e., `pre_resolve_tip == tip_hash` because both are from the same stale snapshot), the guard at line 120 is not entered. `_submit_entry` is called unconditionally, inserting the transaction with a dead input cell.

**Why the `pre_check` Dead-cell guard does not help:**
The guard at `pre_check` lines 292–303 only catches cells that are already dead *in the stale snapshot itself*. A cell that was spent in a block not yet reflected in the stale snapshot appears `Live` to `resolve_tx`, so the `OutPointError::Dead` branch is never reached and the guard never fires. [7](#0-6) 

## Impact Explanation

Invalid transactions admitted to the mempool are relayed to connected peers via the P2P relay protocol. Peers running the same code have the same stale-snapshot window and may also admit the transaction, propagating the invalid entry across the network. When the reorg channel is under backpressure (e.g., rapid block arrival or high tx throughput), the snapshot can remain stale across many blocks, widening the window significantly. An attacker can exploit this to flood the mempool and relay network with invalid transactions at low cost, matching the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

The vulnerability is reachable by any unprivileged RPC caller (`send_transaction`) or P2P transaction sender. Under normal conditions the window is milliseconds, requiring precise timing. However, the window is reliably widened under load: if the reorg notification channel fills (512 slots), `try_send` silently drops notifications and the snapshot stays stale across multiple blocks, making the attack straightforward to execute. No special privileges, keys, or majority hashpower are required.

## Recommendation

1. **Remove the conditional guard in `submit_entry`**: `check_rtx` (which uses `OverlayCellChecker` against the current snapshot) should run unconditionally, not only when `pre_resolve_tip != tip_hash`. This ensures cell liveness is always re-validated under the write lock against the latest snapshot.
2. **Replace `try_send` with a non-lossy send** for the reorg notification, or update the tx-pool snapshot synchronously before the chain service returns, to eliminate the stale-snapshot window entirely.
3. As a defense-in-depth measure, after `_update_tx_pool_for_reorg` completes, sweep remaining pool entries against the new snapshot to evict any entries whose inputs are now dead.

## Proof of Concept

**Setup:** Node N with tx-pool snapshot stale at block height H; actual chain tip is H+1 where cell C was spent by Tx A.

1. Attacker observes Tx A (spending cell C) committed in block H+1.
2. Before the tx-pool snapshot updates to H+1 (either within the normal millisecond window, or during an extended window caused by reorg channel backpressure), attacker submits Tx B (also spending cell C) via `send_transaction` RPC.
3. `pre_check` runs: `resolve_tx_from_pool` uses the stale snapshot at H → cell C appears `Live` → Tx B resolves successfully. The `OutPointError::Dead` branch is never reached.
4. `verify_rtx` runs with stale snapshot → passes.
5. `submit_entry` runs: `pre_resolve_tip == snapshot.tip_hash()` (both are H from the stale snapshot) → the guard at line 120 is **not entered** → `_submit_entry` inserts Tx B unconditionally.
6. Tx B is relayed to peers. Peers with the same stale window also admit it.
7. Eventually `update_tx_pool_for_reorg` runs and `resolve_conflict` evicts Tx B — but the invalid entry was live in the pool and propagated across the network during the window.

To reproduce the extended-window variant: submit blocks rapidly enough to fill the 512-slot reorg channel, then submit double-spend transactions; the stale snapshot persists until the channel drains, making every such submission succeed.
