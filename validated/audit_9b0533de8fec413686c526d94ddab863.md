### Title
Miner-Controlled Timestamp Inflation Constrains Subsequent Block Creation and Manipulates `since` Timelocks — (`verification/src/header_verifier.rs`)

---

### Summary

CKB's `TimestampVerifier` permits block timestamps up to `now + ALLOWED_FUTURE_BLOCKTIME` (15 seconds). A miner who sets a block timestamp at this maximum value inflates the rolling 37-block median time. This creates two concrete downstream effects that are structurally identical to the zkSync finding: (1) it constrains the valid timestamp window for subsequent blocks, and (2) it manipulates `since` timestamp-based timelocks evaluated by `SinceVerifier`.

---

### Finding Description

**Relevant constants and code:**

`ALLOWED_FUTURE_BLOCKTIME = 15,000 ms` (15 seconds): [1](#0-0) 

`MEDIAN_TIME_BLOCK_COUNT = 37`: [2](#0-1) 

`TimestampVerifier::verify()` enforces two bounds on every accepted block header:
- **Lower bound:** `block.timestamp > block_median_time(parent, 37 blocks)`
- **Upper bound:** `block.timestamp <= unix_time_as_millis() + ALLOWED_FUTURE_BLOCKTIME` [3](#0-2) 

`block_median_time()` collects the timestamps of the 37 most recent blocks and returns the median (index `len >> 1`, i.e., the upper-median for even counts): [4](#0-3) 

`SinceVerifier` uses `block_median_time` as the "current time" for both absolute and relative timestamp-based `since` checks: [5](#0-4) [6](#0-5) 

**The structural problem (direct CKB analog):**

A miner who wins a PoW block sets its timestamp to `T_N = now + 15,000 ms`. This is the maximum value accepted by `TimestampVerifier`. Once `T_N` enters the 37-block rolling window and becomes the median value, the next block must satisfy:

```
block_{N+1}.timestamp > T_N = now + 15,000 ms
```

But `block_{N+1}` is simultaneously bounded by:

```
block_{N+1}.timestamp <= now' + 15,000 ms
```

where `now'` is the wall-clock time when block N+1 is mined. For the constraint to be satisfiable, `now' > now` must hold — i.e., real time must advance. If the next block is mined within the same millisecond, no valid timestamp exists. This is the direct structural analog to the zkSync batch-timestamp constraint.

**Impact on `since` relative timestamp timelocks (post-CKB2021):**

After the CKB2021 hardfork, the base timestamp for a relative `since` lock is the **block's actual timestamp** (not the median):

```rust
let base_timestamp = self.data_loader
    .get_header_fields(&info.block_hash)
    .expect("header exist")
    .timestamp;  // block.timestamp, not median
``` [7](#0-6) 

A miner who mines the block containing a cell sets `block.timestamp = now + 15s`. The required unlock time for a relative timestamp lock becomes:

```
required_unlock = block.timestamp + since_duration
                = (now + 15s) + since_duration
```

This extends the effective relative timelock by up to 15 seconds compared to what the transaction author intended. This requires only **one block** from the attacker — the block that includes the cell being locked.

**Impact on `since` absolute timestamp timelocks:**

For absolute timestamp locks, the check is `block_median_time(parent) >= since_timestamp`. If a miner with sufficient hashpower mines 19 of the 37 blocks in the median window with timestamps at `now + 15s`, the median advances up to 15 seconds ahead of real time. This causes absolute timestamp-locked transactions to unlock up to 15 seconds **earlier** than the real-world time the lock author intended. [5](#0-4) 

---

### Impact Explanation

1. **Block creation rate constraint:** A miner who sets a block timestamp at `now + ALLOWED_FUTURE_BLOCKTIME` creates a window where the next block cannot be mined until real time advances past the inflated timestamp. While `ALLOWED_FUTURE_BLOCKTIME = 15s` limits the severity compared to zkSync's 365-day delta, the structural constraint is identical: once the inflated timestamp enters the median, subsequent blocks are constrained.

2. **Relative timelock extension (single-block attack):** Any miner who mines the block containing a cell can extend that cell's relative timestamp timelock by up to 15 seconds. This is a single-block, no-hashpower-majority attack. For time-sensitive DeFi contracts (options expiry, auction deadlines, payment channels), a 15-second manipulation window is exploitable.

3. **Absolute timelock early unlock (majority-hashpower attack):** A miner controlling 19/37 blocks can advance the median time by up to 15 seconds, causing absolute timestamp locks to expire early. This requires significant hashpower.

---

### Likelihood Explanation

- **Relative timelock manipulation:** Any miner who wins a single PoW block can execute this. No majority hashpower required. Likelihood is proportional to the attacker's hashrate share.
- **Absolute timelock manipulation / block creation constraint:** Requires mining 19+ of 37 consecutive blocks. Requires significant (near-majority) hashpower. Lower likelihood.
- The attack is permanent: once a block with an inflated timestamp is committed to the chain, its effect on the median window persists for the next 37 blocks and cannot be reversed.

---

### Recommendation

1. **Reduce `ALLOWED_FUTURE_BLOCKTIME`:** A smaller delta (e.g., 5 seconds) reduces the maximum timestamp inflation a single miner can introduce. [8](#0-7) 

2. **Enforce a stricter upper bound on the median itself:** In addition to bounding individual block timestamps, consider bounding how far the computed `block_median_time` can deviate from the node's local clock. This mirrors the recommendation in the zkSync report to apply L1-side timestamp constraints also on L2.

3. **Use median time (not raw block timestamp) as the base for relative `since` locks:** Post-CKB2021 changed the base to `block.timestamp`, which is directly miner-controlled. Reverting to `parent_median_time` as the base would reduce the single-block manipulation surface. [9](#0-8) 

---

### Proof of Concept

**Relative timelock extension (single block):**

1. Attacker mines block N containing a cell output with a relative timestamp `since` lock of duration D seconds.
2. Attacker sets `block_N.timestamp = unix_time_as_millis() + 15_000` (maximum allowed by `TimestampVerifier`).
3. The block passes `TimestampVerifier::verify()` because `timestamp <= now + ALLOWED_FUTURE_BLOCKTIME`.
4. Post-CKB2021, `SinceVerifier::verify_relative_lock()` computes `base_timestamp = block_N.timestamp = now + 15s`.
5. The required unlock condition becomes `current_median_time >= (now + 15s) + D`, effectively extending the lock by 15 seconds beyond the author's intent.

**Block creation constraint:**

1. Attacker mines 19 consecutive blocks, each with `timestamp = unix_time_as_millis() + 15_000`.
2. The 37-block median now equals `now + 15s`.
3. The next block must satisfy `timestamp > now + 15s` AND `timestamp <= now' + 15s`.
4. If `now' <= now` (blocks mined rapidly), no valid timestamp exists — block production stalls until real time advances past the inflated median. [10](#0-9) [2](#0-1)

    pub fn verify(&self) -> Result<(), Error> {
        // skip genesis block
        if self.header.is_genesis() {
            return Ok(());
        }

        let min = self.data_loader.block_median_time(
            &self.header.data().raw().parent_hash(),
            self.median_block_count,
        );
        if self.header.timestamp() <= min {
            return Err(TimestampError::BlockTimeTooOld {
                min,
                actual: self.header.timestamp(),
            }
            .into());
        }
        let max = self.now + ALLOWED_FUTURE_BLOCKTIME;
        if self.header.timestamp() > max {
            return Err(TimestampError::BlockTimeTooNew {
                max,
                actual: self.header.timestamp(),
            }
            .into());
        }
        Ok(())
    }
```

    fn block_median_time(&self, block_hash: &Byte32, median_block_count: usize) -> u64 {
        let mut timestamps: Vec<u64> = Vec::with_capacity(median_block_count);
        let mut block_hash = block_hash.clone();
        for _ in 0..median_block_count {
            let header_fields = self
                .get_header_fields(&block_hash)
                .expect("parent header exist");
            timestamps.push(header_fields.timestamp);
            block_hash = header_fields.parent_hash;

            if header_fields.number == 0 {
                break;
            }
        }

        // return greater one if count is even.
        timestamps.sort_unstable();
        timestamps[timestamps.len() >> 1]
    }
