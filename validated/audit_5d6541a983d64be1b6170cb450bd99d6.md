### Title
Concurrent `block_in_place` Script Verification via RPC `send_transaction` Exhausts Tokio Worker Thread Pool — (`tx-pool/src/util.rs`)

---

### Summary

The `send_transaction` RPC path spawns an independent tokio task per submission, each of which calls `tokio::task::block_in_place` for full script verification when no cache entry exists and no `command_rx` is provided. Submitting N transactions concurrently can block all tokio worker threads simultaneously, stalling block relay, P2P message handling, and all other async tasks for the full verification duration.

---

### Finding Description

The tx-pool service message loop spawns a **new independent tokio task** for every incoming message: [1](#0-0) 

Each `Message::SubmitLocalTx` task calls `service.process_tx(tx, None)`: [2](#0-1) 

`process_tx` calls `_process_tx` with `command_rx = None`: [3](#0-2) 

Inside `_process_tx`, `verify_rtx` is called with `command_rx = None`. Because there is no cache entry and no `command_rx`, execution falls into the `else` branch: [4](#0-3) 

`tokio::task::block_in_place` blocks the **current tokio worker thread** for the entire synchronous `ContextualTransactionVerifier::verify(max_tx_verify_cycles, false)` call. The tokio runtime is configured with `available_parallelism()` worker threads (equal to CPU count): [5](#0-4) 

If N ≥ worker_thread_count concurrent `send_transaction` RPC calls arrive with uncached, max-cycle scripts, all worker threads are simultaneously blocked. Tokio cannot run any other async tasks until at least one verification completes.

**Contrast with the `VerifyMgr` worker path:** Workers in `verify_mgr.rs` always pass `Some(&mut self.command_rx)` to `_process_tx`, taking the `verify_with_pause` async path that yields cooperatively and never blocks a worker thread: [6](#0-5) 

The `SubmitLocalTx` path entirely bypasses the `VerifyMgr` queue and its cooperative scheduling.

---

### Impact Explanation

All tokio worker threads blocked → async task starvation. Affected tasks include block relay, compact block processing, P2P message handling, and sync. The node becomes unresponsive to the network for the duration of the burst. `max_tx_verify_cycles` defaults to `TWO_IN_TWO_OUT_CYCLES * 20 ≈ 70,000,000` cycles: [7](#0-6) 

On modern hardware, 70M cycles of CKB-VM execution can take hundreds of milliseconds per transaction. With CPU-count concurrent submissions, the stall duration equals the full verification time of one transaction (since threads are freed as verifications complete, but the window of total starvation is real).

Scoped impact: **Low (501–2000)** — performance degradation / async task starvation during burst submission. No funds at risk, no consensus violation.

---

### Likelihood Explanation

- Attacker needs no privileges; `send_transaction` is an unauthenticated RPC endpoint.
- Attacker needs valid UTXOs and fee payment — a real but low barrier (a handful of CKB suffices).
- Scripts near the cycle limit are valid on mainnet (e.g., complex lock scripts).
- No cache entries exist for novel transactions, guaranteeing the `block_in_place` branch.
- Triggering this requires only N = CPU-count concurrent HTTP requests, trivially achievable.

---

### Recommendation

Route `SubmitLocalTx` through the `VerifyMgr` queue (i.e., call `resumeble_process_tx` / `enqueue_verify_queue` instead of `process_tx` with `command_rx=None`). This ensures all script verification uses the cooperative `verify_with_pause` path and is bounded by `max_tx_verify_workers`, preventing unbounded worker thread exhaustion. Alternatively, replace `block_in_place` with `tokio::task::spawn_blocking` so verification runs on a dedicated blocking thread pool that does not consume worker threads.

---

### Proof of Concept

```
1. Prepare N = num_cpus() transactions, each spending a valid UTXO with a script
   that consumes ~70,000,000 cycles (e.g., a tight loop in CKB-VM).
2. Ensure none of these transactions are in the tx verification cache.
3. Concurrently send N HTTP POST requests to the node's RPC endpoint:
     {"method": "send_transaction", "params": [<tx_i>, "passthrough"]}
4. Simultaneously, measure latency of a lightweight async RPC call (e.g., get_tip_block_number)
   issued from a separate client.
5. Observe: the lightweight RPC call stalls for the full script verification duration
   (hundreds of ms) because all tokio worker threads are blocked in block_in_place.
6. Assert: ping-pong latency exceeds 10x baseline during the burst window.
```
