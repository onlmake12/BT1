### Title
Unbounded Iteration Over All Tx-Pool Entries in `get_raw_tx_pool` RPC Causes Excessive Resource Consumption — (File: `tx-pool/src/pool.rs`, `rpc/src/module/pool.rs`)

---

### Summary
The `get_raw_tx_pool` RPC endpoint iterates over every entry in the transaction pool without pagination or a result-count limit. A tx-pool submitter can pre-fill the pool with many minimum-size transactions, then an RPC caller can repeatedly invoke `get_raw_tx_pool(verbose=true)` to trigger an O(n) full-pool scan plus full JSON serialization on each call, exhausting CPU and memory on the node.

---

### Finding Description
`get_raw_tx_pool` in `rpc/src/module/pool.rs` dispatches to either `get_all_entry_info()` (verbose=true) or `get_all_ids()` (verbose=false), both of which iterate the entire pool with no upper bound on the number of entries returned.

`get_all_entry_info()` in `tx-pool/src/pool.rs` performs three unbounded scans:

```rust
// tx-pool/src/pool.rs  lines 464-487
pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
    let pending = self
        .pool_map
        .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
        .map(|entry| (entry.transaction().hash(), entry.to_info()))
        .collect();                          // ← no limit

    let proposed = self
        .pool_map
        .sorted_proposed_iter()
        .map(|entry| (entry.transaction().hash(), entry.to_info()))
        .collect();                          // ← no limit

    let conflicted = self
        .conflicts_cache
        .iter()
        .map(|(_id, tx)| tx.hash())
        .collect();                          // ← no limit
    ...
}
```

`score_sorted_iter_by_statuses` itself iterates the entire multi-index map and filters by status with no early exit:

```rust
// tx-pool/src/component/pool_map.rs  lines 408-416
pub(crate) fn score_sorted_iter_by_statuses(
    &self,
    statuses: Vec<Status>,
) -> impl Iterator<Item = &TxEntry> {
    self.entries
        .iter_by_score()
        .rev()
        .filter_map(move |entry| statuses.contains(&entry.status).then_some(&entry.inner))
}
```

The RPC handler wires this directly to the network-facing endpoint with no rate-limiting or pagination:

```rust
// rpc/src/module/pool.rs  lines 703-718
fn get_raw_tx_pool(&self, verbose: Option<bool>) -> Result<RawTxPool> {
    let tx_pool = self.shared.tx_pool_controller();
    let raw = if verbose.unwrap_or(false) {
        let info = tx_pool.get_all_entry_info()...;   // full scan
        RawTxPool::Verbose(info.into())
    } else {
        let ids = tx_pool.get_all_ids()...;           // full scan
        RawTxPool::Ids(ids.into())
    };
    Ok(raw)
}
```

A secondary unbounded scan exists in `remove_expired()` (`tx-pool/src/pool.rs` lines 271-288), which iterates every pool entry on every block via `_update_tx_pool_for_reorg`. With a full pool this is O(n) work triggered automatically on each new block.

---

### Impact Explanation
The tx-pool is bounded by `max_tx_pool_size` bytes, but with minimum-size transactions (~100–200 bytes each) the pool can hold tens of thousands of entries. Each `get_raw_tx_pool(verbose=true)` call:
- Acquires the pool write-lock (via the async channel) for the full duration of the scan
- Allocates a `HashMap` containing one `TxEntryInfo` struct per entry
- Serializes the entire result to JSON before returning

Repeated calls from a single RPC client can starve the pool's async service loop, delaying or blocking transaction submission, block assembly (`get_block_template`), and reorg processing for all other users. This constitutes a denial-of-service against the node's tx-pool service.

---

### Likelihood Explanation
The attack requires two unprivileged roles that are explicitly in scope:
1. **Tx-pool submitter**: submits many small, valid, low-fee transactions to fill the pool.
2. **RPC caller**: repeatedly calls `get_raw_tx_pool(verbose=true)`.

Both roles require no special privilege. The RPC port is commonly exposed to localhost or a local network; operators who expose it more broadly (e.g., public RPC nodes) face direct external exploitation. Even on a localhost-only node, a compromised co-located process or a script running on the same host can trigger this.

---

### Recommendation
1. Add a `limit` / `after` pagination parameter to `get_raw_tx_pool` and enforce a hard cap (e.g., 1 000 entries per call) in `get_all_entry_info` and `get_all_ids`.
2. In `remove_expired`, consider maintaining a separate time-ordered index of entries so expiry scanning is O(expired) rather than O(total).
3. Apply per-IP or per-connection rate-limiting to expensive pool-dump RPC methods.

---

### Proof of Concept

**Step 1 – Fill the pool** (tx-pool submitter role):
```python
# Submit N minimum-size transactions with fee just above min_fee_rate
for i in range(50_000):
    rpc.send_transaction(build_min_size_tx(i))
```

**Step 2 – Trigger unbounded scan** (RPC caller role):
```python
import time, requests
while True:
    t0 = time.time()
    requests.post("http://node:8114", json={
        "jsonrpc": "2.0", "id": 1,
        "method": "get_raw_tx_pool",
        "params": [True]          # verbose=true → get_all_entry_info()
    })
    print(f"call took {time.time()-t0:.2f}s")
    # Each call holds the pool lock for the full scan duration,
    # blocking all concurrent pool operations.
```

**Observed effect**: Pool service loop is saturated; `send_transaction` and `get_block_template` calls queue up and time out. The `remove_expired` path (`tx-pool/src/pool.rs:271`) compounds the effect on every new block.

        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())

        }
