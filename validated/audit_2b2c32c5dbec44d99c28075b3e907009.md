Audit Report

## Title
Missing `from != to` Identity Check and Absent Sender-Binding in `ConnectionRequestProcess::execute` Enables `pending_delivered` Poisoning and Forced NAT Traversal to Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

`ConnectionRequestProcess::execute` contains no guard enforcing `content.from != content.to`, and neither `ConnectionRequestProcess` nor `ConnectionSyncProcess` validates that `content.from` matches the actual sender's session `PeerId`. An attacker with a single established P2P connection can send a `ConnectionRequest` with both `from` and `to` set to the victim's own `PeerId`, causing the victim to insert attacker-controlled TCP addresses into `pending_delivered` under its own key. A subsequent `ConnectionSync` (also with `from == to == victim_peer_id`) causes the victim to spawn a 30-second NAT traversal retry loop to attacker-controlled endpoints and, on success, register a `raw_session` to the attacker — bypassing normal peer selection and scoring.

## Finding Description

**Root cause — no `from != to` guard and no sender-identity binding:**

In `ConnectionRequestProcess::execute`, after all validation checks, the only routing branch is:

```rust
if self_peer_id == &content.to {
    self.respond_delivered(content.from, &content.to, content.listen_addrs).await
``` [1](#0-0) 

There is no check that `content.from != content.to`, and `content.from` is never validated against the actual session peer ID. `self.peer` is a `PeerIndex` only — it is never resolved to a `PeerId` and compared against `content.from`. [2](#0-1) 

**Step 1 — `pending_delivered` poisoned:**

Inside `respond_delivered`, after filtering `remote_listens` to TCP/IPv4/IPv6 addresses (attacker-controlled TCP addresses pass this filter), the state is written unconditionally:

```rust
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
``` [3](#0-2) 

`pending_delivered` now maps `victim_peer_id → ([attacker_addr], timestamp)`. The only existing guard is a 2-minute cooldown check on the same key, which the attacker satisfies trivially on first use. [4](#0-3) 

**Step 2 — `ConnectionSync` triggers NAT traversal to attacker-controlled addresses:**

In `ConnectionSyncProcess::execute`, when `route` is empty and `self_peer_id == &content.to`, the handler reads back the poisoned entry:

```rust
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
``` [5](#0-4) 

With `content.from = victim_peer_id`, it retrieves the attacker-controlled addresses and spawns NAT traversal tasks: [6](#0-5) 

`try_nat_traversal` retries TCP connections for up to 30 seconds with jitter, consuming victim resources. [7](#0-6) 

**Step 3 — `raw_session` registered to attacker on success:**

```rust
let _ignore = control
    .raw_session(stream, addr, RawSessionInfo::inbound(listen_addr))
    .await;
``` [8](#0-7) 

This registers an inbound session to the attacker's address, bypassing normal peer scoring and selection.

**Critically, `pending_delivered` is never cleared after `ConnectionSync` processes it** — the entry persists until the 5-minute `TIMEOUT` cleanup in `notify`. The `forward_rate_limiter` for `ConnectionSync` is keyed by `(from, to, msg_item_id)` = `(victim_peer_id, victim_peer_id, id)`, allowing 1 message per second. Over the 5-minute lifetime of the poisoned entry, the attacker can spawn up to ~300 concurrent 30-second NAT traversal tasks. [9](#0-8) 

**Existing checks are insufficient:**

- The per-session rate limiter (30 req/s keyed by `(session_id, msg_item_id)`) limits message throughput but does not prevent the attack. [10](#0-9) 
- The forward rate limiter (1 req/s keyed by `(from, to, msg_item_id)`) limits repetition but does not block the first attempt, and with `from == to` the key space is not expanded. [11](#0-10) 
- The `pending_delivered` 2-minute cooldown only prevents re-poisoning the same key within the window; it does not prevent the initial poisoning. [12](#0-11) 

## Impact Explanation

This maps to **High severity — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The attacker forces the victim to: (1) corrupt `pending_delivered` with attacker-controlled addresses keyed under the victim's own `PeerId`; (2) initiate outbound TCP connections to attacker-controlled endpoints for up to 30 seconds per `ConnectionSync` trigger; (3) accumulate up to ~300 concurrent async NAT traversal tasks over a 5-minute window (1/s rate × 5-minute `pending_delivered` lifetime), exhausting async runtime resources; (4) register a `raw_session` to the attacker on TCP handshake success, bypassing normal peer selection and scoring. Across multiple rate-limit windows, the attacker can accumulate unauthorized peer connections, with a credible path toward peer-slot saturation and eclipse-style network isolation.

## Likelihood Explanation

The attack requires only a single established P2P connection — no special privileges, no PoW, no key material. The victim's `PeerId` is its public network identity, known to any connected peer. The two crafted messages (`ConnectionRequest` then `ConnectionSync`) are valid wire-format messages that pass all existing syntactic checks. The `from` field is never verified against the sender's actual session `PeerId`, making identity spoofing trivial. The forward rate limiter allows 1 attempt per second per `(from, to, msg_type)` tuple; the `pending_delivered` cooldown allows re-triggering every 2 minutes. Both constraints are easily managed by a persistent attacker.

## Recommendation

1. Add an explicit identity check immediately after parsing in `ConnectionRequestProcess::execute`:
```rust
if content.from == content.to {
    return StatusCode::InvalidFromPeerId
        .with_context("from and to must be distinct peers");
}
``` [13](#0-12) 

2. Apply the same guard in `ConnectionSyncProcess::execute`. [14](#0-13) 

3. Validate that `content.from` matches the sender's actual session `PeerId` (resolvable from `self.peer` via the peer registry) to prevent all identity spoofing, not just the `from == to` case. [2](#0-1) 

4. Remove the `pending_delivered` entry after it is consumed by `ConnectionSyncProcess` to prevent repeated NAT traversal spawning from a single poisoning event. [5](#0-4) 

## Proof of Concept

```
1. Attacker peer A establishes a P2P connection to victim V (V.peer_id is V's public identity).

2. A sends ConnectionRequest {
       from: V.peer_id,
       to:   V.peer_id,
       listen_addrs: [/ip4/<attacker_ip>/tcp/<attacker_port>],
       max_hops: 1,
       route: [],
   }

3. V processes it:
   - forward_rate_limiter key (V.peer_id, V.peer_id, ConnectionRequest_id): passes (first use)
   - self_peer_id == content.to → respond_delivered() called
   - 2-minute cooldown: passes (first use)
   - remote_listens filtered: TCP/IPv4 passes
   - pending_delivered[V.peer_id] = ([/ip4/<attacker_ip>/tcp/<attacker_port>], now)
   - ConnectionRequestDelivered sent back to A (ignored by attacker)

4. A sends ConnectionSync {
       from: V.peer_id,
       to:   V.peer_id,
       route: [],
   }

5. V processes it:
   - route empty, self_peer_id == content.to → NAT traversal branch
   - pending_delivered.get(V.peer_id) → [/ip4/<attacker_ip>/tcp/<attacker_port>]
   - try_nat_traversal(bind_addr, attacker_addr) spawned (30s retry loop)
   - On TCP accept at attacker_ip:attacker_port:
     raw_session(stream, attacker_addr, RawSessionInfo::inbound(listen_addr)) registered

6. A repeats step 4 at 1/second for 5 minutes (pending_delivered entry persists until TIMEOUT):
   - ~300 concurrent 30-second NAT traversal tasks spawned on V
   - Each successful TCP handshake registers an additional raw_session to the attacker

Assert:
  - pending_delivered contains V.peer_id → attacker_addr (should be absent)
  - V has initiated TCP connections to attacker_ip:attacker_port
  - V has raw inbound sessions registered to the attacker
  - V's async runtime is saturated with up to ~300 concurrent NAT traversal tasks
```
