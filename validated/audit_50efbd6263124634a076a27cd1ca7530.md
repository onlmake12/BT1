### Title
`get_twap_update_fee` Returns Hardcoded Zero Instead of Actual Fee — (`File: target_chains/stylus/contracts/pyth-receiver/src/lib.rs`)

### Summary

The Stylus Rust implementation of the Pyth receiver contract hardcodes `get_twap_update_fee` to return `U256::from(0u8)` (zero) instead of computing the actual fee based on the number of price updates in the submitted data. This is directly analogous to the reported bug where a price oracle returned `1` (1 wei) instead of `1e18` — both are hardcoded wrong values in a critical accounting function.

### Finding Description

In `target_chains/stylus/contracts/pyth-receiver/src/lib.rs`, the `get_twap_update_fee` function is implemented as:

```rust
pub fn get_twap_update_fee(&self, _update_data: Vec<Vec<u8>>) -> U256 {
    U256::from(0u8)
}
```

The `_update_data` parameter is entirely ignored and the function unconditionally returns zero.

Compare this to the correct implementation in the same file for `get_update_fee` (lines 300–318), which properly parses the accumulator update data, counts the number of individual price updates, and multiplies by `single_update_fee_in_wei`:

```rust
Ok(
    U256::from(total_num_updates).saturating_mul(self.single_update_fee_in_wei.get())
        + self.transaction_fee_in_wei.get(),
)
```

The canonical Solidity implementation in `target_chains/ethereum/contracts/contracts/pyth/Pyth.sol` (lines 123–152) also correctly computes the TWAP fee by parsing the first VAA's update count and calling `getTotalFee(totalNumUpdates)`. The `parseTwapPriceFeedUpdates` function in `Pyth.sol` (line 505–506) enforces this fee:

```solidity
uint requiredFee = getTwapUpdateFee(updateData);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
```

Because the Stylus `get_twap_update_fee` returns 0, any caller querying this function to determine the required fee will be told the cost is zero. If `parseTwapPriceFeedUpdates` in the Stylus contract uses `get_twap_update_fee` for its fee enforcement check (consistent with the Solidity pattern), TWAP price feed updates can be submitted for free, bypassing the protocol's fee collection entirely.

### Impact Explanation

- **Fee bypass**: Any unprivileged transaction sender can submit TWAP price feed updates to the Stylus Pyth receiver without paying any fee, since `get_twap_update_fee` returns 0.
- **Protocol revenue loss**: The Pyth protocol collects fees per price update to sustain operations. TWAP updates submitted via the Stylus receiver would generate zero revenue.
- **Incorrect client behavior**: Any off-chain client or on-chain contract that calls `get_twap_update_fee` to pre-compute the required payment will receive 0, causing them to send 0 value. If the actual enforcement path ever gets corrected without fixing this function, all existing integrations would break.

This is a direct analog to the reported bug: a hardcoded wrong value (`0` instead of `fee_per_update × num_updates`) in a fee-reporting function that downstream logic depends on.

### Likelihood Explanation

The entry path is fully permissionless — any transaction sender can call `parseTwapPriceFeedUpdates` on the Stylus receiver. No privileged role, leaked key, or external oracle manipulation is required. The bug is triggered simply by submitting valid TWAP update data with `msg.value = 0`.

### Recommendation

Replace the hardcoded return with the same fee computation logic used in `get_update_fee`:

```rust
pub fn get_twap_update_fee(&self, update_data: Vec<Vec<u8>>) -> Result<U256, PythReceiverError> {
    // TWAP updateData is always length 2 (start + end); count updates from first VAA only
    let first = update_data.first().ok_or(PythReceiverError::InvalidAccumulatorMessage)?;
    let accumulator_update = AccumulatorUpdateData::try_from_slice(first)
        .map_err(|_| PythReceiverError::InvalidAccumulatorMessage)?;
    let num_updates = match accumulator_update.proof {
        Proof::WormholeMerkle { vaa: _, updates } =>
            u64::try_from(updates.len()).map_err(|_| PythReceiverError::TooManyUpdates)?,
    };
    Ok(
        U256::from(num_updates).saturating_mul(self.single_update_fee_in_wei.get())
            + self.transaction_fee_in_wei.get(),
    )
}
```

### Proof of Concept

1. Deploy the Stylus `PythReceiver` contract with `single_update_fee_in_wei = 100` and `transaction_fee_in_wei = 32`.
2. Construct valid TWAP update data containing N price feeds.
3. Call `get_twap_update_fee(update_data)` — it returns `0` regardless of N.
4. Call `parseTwapPriceFeedUpdates` with `msg.value = 0` — the fee check passes because `0 >= 0`, and the TWAP update is processed for free.

                revert PythErrors.InvalidUpdateData();

                offset
            );

            revert PythErrors.InvalidUpdateData();
