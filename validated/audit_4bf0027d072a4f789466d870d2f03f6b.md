Audit Report

## Title
Global `TimeAnalyzer` Thresholds Poisonable by Fast Peer, Leading to Legitimate Peer Disconnection — (`sync/src/types/mod.rs`)

## Summary

`InflightBlocks` holds a single, process-wide `TimeAnalyzer` whose three thresholds (`fast_time`, `normal_time`, `low_time`) are updated via an EMA over a sliding window of block-download latencies contributed by **all** peers. A peer that responds to block requests with near-zero latency can fill the shared buffer with sub-millisecond samples, driving `low_time` toward zero. Once `low_time` is near-zero, every block delivery from a legitimate peer is classified `UpperToMax`, causing repeated `decrease` calls that drain `task_count` to zero and disconnect the peer.

## Finding Description

**Root cause — shared, attacker-writable `TimeAnalyzer`**

`TimeAnalyzer` is a field of `InflightBlocks`, which is a single shared instance. [1](#0-0) 

Every block receipt calls `remove_by_block`, which measures elapsed time and feeds it into the global analyzer: [2](#0-1) 

When the buffer (`TIME_TRACE_SIZE = MAX_BLOCKS_IN_TRANSIT_PER_PEER * 4`) is full, `push_time` sorts the trace and updates thresholds via EMA: [3](#0-2) 

`LOW_INDEX = TIME_TRACE_SIZE * 9 / 10` (90th percentile of the sorted window). [4](#0-3) 

If an attacker fills the entire buffer with near-zero samples, `trace[LOW_INDEX]` ≈ 0, and the EMA converges: `low_time_new = (low_time_old + 0) >> 1`. Starting from the default of 1500 ms, after ~10 buffer cycles `low_time` reaches 0.

**How a poisoned `low_time` harms legitimate peers**

`push_time` returns `UpperToMax` when `time > self.low_time`. With `low_time ≈ 0`, every non-zero response time triggers this branch: [5](#0-4) 

`UpperToMax` calls `set.decrease(2)` in `remove_by_block`: [6](#0-5) 

`decrease` decrements `task_count` by 1 on every invocation (the `timeout_count` field is overwritten with `task_count + num` each call, which is always > 2, so the decrement fires unconditionally): [7](#0-6) 

Starting from `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32), after 32 block deliveries `task_count` reaches 0 and the peer is added to `disconnect_list`: [8](#0-7) 

A second, faster disconnection path exists in `prune` via `punish(1)` (halves `task_count`) when `now > low_time + *time`, but this path only activates when `trace_number` is non-empty (anomalous sync conditions), so it is a secondary concern. [9](#0-8) 

**Existing guards and why they are insufficient**

- `should_punish = download_schedulers.len() > protect_num`: punishment is suppressed when peer count is low. However, an attacker running multiple Sybil connections trivially bypasses this guard, and during normal post-IBD operation the node has enough peers for `should_punish` to be true. [10](#0-9) 

- The `adjustment` flag can disable penalties globally, but it defaults to `true` and is not toggled by the attack. [11](#0-10) 

- There is no floor on `low_time`, no per-peer sample quota, and no outlier rejection.

## Impact Explanation

Once `low_time` is poisoned, the victim node systematically disconnects every legitimate peer whose block-delivery latency exceeds the artificially low threshold. With honest peers gone, block synchronization halts: the node cannot advance its chain tip, validate new blocks, or relay transactions. Because the poisoned EMA persists in memory and legitimate peers are evicted before they can contribute corrective samples, recovery is self-blocking. This constitutes a targeted, sustained DoS against a CKB node — mapping to **High: Vulnerabilities which could easily crash a CKB node** (render it non-functional/isolated).

## Likelihood Explanation

- Any unprivileged peer can connect; no key or stake is required.
- Responding in < 1 ms is trivially achievable by pre-downloading the chain and serving blocks from RAM.
- During IBD the node syncs primarily from one selected peer, giving the attacker near-total control over all `TIME_TRACE_SIZE` samples in a single buffer cycle.
- After IBD, the attacker exploits the feedback loop: fast responses earn `increase` rewards, raising `task_count` and causing the scheduler to assign more blocks to the attacker, further dominating the sample buffer.
- A single Sybil connection is sufficient during IBD; a handful of Sybil connections bypasses the `should_punish` guard post-IBD.
- The poisoned EMA is persistent: once driven to near-zero, `low_time` recovers only if enough legitimate peers survive long enough to contribute high-latency samples — which the attack prevents.

## Recommendation

1. **Per-peer time tracking**: Maintain a separate `TimeAnalyzer` per peer. Peer-specific thresholds prevent one peer from poisoning the evaluation of others.
2. **Hard floor on `low_time`**: Enforce `low_time = max(low_time, MIN_LOW_TIME)` (e.g., `BLOCK_DOWNLOAD_TIMEOUT / 4`) after every EMA update so the threshold cannot drop below a safe operational minimum.
3. **Per-peer sample quota**: Cap how many samples a single peer can contribute to the global analyzer per time window, preventing a single fast peer from dominating the distribution.
4. **Outlier rejection / trimmed mean**: Discard samples below the 5th percentile before updating the EMA, or use a trimmed mean that ignores the bottom K% of the window.

## Proof of Concept

```
Preconditions:
  - Victim node is in or has recently completed IBD.
  - Attacker controls one outbound sync peer with the full chain pre-downloaded.
  - TIME_TRACE_SIZE = MAX_BLOCKS_IN_TRANSIT_PER_PEER * 4 (e.g., 128 if MAX = 32).

Steps:
1. Attacker peer responds to every GetBlocks request in < 1 ms (blocks served from RAM).

2. After TIME_TRACE_SIZE responses, push_time triggers an EMA update:
     trace.sort_unstable()  →  trace[LOW_INDEX=115] ≈ 0
     low_time = (1500 + 0) >> 1 = 750

3. Repeat for ~10 buffer cycles:
     750 → 375 → 187 → 93 → 46 → 23 → 11 → 5 → 2 → 1 → 0

4. Legitimate peers now connect (post-IBD). Their block responses take 200–500 ms.
   push_time returns UpperToMax for every delivery (200 > 0).
   remove_by_block calls decrease(2) → task_count decrements by 1 per block.

5. After INIT_BLOCKS_IN_TRANSIT_PER_PEER (32) block deliveries:
     task_count == 0
     prune() → disconnect_list.insert(legitimate_peer)

6. All honest peers are disconnected; victim node is isolated and sync-stalled.

Verification: add a unit test to inflight_blocks.rs that:
  a. Creates an InflightBlocks with two peers (attacker + victim).
  b. Simulates TIME_TRACE_SIZE * 10 near-zero-latency remove_by_block calls for the attacker.
  c. Asserts low_time < 5.
  d. Simulates 32 remove_by_block calls with 300 ms elapsed for the victim peer.
  e. Asserts the victim peer appears in the disconnect_list returned by prune().
```
