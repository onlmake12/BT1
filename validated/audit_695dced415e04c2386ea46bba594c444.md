### Title
Missing Caller Authentication in `advanceProviderCommitment` Allows Any Unprivileged Actor to Advance Any Provider's Hash Chain State — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.advanceProviderCommitment` is a `public` function that modifies a provider's on-chain commitment state (`currentCommitment`, `currentCommitmentSequenceNumber`, and conditionally `sequenceNumber`) but contains **no check that `msg.sender == provider`**. Any unprivileged caller who can supply a valid hash-chain preimage can advance any registered provider's commitment without the provider's knowledge or consent.

---

### Finding Description

The function signature is:

```solidity
function advanceProviderCommitment(
    address provider,
    uint64 advancedSequenceNumber,
    bytes32 providerContribution
) public override {
``` [1](#0-0) 

The only validation performed is:

1. `advancedSequenceNumber > currentCommitmentSequenceNumber` (not stale)
2. `advancedSequenceNumber < endSequenceNumber` (within chain bounds)
3. `constructProviderCommitment(numHashes, providerContribution) == currentCommitment` (valid preimage) [2](#0-1) 

There is **no** `require(msg.sender == provider)` guard. Compare this with `register`, `withdraw`, `setMaxNumHashes`, and `setDefaultGasLimit`, all of which use `msg.sender` as the implicit provider identity: [3](#0-2) 

The hash-chain preimage required to pass the cryptographic check is **publicly available**: the Fortuna service exposes a `/revelations/{sequence}` endpoint that returns `x_j` for any sequence number `j` that has been requested on-chain. [4](#0-3) 

Additionally, every `Revealed` / `RevealedWithCallback` event emits `providerContribution` in plaintext, making past preimages trivially observable from chain history. [5](#0-4) 

---

### Impact Explanation

**Unauthorized state mutation of any provider.** An attacker can:

1. Request a sequence number `j` from any provider (paying the fee).
2. Immediately fetch `x_j` from the Fortuna API (the endpoint returns it once the request is on-chain).
3. Call `advanceProviderCommitment(provider, j, x_j)` — no provider signature or consent required.

This updates `currentCommitmentSequenceNumber = j` and `currentCommitment = x_j` for the targeted provider without the provider ever calling the function.

**Capacity-burning DoS.** The contract's own comment documents the critical branch:

```solidity
// This means the provider called the function with a sequence number that was not yet requested.
// Providers should never do this and we consider such an implementation flawed.
// Assuming this is landed on-chain it's better to bump the sequence number and never use that range
// for future requests. Otherwise, someone can use the leaked revelation to derive favorable random numbers.
providerInfo.sequenceNumber =
    providerInfo.currentCommitmentSequenceNumber + 1;
``` [6](#0-5) 

If an attacker can supply `x_j` for `j >= sequenceNumber` (a future, unrequested slot), the contract permanently burns that slot by bumping `sequenceNumber`. Because the Fortuna API only returns values for on-chain-requested sequence numbers, triggering this branch requires the attacker to first request the slot themselves (paying the fee), then immediately call `advanceProviderCommitment` before the provider fulfills it — advancing to exactly `sequenceNumber - 1` does not trigger the branch, but advancing to `sequenceNumber` (the slot the attacker just requested, before `sequenceNumber` is incremented by the request) is not reachable in a single atomic transaction. The capacity-burning path therefore requires a race condition or a provider with a very slow fulfillment loop, making it a realistic griefing vector under congestion or when the Fortuna keeper is delayed.

**Disruption of provider gas-optimization strategy.** Even without burning capacity, an attacker can advance the commitment to any in-flight sequence number at will, overriding the provider's own `update_commitments_loop` logic and potentially causing the provider's keeper to behave unexpectedly. [7](#0-6) 

---

### Likelihood Explanation

- The function is `public` with no access control.
- The required preimage is publicly available from the Fortuna HTTP API and from on-chain events.
- No special privileges, leaked keys, or governance majority are needed.
- Any unprivileged Ethereum address can call this function at any time.

---

### Recommendation

Add a caller-authentication guard identical to the pattern used by every other provider-mutating function in the contract:

```solidity
function advanceProviderCommitment(
    address provider,
    uint64 advancedSequenceNumber,
    bytes32 providerContribution
) public override {
+   if (msg.sender != provider) revert EntropyErrors.Unauthorized();
    ...
}
```

This is consistent with `register`, `withdraw`, `setMaxNumHashes`, `setDefaultGasLimit`, and `setProviderFee`, all of which implicitly authenticate via `msg.sender`.

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

interface IEntropy {
    function advanceProviderCommitment(
        address provider,
        uint64 advancedSequenceNumber,
        bytes32 providerRevelation
    ) external;
}

contract AttackerAdvanceCommitment {
    // 1. Attacker calls requestWithCallback(provider, userContribution) to get seq j.
    // 2. Attacker fetches x_j from Fortuna API: GET /v1/chains/{chain_id}/revelations/{j}
    // 3. Attacker calls this function — no provider key or signature needed.
    function attack(
        address entropy,
        address provider,
        uint64 seqNumber,
        bytes32 providerReveal
    ) external {
        IEntropy(entropy).advanceProviderCommitment(
            provider,      // victim provider — attacker does NOT need to be this address
            seqNumber,
            providerReveal // obtained from public Fortuna API
        );
        // provider's currentCommitment and currentCommitmentSequenceNumber are now
        // modified without the provider's knowledge or consent.
    }
}

        send_and_confirm(contract_call).await?;
