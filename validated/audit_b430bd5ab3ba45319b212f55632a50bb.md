### Title
`_state.pyth` Oracle Address Cannot Be Updated After Initialization — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
The `Echo` contract stores the Pyth oracle address in `_state.pyth` during `_initialize()` with no admin-accessible setter to update it. If the Pyth oracle contract is ever redeployed or migrated to a new address, the `Echo` contract becomes permanently broken: all `executeCallback` calls revert, pending requests can never be fulfilled, and user-paid fees are locked in the contract. The same issue exists in `Scheduler.sol` (`_state.pyth`).

### Finding Description
In `Echo._initialize()`, the Pyth oracle address is written to storage once:

```solidity
// Echo.sol line 30
_state.pyth = pythAddress;
```

The `executeCallback` function uses this address to parse and verify price updates:

```solidity
IPyth(_state.pyth).parsePriceFeedUpdates(...)
```

Neither `Echo.sol`, `EchoUpgradeable.sol`, nor the `IEcho` interface exposes any function to update `_state.pyth`. The governance contract (`EchoGovernance` / admin functions) provides setters for `pythFeeInWei`, `defaultProvider`, `exclusivityPeriod`, and `admin`, but **no `setPythAddress`**.

The identical pattern exists in `Scheduler._initialize()`:

```solidity
// Scheduler.sol line 25
_state.pyth = pythAddress;
```

`SchedulerGovernance.sol` exposes `setSingleUpdateKeeperFeeInWei` and `setMinimumBalancePerFeed` but no setter for `_state.pyth`.

### Impact Explanation
If the Pyth oracle is redeployed to a new address (e.g., due to a critical bug fix, upgrade, or chain migration), `Echo.executeCallback` will permanently revert on every call because `IPyth(_state.pyth).parsePriceFeedUpdates(...)` will fail against the stale address. All in-flight requests — for which users already paid fees — can never be fulfilled. The fees are locked in the contract with no recovery path short of a full redeployment of `Echo` and all dependent contracts. This constitutes a permanent denial-of-service for the Echo protocol and loss of user funds.

### Likelihood Explanation
Pyth oracle contracts are upgradeable proxies on most chains, so a direct address change is unlikely in normal operation. However, Pyth has historically deployed new contract versions (e.g., the move from legacy to UUPS proxy), and chain-specific incidents could force a redeployment. The `Echo` contract is a newer production contract with no migration path for `_state.pyth`, making this a realistic operational risk with high impact when triggered.

### Recommendation
Add a privileged setter for the Pyth oracle address in `Echo` (and `Scheduler`), callable only by the owner or admin, analogous to the existing `setDefaultProvider`:

```solidity
function setPythAddress(address newPyth) external {
    require(msg.sender == owner() || msg.sender == _state.admin, "Unauthorized");
    require(newPyth != address(0), "zero address");
    address old = _state.pyth;
    _state.pyth = newPyth;
    emit PythAddressUpdated(old, newPyth);
}
```

Apply the same fix to `Scheduler.sol`.

### Proof of Concept

1. `EchoUpgradeable` is deployed with `pythAddress = 0xABC...` (current Pyth oracle).
2. A user calls `requestPriceUpdatesWithCallback(...)`, paying a fee. The fee is stored in `req.fee` and `_state.accruedFeesInWei` is incremented.
3. Pyth Network redeploys its oracle to `0xDEF...` (old address is now defunct).
4. The off-chain executor calls `executeCallback(...)`.
5. Inside `executeCallback`, `IPyth(_state.pyth).parsePriceFeedUpdates(...)` is called against `0xABC...` which is now defunct — the call reverts.
6. No admin function exists to update `_state.pyth` to `0xDEF...`.
7. All pending requests are permanently stuck; user fees are unrecoverable without a full contract redeployment.
