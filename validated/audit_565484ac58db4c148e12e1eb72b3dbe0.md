### Title
Rich-Indexer `insert_transaction` Premature `break` on `spend_cell` Miss Causes Indexed Cells to Appear Unspent — (`util/rich-indexer/src/indexer/mod.rs`)

### Summary

When `cell_filter` is enabled, the input-processing loop in `insert_transaction` calls `spend_cell` and **breaks** on a `false` return. `spend_cell` returns `false` whenever the referenced output is not in the DB — which is the normal case for any input spending a cell that was never indexed (because it didn't match the filter). If such an unindexed cell appears as `input[0]`, the loop exits immediately and all subsequent inputs — including those that spend previously-indexed, filter-matching cells — are never processed. Those cells are never marked `is_spent = 1` and their spend records are never written to the `input` table.

### Finding Description

In `util/rich-indexer/src/indexer/mod.rs` lines 229–248:

```rust
for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
    let out_point = input.previous_output();
    if !spend_cell(&out_point, tx).await? {
        break;          // ← exits the entire loop
    }
    if self.custom_filters.is_cell_filter_enabled() {
        if let Some((output_id, output, output_data)) =
            query_output_cell(&out_point, tx).await?
            && self.custom_filters.is_cell_filter_match(...)
        {
            build_input_rows(output_id, &input, input_index, &mut input_rows);
            is_tx_matched = true;
        }
    }
    ...
}
``` [1](#0-0) 

`spend_cell` issues `UPDATE output SET is_spent = 1 WHERE tx_id = ... AND output_index = ...` and returns `Ok(false)` when `rows_affected == 0` — i.e., the output is simply not in the DB. [2](#0-1) 

With `cell_filter` active, only outputs matching the filter are ever inserted into the `output` table. A transaction spending a mix of filtered and unfiltered cells will have some inputs whose `out_point` resolves to nothing in the DB. If any such unindexed input appears **before** a filter-matching input in the input vector, the `break` fires and the filter-matching spend is silently dropped.

### Impact Explanation

- The `output` row for the filter-matching cell retains `is_spent = 0` — it appears live/unspent.
- No row is written to the `input` table for that spend.
- `get_cells` returns the cell as a live UTXO.
- `get_cells_capacity` returns an inflated balance.
- `get_transactions` omits the spending transaction from the cell's history.

Any wallet or application relying on the rich-indexer RPC for UTXO management will observe phantom unspent cells and incorrect balances.

### Likelihood Explanation

The precondition is trivially reachable: a valid CKB transaction may spend any combination of cells regardless of whether they match the indexer's filter. The input ordering is determined by the transaction author, not the indexer. Any block containing such a transaction — relayed through normal P2P block propagation — triggers the bug. No special privilege is required.

### Recommendation

Replace `break` with `continue` at line 231. When `spend_cell` returns `false`, the cell is simply not in the DB (expected under filtering); the loop must continue to process remaining inputs rather than abort. [3](#0-2) 

### Proof of Concept

1. Enable `cell_filter` matching script `S`.
2. Index a transaction `T1` that creates `output[0]` with script `S` (cell is now in DB, `is_spent = 0`).
3. Construct transaction `T2` with:
   - `input[0]`: spends any output **not** in the DB (e.g., a cell with a non-matching script, never indexed).
   - `input[1]`: spends `T1.output[0]` (the indexed, filter-matching cell).
4. Include `T2` in a valid block and relay it.
5. After indexing, query `get_cells` for script `S` — `T1.output[0]` is returned as unspent. Query `get_transactions` for script `S` — `T2` does not appear as a spending transaction.

The invariant that all filter-matching input spends are recorded is violated.

```rust
pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
}
