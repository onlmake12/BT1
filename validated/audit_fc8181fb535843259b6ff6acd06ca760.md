### Title
Missing Staleness and Positive-Price Safety Checks in Chainlink-Compatible Adapter — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official SDK contract that implements Chainlink's `AggregatorV3Interface` as a drop-in replacement for protocols migrating from Chainlink. Every price-returning function in the contract calls `pyth.getPriceUnsafe()` — which explicitly carries no staleness guarantee — instead of `pyth.getPriceNoOlderThan()`. Additionally, no function validates that the returned `price.price` is positive before casting it to `int256`. Any downstream protocol that deploys this adapter and relies on `latestRoundData()`, `getRoundData()`, or `latestAnswer()` will silently receive an arbitrarily stale or zero/negative price with no revert.

---

### Finding Description

`PythAggregatorV3` implements the full Chainlink `AggregatorV3Interface`. All three price-returning entry points share the same root cause:

**`latestRoundData()` (lines 110–118):**
```solidity
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
roundId = uint80(price.publishTime);
return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
```

**`getRoundData()` (lines 89–96):**
```solidity
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
return (_roundId, int256(price.price), price.publishTime, price.publishTime, _roundId);
```

**`latestAnswer()` (lines 54–56):**
```solidity
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
return int256(price.price);
``` [1](#0-0) [2](#0-1) [3](#0-2) 

`getPriceUnsafe` is explicitly documented in `IPyth.sol` as returning "the most recent price update in this contract **without any recency checks**" and that "the returned price update may be **arbitrarily far in the past**." [4](#0-3) 

The safe alternative, `getPriceNoOlderThan(id, age)`, is available in the same interface and reverts with `StalePrice` if the price is too old. [5](#0-4) 

`PythStructs.Price.price` is typed `int64`, meaning it can be zero or negative. The adapter casts it directly to `int256` with no `> 0` guard. By contrast, the Aave-specific adapter `PythPriceOracleGetter.sol` correctly uses `getPriceNoOlderThan` and explicitly reverts on `price.price <= 0`. [6](#0-5) 

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a Chainlink drop-in (the documented migration path) and calls `latestRoundData()` or `getRoundData()` will:

1. **Receive a stale price with no revert** — if the Pyth feed has not been updated recently (e.g., network outage, keeper failure), the returned `answer` and `updatedAt` reflect the last cached value, which may be hours or days old. Lending protocols, perps, and AMMs that rely on this for collateral valuation or liquidation thresholds will operate on incorrect prices.

2. **Receive a zero or negative price with no revert** — `int256(price.price)` is returned directly. A zero price in a lending protocol can allow infinite borrowing; a negative price can cause arithmetic underflows or incorrect collateral calculations in protocols that do not independently validate the sign.

The Pyth documentation explicitly recommends `PythAggregatorV3` for Chainlink migration and states that protocols should use `getPriceNoOlderThan`, but the adapter itself does not enforce this, creating a silent footgun for every integrator. [7](#0-6) 

---

### Likelihood Explanation

The contract is the official Pyth SDK artifact for Chainlink migration, published under `@pythnetwork/pyth-sdk-solidity`. The migration guide explicitly instructs protocols to deploy it as a drop-in replacement. [8](#0-7) 

Protocols migrating from Chainlink expect `latestRoundData()` to behave safely (Chainlink's own `AggregatorFacade` includes staleness guards). The absence of any staleness check in the adapter is a silent behavioral difference that most integrators will not notice. The trigger condition — a Pyth feed going stale — is a realistic network event explicitly acknowledged in Pyth's own best-practices documentation.

---

### Recommendation

1. Replace all `pyth.getPriceUnsafe(priceId)` calls in `latestRoundData()`, `getRoundData()`, and `latestAnswer()` with `pyth.getPriceNoOlderThan(priceId, validTimePeriod)`, where `validTimePeriod` is a constructor-configurable parameter (similar to `PythPriceOracleGetter`).

2. Add a `require(price.price > 0, "invalid price")` guard before casting `price.price` to `int256` in all price-returning functions.

3. Alternatively, add a prominent NatSpec warning on each function that the caller **must** independently validate `updatedAt` against a staleness threshold, and update the migration guide to make this mandatory.

### Proof of Concept

```solidity
// Attacker scenario: Pyth feed goes stale (no update for 24h)
// PythAggregatorV3 is deployed and used by a lending protocol

PythAggregatorV3 adapter = PythAggregatorV3(deployedAdapterAddress);

// latestRoundData() returns the 24h-old price with NO revert
(uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
    = adapter.latestRoundData();

// answer = stale price from 24 hours ago
// updatedAt = publishTime from 24 hours ago
// No StalePrice revert was thrown — the lending protocol proceeds with the stale price

// If the lending protocol checks: require(block.timestamp - updatedAt < 3600)
// it will correctly revert. But if it trusts the adapter (as Chainlink integrators do),
// it will use the stale price, enabling under-collateralized borrows or
// blocking legitimate liquidations.
```

    function getPriceNoOlderThan(
        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
        price = getPriceUnsafe(id);

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();

        return price;
    }
