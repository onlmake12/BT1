### Title
`ScriptHashTypeVerifier` Only Checks Lock Script Hash Type, Ignores Type Script Hash Type - (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` is documented as verifying that the `ScriptHashType` of **all** transaction outputs is within the range permitted by current consensus rules. However, the implementation only inspects `output.lock().hash_type()` and never checks `output.type_().hash_type()`. A transaction output carrying a type script with a not-yet-activated `ScriptHashType` (e.g., `Data2` before the CKB2023 hardfork) silently passes this verifier, bypassing the intended early-rejection and "malformed transaction" classification.

---

### Finding Description

`ScriptHashTypeVerifier` in `verification/src/transaction_verifier.rs` is the dedicated consensus guard that rejects transactions whose outputs use a `ScriptHashType` value not yet enabled by the current hardfork epoch. The comment above the struct reads:

> "Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules."

The implementation iterates over every output and checks only the **lock script**:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

There is no corresponding check on `output.type_()`. A `CellOutput` can carry both a lock script and an optional type script; the type script's `hash_type` field is never validated here.

`ScriptHashType` values are defined in `util/gen-types/src/core.rs`. The structural validity check (`check_data` in `util/gen-types/src/extension/check_data.rs`) only verifies that the raw byte is even or equals 1 — it does **not** check whether the value is enabled by consensus. `ENABLED_SCRIPT_HASH_TYPE` (from `util/constant/src/consensus.rs`) is the consensus-level gate, and `ScriptHashTypeVerifier` is the only place it is applied to transaction outputs. Since the type script field is skipped, a type script with `ScriptHashType::Data2` (value `4`) or any future not-yet-activated variant passes `ScriptHashTypeVerifier` unchallenged before the relevant hardfork epoch.

---

### Impact Explanation

1. **Malformed-tx classification bypass.** `TransactionError::ScriptHashTypeNotPermitted` is listed as a malformed-tx error in `is_malformed_tx()`. Malformed transactions are treated specially by the tx-pool and peer-scoring subsystems. A transaction with an invalid type-script hash type escapes this classification and instead fails later at script execution with `ScriptError::InvalidVmVersion`, a different error path with different peer-scoring consequences.

2. **Tx-pool pollution.** Because the early rejection gate is incomplete, such transactions can be admitted to the pending pool and consume validation resources before being rejected during block assembly or script execution.

3. **Defense-in-depth failure.** The verifier is the designated consensus-rule enforcement point for hash-type gating. Its incompleteness means the type script field of every output is unguarded at the verification layer, relying entirely on the script execution engine as the sole backstop.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a `CellOutput` with a type script whose `hash_type` byte is set to a not-yet-activated value (e.g., `Data2 = 4` before CKB2023, or any `DataN` for N ≥ 3 on current mainnet). The structural `check_data` pass accepts even-valued bytes, so the transaction is not dropped at the P2P deserialization layer. It reaches `ScriptHashTypeVerifier`, passes, and proceeds into the tx-pool. This is trivially reachable with a standard RPC `send_transaction` call.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type for each output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Check lock script hash type (existing)
        check_hash_type(output.lock().hash_type())?;

        // Check type script hash type (missing — add this)
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the `TryInto<ScriptHashType>` + `ENABLED_SCRIPT_HASH_TYPE.contains` logic already present for the lock script.

// Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}

    /// Specifies how the script `code_hash` is used to match the script code and how to run the code.
    /// The hash type is split into the high 7 bits and the low 1 bit,
    /// when the low 1 bit is 1, it indicates the type,
    /// when the low 1 bit is 0, it indicates the data,
    /// and then it relies on the high 7 bits to indicate
    /// that the data actually corresponds to the version.
     #[derive(Default, Clone, Copy, PartialEq, Eq, Debug, Hash, FromRepr)]
     #[repr(u8)]
    pub enum ScriptHashType {
        /// Type "type" matches script code via cell type script hash.
        Type = 1,
        /// Type "data" matches script code via cell data hash, and run the script code in v0 CKB VM.
        #[default]
        Data = 0,
        /// Type "data1" matches script code via cell data hash, and run the script code in v1 CKB VM.
        Data1 = 2,
        /// Type "data2" matches script code via cell data hash, and run the script code in v2 CKB VM.
        Data2 = 4,
