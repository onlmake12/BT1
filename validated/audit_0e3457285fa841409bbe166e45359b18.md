### Title
TOCTOU Race in `process_fetch_cmd` / `can_start`: `assume_valid_targets` Cleared Between Lock Acquisitions Causes Unconditional Panic in BlockDownload Thread — (`sync/src/synchronizer/mod.rs`)

### Summary
A time-of-check/time-of-use race between the `BlockFetchCMD` sync thread and the chain verification thread allows `assume_valid_targets` to be set to `None` between `can_start()` releasing the mutex (returning `AssumeValidNotFound`) and `process_fetch_cmd` re-acquiring it for the unconditional `.expect()` call, causing a panic that halts IBD block fetching.

### Finding Description
`can_start()` acquires `shared.assume_valid_targets()` (a `MutexGuard`), observes `Some([last_target])` with the target absent from `header_map`, releases the guard, and returns `CanStart::AssumeValidNotFound`. [6](#0-5) 

`process_fetch_cmd` then matches `AssumeValidNotFound` and immediately performs a **second, independent** lock acquisition at line 144, calling `.expect("assume valid target must exist")` unconditionally. [7](#0-6) 

Between these two lock acquisitions, the chain verification thread in `verify_block()` can acquire the same mutex, remove the last target entry, and call `assume_valid_targets.take()`, setting the `Option` to `None`. [8](#0-7) 

When `process_fetch_cmd` then acquires the mutex and finds `None`, the chain `.and_then(|targets| targets.first())` evaluates to `None`, and `.expect()` panics, terminating the `BlockDownload` thread.

### Impact Explanation
The `BlockDownload` thread panic halts all IBD block fetching. The node stalls during initial sync and cannot make further chain progress. This is remotely triggerable by any sync peer that provides valid blocks causing the chain thread to process the last assume_valid_target concurrently with the sync thread's logging path.

### Likelihood Explanation
The race window is narrow (two sequential mutex acquisitions with no atomic guarantee between them), but the condition is naturally reached during normal mainnet IBD when the chain thread processes the last hardcoded assume_valid_target block. No special peer behavior is required beyond providing valid PoW blocks in the normal sync protocol. The default target list contains 20 mainnet checkpoints; the race applies specifically when the last one is being processed. [9](#0-8) 

### Recommendation
Eliminate the second lock acquisition in the `AssumeValidNotFound` branch of `process_fetch_cmd`. The `assume_valid_target` value needed for logging should be captured **inside** `can_start()` while the mutex is still held and returned alongside the `CanStart` variant (e.g., `AssumeValidNotFound(Byte32)`), or the `AssumeValidNotFound` branch should re-check for `None` and gracefully transition to `Ready` instead of calling `.expect()`.

### Proof of Concept
Spawn two threads sharing the same `Shared` instance initialized with `assume_valid_targets = Some([target_hash])`. Thread A calls `can_start()` (which returns `AssumeValidNotFound` because target is absent from `header_map`), then sleeps. Thread B calls `verify_block()` with the target block, triggering `assume_valid_targets.take()`. Thread A resumes and enters the `AssumeValidNotFound` match arm, calling `.expect()` on the now-`None` guard — assert panic occurs.
