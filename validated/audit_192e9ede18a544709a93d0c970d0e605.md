Audit Report

## Title
Copy-Paste Bug in `script_exists_in_output` Always Returns `false` for Type-Script Check on PostgreSQL, Causing Permanent Script-Table Corruption During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary
`script_exists_in_output` fetches two rows (`row_lock`, `row_type`) but the final `match` on line 252 reads `row_lock` again instead of `row_type`. On PostgreSQL, because execution only reaches line 252 when `row_lock` is already `false`, the function always returns `false` for the type-script existence check. During `rollback_block`, every type-script ID is therefore unconditionally added to the deletion list, permanently removing script rows that are still referenced by surviving outputs.

## Finding Description
`script_exists_in_output` (lines 204–257) executes two `SELECT EXISTS` queries. After the first query (`row_lock`), a `match` correctly returns `Ok(true)` early if the lock-script exists. After the second query (`row_type`), line 252 mistakenly matches on `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, `row_lock` holds a BOOLEAN `false` at this point (the `true` case already returned), so `Ok(false)` is always returned regardless of `row_type`'s actual value. The caller in `rollback_block` (lines 33–37) pushes `type_script_id` into `script_id_list_to_remove` whenever the function returns `false`, then deletes all collected IDs from the `script` table. Type-script rows still referenced by outputs in the surviving chain are thus permanently deleted. SQLite is unaffected because `try_get::<bool, _>` returns `Err(_)` there, causing the fallthrough branch to correctly read `row_type`.

## Impact Explanation
The rich-indexer is CKB's state storage mechanism for cell and transaction queries. Permanent deletion of live type-script rows from the `script` table after a reorg means any subsequent `get_cells` or `get_transactions` RPC call filtered by those type scripts silently returns empty or wrong results. The corruption is not self-healing. This constitutes a suboptimal (broken) implementation of the CKB state storage mechanism, matching the **Medium (2001–10000 points)** bounty impact: "Suboptimal implementation of CKB state storage mechanism."

## Likelihood Explanation
A 1-block reorg is a normal, unprivileged blockchain event requiring only a competing valid block at the same height. No special attacker capability is needed. The bug is deterministic and triggers on every reorg on any PostgreSQL-backed rich-indexer deployment. PostgreSQL is the recommended production backend, so production deployments are directly at risk.

## Recommendation
Change line 252 from:
```rust
match row_lock.try_get::<bool, _>(0) {
```
to:
```rust
match row_type.try_get::<bool, _>(0) {
```

## Proof of Concept
1. Start a CKB node with the rich-indexer backed by PostgreSQL.
2. Index a block containing outputs with type scripts.
3. Trigger a 1-block reorg (mine a competing block at the same height with higher total difficulty).
4. Observe `rollback_block` is called; `script_exists_in_output` is invoked for each type-script ID and always returns `false`.
5. Query `SELECT COUNT(*) FROM script` — type-script rows still referenced by surviving outputs will be absent.
6. Issue a `get_cells` RPC filtered by one of those type scripts — the result will be empty despite live cells existing.
7. Apply the one-line fix, repeat steps 1–6, and confirm the script rows are retained.
