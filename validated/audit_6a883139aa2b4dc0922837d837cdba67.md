### Title
Incorrect `total_number_of_updates` Accounting for `BatchAttestation` in Fuel Pyth Contract — (`File: target_chains/fuel/contracts/pyth-contract/src/main.sw`)

---

### Summary

In the Fuel Pyth contract, both `update_fee` and `update_price_feeds` always increment `total_number_of_updates` by exactly `1` for a `BatchAttestation` update, regardless of how many individual price feeds the batch actually contains. Because the fee is computed as `total_number_of_updates * single_update_fee`, a caller can submit a single `BatchAttestation` VAA carrying up to 65 535 price feeds and pay only one `single_update_fee` instead of N fees — a direct fee-accounting undercount analogous to the reference report's missing counter updates in batch operations.

---

### Finding Description

`UpdateType::determine_type` in `update_type.sw` classifies any data whose first four bytes are not the accumulator magic (`0x504e4155`) as `UpdateType::BatchAttestation`. The `BatchAttestationUpdate.update_price_feeds` implementation in `batch_attestation_update.sw` then iterates over all `number_of_attestations` entries (a `u16` field, so up to 65 535) and writes each one to storage.

However, in `main.sw` both the fee-estimation path and the execution path hard-code `+= 1` for every `BatchAttestation` element, regardless of `number_of_attestations`:

**`update_fee` (fee estimation, read-only):**
```sway
UpdateType::BatchAttestation => {
    total_number_of_updates += 1;   // always 1, ignores number_of_attestations
},
```

**`update_price_feeds` (execution, read-write):**
```sway
UpdateType::BatchAttestation(batch_attestation_update) => {
    let _updated_ids = batch_attestation_update.update_price_feeds(...);
    total_number_of_updates += 1;   // always 1, ignores number_of_attestations
},
```

The `Accumulator` branch, by contrast, correctly queries the actual count:
```sway
UpdateType::Accumulator(accumulator_update) => {
    let proof_size_offset = accumulator_update.verify();
    total_number_of_updates += accumulator_update.total_updates(proof_size_offset);
},
```

The fee is then enforced as:
```sway
let required_fee = total_fee(total_number_of_updates, storage.single_update_fee);
require(msg_amount() >= required_fee, PythError::InsufficientFee);
```

where `total_fee = total_number_of_updates * single_update_fee.read()`.

---

### Impact Explanation

A caller who submits one `BatchAttestation` blob containing N price feeds pays `1 × single_update_fee` instead of `N × single_update_fee`. For a batch of 100 feeds the protocol collects 1 % of the intended revenue; for the maximum 65 535 feeds it collects ~0.0015 %. This is a direct, quantifiable loss of protocol fee revenue on every `BatchAttestation` call. Because `BatchAttestation` is still a fully accepted update path (the `else` branch of `determine_type`), the undercharge is reachable by any unprivileged caller today.

---

### Likelihood Explanation

Medium. The `BatchAttestation` format is the legacy Wormhole P2W format and is still accepted by the contract without restriction. Any caller who constructs or relays a multi-feed batch attestation VAA triggers the undercount. No privileged role or leaked key is required.

---

### Recommendation

Mirror the `Accumulator` branch: parse `number_of_attestations` from the batch header and add it to `total_number_of_updates` in both `update_fee` and `update_price_feeds`.

In `update_fee`:
```sway
UpdateType::BatchAttestation(batch_attestation_update) => {
    // parse header to get the actual attestation count
    let vm = WormholeVM::parse_and_verify_pyth_vm(...);
    let (_, number_of_attestations, _) =
        parse_and_verify_batch_attestation_header(vm.payload);
    total_number_of_updates += number_of_attestations.as_u64();
},
```

In `update_price_feeds`:
```sway
UpdateType::BatchAttestation(batch_attestation_update) => {
    let updated_ids = batch_attestation_update.update_price_feeds(...);
    total_number_of_updates += updated_ids.len();   // or number_of_attestations
},
```

---

### Proof of Concept

1. Construct a valid Wormhole VAA whose payload is a `BatchAttestation` with `number_of_attestations = 100` (100 distinct price-feed entries, each `attestation_size` bytes).
2. Call `update_fee([vaa_bytes])` on the Fuel Pyth contract.
   - `determine_type` returns `UpdateType::BatchAttestation` (magic ≠ `0x504e4155`).
   - `total_number_of_updates` is incremented by `1`.
   - Returned fee = `1 × single_update_fee`.
3. Call `update_price_feeds([vaa_bytes])` with `msg_amount = 1 × single_update_fee`.
   - `batch_attestation_update.update_price_feeds` iterates all 100 attestations and writes 100 price feeds to storage.
   - `total_number_of_updates` is again `1`; `required_fee = 1 × single_update_fee`; the `require` passes.
4. Result: 100 price feeds updated, only 1 update fee paid — 99 updates obtained for free.
