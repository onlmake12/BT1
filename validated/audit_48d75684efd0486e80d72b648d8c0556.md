**Concrete consequence:**
- `input[1]`'s previous output never has `is_spent` set to `1` in the `output` table.
- No row is inserted into the `input` table for `input[1]`.
- The indexer permanently believes that cell is still live (unspent).

1. Rich-indexer is configured with a `cell_filter` (non-default but documented/supported configuration).
2. A valid on-chain transaction has `input[0]` spending a filtered-out cell and `input[1]` spending a filtered-in cell.

Any unprivileged user who owns a filtered-out cell (e.g., a cell with a lock script excluded by the filter) can craft such a transaction and get it included in a block through normal mining. No special privileges are needed.

### Title
Incorrect `break` in `insert_transaction` input loop causes phantom live cells when cell_filter excludes some inputs' previous outputs — (`util/rich-indexer/src/indexer/mod.rs`)

### Summary
In `AsyncRichIndexer::insert_transaction`, when iterating over a transaction's inputs, `spend_cell` is called for each input's previous output. If `spend_cell` returns `false` (the previous output is not in the index due to cell filtering), the loop unconditionally `break`s. This silently skips all remaining inputs, leaving their previous outputs permanently marked as unspent in the indexer database.

### Finding Description
The loop at [1](#0-0)  calls `spend_cell` and breaks on `false`. The `spend_cell` function at [2](#0-1)  returns `false` when the referenced output row does not exist in the `output` table — which is the normal case for any cell excluded by the configured `cell_filter`. The `break` should be `continue`: a missing previous output simply means there is nothing to mark as spent in the local index, but subsequent inputs must still be processed.

### Impact Explanation
Any input after the first unindexed input in a transaction is silently dropped:
- Its previous output's `is_spent` flag is never set to `1` in the `output` table. [3](#0-2) 
- No row is inserted into the `input` table for it. [4](#0-3) 

Consumers of the indexer (wallets, dApps using DAO or Secp256k1 lock scripts) will see those cells as live/unspent indefinitely, creating phantom spendable UTXOs. Attempts to spend them will fail on-chain, and balance calculations will be inflated.

### Likelihood Explanation
The precondition — a `cell_filter` that excludes at least one script type — is a standard, documented deployment configuration for operators who want to index only specific scripts (e.g., only DAO cells, or only a specific dApp's cells). Any user who holds a cell matching the *excluded* script type can craft a valid two-input transaction where `input[0]` spends their filtered-out cell and `input[1]` spends a filtered-in cell. No elevated privileges, no majority hashpower, and no special network access are required beyond normal transaction submission.

### Recommendation
Replace `break` with `continue` at [5](#0-4)  so that a missing previous output in the index is treated as "nothing to update here" and the loop proceeds to the next input.

```rust
// Before (buggy):
if !spend_cell(&out_point, tx).await? {
    break;
}

// After (correct):
if !spend_cell(&out_point, tx).await? {
    continue;
}
```

### Proof of Concept
1. Configure rich-indexer with a `cell_filter` that matches only cells with lock script `A` (e.g., Secp256k1).
2. Create cell `C1` with lock script `B` (excluded by filter) and cell `C2` with lock script `A` (included).
3. Construct a transaction `T` with `input[0] = C1` and `input[1] = C2`. Submit and get it mined.
4. After `IndexerSync::append` processes the block, query the `output` table for `C2`: `is_spent` will be `0`.
5. Query `get_transactions` / live cells for the address owning `C2`: it appears as a live cell despite being spent on-chain.

pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
}
