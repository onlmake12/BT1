### Title
`ScriptHashTypeVerifier` Only Validates Lock Script Hash Type, Ignores Type Script Hash Type — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates that the **lock script** hash type is within the consensus-permitted set (`ENABLED_SCRIPT_HASH_TYPE`). It completely omits the same check for the **type script**. This is a direct structural analog to the external report: a function iterates over a collection but only processes one variant of a two-variant field, silently ignoring the other.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, `ScriptHashTypeVerifier::verify()` loops over every output and checks only `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

The type script field (`output.type_().to_opt()`) is never inspected. A `CellOutput` carries both a mandatory lock script and an optional type script, each with its own `hash_type` byte. The consensus constant `ENABLED_SCRIPT_HASH_TYPE` gates which hash-type values are currently active (e.g., `Data2` / value `4` is only valid after the CKB2023 hardfork). The verifier enforces this gate for lock scripts but not for type scripts.

`ScriptHashTypeVerifier` is composed into `NonContextualTransactionVerifier`, which is the **cheap, early-rejection** layer called before any script execution: [2](#0-1) 

The same non-contextual path is used during block relay verification. The expensive contextual path (`verify_rtx`) does eventually catch the invalid hash type inside `extract_script_and_dep_index` when it hits the `hash_type => Err(ScriptError::InvalidScriptHashType(...))` arm: [3](#0-2) 

But by that point, full script-group construction and VM setup have already been initiated.

---

### Impact Explanation

Any unprivileged transaction sender can craft a transaction whose output carries a type script with a currently-disabled hash type (e.g., `Data2 = 4` on a pre-CKB2023 node, or any future reserved value). Such a transaction:

1. **Passes `NonContextualTransactionVerifier`** — the missing type-script hash-type check means it is not rejected cheaply.
2. **Enters the tx-pool admission pipeline** — `verify_rtx` is invoked, consuming CPU for full contextual verification including script-group construction.
3. **Fails only at script execution** — the transaction is ultimately rejected, but only after expensive work has been done.
4. **During block relay** — a block containing such transactions passes `NonContextualBlockTxsVerifier` and forces the receiving node to run contextual block verification before the block is rejected, amplifying the wasted work.

The net effect is a **tx-pool and block-relay DoS vector**: an attacker can continuously submit or relay transactions/blocks with invalid type-script hash types, forcing full contextual verification on every peer without paying the cost of valid transactions.

---

### Likelihood Explanation

The entry path requires no privilege: `send_transaction` RPC or the P2P relay protocol accepts transactions from any peer. Crafting a transaction with an output whose type script has `hash_type = 4` (or any other non-enabled byte) is trivial. The attack is cheap for the sender and expensive for the receiver, making it a realistic DoS amplification.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type when present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Existing lock script check
        let lock_hash_type = output.lock().hash_type();
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(lock_hash_type) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType { hash_type: lock_hash_type }.into());
        }

        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_hash_type = type_script.hash_type();
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_hash_type) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType { hash_type: type_hash_type }.into());
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Construct a `TransactionView` with one output whose **type script** has `hash_type = 4` (`Data2`) on a node running before the CKB2023 hardfork (where `ENABLED_SCRIPT_HASH_TYPE` does not include `4`).
2. Submit via `send_transaction` RPC or inject via P2P relay.
3. Observe that `NonContextualTransactionVerifier::verify()` returns `Ok(())` — the transaction passes the cheap gate.
4. Observe that `verify_rtx` / contextual verification eventually rejects it with `ScriptError::InvalidScriptHashType`, but only after full script-group construction.
5. Repeat at high rate to exhaust node verification resources.

        .verify()
        .map_err(Reject::Verification)?;

        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
