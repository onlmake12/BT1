### Title
Unbounded `tx.gasprice` in Keeper Fee Calculation Allows Any Caller to Drain Subscription Balance - (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `Scheduler` contract's `updatePriceFeeds` function is permissionless and pays the caller a keeper fee calculated directly from `tx.gasprice` with no upper bound. A malicious caller can set an arbitrarily high gas price to drain the entire subscription balance in a single transaction.

### Finding Description

`updatePriceFeeds` has no access control — any address can call it. Inside `_processFeesAndPayKeeper`, the keeper fee is computed as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
```

`tx.gasprice` is fully controlled by the transaction sender. There is no cap or sanity check on it. The only guard is:

```solidity
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
```

This means the attacker can set `tx.gasprice` to a value such that `totalKeeperFee` equals exactly `status.balanceInWei`, draining the entire subscription balance in one call. The attacker pays `gasUsed * tx.gasprice` in actual gas but receives `(gasUsed + GAS_OVERHEAD) * tx.gasprice + keeperSpecificFee`, always netting a profit of at least `GAS_OVERHEAD * tx.gasprice + keeperSpecificFee`.

The root cause is the absence of any maximum gas price validation in `_processFeesAndPayKeeper`. [1](#0-0) [2](#0-1) 

### Impact Explanation

A subscription manager deposits ETH into a subscription to fund ongoing price feed updates. A malicious keeper can call `updatePriceFeeds` with a crafted `tx.gasprice` that causes `totalKeeperFee` to equal the full subscription balance, transferring all deposited ETH to themselves. The subscription manager loses all deposited funds. This is a direct theft of user funds with no recovery path. [3](#0-2) 

### Likelihood Explanation

`updatePriceFeeds` is intentionally permissionless to allow any keeper to push updates. Valid `updateData` is freely available from the public Hermes API. The attacker only needs to wait for update conditions to be satisfied (heartbeat elapsed or price deviation threshold crossed), then submit the transaction with an inflated gas price. This requires no special privileges, no leaked keys, and no governance manipulation. Any EOA can execute this attack. [4](#0-3) 

### Recommendation

Cap the effective gas price used in the keeper fee calculation to a protocol-defined maximum (e.g., a governance-settable `maxKeeperGasPriceInWei`):

```solidity
uint256 effectiveGasPrice = tx.gasprice > maxKeeperGasPrice
    ? maxKeeperGasPrice
    : tx.gasprice;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, compute the keeper fee as a fixed amount per price feed rather than a gas-reimbursement model, removing the dependency on `tx.gasprice` entirely. [5](#0-4) 

### Proof of Concept

1. Subscription manager creates a subscription with 1 ETH balance and 2 price feeds.
2. Heartbeat period elapses; update conditions are met.
3. Attacker fetches valid `updateData` from Hermes API.
4. Attacker computes `targetGasPrice = (subscriptionBalance - keeperSpecificFee) / (gasUsed + GAS_OVERHEAD)` where `gasUsed` is estimated for the call.
5. Attacker calls `scheduler.updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = targetGasPrice`.
6. `_processFeesAndPayKeeper` computes `totalKeeperFee ≈ subscriptionBalance`.
7. The balance check passes; the attacker receives the full subscription balance.
8. Subscription balance is zero; the subscription manager's deposited ETH is stolen. [6](#0-5)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);
