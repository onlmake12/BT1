### Title
`buffer::update` Increments `num_prices` Before Bounds Check, Corrupting Buffer State on `NotEnoughSpace` — (`target_chains/solana/programs/pyth-price-store/src/accounts/buffer.rs`)

### Summary

`buffer::update` mutates `header.num_prices` on lines 163–166 **before** performing the `get_mut` bounds check on lines 168–170. If the bounds check fails and `NotEnoughSpace` is returned via `?`, the mutation is not rolled back. The on-chain buffer is left with an inflated `num_prices` that points beyond the allocated space, causing every subsequent `read()` call for that publisher to return `ReadAccountError::InvalidNumPrices` for the remainder of the slot.

### Finding Description

In `buffer::update`:

```
163:    header.num_prices = header
164:        .num_prices
165:        .checked_add(num_new_prices)
166:        .expect("unexpected overflow");   // ← mutation committed here
167:
168:    let destination = prices
169:        .get_mut(start..end)
170:        .ok_or(ExtendError::NotEnoughSpace)?;  // ← error returned here, mutation NOT rolled back
``` [1](#0-0) 

The `?` operator on line 170 causes an early return, but Rust has no implicit rollback of the `header.num_prices` write that already occurred on line 163. The `BufferHeader` is a `Pod` struct written directly into the account's data slice; there is no transaction-level undo within the same Solana instruction.

The `read()` function then enforces:

```rust
let expected_len = num_prices.saturating_mul(size_of::<BufferedPrice>());
if expected_len > prices_bytes.len() {
    return Err(ReadAccountError::InvalidNumPrices);
}
``` [2](#0-1) 

With the inflated `num_prices`, `expected_len` exceeds the buffer's actual capacity, so `InvalidNumPrices` is returned for every read until the next slot resets the counter.

The slot-reset guard that would eventually recover the state is:

```rust
if header.slot != current_slot {
    header.slot = current_slot;
    header.num_prices = 0;
}
``` [3](#0-2) 

Recovery only occurs when the publisher submits prices in a **subsequent** slot. If the publisher does not submit in the next slot, the corrupted state persists.

### Impact Explanation

After the failed second call, the validator's `read()` returns `InvalidNumPrices` for the affected publisher's buffer. All prices accumulated in the current slot — including those successfully written before the overflow attempt — become unreadable. The validator skips the publisher's entire price contribution for that slot. Any downstream DeFi protocol consuming that publisher's feed receives no update for the affected slot.

The impact is **temporary** (bounded to the current slot, self-recovering on the next publisher submission) and **scoped to a single publisher's buffer**. It does not affect other publishers.

### Likelihood Explanation

The trigger condition — buffer filled exactly to capacity followed by one additional price in the same slot — is reachable in normal operation if a publisher submits prices in multiple transactions within a single slot and the buffer is sized tightly. It is also trivially triggerable intentionally by the publisher themselves. No privileged access, leaked keys, or external oracle cooperation is required; the publisher's own signing key is sufficient.

### Recommendation

Move the `header.num_prices` increment to **after** the successful `get_mut` check, so the mutation only occurs when the write is guaranteed to succeed:

```rust
let destination = prices
    .get_mut(start..end)
    .ok_or(ExtendError::NotEnoughSpace)?;

header.num_prices = header
    .num_prices
    .checked_add(num_new_prices)
    .expect("unexpected overflow");

// ... memcpy ...
```

This makes `update` atomic with respect to `num_prices`: the counter is only advanced when the data has been (or is about to be) written.

### Proof of Concept

State-transition test (no-`solana-program` feature, using the existing test harness pattern):

```rust
#[test]
fn test_overflow_does_not_corrupt_num_prices() {
    // Buffer sized for exactly 2 prices (2 * 20 = 40 bytes of price space)
    let mut buf = vec![0u8; size(2)];
    create(&mut buf, Default::default()).unwrap();

    let (header, prices) = read_mut(&mut buf).unwrap();

    // Fill buffer exactly to capacity: 2 prices = 40 bytes
    update(header, prices, 1, &vec![1u8; 40]).unwrap();
    assert_eq!(header.num_prices, 2);

    // Attempt one more price in the same slot — must fail
    let result = update(header, prices, 1, &vec![2u8; 20]);
    assert!(matches!(result, Err(ExtendError::NotEnoughSpace)));

    // INVARIANT: num_prices must still be 2, not 3
    assert_eq!(header.num_prices, 2,
        "num_prices was corrupted by a failed update");

    // INVARIANT: read() must still succeed and return 2 prices
    let read_result = read(&buf);
    assert!(read_result.is_ok(),
        "read() returned error after failed update: {:?}", read_result);
    assert_eq!(read_result.unwrap().1.len(), 2);
}
```

With the current code, `header.num_prices` will be `3` after the failed call, and `read()` will return `Err(InvalidNumPrices)`, demonstrating the invariant violation. [4](#0-3)

```rust
pub fn update(
    header: &mut BufferHeader,
    prices: &mut [u8],
    current_slot: u64,
    new_prices: &[u8],
) -> Result<(), ExtendError> {
    if new_prices.len() % size_of::<BufferedPrice>() != 0 {
        return Err(ExtendError::InvalidLength);
    }
    // We expect the validator to apply the buffered prices at the end of each slot,
    // so when we observe a new slot we remove previously stored prices.
    if header.slot != current_slot {
        header.slot = current_slot;
        header.num_prices = 0;
    }
    let num_new_prices = (new_prices.len() / size_of::<BufferedPrice>())
        .try_into()
        .expect("unexpected overflow");
    let num_prices: usize = header.num_prices.try_into().unwrap();
    let start = size_of::<BufferedPrice>() * num_prices;
    let end = size_of::<BufferedPrice>() * num_prices + new_prices.len();
    header.num_prices = header
        .num_prices
        .checked_add(num_new_prices)
        .expect("unexpected overflow");

    let destination = prices
        .get_mut(start..end)
        .ok_or(ExtendError::NotEnoughSpace)?;

    // We don't validate the values of `new_prices` to make the publishing process
    // more efficient. They will be validated when applied in the validator.
    #[cfg(feature = "solana-program")]
    solana_program::program_memory::sol_memcpy(destination, new_prices, new_prices.len());
    #[cfg(not(feature = "solana-program"))]
    destination.copy_from_slice(new_prices);
    Ok(())
}
