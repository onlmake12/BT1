### Title
Unconstrained `slash_custody` Account in `initialize_pool` Allows Redirection of Slashed Tokens — (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `initialize_pool` instruction in the integrity-pool program accepts a `slash_custody` token account with **no authority, mint, or ownership constraints**. This account is stored in `PoolConfig` and later used as the destination for all slashed PYTH tokens. Because `initialize_pool` has no admin/authority signer requirement (only a generic `payer`), an unprivileged actor who calls it first can supply an attacker-controlled account as `slash_custody`, permanently redirecting all future slashed tokens to themselves.

---

### Finding Description

The `initialize_pool` instruction's account list, as reflected in the on-chain IDL, is:

```
payer          (signer, writable)
pool_data      (writable, no PDA constraint)
pool_config    (PDA, writable — initialized here)
slash_custody  (no constraints at all)
system_program
``` [1](#0-0) 

There is no `authority` or `admin` signer beyond the fee-paying `payer`. The `pool_config` PDA is created during this call, meaning it can only be initialized once — whoever calls `initialize_pool` first permanently sets `slash_custody` in `PoolConfig`:

```json
"PoolConfig": {
  "fields": [
    { "name": "pool_data",                "type": "pubkey" },
    { "name": "reward_program_authority", "type": "pubkey" },
    { "name": "pyth_token_mint",          "type": "pubkey" },
    { "name": "y",                        "type": "u64"    },
    { "name": "slash_custody",            "type": "pubkey" }
  ]
}
``` [2](#0-1) 

The `slash_custody` field is later consumed by the `slash` instruction, which transfers slashed tokens directly to it:

```json
{
  "name": "slash_custody",
  "relations": ["slash_event"],
  "writable": true
}
``` [3](#0-2) 

No validation is performed on `slash_custody` during `initialize_pool` to confirm:
- It is a valid SPL token account
- Its mint matches `pyth_token_mint`
- Its authority is the program or a trusted PDA

The SDK client itself passes `getStakeAccountCustodyAddress(poolData)` as `slashCustody`, but this is only a client-side convention — the on-chain program enforces nothing: [4](#0-3) 

---

### Impact Explanation

An attacker who front-runs `initialize_pool` (or is the first to call it on a new deployment) can supply their own token account as `slash_custody`. All subsequent `slash` calls — which penalize misbehaving publishers by transferring their staked PYTH tokens — will send those tokens to the attacker's account instead of the intended protocol-controlled custody. This completely neutralizes the slashing mechanism and allows the attacker to profit from every slash event.

---

### Likelihood Explanation

The `initialize_pool` instruction has no privileged signer requirement. Any transaction sender can call it. Since `pool_config` is a PDA initialized exactly once, the first caller wins. On a new deployment or redeployment, this is a straightforward race condition exploitable by any on-chain observer watching the mempool or deployment transactions.

---

### Recommendation

1. **Add an authority signer** to `initialize_pool` (e.g., a `governance_authority` that must sign), analogous to how other admin instructions in the staking program require `governance_authority`.
2. **Validate `slash_custody`** with Anchor token account constraints:
   ```rust
   #[account(
       token::mint = config_account.pyth_token_mint,
       token::authority = pool_config,  // or a program-controlled PDA
   )]
   pub slash_custody: Account<'info, TokenAccount>,
   ```
3. **Alternatively, derive `slash_custody` as a PDA** seeded from `pool_config`, so it is exclusively controlled by the program and cannot be substituted by an external actor.

---

### Proof of Concept

1. Deploy or observe a fresh deployment of the integrity-pool program.
2. Before the legitimate admin calls `initialize_pool`, submit a transaction calling `initialize_pool` with:
   - `payer`: attacker's keypair
   - `pool_data`: any writable account
   - `pool_config`: the deterministic PDA (computed from seed `pool_config`)
   - `slash_custody`: attacker's own PYTH token account
3. The `pool_config` PDA is now initialized with `slash_custody = attacker_account`.
4. Any future `slash` instruction will transfer slashed PYTH tokens to the attacker's account. [1](#0-0) [5](#0-4)
