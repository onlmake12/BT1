### Title
`payable.transfer()` in `verifyUpdate()` Causes Revert for Contract Callers with Non-Trivial Fallback — (`File: lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` uses `.transfer()` to refund excess ETH to `msg.sender`. Since `.transfer()` forwards only 2,300 gas, any contract caller whose `receive()` or `fallback()` function consumes more than 2,300 gas (e.g., due to SLOAD costs introduced by EIP-1884 in the Istanbul hard fork) will cause the entire `verifyUpdate()` call to revert. This permanently blocks such callers from using the Lazer price verification service.

### Finding Description
In `PythLazer.sol`, the `verifyUpdate()` function accepts ETH as a fee and refunds any excess to the caller:

```solidity
// lazer/contracts/evm/src/PythLazer.sol, lines 74-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
```

The Solidity `.transfer()` primitive forwards exactly 2,300 gas to the recipient. After EIP-1884 (Istanbul hard fork), the cost of `SLOAD` increased from 200 to 800 gas, meaning many standard contract `receive()` or `fallback()` implementations (e.g., those that read a storage variable, emit an event, or are proxy contracts) now exceed the 2,300 gas stipend. When the refund `.transfer()` call runs out of gas, it reverts, and because there is no try/catch around it, the entire `verifyUpdate()` transaction reverts. [1](#0-0) 

By contrast, the Entropy, Scheduler, and Echo contracts in the same repository all correctly use `.call{value: amount}("")` with the checks-effects-interactions pattern: [2](#0-1) [3](#0-2) 

### Impact Explanation
Any smart contract that integrates with `PythLazer.verifyUpdate()` and sends more ETH than `verification_fee` will have its transaction permanently reverted if its `receive()` or `fallback()` function costs more than 2,300 gas. This is a denial-of-service against contract-based Lazer updaters/consumers. The excess ETH is not lost (the transaction reverts), but the caller is completely unable to use the `verifyUpdate()` function unless it sends the exact fee amount — which is fragile since `verification_fee` can be changed by the owner.

### Likelihood Explanation
Medium. Any contract caller (e.g., a DeFi protocol, a proxy-based wallet, or any contract that does non-trivial work in its `receive()`) is affected. Proxy contracts in particular always exceed 2,300 gas in their fallback. The only safe callers are EOAs or contracts with empty `receive()` functions. The `verification_fee` is currently set to `1 wei`, making overpayment extremely common in practice. [4](#0-3) 

### Recommendation
Replace `.transfer()` with `.call{value: ...}("")` and check the return value, following the checks-effects-interactions pattern already used elsewhere in the codebase:

```solidity
if (msg.value > verification_fee) {
    (bool sent, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(sent, "Refund failed");
}
```

### Proof of Concept
1. Deploy a contract `AttackerProxy` that calls `PythLazer.verifyUpdate()` with `msg.value = verification_fee + 1 wei`, and whose `receive()` function performs a single `SLOAD` (e.g., reads a storage variable). This costs ~800 gas, exceeding the 2,300 stipend.
2. Call `AttackerProxy.callVerifyUpdate{value: verification_fee + 1}(update)`.
3. The `.transfer()` at line 76 of `PythLazer.sol` forwards 2,300 gas to `AttackerProxy.receive()`, which runs out of gas.
4. The entire `verifyUpdate()` call reverts with an out-of-gas error.
5. The contract-based caller is permanently unable to use `verifyUpdate()` unless it sends the exact fee — which is fragile since `verification_fee` is owner-settable. [5](#0-4)
