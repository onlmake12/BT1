### Title
Provider Fee Instant Update Enables Front-Running of Entropy Requests — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol` allows any registered provider (or their designated fee manager) to call `setProviderFee` / `setProviderFeeAsFeeManager` with no time-lock, no upper bound, and immediate on-chain effect. A provider can front-run a user's pending `requestWithCallback` / `requestV2` transaction by raising `feeInWei` to an arbitrarily high value in the same block, causing the user's transaction to revert with `InsufficientFee` and consuming the user's gas. Because excess `msg.value` is explicitly **not** refunded, a provider can also extract surplus ETH from users who add a safety buffer to their fee estimate.

---

### Finding Description

`setProviderFee` writes the new fee directly to storage with no delay:

```solidity
// Entropy.sol L809-827
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;          // ← immediate, unbounded
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
}
```

`setProviderFeeAsFeeManager` (L829-855) has the same property for the delegated fee-manager role.

The fee is consumed in `requestHelper` (L234-235):

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
```

`getFeeV2` (L760-764) reads `provider.feeInWei` live from storage at execution time:

```solidity
function getFeeV2(address provider, uint32 gasLimit)
    public view override returns (uint128 feeAmount) {
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
```

There is no minimum delay, no pending/commit pattern, and no cap on the new fee value. The IEntropyV2 interface documentation itself acknowledges the race: *"Note that the fee can change over time"* — but provides no on-chain protection against same-block manipulation.

---

### Impact Explanation

1. **Griefing / DoS of user requests**: A provider monitors the mempool for `requestWithCallback` calls targeting their address. They submit `setProviderFee(type(uint128).max)` with higher gas, front-running the user. The user's transaction reverts with `InsufficientFee`, wasting gas and failing to obtain randomness.

2. **Excess ETH extraction**: Users who add a safety buffer (e.g., `fee * 110 / 100`) to guard against fee changes are not refunded excess value per the explicit contract comment: *"excess value is **not** refunded to the caller"* (L321, L344). A provider can raise the fee to exactly consume the user's buffer, extracting ETH beyond the legitimate fee.

3. **Repeated griefing**: The provider can lower the fee back after the attack, making the pattern repeatable with no lasting on-chain evidence beyond events.

---

### Likelihood Explanation

- The attacker role is a **registered Entropy provider**, explicitly listed as an in-scope attacker class ("Entropy user/provider").
- No privileged access beyond provider registration is required.
- The attack is executable on any EVM chain where Entropy is deployed.
- The Fortuna keeper (`apps/fortuna/src/keeper/fee.rs`) already demonstrates automated fee adjustment logic, confirming providers routinely update fees programmatically — the same infrastructure can be weaponized.
- The `setProviderFeeAsFeeManager` path (L829-855) extends the attack surface to any address the provider delegates as fee manager, further lowering the barrier.

---

### Recommendation

Introduce a two-step fee update with a mandatory time-lock (e.g., 24 hours):

1. **Announce**: Provider calls `proposeProviderFee(newFee)`, which stores the pending fee and the earliest activation timestamp.
2. **Commit**: After the time window elapses, any call to `applyProviderFee()` (or the next `requestHelper` automatically applies it) finalizes the change.

Additionally, enforce a reasonable upper bound on `feeInWei` to prevent griefing via extreme values.

---

### Proof of Concept

1. User calls `getFee(provider)` off-chain → receives `X wei`.
2. User submits `requestWithCallback{value: X}(provider, commitment)` to the mempool.
3. Provider observes the pending transaction and submits `setProviderFee(X + 1)` with higher gas priority.
4. Provider's `setProviderFee` is mined first; `provider.feeInWei` is now `X + 1`.
5. User's `requestWithCallback` executes: `getFeeV2` returns `X + 1`; `msg.value (X) < requiredFee (X+1)` → `revert InsufficientFee`.
6. User loses gas; randomness request fails. Provider resets fee to `X` for normal operation.

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei

        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;
        providerInfo.feeInWei = newFeeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);

            oldFeeInWei,
            newFeeInWei,
