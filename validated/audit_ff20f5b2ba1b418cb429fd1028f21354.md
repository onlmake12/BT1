### Title
Unbounded Iteration in `get_raw_tx_pool` RPC Dumps Entire Tx-Pool Without Pagination - (`rpc/src/module/pool.rs`)

### Summary
The `get_raw_tx_pool` RPC endpoint iterates over every pending, proposed, and conflicted transaction in the tx-pool in a single call with no `limit` or pagination parameter. This is the direct CKB analog to M-03: a state-reading function that traverses an unbounded collection in one shot, causing proportional memory allocation, CPU serialization work, and extended read-lock hold time — all triggerable by any RPC caller.

### Finding Description

`get_raw_tx_pool` is defined in `rpc/src/module/pool.rs`:

```rust
fn get_raw_tx_pool(&self, verbose: Option<bool>) -> Result<RawTxPool> {
    let tx_pool = self.shared.tx_pool_controller();
    let raw = if verbose.unwrap_or(false) {
        let info = tx_pool.get_all_entry_info()...;
        RawTxPool::Verbose(info.into())
    } else {
        let ids = tx_pool.get_all_ids()...;
        RawTxPool::Ids(ids.into())
    };
    Ok(raw)
}
``` [1](#0-0) 

The underlying `get_all_entry_info()` in `tx-pool/src/pool.rs` unconditionally collects every entry across all three pools — pending/gap, proposed, and conflicted — into heap-allocated `HashMap`s and `Vec`s with no iteration limit:

```rust
pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
    let pending = self.pool_map
        .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
        .map(|entry| (entry.transaction().hash(), entry.to_info()))
        .collect();
    let proposed = self.pool_map
        .sorted_proposed_iter()
        .map(|entry| (entry.transaction().hash(), entry.to_info()))
        .collect();
    let conflicted = self.conflicts_cache.iter()
        .map(|(_id, tx)| tx.hash())
        .collect();
    TxPoolEntryInfo { pending, proposed, conflicted }
}
``` [2](#0-1) 

The same pattern applies to `get_ids()` (the non-verbose path): [3](#0-2) 

This is dispatched from the tx-pool service while holding the pool's async read lock: [4](#0-3) 

The RPC signature accepts only a single `verbose` boolean — there is no `limit`, `after`, or any pagination parameter: [5](#0-4) 

Contrast this with the indexer's `get_cells` and `get_transactions` RPCs, which do enforce a `limit` and `after` cursor: [6](#0-5) 

No equivalent guard exists for `get_raw_tx_pool`.

### Impact Explanation

The tx-pool is bounded in total byte size by `max_tx_pool_size` (default 180 MB per `resource/ckb.toml`). With minimum-size transactions (~100 bytes each), the pool can hold on the order of hundreds of thousands of entries. A single `get_raw_tx_pool(verbose=true)` call will:

1. Allocate a `HashMap<Byte32, TxEntryInfo>` for every pending and proposed entry — proportional to pool occupancy.
2. Serialize the entire result to JSON in the RPC thread.
3. Hold the tx-pool async read lock for the full duration of steps 1–2, blocking concurrent pool writes (new transaction admission, block commit) until the response is built.

The response payload itself can be tens to hundreds of megabytes, stressing both the node's memory and the network connection to the caller.

### Likelihood Explanation

The RPC is reachable by any supported local RPC caller (the objective explicitly lists "RPC caller" as a valid entry path). No authentication is required beyond network access to the RPC port. An adversary with RPC access — or a legitimate user on a heavily loaded node — can repeatedly call `get_raw_tx_pool(true)` to amplify resource consumption. The pool is most vulnerable during periods of high mempool congestion, which is precisely when the node is already under load.

### Recommendation

Add a `limit` and `after` (cursor) parameter to `get_raw_tx_pool`, mirroring the pagination design already used by `get_cells` and `get_transactions`. The internal `get_ids()` and `get_all_entry_info()` helpers should accept a count cap and a starting cursor so that callers must page through the pool rather than dumping it in one shot. A reasonable default cap (e.g., 500 entries) should be enforced when no limit is supplied.

### Proof of Concept

```
# Fill the tx-pool with many small transactions, then:
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[true]}'
```

With a large pool this single call forces the node to allocate and serialize the full pool contents — potentially hundreds of megabytes — while holding the pool read lock, delaying all concurrent pool operations until the response is complete.
