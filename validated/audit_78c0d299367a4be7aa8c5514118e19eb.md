Audit Report

## Title
Remote Panic via `get_block_uncles` on Frozen-Block Transactions in `GetTransactionsProof` Handler — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

The `GetTransactionsProofProcess::execute` handler calls `snapshot.get_block_uncles(&block_hash).expect("block uncles must be stored")` at line 121 after successfully retrieving a frozen block via `get_block()`. For frozen blocks, `get_block()` takes an early-return path through the freezer and never reads `COLUMN_BLOCK_UNCLE`, but `get_block_uncles()` has no freezer fallback and reads only from the in-memory cache and `COLUMN_BLOCK_UNCLE` — which was deleted by `delete_block_body()` during freezer migration. The result is an unrecoverable `panic!` that terminates the node process, triggerable by any unauthenticated light-client peer with a single message.

## Finding Description

**Root cause:** `get_block_uncles()` at `store/src/store.rs:205–224` has no freezer fallback path. It checks only the in-memory cache and `COLUMN_BLOCK_UNCLE`. However, `delete_block_body()` at `store/src/write_batch.rs:97` deletes `COLUMN_BLOCK_UNCLE` for every frozen block during migration. The in-memory cache is volatile and will not contain entries for frozen blocks after a node restart.

**Exploit flow:**

1. Attacker sends `GetTransactionsProof { tx_hashes: [<tx_hash from frozen block>], last_hash: <any valid tip> }`.

2. At `get_transactions_proof.rs:59–64`, `get_transaction_info(tx_hash)` succeeds because `COLUMN_TRANSACTION_INFO` is **not** deleted by `delete_block_body()`. `is_main_chain(&tx_info.block_hash)` also returns `true` because `COLUMN_INDEX` is not deleted. The tx_hash enters the `found` partition. [1](#0-0) 

3. At `get_transactions_proof.rs:83–85`, `get_block(&block_hash)` is called. For a frozen block, `store.rs:44–53` detects `header.number() < freezer.number()`, reads the raw block from the freezer, and returns a full `BlockView` — **without ever touching `COLUMN_BLOCK_UNCLE`**. [2](#0-1) 

4. At `get_transactions_proof.rs:119–121`, `get_block_uncles(&block_hash)` is called on the same frozen block hash. This function checks only the in-memory cache and `COLUMN_BLOCK_UNCLE` — there is no freezer fallback. [3](#0-2) 

5. `COLUMN_BLOCK_UNCLE` was deleted during freezer migration by `delete_block_body()`. [4](#0-3) 

6. `get_block_uncles()` returns `None`. The `.expect("block uncles must be stored")` at line 121 panics, terminating the node process. [5](#0-4) 

**Why existing guards fail:**

- `GET_TRANSACTIONS_PROOF_LIMIT` limits count only, not block age.
- The `is_main_chain` check passes for frozen blocks because `COLUMN_INDEX` is preserved.
- `get_block()` at line 83 succeeds for frozen blocks via the freezer early-return path, masking the missing `COLUMN_BLOCK_UNCLE` and allowing execution to reach line 119. [6](#0-5) 

## Impact Explanation

**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

A single well-formed `GetTransactionsProof` message causes an unrecoverable `panic!` in the node process. The panic is not caught; it terminates the entire node. No authentication or privilege is required. The attack is repeatable and prevents the node from staying online.

## Likelihood Explanation

- All historical `tx_hash` values are publicly available via block explorers; no special knowledge is needed.
- The freezer activates automatically on any long-running CKB mainnet/testnet node once the chain grows past the freezer threshold — a standard configuration.
- The light-client protocol server requires no authentication from peers.
- The attack requires sending a single well-formed protocol message.

## Recommendation

At `get_transactions_proof.rs:119–121`, replace the separate `get_block_uncles()` call with `block.uncles()`, using the `BlockView` already retrieved at line 83. For frozen blocks, `get_block()` reconstructs the full `BlockView` (including uncles) from the freezer's raw block data, so `block.uncles()` is correct and always available:

```rust
// Replace:
let uncles = snapshot
    .get_block_uncles(&block_hash)
    .expect("block uncles must be stored");

// With:
let uncles = block.uncles();
```

This eliminates the dependency on the deleted `COLUMN_BLOCK_UNCLE` entry and is consistent with how `get_block()` itself handles frozen blocks.

## Proof of Concept

1. Run a CKB node with a freezer configured until blocks are frozen (or use an existing mainnet/testnet node past the freezer threshold).
2. Obtain any `tx_hash` from a frozen block (block number < `freezer.number()`) from a public block explorer.
3. Connect as a light-client peer to the node.
4. Send:
   ```
   GetTransactionsProof {
       last_hash: <any recent main-chain block hash>,
       tx_hashes: [<tx_hash from frozen block>],
   }
   ```
5. The node panics at `get_transactions_proof.rs:121` with message `"block uncles must be stored"` and the process terminates.
