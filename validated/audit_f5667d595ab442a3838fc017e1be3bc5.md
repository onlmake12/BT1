Audit Report

## Title
Peer-Supplied `declared_cycles = 1` Poisons Relay `tx_filter` via Misclassified Cycle-Exceeded Error, Blocking Honest Tx Propagation — (`File: sync/src/relayer/transactions_process.rs`, `tx-pool/src/process.rs`)

## Summary
The CKB relay protocol accepts a peer-supplied `declared_cycles` value and validates only an upper bound, allowing `declared_cycles = 1` to pass. The tx hash is marked as known in the relay bloom filter before pool admission. When the VM aborts with `ExceededMaximumCycles`, the error is classified as a malformed-tx `Script` error, causing `is_allowed_relay()` to return `false`, suppressing the `TxVerificationResult::Reject` signal that would trigger `remove_from_known_txs`. The hash remains in `tx_filter` for the full TTL, silently dropping all subsequent honest announcements of the same hash.

## Finding Description

**No lower-bound on `declared_cycles`:**
`TransactionsProcess::execute()` only validates an upper bound: [1](#0-0) 
A value of `1` passes this check. The tx hash is then immediately marked as known in the relay filter **before** pool admission: [2](#0-1) 

**`declared_cycles` used directly as VM cap:**
In `_process_tx()`, `max_cycles` is set to `declared_cycles` with no floor: [3](#0-2) 
If `declared_cycles = 1`, `verify_rtx` returns a cycle-limit-exceeded error immediately. The `try_or_return_with_snapshot!` macro returns early: [4](#0-3) 
The `DeclaredWrongCycles` check at L736–748 is **never reached**: [5](#0-4) 

**Misclassification as malformed:**
`VMInternalError::CyclesExceeded` maps to `ScriptError::ExceededMaximumCycles`, which carries `ErrorKind::Script`: [6](#0-5) [7](#0-6) 
`is_malformed_from_verification` returns `true` for `ErrorKind::Script` unless the message contains `ARGV_TOO_LONG_TEXT` — which `ExceededMaximumCycles` does not: [8](#0-7) 
This makes `is_malformed_tx()` return `true`: [9](#0-8) 

**`is_allowed_relay()` returns `false`, blocking filter cleanup:** [10](#0-9) 
Because `is_allowed_relay()` is `false`, `TxVerificationResult::Reject` is never sent to the relayer: [11](#0-10) 
`remove_from_known_txs` is therefore never called, and the tx hash remains in `tx_filter`.

**Subsequent honest announcements silently dropped:**
Any later announcement of the same hash is filtered out: [12](#0-11) 

## Impact Explanation
This is a targeted transaction relay denial-of-service. An attacker can prevent any specific transaction from propagating through a target node for the full `TtlFilter` TTL duration. Applied across multiple nodes simultaneously, this matches the allowed impact class: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** — **High severity (10001–15000 points)**. The attacker expends only a single peering connection and one malformed relay message per target node per TTL window.

## Likelihood Explanation
The attacker must be peered with the target node and must announce the tx hash before the legitimate originator — both achievable for a well-connected attacker monitoring the P2P gossip layer. The attacker is banned after a single attempt (3-day ban per node), but the filter poisoning persists for the full TTL regardless. The attack requires no key material, no hashpower, and no special privilege. It is repeatable across different target nodes using different attacker identities.

## Recommendation
1. **Add a lower-bound validation on `declared_cycles`** in `TransactionsProcess::execute()`. Reject and ban any peer declaring `declared_cycles = 0` or below a configurable minimum (e.g., `TWO_IN_TWO_OUT_CYCLES`).
2. **Distinguish cycle-limit-exceeded from structurally malformed.** A cycle-exceeded error means the declared value was insufficient, not that the transaction is structurally invalid. It should be treated equivalently to `DeclaredWrongCycles` so that `is_allowed_relay()` returns `true` and `remove_from_known_txs` is called.
3. **Do not mark the tx hash as known before pool admission succeeds.** Move `mark_as_known_txs` to after successful pool insertion, or ensure all non-peer-fault rejection paths clear the filter entry.

## Proof of Concept
1. Attacker connects to target node via RelayV3.
2. Attacker sends `RelayTransactionHashes` containing hash `H` of a valid pending transaction.
3. Target node adds `H` to `unknown_tx_hashes` and sends `GetRelayTransactions` back to attacker.
4. Attacker responds with `RelayTransactions` containing the valid transaction body but with `cycles = 1`.
5. `TransactionsProcess::execute()` passes the upper-bound check (`1 ≤ max_block_cycles`), marks `H` as known in `tx_filter`, and calls `submit_remote_tx(tx, 1, peer)`.
6. `_process_tx` sets `max_cycles = 1`; VM aborts immediately with `ScriptError::ExceededMaximumCycles` → `Reject::Verification(Script error)`.
7. `after_process`: `is_malformed_tx() = true` → `ban_malformed(peer)` called; `is_allowed_relay() = false` → `TxVerificationResult::Reject` not sent → `remove_from_known_txs` NOT called; `H` remains in `tx_filter`.
8. Legitimate peer later sends `RelayTransactionHashes` with `H`; `TransactionHashesProcess::execute()` filters it out (`tx_filter.contains(H) = true`); node never requests the tx from the honest peer.
9. Transaction is blocked from propagating through this node for the duration of the `TtlFilter` TTL.
