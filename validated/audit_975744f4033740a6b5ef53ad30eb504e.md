Audit Report

## Title
Quadratic Work in Indexer `append` Due to Unbounded Linear Scan Over Block Transactions for Within-Block Cell Dependencies — (File: util/indexer/src/indexer.rs)

## Summary
The `Indexer::append` function performs an O(T) linear scan over all block transactions for every input whose referenced cell was produced within the same block. Because `self.store.get` reads only from the committed store and the batch is not committed until the end of `append`, any cell written earlier in the same call is invisible to the store lookup, causing the `or_else` fallback scan to execute unconditionally for all within-block dependencies. For a block with T chained transactions this produces O(T²) total work.

## Finding Description
In `util/indexer/src/indexer.rs`, `append` creates a batch at line 318 (`let mut batch = self.store.batch()?`) and commits it only at line 511 (`batch.commit()?`). During the intervening input-processing loop (lines 340–421), for each input the code first attempts a committed-store lookup:

```rust
if let Some(stored_live_cell) = self.store.get(&key_vec)?.or_else(|| {
    transactions
        .iter()
        .enumerate()
        .find(|(_i, tx)| tx.hash() == out_point.tx_hash())
        ...
})
```

Outputs written to the batch by earlier transactions in the same block (lines 473–476: `batch.put_kv(Key::OutPoint(&out_point), Value::Cell(...))`) are not yet visible to `self.store.get`, which returns `None`. This unconditionally triggers the `or_else` closure, which performs a full O(T) `.find()` scan over all block transactions. For a block containing T transactions each with I within-block inputs, total complexity is O(T² × I). No hash-map cache, early-exit, or guard exists to prevent this.

## Impact Explanation
This matches **Low (501–2000 points): Any other important performance improvements for CKB**. The indexer is a local service; the CKB node, consensus, and network are unaffected. The concrete impact is the indexer falling behind the chain tip, making it unavailable to wallets, dApps, and tooling that depend on it for cell and transaction queries.

## Likelihood Explanation
Any unprivileged transaction sender can trigger worst-case behavior by constructing a chain of N transactions (each spending the previous transaction's output) and having them included in a single block. This is a standard, economically rational UTXO pattern (consolidation, batch payments) indistinguishable from legitimate use. The attacker pays O(N) in fees while imposing O(N²) indexer work. No special privileges, majority hashpower, or victim mistakes are required.

## Recommendation
Build an in-memory `HashMap<Vec<u8>, Vec<u8>>` from the current block's outputs before the main input-processing loop. Populate it once in O(T × outputs_per_tx) and consult it in O(1) per input lookup, replacing the `or_else` linear scan:

```rust
let mut block_cell_map: HashMap<Vec<u8>, Vec<u8>> = HashMap::new();
for (i, tx) in transactions.iter().enumerate() {
    for (idx, (output, data)) in tx.outputs_with_data_iter().enumerate() {
        let out_point = OutPoint::new(tx.hash(), idx as u32);
        let key = Key::OutPoint(&out_point).into_vec();
        let val: Vec<u8> = Value::Cell(block_number, i as u32, &output, &data).into();
        block_cell_map.insert(key, val);
    }
}
// In the input loop, replace the or_else closure:
if let Some(stored_live_cell) = self.store.get(&key_vec)?
    .or_else(|| block_cell_map.get(&key_vec).cloned())
{ ... }
```

This reduces total complexity to O(T × I).

## Proof of Concept
1. Construct a chain of N = 3,000 transactions: `tx[0]` creates one output; `tx[k]` spends `tx[k-1]`'s output.
2. Submit all 3,000 transactions to the mempool and have them included in a single block (valid by CKB consensus).
3. When the indexer calls `append` on this block, for each `tx[k]` (k = 1…2999): `store.get` returns `None` (cell not yet committed), triggering the `or_else` scan over all 3,000 transactions → 3,000 hash comparisons per input.
4. Total: ~9,000,000 hash comparisons for one block instead of ~3,000.
5. Measure wall-clock time of `append` with and without the chained pattern to confirm quadratic growth.
