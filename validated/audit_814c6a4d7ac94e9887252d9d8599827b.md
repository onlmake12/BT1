### Title
Silent u128→u64 Truncation Returns Wrong Withdrawal Capacity in DAO Calculator — (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the DAO interest-adjusted withdrawal capacity using a `u128` intermediate value, then silently truncates it to `u64` with an `as u64` cast. Every other `u128`→`u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. Only this one uses the silent truncating cast. When the intermediate value overflows `u64` but the truncated remainder plus `occupied_capacity` does not, the function returns a **wrong (much smaller) capacity** with no error — directly analogous to storing the pre-fee amount instead of the post-fee amount.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the withdrawable capacity as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the high bits of `withdraw_counted_capacity` if it exceeds `u64::MAX`. The `safe_add` that follows only catches overflow of the *truncated* value plus `occupied_capacity` — it does not detect the prior truncation. If `withdraw_counted_capacity > u64::MAX` but `(withdraw_counted_capacity % 2^64) + occupied_capacity ≤ u64::MAX`, the function returns `Ok(wrong_small_value)` with no indication of error.

Every other `u128`→`u64` narrowing in the same file uses the checked form:

```rust
// secondary_block_reward (line 204)
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch (line 244-245)
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);

// dao_field_with_current_epoch (line 258)
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The `as u64` in `calculate_maximum_withdraw` is the sole exception and is inconsistent with the established pattern.

The existing overflow test (`check_withdraw_calculation_overflows`) only exercises the path where `safe_add` itself overflows — it does not cover the silent-truncation window where `withdraw_counted_capacity % 2^64` is small enough that `safe_add` succeeds with a wrong value: [5](#0-4) 

---

### Impact Explanation

`calculate_maximum_withdraw` is called from three production paths:

1. **`transaction_maximum_withdraw` → `transaction_fee`** (`util/dao/src/lib.rs` lines 30–36): fee = `maximum_withdraw − outputs_capacity`. A truncated `maximum_withdraw` produces a wrong (too-small) fee. If the user crafts the withdrawal transaction with `outputs_capacity = truncated_value`, the node accepts it and the depositor receives far less than owed.

2. **`withdrawed_interests` → `dao_field_with_current_epoch` → `current_s`** (`util/dao/src/lib.rs` lines 312–333, 252–254): `withdrawed_interests` is underestimated, so `current_s` (the NervosDAO savings field `S`) stored in every subsequent block header is inflated. This corrupts the on-chain DAO accounting state.

3. **`calculate_dao_maximum_withdraw` RPC** (`rpc/src/module/experiment.rs` lines 259–267, 288–294): The RPC returns the wrong (too-small) capacity to the caller, who then builds a withdrawal transaction claiming less than they are entitled to. [6](#0-5) [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

The truncation fires when:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX  (≈ 1.844 × 10¹⁹)
```

`counted_capacity` is bounded by the total CKB supply (≈ 3.36 × 10¹⁸ shannons), so the condition requires:

```
withdrawing_ar / deposit_ar  >  1.844×10¹⁹ / 3.36×10¹⁸  ≈  5.49
```

The accumulation rate (`ar`) starts at `10_000_000_000_000_000` and grows slowly with secondary issuance. A 5.49× growth is not achievable on any realistic timeline under current protocol parameters. **Likelihood is therefore very low** — this is a latent defect rather than an immediately exploitable condition. It is reported because the root cause (wrong variable used — `as u64` instead of `u64::try_from`) is a real code error with a real, reachable impact path, directly matching the "wrong amount stored" vulnerability class.

---

### Recommendation

Replace the silent cast with the checked conversion already used everywhere else in the file:

```rust
// Before (silent truncation):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (consistent with the rest of the file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
```

Add a test that places `withdraw_counted_capacity` in the silent-truncation window (i.e., `> u64::MAX` but truncated remainder `+ occupied_capacity ≤ u64::MAX`) and asserts `Err(DaoError::Overflow)`.

---

### Proof of Concept

**Trigger condition** (arithmetic):

```
counted_capacity  = C   (e.g., 3.36 × 10¹⁸ shannons — full CKB supply)
withdrawing_ar    = W
deposit_ar        = D

withdraw_counted_capacity = C × W / D

For silent truncation:
  C × W / D  >  u64::MAX                          (truncation occurs)
  (C × W / D) % 2^64  +  occupied_capacity  ≤  u64::MAX   (safe_add succeeds)
```

**Code path** (reachable by any RPC caller today):

```
RPC caller → calculate_dao_maximum_withdraw (rpc/src/module/experiment.rs:235)
           → DaoCalculator::calculate_maximum_withdraw (util/dao/src/lib.rs:127)
           → withdraw_counted_capacity as u64   ← wrong value
           → safe_add succeeds
           → Ok(wrong_small_capacity) returned to caller
```

The caller uses this value to set `outputs_capacity` in the withdrawal transaction. The node's `transaction_fee` path (`util/dao/src/lib.rs:30`) recomputes `maximum_withdraw` with the same truncated value, accepts the transaction as valid, and the depositor permanently loses the difference between the correct and truncated withdrawal amounts.     fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
            DaoWithdrawingCalculationKind::WithdrawingOutPoint(withdrawing_out_point) => {
                let (_tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                let withdrawing_out_point: packed::OutPoint = withdrawing_out_point.into();
                let (withdrawing_tx, withdrawing_header_hash) = snapshot
                    .get_transaction(&withdrawing_out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid withdrawing_out_point"))?;

                let output = withdrawing_tx
                    .outputs()
                    .get(withdrawing_out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid withdrawing_out_point"))?;
                let output_data = withdrawing_tx
                    .outputs_data()
                    .get(withdrawing_out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid withdrawing_out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash,
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
        }
    }
