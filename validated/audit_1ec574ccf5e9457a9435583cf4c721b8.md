### Title
`ScriptHashTypeVerifier` Enforces Hash-Type Consensus Rule Only on Lock Scripts, Silently Skipping Type Scripts — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier` is the non-contextual gate that enforces the consensus rule "only permitted `ScriptHashType` values may appear in transaction outputs." However, its implementation iterates over outputs and checks only `output.lock().hash_type()`, completely omitting `output.type_().hash_type()`. This mirrors the external report's pattern: one component enforces an authorization rule using mechanism X (lock-script path), while the actual execution engine enforces it using mechanism Y (all-script path), leaving the type-script path unguarded at the cheap filter stage.

---

### Finding Description

`ScriptHashTypeVerifier::verify` in `verification/src/transaction_verifier.rs` reads:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

The consensus-permitted set is:

```rust
pub static ENABLED_SCRIPT_HASH_TYPE: Set<u8> = phf_set! {
    0u8, // ScriptHashType::Data
    1u8, // ScriptHashType::Type
    2u8, // ScriptHashType::Data1
    4u8, // ScriptHashType::Data2
};
``` [2](#0-1) 

Values `3`, `5`, `6`, `7`, … are absent from the set and are therefore disallowed by consensus. The verifier correctly rejects a **lock** script carrying such a value, but it never reads `output.type_()` at all. A transaction whose outputs carry a type script with `hash_type = 3` (or any other disallowed byte) passes `ScriptHashTypeVerifier` without error.

The struct is wired into `NonContextualTransactionVerifier`, which is the first, cheapest admission gate for both the tx-pool and block-tx verification:

```rust
pub struct NonContextualTransactionVerifier<'a> {
    ...
    pub(crate) script_hash_type: ScriptHashTypeVerifier<'a>,
}
``` [3](#0-2) 

The module-level comment for `NonContextualTransactionVerifier` even documents the gap explicitly — it lists only "Check whether output **lock** hash type within enabled range," confirming the type-script path was never added. [4](#0-3) 

---

### Impact Explanation

**Resource amplification / tx-pool DoS.** The non-contextual verifier is intentionally cheap — it runs before any cell resolution or VM execution. When it passes a transaction, the node admits it to the tx-pool, propagates it to peers via the relay protocol, and eventually runs the full `TransactionScriptsVerifier` (expensive VM execution) against it. A transaction with a disallowed type-script `hash_type` will ultimately be rejected by the VM, but only after consuming:

- tx-pool admission work on every receiving node,
- relay bandwidth to every connected peer,
- VM setup and early-exit cycles inside `TransactionScriptsVerifier`.

Because the cheap gate is bypassed, an unprivileged attacker can craft a stream of syntactically valid transactions (valid lock scripts, valid capacity, valid structure) that each carry a type script with `hash_type = 3`. Each transaction passes `NonContextualTransactionVerifier`, floods the relay layer, and forces every peer to spin up the script verifier before discarding it. The attacker pays no on-chain fee because the transactions are never mined.

---

### Likelihood Explanation

The entry path requires no privilege: any RPC caller or P2P peer can submit a raw transaction. Constructing a transaction with an arbitrary `hash_type` byte in a type script is trivial — it is a single-byte field in the molecule-encoded `Script` struct. The attack is repeatable at negligible cost to the attacker and scales linearly with the number of peers.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify` to also validate the type script of every output, mirroring the existing lock-script check:

```rust
// after the existing lock check, inside the same loop:
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        })
        .into());
    }
}
```

Update the `NonContextualTransactionVerifier` doc-comment to reflect that both lock and type hash types are checked.

---

### Proof of Concept

1. Craft a `TransactionView` whose single output has:
   - a valid lock script (`hash_type = 0`, always-success code hash),
   - a type script with `hash_type = 3` (not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via RPC `send_raw_transaction` or relay over P2P.
3. `NonContextualTransactionVerifier::verify` → `ScriptHashTypeVerifier::verify` iterates the output, checks `output.lock().hash_type() == 0` ✓, never reads `output.type_()`, returns `Ok(())`.
4. The transaction enters the tx-pool and is relayed to peers.
5. `TransactionScriptsVerifier` eventually fails when it tries to resolve the type script's code hash under hash-type `3`, but only after full VM setup cost has been paid on every node.

The lock-script path is correctly blocked: replacing the type script with a lock script carrying `hash_type = 3` causes `ScriptHashTypeVerifier` to return `ScriptHashTypeNotPermitted` immediately, confirming the asymmetry. [5](#0-4) [2](#0-1)
