### Title
No `maxFeeAccepted` Guard in `requestHelper` Silently Consumes Excess ETH — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` function in `Entropy.sol` unconditionally credits all `msg.value` above the provider fee to `accruedPythFeesInWei` with no refund and no caller-supplied upper-bound parameter. This is the direct analog of the ERC4626 "no `maxAssetsIn`" slippage gap: a user who sends more ETH than the current fee (a common defensive pattern against fee volatility) permanently loses the surplus to the protocol with no recourse.

---

### Finding Description

Every `requestV2` variant routes through `requestHelper`:

```solidity
// Entropy.sol lines 233-239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The arithmetic `msg.value - providerFee` is assigned entirely to `accruedPythFeesInWei`. Because `requiredFee = providerFee + pythFee`, any `msg.value` above `requiredFee` inflates `accruedPythFeesInWei` by exactly that surplus — it is never returned to the caller.

The `IEntropyV2` interface documents this but provides no mitigation:

> "Note that the fee can change over time … excess value is *not* refunded to the caller." [2](#0-1) 

None of the four `requestV2` overloads accept a `maxFeeAccepted` parameter: [3](#0-2) 

The provider fee is mutable at any time via `setProviderFee` / `setProviderFeeAsFeeManager`: [4](#0-3) 

---

### Impact Explanation

A consumer contract that forwards user ETH directly — the canonical pattern shown in Pyth's own documentation:

```solidity
uint256 fee = entropy.getFeeV2();
entropy.requestV2{ value: fee }();
``` [2](#0-1) 

…must either:

1. Send exactly `getFeeV2()` — and risk an `InsufficientFee` revert if the fee rises between the off-chain read and on-chain execution, or
2. Send a buffer (e.g. `msg.value` passed through from the end user) — and permanently lose the surplus to `accruedPythFeesInWei`.

There is no third option. The contract provides no way for a caller to express "revert if the fee exceeds X" — the exact protection the ERC4626 `maxAssetsIn` parameter provides. The lost ETH accrues to the Pyth protocol treasury and is irrecoverable by the user.

---

### Likelihood Explanation

- The fee is explicitly documented as dynamic and gas-price-dependent; it changes on every keeper `adjust_fee_if_necessary` cycle.
- Consumer contracts that pass `msg.value` through (e.g. `entropy.requestV2{value: msg.value}(...)`) are the natural implementation pattern and are shown in Pyth's own tutorial code.
- Any end user who sends a small buffer to avoid reverts (a standard UX practice) silently loses that buffer.
- No privileged role is required on the attacker side; the loss is triggered by ordinary usage.

---

### Recommendation

Add a `maxFeeAccepted` parameter to `requestHelper` (and expose it through at least one `requestV2` overload):

```solidity
function requestHelper(
    address provider,
    bytes32 userCommitment,
    bool useBlockhash,
    bool isRequestWithCallback,
    uint32 callbackGasLimit,
    uint128 maxFeeAccepted   // NEW
) internal returns (...) {
    uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
    if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
    if (requiredFee > maxFeeAccepted) revert EntropyErrors.FeeTooHigh();
    // refund surplus
    uint128 surplus = SafeCast.toUint128(msg.value) - requiredFee;
    if (surplus > 0) {
        (bool ok,) = msg.sender.call{value: surplus}("");
        require(ok, "refund failed");
    }
    ...
}
```

Alternatively, always refund `msg.value - requiredFee` unconditionally, which eliminates the surplus-loss issue without requiring a new parameter.

---

### Proof of Concept

1. Provider fee is currently 0.001 ETH; `pythFeeInWei` = 0.0001 ETH; `getFeeV2()` = 0.0011 ETH.
2. Consumer contract user sends 0.002 ETH to `requestRandomNumber()`.
3. Consumer calls `entropy.requestV2{value: 0.002 ether}()`.
4. Inside `requestHelper`:
   - `requiredFee` = 0.0011 ETH ✓ (no revert)
   - `providerInfo.accruedFeesInWei` += 0.001 ETH
   - `_state.accruedPythFeesInWei` += 0.002 − 0.001 = **0.001 ETH** (includes 0.0009 ETH surplus that belongs to the user)
5. User's 0.0009 ETH surplus is permanently credited to the Pyth treasury. No event is emitted for the surplus; no refund path exists. [5](#0-4)
