### Title
Deviation-Only Subscriptions Permanently Blocked When All Price Feeds Report Zero Prices — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler._validateShouldUpdatePrices`, when a subscription is configured with `updateOnDeviation = true` and `updateOnHeartbeat = false`, the deviation loop silently skips every price feed whose `previousPrice` or `currentPrice` is zero via a `continue` statement. If all feeds in the subscription have a zero price, the loop exhausts without ever returning a valid timestamp, and the function unconditionally reverts with `UpdateConditionsNotMet`. The subscription is permanently blocked from receiving updates, mirroring the GMX `Compound()` bug where the function only executes when one specific token (ARB) is present and silently fails when only the other tokens are present.

---

### Finding Description

`_validateShouldUpdatePrices` evaluates two independent criteria in sequence:

```solidity
// If updateOnHeartbeat is enabled and the heartbeat interval has passed, trigger update
if (params.updateCriteria.updateOnHeartbeat) { ... return updateTimestamp; }

// If updateOnDeviation is enabled, check if any price has deviated enough
if (params.updateCriteria.updateOnDeviation) {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        ...
        // Skip if either price is zero to avoid division by zero
        if (previousPrice == 0 || currentPrice == 0) {
            continue;                          // ← silently skips the feed
        }
        ...
        if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
            return updateTimestamp;
        }
    }
}

revert SchedulerErrors.UpdateConditionsNotMet();   // ← always reached when all feeds are skipped
``` [1](#0-0) 

When `updateOnHeartbeat` is `false` and `updateOnDeviation` is `true`, the heartbeat block is never entered. The deviation loop then iterates all feeds. For any feed where `previousPrice == 0 || currentPrice == 0`, the feed is skipped with `continue`. If every feed in the subscription satisfies this condition, the loop terminates without returning, and execution falls through to the unconditional revert. [2](#0-1) 

The `UpdateCriteria` struct explicitly supports deviation-only subscriptions (`updateOnHeartbeat = false`, `updateOnDeviation = true`), and `_validateSubscriptionParams` permits this configuration as long as `deviationThresholdBps > 0`. [3](#0-2) 

The `price` field in `PythStructs.PriceFeed` is `int64`, which can legitimately be zero for certain exotic or synthetic assets, or transiently during market halts. Once a zero price is stored as the `previousFeed` for a subscription, every subsequent `updatePriceFeeds` call for that subscription will skip all feeds and revert.

---

### Impact Explanation

- **Subscription permanently blocked:** A deviation-only subscription whose stored prices are all zero can never be updated. Every keeper call to `updatePriceFeeds` reverts with `UpdateConditionsNotMet`, regardless of how much the actual price has moved.
- **Subscription balance drained by Pyth fee:** The Pyth fee is deducted from `status.balanceInWei` and paid to the Pyth contract *before* `_validateShouldUpdatePrices` is called. If the transaction reverts, the entire state change reverts — but the keeper still pays gas for the failed call on every attempt. [4](#0-3) 

- **Keeper gas wasted:** Any permissionless keeper monitoring the subscription will repeatedly submit failing transactions, losing gas with no compensation, since `_processFeesAndPayKeeper` is never reached.
- **Stale prices for readers:** Downstream consumers of the subscription's price data receive permanently stale prices, which can cause incorrect protocol behavior in any DeFi protocol reading from this Scheduler subscription.

---

### Likelihood Explanation

- Pyth's `int64 price` field can be zero for assets with extremely low prices, assets in a halted state, or synthetic instruments. A single zero-price update stored as `previousFeed` is sufficient to trigger the block.
- The first `updatePriceFeeds` call for a deviation-only subscription always succeeds (because `previousFeed.id == bytes32(0)` triggers an early return). If that first update stores a zero price, all subsequent calls are permanently blocked.
- Any unprivileged keeper can call `updatePriceFeeds` with valid Pyth update data containing zero prices, causing the zero price to be stored and the subscription to be blocked. [5](#0-4) 

---

### Recommendation

Replace the `continue` with logic that treats a zero-price feed as "deviation threshold met" (since a price moving from zero to non-zero is an infinite deviation), or count skipped feeds and treat a fully-skipped loop as a trigger condition:

```solidity
if (previousPrice == 0 || currentPrice == 0) {
    // A price transitioning from/to zero is always a significant deviation
    return updateTimestamp;
}
```

Alternatively, mirror the GMX report's recommendation: if the loop completes with all feeds skipped (i.e., no feed was evaluated), treat this as a trigger rather than a non-trigger, since the inability to compute deviation should not permanently block updates.

---

### Proof of Concept

1. Deploy `Scheduler` with a valid `IPyth` contract.
2. Create a subscription with `updateOnHeartbeat = false`, `updateOnDeviation = true`, `deviationThresholdBps = 100`.
3. Call `updatePriceFeeds` with update data where all price feeds have `price = 0`. This succeeds (first update, `previousFeed.id == bytes32(0)`).
4. Call `updatePriceFeeds` again with any update data (zero or non-zero prices). For each feed, `previousPrice == 0` → `continue`. Loop exhausts → `revert UpdateConditionsNotMet()`.
5. The subscription is now permanently blocked. All subsequent keeper calls revert, wasting gas, and the subscription's price data is permanently stale. [6](#0-5)

    function _validateShouldUpdatePrices(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams storage params,
        SchedulerStructs.SubscriptionStatus storage status,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal view returns (uint256) {
        // Use the most recent timestamp, as some asset markets may be closed.
        // Closed markets will have a publishTime from their last trading period.
        // Since we verify all updates share the same Pythnet slot, we still ensure
        // that all price feeds are synchronized from the same update cycle.
        uint256 updateTimestamp = 0;
        for (uint8 i = 0; i < priceFeeds.length; i++) {
            if (priceFeeds[i].price.publishTime > updateTimestamp) {
                updateTimestamp = priceFeeds[i].price.publishTime;
            }
        }

        // Calculate the minimum acceptable timestamp (clamped at 0)
        // The maximum acceptable timestamp is enforced by the parsePriceFeedUpdatesWithSlots call
        uint256 minAllowedTimestamp = (block.timestamp >
            PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            : 0;

        // Validate that the update timestamp is not too old
        if (updateTimestamp < minAllowedTimestamp) {
            revert SchedulerErrors.TimestampTooOld(
                updateTimestamp,
                block.timestamp
            );
        }

        // Reject updates if they're older than the latest stored ones
        if (
            status.priceLastUpdatedAt > 0 &&
            updateTimestamp <= status.priceLastUpdatedAt
        ) {
            revert SchedulerErrors.TimestampOlderThanLastUpdate(
                updateTimestamp,
                status.priceLastUpdatedAt
            );
        }

        // If updateOnHeartbeat is enabled and the heartbeat interval has passed, trigger update
        if (params.updateCriteria.updateOnHeartbeat) {
            uint256 lastUpdateTime = status.priceLastUpdatedAt;

            if (
                lastUpdateTime == 0 ||
                updateTimestamp >=
                lastUpdateTime + params.updateCriteria.heartbeatSeconds
            ) {
                return updateTimestamp;
            }
        }

        // If updateOnDeviation is enabled, check if any price has deviated enough
        if (params.updateCriteria.updateOnDeviation) {
            for (uint8 i = 0; i < priceFeeds.length; i++) {
                // Get the previous price feed for this price ID using subscriptionId
                PythStructs.PriceFeed storage previousFeed = _state
                    .priceUpdates[subscriptionId][priceFeeds[i].id];

                // If there's no previous price, this is the first update
                if (previousFeed.id == bytes32(0)) {
                    return updateTimestamp;
                }

                // Calculate the deviation percentage
                int64 currentPrice = priceFeeds[i].price.price;
                int64 previousPrice = previousFeed.price.price;

                // Skip if either price is zero to avoid division by zero
                if (previousPrice == 0 || currentPrice == 0) {
                    continue;
                }

                // Calculate absolute deviation basis points (scaled by 1e4)
                uint256 numerator = SignedMath.abs(
                    currentPrice - previousPrice
                );
                uint256 denominator = SignedMath.abs(previousPrice);
                uint256 deviationBps = Math.mulDiv(
                    numerator,
                    10_000,
                    denominator
                );

                // If deviation exceeds threshold, trigger update
                if (
                    deviationBps >= params.updateCriteria.deviationThresholdBps
                ) {
                    return updateTimestamp;
                }
            }
        }

        revert SchedulerErrors.UpdateConditionsNotMet();
    }
