### Title
Zero Price Silently Skips Deviation Check, Permanently Blocking Price Updates for Deviation-Only Subscriptions - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler._validateShouldUpdatePrices`, when either the stored previous price or the incoming current price is zero, the deviation check is silently skipped via `continue`. For subscriptions configured with `updateOnDeviation: true` and `updateOnHeartbeat: false`, this causes every `updatePriceFeeds` call to revert with `UpdateConditionsNotMet` whenever any price feed reaches zero — permanently blocking price updates at the most critical moment.

---

### Finding Description

In `_validateShouldUpdatePrices`, the deviation loop at lines 428–431 reads:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
``` [1](#0-0) 

When `currentPrice == 0` (price has crashed to zero) or `previousPrice == 0` (stored price is zero and a new price arrives), the feed is silently skipped. If all feeds in the subscription hit this condition, the loop exhausts without returning, and execution falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [2](#0-1) 

This means `updatePriceFeeds` always reverts for deviation-only subscriptions when any price is zero. The Scheduler's stored price is never updated to reflect the zero price.

The subscription validation only requires at least one criterion to be enabled:

```solidity
if (
    !params.updateCriteria.updateOnHeartbeat &&
    !params.updateCriteria.updateOnDeviation
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [3](#0-2) 

So a subscription with `updateOnDeviation: true, updateOnHeartbeat: false` is fully valid and deployable. The `updatePriceFeeds` function is permissionless — any Keeper can call it: [4](#0-3) 

Two concrete scenarios are affected:

1. **Price crashes to zero:** `currentPrice == 0`, `previousPrice != 0` → skip → revert. The Scheduler retains the old non-zero price indefinitely.
2. **Price recovers from zero:** `previousPrice == 0` (stored from a prior first-update), `currentPrice != 0` → skip → revert. The Scheduler cannot push the recovery price.

---

### Impact Explanation

Downstream protocols (DeFi lending, perpetuals, etc.) that consume prices from the Pyth Pulse Scheduler via `getPricesUnsafe` or `getPricesNoOlderThan` will receive the stale pre-crash price instead of zero. This directly mirrors the GMX M-20 scenario: at the exact moment a price reaches zero — when liquidations are most urgently needed — the price oracle layer stops updating. Protocols relying on Pulse for push-style price feeds will:

- Fail to liquidate underwater positions (stale price makes positions appear solvent).
- Allow continued borrowing against worthless collateral.
- Accumulate bad debt until the subscription's heartbeat fires (if one exists) or manual intervention occurs.

For deviation-only subscriptions (no heartbeat), the block is **permanent** until the subscription parameters are manually changed by the Manager.

---

### Likelihood Explanation

Pyth price feeds carry `int64` prices and explicitly support zero as a valid value — the core Pyth contract does not reject zero prices. DAO governance tokens, newly listed assets, and de-pegged stablecoins can realistically reach zero. The Scheduler README explicitly targets DeFi protocols as the primary consumer. Any subscription created with `updateOnDeviation: true, updateOnHeartbeat: false` (a natural configuration for latency-sensitive protocols that want updates only on significant moves) is permanently vulnerable the moment any subscribed feed hits zero.

---

### Recommendation

Replace the `continue` with an unconditional trigger when either price is zero, since a transition to or from zero represents an infinite deviation:

```solidity
// A zero price represents maximum deviation — always trigger update
if (previousPrice == 0 || currentPrice == 0) {
    return updateTimestamp;
}
```

This ensures that any price movement involving zero is treated as a threshold-exceeding deviation, consistent with the intent of the deviation trigger.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` and create a subscription with:
   - `updateOnHeartbeat: false`
   - `updateOnDeviation: true`, `deviationThresholdBps: 100` (1%)
2. Call `updatePriceFeeds` with `price = 1000` → succeeds (first update, `previousFeed.id == bytes32(0)` path).
3. Call `updatePriceFeeds` with `price = 0` (asset crashes to zero):
   - `currentPrice == 0` → `continue` for all feeds
   - Loop exhausts → `revert UpdateConditionsNotMet()`
4. The Scheduler still reports `price = 1000`. Downstream consumers see no crash.
5. Any subsequent call with `price != 0` also reverts because `previousPrice == 0` → `continue` → same revert.

The stored price is permanently frozen at `1000` for a deviation-only subscription, with no on-chain mechanism to recover without the Manager changing subscription parameters. [5](#0-4)
