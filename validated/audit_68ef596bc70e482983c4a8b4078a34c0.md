### Title
Unprotected Filter Protocol Enables Asymmetric DB-Exhaustion DoS via `GetBlockFilterCheckPoints` — (`sync/src/filter/get_block_filter_check_points_process.rs`, `sync/src/filter/mod.rs`)

---

### Summary

The `BlockFilter` protocol handler has no per-peer rate limiter, while the `Relayer` protocol enforces a `governor::RateLimiter` at 30 req/s per `(PeerIndex, message_type)`. An unprivileged remote peer can exploit this asymmetry by flooding a Filter-enabled node with `GetBlockFilterCheckPoints` messages, each triggering up to 4,000 synchronous DB reads on the shared async runtime, starving relay/sync tasks and degrading block propagation.

---

### Finding Description

**Relayer has rate limiting; Filter does not.**

`Relayer::try_process` gates every non-`CompactBlock` message through a keyed rate limiter before dispatching: [1](#0-0) [2](#0-1) [3](#0-2) 

`BlockFilter::received` has no equivalent check — it parses the message and immediately calls `self.process(...)`: [4](#0-3) 

**Each `GetBlockFilterCheckPoints` message triggers up to 4,000 DB reads.**

`GetBlockFilterCheckPointsProcess::execute` loops up to `BATCH_SIZE = 2000` times, performing two DB lookups per iteration (`get_block_hash` + `get_block_filter_hash`): [5](#0-4) [6](#0-5) 

`GetBlockFilterHashes` has the same `BATCH_SIZE = 2000` and the same two-read-per-iteration pattern, also unprotected: [7](#0-6) [8](#0-7) 

`GetBlockFilters` uses `BATCH_SIZE = 1000` with two reads per iteration (up to 2,000 reads), also unprotected: [9](#0-8) [10](#0-9) 

---

### Impact Explanation

All three Filter request handlers (`GetBlockFilterCheckPoints`, `GetBlockFilterHashes`, `GetBlockFilters`) and the Relay handlers share the same tokio async runtime. An attacker sending `N` `GetBlockFilterCheckPoints` messages per second forces up to `N × 4,000` DB reads per second into the shared runtime. Because the Relay protocol's rate limiter caps relay message processing at 30 req/s per peer, the Filter protocol becomes the only unbounded path. Saturating the DB and async task pool on a targeted node degrades its ability to process `CompactBlock` and relay messages, slowing block propagation from that node.

---

### Likelihood Explanation

- Requires only a standard P2P connection to a Filter-enabled node — no credentials, no PoW, no privileged access.
- The message payload is minimal (a single `start_number` field), so bandwidth cost to the attacker is negligible.
- The asymmetry is structural: the Relayer explicitly documents its rate limiter as a "hard cap with buffer" (line 90 comment), but no equivalent was added to the Filter protocol.
- Filter is an opt-in feature, so only nodes that enable it are exposed; however, light-client infrastructure nodes are likely to enable it, making them high-value targets. [11](#0-10) 

---

### Recommendation

Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`) and check it at the top of `BlockFilter::try_process` before dispatching to any of the three request handlers. The same 30 req/s quota used by the Relayer is a reasonable starting point; a lower value (e.g., 5 req/s) is defensible given the much higher per-message DB cost.

---

### Proof of Concept

```
# Differential test (pseudocode)
for i in 0..1000:
    send_to_filter_protocol(peer, GetBlockFilterCheckPoints { start_number: 0 })
    send_to_relay_protocol(peer, GetRelayTransactions { tx_hashes: [...] })

# Expected with current code:
#   Filter: all 1000 messages processed → up to 4,000,000 DB reads
#   Relay:  ~30 messages processed, remainder dropped by rate limiter

# Observe: node's block relay latency increases while filter flood is active;
# relay latency returns to baseline after flood stops.
```
