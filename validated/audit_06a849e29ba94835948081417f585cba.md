### Title
`DaoCalculator::transaction_maximum_withdraw` Uses Full `u64` Witness Index While On-Chain DAO C Script Uses Only the Lowest Byte, Causing Valid DAO Withdrawal Transactions to Be Rejected — (`util/dao/src/lib.rs`)

---

### Summary

The Rust off-chain `DaoCalculator` in `util/dao/src/lib.rs` reads the `header_dep_index` from the witness `input_type` field as a full `u64` and uses it directly to index into `header_deps`. The on-chain DAO C script (`dao.c`) uses only the **lowest byte** of that same 8-byte field as the index. When a DAO withdrawal transaction encodes an index whose full `u64` value exceeds 255 but whose lowest byte correctly identifies the deposit block, the Rust verifier resolves a different (wrong) `header_dep` entry than the C VM does. The Rust verifier then rejects the transaction as invalid, while the C VM would accept it. This is a direct analog to the external report's pattern: the wrong object/value is used for a critical lookup, causing a DoS.

---

### Finding Description

**Root cause — `util/dao/src/lib.rs`, lines 91–99:**

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```
followed by:
```rust
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // ← full u64 used as index
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

The Rust code reads the full 8-byte little-endian `u64` from `WitnessArgs.input_type` and uses it as the array index into `header_deps`. The on-chain DAO C script at `dao.c` reads the same 8-byte field but interprets only the **lowest byte** as the index (effectively treating it as a `u8`).

**Discrepancy documented in the codebase itself:**

The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this split:

```rust
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
let mut header_deps = vec![dummy; 258];
header_deps[1] = deposit_block.hash();
header_deps[257] = withdraw_block.hash();

// input_type = 257, lowest byte = 1
let witness = WitnessArgs::new_builder()
    .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
    .build();
```

The test then asserts `result.is_err()` — confirming the Rust verifier rejects a transaction the C VM would accept.

**Exploit flow:**

1. Attacker (or any DAO user) crafts a DAO withdrawal transaction with ≥ 258 `header_deps`.
2. The witness `input_type` encodes the value `257` (bytes: `0x01 0x01 0x00 0x00 0x00 0x00 0x00 0x00`).
3. The on-chain C VM reads the lowest byte = `1`, resolves `header_deps[1]` = correct deposit block → script execution succeeds.
4. The Rust `DaoCalculator::transaction_maximum_withdraw` reads the full `u64` = `257`, resolves `header_deps[257]` = a different block (e.g., the withdraw block, number 200).
5. The block number check at line 105 (`deposit_header.number() != deposited_block_number`) fails because the resolved header is the wrong block.
6. `DaoCalculator::transaction_fee` returns `Err(DaoError::InvalidOutPoint)`.
7. `check_tx_fee` in `tx-pool/src/util.rs` (line 34–41) propagates this as `Reject::Malformed`, blocking the transaction from the tx-pool.

---

### Impact Explanation

**Tx-pool DoS for valid DAO withdrawals:** Any DAO withdrawal transaction with `header_dep_index > 255` (where the lowest byte correctly identifies the deposit block but the full `u64` does not) is permanently rejected by `check_tx_fee` in the tx-pool, even though the on-chain C VM would accept it. The user cannot withdraw their DAO deposit via this transaction format.

**Consensus split:** If a miner includes such a transaction in a block (bypassing the tx-pool), nodes running the Rust verifier will reject the block at the `DaoHeaderVerifier` stage in `verification/contextual/src/contextual_block_verifier.rs` (line 300–319), while nodes running only the C VM would accept it. This splits the network.

---

### Likelihood Explanation

The condition requires a transaction with ≥ 256 `header_deps` and a witness index whose lowest byte differs from its full `u64` value. This is an unusual but reachable construction — any transaction sender with a DAO deposit can submit such a transaction via `send_transaction` RPC. No privileged access is required. The discrepancy is already documented in the production test suite, confirming the developers are aware of the behavioral difference.

---

### Recommendation

In `transaction_maximum_withdraw`, truncate the `header_dep_index` to a `u8` before using it as an array index, to match the on-chain C VM behavior:

```rust
// Read as u64 but use only the lowest byte, matching dao.c behavior
let index_u8 = (LittleEndian::read_u64(&header_deps_index_data.unwrap()) & 0xFF) as usize;
rtx.transaction
    .header_deps()
    .get(index_u8)
    ...
```

Alternatively, add an explicit validation that rejects any witness index exceeding `u8::MAX`, and document this as a protocol-level constraint enforced by both the C VM and the Rust verifier.

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` is the PoC. It constructs a transaction where:

- `header_deps[1]` = correct deposit block (block 100)
- `header_deps[257]` = withdraw block (block 200)
- Witness `input_type` = `257u64` (lowest byte = 1)
- Cell data = `100u64` (deposited at block 100)

The C VM resolves index `1` → deposit block (number 100) → matches cell data → **accepts**.
The Rust `DaoCalculator` resolves index `257` → withdraw block (number 200) → does not match cell data (100) → **rejects** with `DaoError::InvalidOutPoint`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
