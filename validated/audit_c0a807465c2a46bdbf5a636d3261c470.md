Audit Report

## Title
`ScriptHashTypeVerifier::verify()` Skips Output Type Script Hash Type Validation, Enabling Consensus Split During Hardfork Window — (File: `verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs but validates only the lock script's `hash_type` against `ENABLED_SCRIPT_HASH_TYPE`. The type script field is never read. During CKB's staged hardfork preparation window — when the VM binary already supports a new version but consensus has not yet activated it — an attacker can embed a disallowed hash type (e.g., `Data3 = 6`) in an output's type script, pass this verifier silently, and cause updated nodes to accept the transaction while older nodes reject it, producing a consensus split.

## Finding Description

**Root cause — `ScriptHashTypeVerifier::verify()` lines 796–814:**

The loop body calls `output.lock().hash_type()` and checks it against `ENABLED_SCRIPT_HASH_TYPE`. There is no corresponding call to `output.type_().to_opt()`. The type script hash type is entirely invisible to this verifier. [1](#0-0) 

`ENABLED_SCRIPT_HASH_TYPE` is `{0, 1, 2, 4}` — Data, Type, Data1, Data2 — and explicitly excludes `Data3 = 6`: [2](#0-1) 

The struct's own comment states the intent is to verify "the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules," but only half of each output (the lock script) is checked: [3](#0-2) 

The existing tests cover only the lock script path (`test_unknown_hash_type_output_lock`, `test_not_enabled_hash_type_output_lock`); no test exercises a disallowed hash type on a type script: [4](#0-3) 

**Exploit flow:**

1. Attacker constructs a transaction output with lock `hash_type = 0` (Data, allowed) and type script `hash_type = 6` (Data3, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. `ScriptHashTypeVerifier::verify()` reads only the lock hash type, finds `0 ∈ ENABLED_SCRIPT_HASH_TYPE`, returns `Ok(())`.
3. On a node whose CKB-VM binary already supports VM version 3 (normal during hardfork prep), `TransactionScriptsVerifier` runs the type script group, `select_version` succeeds — transaction accepted.
4. On a node whose binary does not yet support VM version 3, `select_version` returns `InvalidVmVersion(3)` — transaction rejected.
5. The two nodes diverge on chain state.

No other verifier in the transaction verification pipeline was found to compensate for this omission.

## Impact Explanation

**Critical (15001–25000 points) — consensus deviation.** `ScriptHashTypeVerifier` is the designated enforcement boundary that keeps VM binary support and consensus activation in sync during hardfork preparation. Because it ignores type scripts, any transaction sender can unilaterally exercise a future VM version for type scripts during the preparation window, splitting the network between nodes that have and have not yet upgraded their binary.

## Likelihood Explanation

The attacker is an ordinary transaction sender with no privileged access, no keys, and no majority hashpower. The only precondition is that the CKB-VM binary already contains support for the target VM version — the normal and expected state during any hardfork preparation window. The crafted transaction is submitted via the standard `send_transaction` RPC. The gap is reachable on every transaction that includes an output with a type script.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to validate the hash type of each output's type script in addition to its lock script:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        Self::check_hash_type(output.lock().hash_type())?;
        if let Some(type_script) = output.type_().to_opt() {
            Self::check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}

fn check_hash_type(raw: packed::Byte) -> Result<(), Error> {
    match TryInto::<ScriptHashType>::try_into(raw) {
        Ok(hash_type) => {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
            Ok(())
        }
        Err(_) => Err((TransactionError::InvalidScriptHashType { hash_type: raw }).into()),
    }
}
```
