### Title
Governance VAA Out-of-Order Relay Permanently Blocks Lower-Sequence Governance Actions — (`target_chains/sui/contracts/sources/governance/governance.move`)

### Summary
The sequence guard in `execute_governance_instruction` only enforces `sequence > last_executed`, not `sequence == last_executed + 1`. Any unprivileged relayer holding multiple legitimately-signed governance VAAs can relay a high-sequence VAA first, permanently making all lower-sequence VAAs unexecutable.

### Finding Description

In `governance::execute_governance_instruction`, the only sequence check is:

```move
assert!(sequence > state::get_last_executed_governance_sequence(pyth_state),
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);
state::set_last_executed_governance_sequence(&latest_only, pyth_state, sequence);
``` [1](#0-0) 

The same pattern is duplicated in `contract_upgrade::authorize_upgrade`: [2](#0-1) 

The guard enforces strict monotonicity (`new > old`) but **not strict sequentiality** (`new == old + 1`). After `set_last_executed_governance_sequence` stores the high sequence number: [3](#0-2) 

every VAA with a sequence number below the stored value will permanently fail the `assert!` on re-submission.

**Attack path:**

1. Pyth governance emits VAAs with sequences 1, 2, 3 (e.g., `set_data_sources`, `set_update_fee`, `set_stale_price_threshold`). All three are publicly observable on the Wormhole guardian network.
2. A malicious relayer calls `wormhole::vaa::parse_and_verify` on VAA #3, obtaining a `VAA` object, then calls `governance::verify_vaa` (which only checks the emitter chain/address): [4](#0-3) 

3. The relayer calls `execute_governance_instruction` with the receipt for sequence 3. The check `3 > 0` passes; `last_executed_governance_sequence` is set to 3.
4. VAAs #1 and #2 now permanently fail: `1 > 3` is false, `2 > 3` is false. They can never be executed.

No privileged access is required. Wormhole-signed VAAs are publicly available to any observer.

### Impact Explanation

Any set of pending governance VAAs (data source updates, fee changes, stale-price threshold changes, contract upgrades) can be permanently censored by any unprivileged relayer who submits a higher-sequence VAA first. The skipped governance actions are irrecoverable without issuing entirely new VAAs from the governance source — and even those new VAAs could be attacked the same way.

### Likelihood Explanation

Governance VAAs are publicly observable on the Wormhole guardian network. Any time Pyth governance issues more than one VAA in a short window (e.g., a batch of parameter updates or a contract upgrade paired with a config change), a griefing relayer can exploit the ordering gap. No key material, privileged role, or off-chain coordination is needed.

### Recommendation

Replace the `>` check with a strict next-sequence check:

```move
assert!(
    sequence == state::get_last_executed_governance_sequence(pyth_state) + 1,
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER
);
```

Apply the same fix in `contract_upgrade::authorize_upgrade`. This enforces that governance VAAs must be executed in the exact order they were issued, preventing any relayer from skipping intermediate actions.

### Proof of Concept

Invariant test sketch (Move pseudocode):

```move
// Setup: last_executed_governance_sequence = 0
// Governance has issued VAAs with sequences 1 and 1000.

// Step 1: relay sequence=1000 first
let receipt_1000 = governance::verify_vaa(pyth_state, vaa_seq_1000);
governance::execute_governance_instruction(pyth_state, receipt_1000);
// last_executed_governance_sequence is now 1000

// Step 2: attempt to relay sequence=1
let receipt_1 = governance::verify_vaa(pyth_state, vaa_seq_1);
governance::execute_governance_instruction(pyth_state, receipt_1);
// ABORTS with E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER
// because 1 > 1000 is false — VAA #1 is permanently blocked.
```
