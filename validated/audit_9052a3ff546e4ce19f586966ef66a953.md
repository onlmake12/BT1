### Title
DAO Withdrawal Witness Index Width Mismatch Between Rust `DaoCalculator` and On-Chain C Script Causes Fund Lockout and Consensus Split Risk — (File: `util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw()` reads the DAO withdrawal witness `input_type` field as a full **u64** to index into `header_deps`, while the on-chain C DAO script (`dao.c`) reads only the **lowest byte (u8)**. A transaction sender can craft a DAO withdrawal with a witness index > 255 that the C script accepts but the Rust node rejects at both tx-pool admission and block verification, permanently locking the user's DAO funds in the prepare cell and creating a latent consensus split vector.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw()` decodes the witness `input_type` field as a little-endian u64 and uses it directly as an index into `header_deps`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
rtx.transaction.header_deps().get(header_dep_index as usize)
``` [1](#0-0) 

The on-chain C DAO script (referenced at `dao.c#L81`) reads only the lowest byte of the same 8-byte field, treating it as a u8 index. The test `check_dao_withdraw_header_dep_index_exceeds_u8` explicitly documents this discrepancy:

- `header_deps[1]` = deposit block (what the C VM resolves via lowest byte of `0x101`)
- `header_deps[257]` = withdraw block (what Rust resolves via full u64 `257`) [2](#0-1) 

The test asserts `result.is_err()` for the Rust path, confirming the divergence is present and unresolved.

This discrepancy propagates through two critical production paths:

**Path 1 — Tx-pool admission (`tx-pool/src/util.rs`):**
`check_tx_fee()` calls `DaoCalculator::transaction_fee()`. If the u64 index resolves to a non-deposit block, the call returns `Err`, and the transaction is rejected as `Reject::Malformed` before entering the pool. [3](#0-2) 

**Path 2 — Block verification (`verification/contextual/src/contextual_block_verifier.rs`):**
`DaoHeaderVerifier::verify()` calls `DaoCalculator::dao_field()` → `withdrawed_interests()` → `transaction_maximum_withdraw()`. If this fails, the block is rejected with `BlockErrorKind::InvalidDAO`. [4](#0-3) 

---

### Impact Explanation

**User fund lockout (direct):** A user who constructs a DAO withdrawal transaction with witness index `N > 255` (e.g., `N = 257`, lowest byte `= 1` pointing to the correct deposit block) will have their transaction rejected by every Rust node's tx-pool via `check_tx_fee()`. The C DAO script running in CKB-VM would accept the same transaction. The user's CKB is locked in the prepare cell with no path to withdrawal using that transaction. They must reconstruct the transaction with index `< 256`, but if the wallet or tooling auto-generates the index from a large `header_deps` list, the user may not understand why the withdrawal fails.

**Consensus split (indirect):** A miner using a non-standard block assembler that bypasses the Rust `DaoCalculator` (e.g., one that mirrors the C script's u8 semantics) could include such a transaction in a block. The block's DAO field would be computed using u8 index semantics. When Rust nodes receive this block, `DaoHeaderVerifier::verify()` recomputes the DAO field using u64 index semantics, obtains a different `withdrawed_interests` value, and rejects the block with `InvalidDAO`. The C DAO script execution via CKB-VM would have accepted the transaction. This is a consensus split between the on-chain script and the Rust verifier.

---

### Likelihood Explanation

The fund lockout path is reachable by any unprivileged transaction sender who submits a DAO withdrawal with `header_deps` containing 256+ entries and a witness index whose full u64 value exceeds 255. This can occur accidentally when a wallet builds `header_deps` from a large set and the deposit block lands at position > 255, or intentionally by an attacker targeting a specific user's prepare cell. The consensus split path requires a custom miner but no privileged access.

---

### Recommendation

1. **Align the Rust `DaoCalculator` with the C DAO script**: In `transaction_maximum_withdraw()`, truncate the decoded u64 index to its lowest byte before indexing into `header_deps`, matching the C script's behavior:
   ```rust
   let header_dep_index = LittleEndian::read_u64(&...) & 0xFF;
   ``` [1](#0-0) 

2. **Add a non-contextual validation rule** rejecting DAO withdrawal transactions whose witness `input_type` u64 value exceeds 255, so the tx-pool and block verifier agree with the C script's reachable index space.

3. **Update the test** `check_dao_withdraw_header_dep_index_exceeds_u8` to assert `result.is_ok()` after the fix, confirming alignment with the C script.

---

### Proof of Concept

The existing test in `util/dao/src/tests.rs` directly demonstrates the discrepancy:

```
header_deps[1]   = deposit_block.hash()   // C VM resolves here (byte 0 of 0x101 = 1)
header_deps[257] = withdraw_block.hash()  // Rust resolves here (full u64 = 257)
witness input_type = 257u64 (LE bytes)
``` [5](#0-4) 

The Rust `DaoCalculator` resolves index 257 → withdraw block (number 200), but cell data encodes deposit at block 100. The block-number check at line 105 of `lib.rs` fires `DaoError::InvalidOutPoint`, causing `check_tx_fee()` to return `Reject::Malformed` and `DaoHeaderVerifier::verify()` to return `BlockErrorKind::InvalidDAO`. [6](#0-5) 

A real attacker submits a DAO withdrawal transaction via `send_transaction` RPC with 258 `header_deps` entries, placing the deposit block at index 1 and any other block at index 257, and sets the witness `input_type` to `257u64` in little-endian. The C DAO script accepts it; every Rust node rejects it at fee-check, permanently blocking the withdrawal.
