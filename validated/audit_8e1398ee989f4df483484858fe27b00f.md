### Title
Wrong Row Variable in `script_exists_in_output` Causes Incorrect Script Deletion on PostgreSQL During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

In `script_exists_in_output`, the second `match` block at line 252 reads from `row_lock` instead of `row_type`. On PostgreSQL, this makes the type-script existence check always return the lock-script query result (which is already known to be `false` at that point), causing every script that is referenced **only** as a `type_script_id` in remaining outputs to be unconditionally deleted during `rollback_block`.

---

### Finding Description

`script_exists_in_output` is supposed to return `true` if `script_id` appears in any remaining `output` row as either `lock_script_id` or `type_script_id`. [1](#0-0) 

The function executes two separate SQL `EXISTS` queries:

1. **`row_lock`** — checks `WHERE lock_script_id = $1` (lines 208–220)
2. **`row_type`** — checks `WHERE type_script_id = $1` (lines 237–249)

The first `match` block (lines 222–235) correctly short-circuits with `return Ok(true)` if the lock query is positive. If it is negative, execution falls through to the type query. The second `match` block (lines 251–256) is supposed to evaluate `row_type`, but it mistakenly reads `row_lock` again:

```rust
// pg type is BOOLEAN
match row_lock.try_get::<bool, _>(0) {   // ← BUG: should be row_type
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [2](#0-1) 

**PostgreSQL path:** `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL returns `BOOLEAN` for `EXISTS`). It returns `Ok(r)` where `r` is the lock-query result — which is already `false` (the early-return at line 225–227 would have fired otherwise). The type query result stored in `row_type` is **never consulted**. The function returns `false` unconditionally for any script that has no lock-script references, regardless of type-script references.

**SQLite path:** `row_lock.try_get::<bool, _>(0)` fails (SQLite returns `INTEGER`), so the `Err(_)` branch executes `row_type.get::<i64, _>(0) == 1`, which correctly reads `row_type`. SQLite is unaffected.

---

### Impact Explanation

During `rollback_block`, after the rolled-back `output` rows are deleted, `script_exists_in_output` is called for every `type_script_id` collected from those outputs: [3](#0-2) 

On PostgreSQL, the function always returns `false` for any script that has no remaining `lock_script_id` references. Such scripts are pushed into `script_id_list_to_remove` and deleted from the `script` table — even when other `output` rows still hold a `type_script_id` foreign key pointing to them.

The result is dangling `type_script_id` references in the `output` table. Any subsequent indexer query that JOINs `output` with `script` on `type_script_id` (e.g., `get_cells`, `get_transactions`) will silently return `NULL` for the type-script fields of those outputs, producing incorrect query results for all consumers of the rich-indexer API.

---

### Likelihood Explanation

- The bug fires on **every** PostgreSQL-backed `rollback_block` call where any rolled-back output had a type script that is still referenced by a surviving output.
- No attacker action is required; natural chain reorgs trigger it automatically.
- An unprivileged peer can also deliberately trigger reorgs by relaying valid competing-chain blocks over P2P, which is a standard, unauthenticated operation.
- PostgreSQL is an explicitly supported production backend (separate DDL files exist for it).

---

### Recommendation

Change line 252 from `row_lock` to `row_type`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // ← fix
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [2](#0-1) 

---

### Proof of Concept

1. Start a CKB node with the rich-indexer backed by PostgreSQL.
2. Submit a block containing two outputs: output A with a unique type script S (and any lock script L1), output B with the same type script S (and a different lock script L2).
3. Trigger a rollback of that block (e.g., by feeding a longer competing chain).
4. After rollback, query `SELECT id FROM script WHERE id = <S_id>` — the row for S is gone, even though no output referencing S remains (correct in this case).
5. Now repeat with a block containing output A (type script S, lock L1) and a **pre-existing** output C (type script S, lock L3) that was indexed in an earlier, non-rolled-back block.
6. Roll back only the block containing A.
7. Query `SELECT id FROM script WHERE id = <S_id>` — the row for S is **deleted**, even though output C still holds `type_script_id = S_id`.
8. Query `get_cells` filtering by script S — the type-script fields for output C are returned as `NULL`, demonstrating the inconsistency.

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
