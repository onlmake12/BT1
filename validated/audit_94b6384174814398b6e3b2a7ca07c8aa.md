Audit Report

## Title
Missing IBD State Guard in `GetBlocksProcess::execute()` Allows Peers to Bypass IBD Isolation - (File: `sync/src/synchronizer/get_blocks_process.rs`)

## Summary
The module-level documentation for the CKB synchronizer explicitly states that a node in IBD mode must respond with `InIBD` to both `GetHeaders` and `GetBlocks` requests. `GetHeadersProcess` correctly implements this guard, but `GetBlocksProcess::execute()` has no IBD check and proceeds directly to serving full block bodies to any requesting peer. Any unprivileged inbound peer can exploit this to force an IBD node to perform block lookups and serialization, wasting CPU and bandwidth during the critical sync phase.

## Finding Description
The module documentation at `sync/src/synchronizer/mod.rs` lines 1–4 states: *"When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` and `GetBlocks` requests."*

`GetHeadersProcess::execute()` (`sync/src/synchronizer/get_headers_process.rs`, lines 53–66) correctly calls `active_chain.is_initial_block_download()`, sends `InIBD`, and returns `Status::ignored()`.

`GetBlocksProcess::execute()` (`sync/src/synchronizer/get_blocks_process.rs`, lines 33–97) has no such check. It immediately reads `block_hashes`, acquires `active_chain`, iterates up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` hashes, calls `active_chain.get_block()` for each, serializes the result into a `SendBlock` message, and spawns an async task to send it — all without ever consulting the IBD state.

Both handlers are dispatched from the same `try_process` match arm (`mod.rs` lines 396–422), so the asymmetry is not a dispatch-level guard. The `Relayer` protocol gates all relay messages behind an IBD check, making `GetBlocksProcess` the sole handler in the sync protocol that ignores IBD state.

Exploit path:
1. Node A is in IBD mode (tip timestamp > `MAX_TIP_AGE` behind wall clock).
2. Attacker peer B establishes an inbound P2P connection (no privileges required).
3. B sends `SyncMessage::GetBlocks` with up to `MAX_HEADERS_LEN` (2000) valid block hashes.
4. `try_process` dispatches to `GetBlocksProcess::execute()`.
5. No IBD check occurs; the node iterates hashes, looks up blocks, serializes them, and sends `SendBlock` responses to B.
6. B repeats indefinitely; the node continues serving block data while it should be ignoring such requests.
7. Contrast: a `GetHeaders` message from B would immediately receive `InIBD` and return `Status::ignored()`.

## Impact Explanation
This is a **Low** severity finding matching the allowed impact: *"Any other important performance improvements for CKB."* The missing guard causes an IBD node to perform non-trivial block lookup and serialization work for arbitrary inbound peers, wasting CPU and outbound bandwidth during the phase when the node is already under maximum load from its own chain download. The work per request is bounded by `INIT_BLOCKS_IN_TRANSIT_PER_PEER`, so the node will not crash, but sustained repeated requests from one or more peers can meaningfully degrade IBD throughput. The finding also represents a direct violation of the documented and partially-implemented IBD isolation contract.

## Likelihood Explanation
Any peer that can establish a P2P connection to the node can trigger this. CKB nodes accept inbound connections during IBD. No keys, hashpower, or special privileges are required. Valid block hashes are public on-chain data. The attacker can repeat the attack indefinitely at low cost, as each `GetBlocks` message is cheap to construct and send.

## Recommendation
Add an IBD guard at the start of `GetBlocksProcess::execute()`, mirroring `GetHeadersProcess`:

```rust
pub fn execute(self) -> Status {
    let active_chain = self.synchronizer.shared.active_chain();

    if active_chain.is_initial_block_download() {
        // Optionally send InIBD response here, matching GetHeadersProcess behavior
        return Status::ignored();
    }

    let block_hashes = self.message.block_hashes();
    // ... rest of existing logic
}
```

This aligns the implementation with the documented contract in `sync/src/synchronizer/mod.rs` lines 1–4 and the existing pattern in `sync/src/synchronizer/get_headers_process.rs` lines 53–66.

## Proof of Concept
1. Start a CKB node whose tip timestamp is more than `MAX_TIP_AGE` behind the wall clock (IBD mode).
2. Connect an attacker peer via the Sync P2P protocol.
3. Send a `SyncMessage::GetBlocks` message containing up to 2000 known valid block hashes.
4. Observe that the node responds with `SendBlock` messages for each found block, rather than `InIBD`.
5. Repeat step 3 in a loop; observe sustained CPU and bandwidth consumption on the IBD node.
6. For contrast, send a `SyncMessage::GetHeaders` message and observe the node responds with `InIBD` and ignores the request immediately.
7. A unit test can be written analogously to existing `GetHeadersProcess` IBD tests: construct a `Synchronizer` with a mocked chain whose tip is behind `MAX_TIP_AGE`, call `GetBlocksProcess::new(...).execute()`, and assert the return value is `Status::ignored()` rather than `Status::ok()`.
