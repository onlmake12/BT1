Audit Report

## Title
Integer Division in `check_purge` Allows `Group::None` Saturation to Permanently Block Peer Discovery — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

When all 16 384 slots in `AddrManager` are occupied by addresses that resolve to `Group::None`, the group-based eviction path in `check_purge` computes `take(len / 2)` where `len = 1`, yielding `take(0)` due to integer truncation. No candidates are selected, and `Err(PeerStoreError::EvictionFailed)` is returned unconditionally. Every subsequent call to `add_addr` — including calls for legitimate IPv4/IPv6 peers — propagates this error, permanently blocking new peer discovery until the node is restarted.

## Finding Description

**Entry point:** `DiscoveryAddressManager::add_new_addrs` in `network/src/protocols/discovery/mod.rs` calls `peer_store.add_addr` for every address received from a remote peer. The only pre-filter is `is_valid_addr`, which explicitly returns `true` for addresses where `multiaddr_to_socketaddr` returns `None` (line 338: `None => true`), meaning bare `/p2p/<peer-id>` multiaddreses with no IP component pass through unfiltered. [1](#0-0) 

**`add_addr` propagates `check_purge` errors:** `add_addr` calls `self.check_purge()?` and returns the error directly to the caller. [2](#0-1) 

**Stage 1 eviction fails for fresh addresses:** `check_purge` first collects addresses where `!addr.is_connectable(now_ms)`. Addresses inserted via `add_addr` are created with `last_connected_at_ms = 0` and `attempts_count = 0`. Since `attempts_count (0) < ADDR_MAX_RETRIES (3)`, `is_connectable` returns `true` for all of them, so Stage 1 finds nothing to evict. [3](#0-2) 

**Stage 2 — integer division root cause:** When Stage 1 finds nothing, Stage 2 groups addresses by `Group`. Any multiaddr without a resolvable IP component (bare `/p2p/<peer-id>`, DNS names, Onion3, etc.) maps to `Group::None`. [4](#0-3) 

When all 16 384 stored addresses are `Group::None`, `peers_by_network_group` has exactly **one** key. The critical line:

```rust
let len = peers_by_network_group.len();  // = 1
peers.into_iter().take(len / 2)          // take(1/2) = take(0)
``` [5](#0-4) 

`take(0)` yields an empty iterator. `candidate_peers` is empty, and the function returns `Err(PeerStoreError::EvictionFailed)`. [6](#0-5) 

Note: even if `take(len/2)` were fixed to `take(1)`, the single `Group::None` group would contain all 16 384 entries (well above the `addrs.len() > 4` threshold), so 2 would be evicted — confirming the integer division is the sole root cause.

**Capacity constant:** [7](#0-6) 

## Impact Explanation

Once the peer store is saturated with `Group::None` entries, `add_addr` returns `EvictionFailed` for every subsequent call, including calls for valid IPv4/IPv6 peers received via discovery, identify, or DNS seeding. The node can no longer learn about new peers. Existing connections are unaffected, but peer rotation and recovery from disconnections are broken. If all existing connections drop, the node becomes fully isolated from the network with no ability to reconnect without a manual restart. This matches **High: Vulnerabilities which could easily crash a CKB node** (network isolation = effective node failure).

## Likelihood Explanation

The discovery protocol accepts peer-advertised addresses with no IP-format pre-filter — `is_valid_addr` explicitly passes through addresses with no socket representation. An attacker operating a single persistent node can advertise batches of unique `/p2p/<random-peer-id>` multiaddreses across multiple discovery rounds. Filling 16 384 slots requires sending that many unique addresses, achievable over time without any special privilege, majority hashpower, or leaked key. The attack is repeatable after each node restart.

## Recommendation

1. **Fix the integer division:** Change `take(len / 2)` to `take(len.saturating_add(1) / 2)` (ceiling division) so that a single-group store still evicts from that group.
2. **Enforce a per-group cap:** Reject or evict addresses that would push any single `Group` (including `Group::None`) beyond a configurable threshold (e.g., 256 entries).
3. **Pre-filter non-socket-parseable addresses:** In `add_new_addrs`, reject addresses that produce `Group::None` unless they are explicitly supported types (e.g., Onion3), preventing the store from being saturated with unroutable entries. The current `is_valid_addr` returning `true` for `None` socket addresses is the permissive gate that enables this attack.

## Proof of Concept

```rust
// Fill AddrManager with 16384 unique bare /p2p/<id> addresses (Group::None)
let mut peer_store = PeerStore::default();
for i in 0u64..16384 {
    let mut addr = Multiaddr::empty();
    addr.push(Protocol::P2P(fake_peer_id(i).into()));
    // first 16383 succeed; 16384th triggers check_purge
    let _ = peer_store.add_addr(addr, Flags::COMPATIBILITY);
}
// Now attempt to add a legitimate IPv4 peer
let good_addr: Multiaddr = "/ip4/1.2.3.4/tcp/8115".parse().unwrap();
let result = peer_store.add_addr(good_addr, Flags::COMPATIBILITY);
// Returns Err(EvictionFailed) — legitimate peer permanently rejected
assert!(result.is_err());
```

The `addrs.len() > 4` guard inside Stage 2's `flat_map` does not protect against this: with all 16 384 entries in one group, that group's length is 16 384 >> 4, but `take(0)` prevents the group from ever being reached.
