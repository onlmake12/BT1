Audit Report

## Title
Off-by-One in Decompressed-Length Guard Allows 8 MB Heap Allocation Per P2P Message — (File: network/src/compress.rs)

## Summary
`MAX_UNCOMPRESSED_LEN` is defined as `1 << 23 = 8,388,608`. Both guard checks use strict `>` instead of `>=`, so a payload whose `decompress_len()` equals exactly 8,388,608 bypasses the guard and triggers an 8 MB heap allocation before any application-layer validation. Any unauthenticated peer can trigger this repeatedly, producing unbounded heap pressure proportional to the number of connected peers.

## Finding Description
`MAX_UNCOMPRESSED_LEN = 1 << 23 = 8,388,608` is defined at line 13. [1](#0-0) 

In `LengthDelimitedCodecWithCompress::decode`, line 235 uses `if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN`. When `decompressed_bytes_len == 8,388,608`, the condition `8388608 > 8388608` evaluates to `false`, the guard is bypassed, and `BytesMut::zeroed(8388608)` is called at line 242. [2](#0-1) 

The identical flaw exists in `Message::decompress` at line 74 (`>` guard), with `vec![0; decompressed_bytes_len]` allocated at line 81. [3](#0-2) 

The decoder at line 232 checks `(data[0] & COMPRESS_FLAG) != 0` with no check on `self.enable_compress`, meaning any peer can send a compressed frame on any protocol channel regardless of whether compression is enabled locally. [4](#0-3) 

The frame-size limit does not prevent the attack: snappy compresses 8 MB of zero bytes to a few hundred bytes, well within any protocol's `max_frame_length`. The compressed payload passes the frame-length check, `decompress_len()` returns exactly 8,388,608, the guard `8388608 > 8388608` is false, and 8 MB is allocated.

## Impact Explanation
**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.** Each connected peer can force an 8 MB heap allocation per message at negligible cost. With the default peer limit of 125, 125 simultaneous peers each sending one such message produces ~1 GB of concurrent heap pressure. At the service-level ceiling of 1024 connections, the upper bound is ~8 GB. This causes OOM kills or severe allocator pressure, crashing or stalling the node and preventing it from participating in block sync or relay.

## Likelihood Explanation
The attack requires only a TCP connection to the node's P2P port — no authentication, no stake, no proof-of-work. The crafted payload is trivial to produce (snappy-compress exactly 8,388,608 bytes of zeros). The off-by-one is a single-character difference (`>` vs `>=`) present in both code paths. The connection is not dropped after the allocation if the decompressed content is valid snappy data, so the attacker can send messages continuously.

## Recommendation
Change both guards from strict `>` to `>=`:

```rust
// network/src/compress.rs, line 74
if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN {

// network/src/compress.rs, line 235
if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN {
```

Additionally: (1) check `self.enable_compress` in the decoder before processing compressed frames; (2) add per-peer rate limiting on decompression; (3) add a node-wide cap on total concurrent decompression memory.

## Proof of Concept
```python
import snappy, struct, socket

# Craft payload: exactly MAX_UNCOMPRESSED_LEN bytes, snappy-compressed
raw = b'\x00' * 8388608           # exactly 1 << 23
compressed = snappy.compress(raw)  # compresses to ~few hundred bytes

# Build frame: [4-byte big-endian length][0x80 compress flag][compressed payload]
payload = b'\x80' + compressed
frame = struct.pack('>I', len(payload)) + payload

# Send on any CKB P2P protocol channel (e.g., port 8115)
s = socket.create_connection(('TARGET_IP', 8115))
# ... complete tentacle/secio handshake, then send frame ...
s.sendall(frame)
# Node executes BytesMut::zeroed(8388608) — 8 MB allocated before any app logic
```

Sending from 125 concurrent connections causes ~1 GB of simultaneous heap pressure. The guard `8388608 > 8388608` evaluates to `false`, confirming the off-by-one is the direct root cause. A unit test can confirm this by constructing a snappy-compressed buffer whose `decompress_len()` returns exactly `MAX_UNCOMPRESSED_LEN` and asserting that the current code allocates rather than rejects it.
