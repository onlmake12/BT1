### Title
`ScriptHashTypeVerifier` Does Not Check Output Type Script Hash Type Against Enabled Set — (`File: verification/src/transaction_verifier.rs`)

### Summary

`ScriptHashTypeVerifier::verify()` is the non-contextual guard that enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist on transaction outputs. However, it only inspects each output's **lock** script hash type and silently ignores the **type** script hash type. An unprivileged RPC caller can craft a transaction whose type script carries a non-enabled `ScriptHashType` (e.g., `Data3` = 6), bypass this early-rejection gate, and have the transaction admitted to the tx-pool. The transaction is only rejected later during expensive script execution, wasting node resources and enabling tx-pool pollution.

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` defines the currently permitted hash-type values (`Data`=0, `Type`=1, `Data1`=2, `Data2`=4). [1](#0-0) 

`ScriptHashTypeVerifier::verify()` iterates over every output and validates only `output.lock().hash_type()` against this set. The type script (`output.type_()`) is never consulted:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { ... }
        } else { ... }
    }
    Ok(())
}
``` [2](#0-1) 

The `ScriptHashType` enum is defined with a `seq!` macro generating `Data3`…`Data127` (values 6, 8, …, 254). All even values pass `ScriptHashType::verify_value()` (the structural check used by `check_data`), so a type script with `hash_type = 6` (`Data3`) is structurally valid but not consensus-enabled. [3](#0-2) 

The `check_data` path used for P2P message validation only checks structural validity (even or 1), not the enabled set: [4](#0-3) 

`NonContextualTransactionVerifier` — which includes `ScriptHashTypeVerifier` — is the gate used before script execution: [5](#0-4) 

When a miner later attempts to execute the type script, `select_version()` returns `ScriptError::InvalidScriptHashType` for any non-enabled hash type, rejecting the transaction — but only after expensive VM setup: [6](#0-5) 

**Exploit path:**
1. Attacker constructs a transaction with a valid lock script (`Data`/`Type`) and a type script with `hash_type = 6` (`Data3`).
2. Submits via `send_transaction` RPC.
3. `check_data` passes (6 is even → structurally valid).
4. `ScriptHashTypeVerifier` passes (only checks lock script).
5. Transaction enters the tx-pool.
6. Miner picks it up; script execution fails in `select_version` → block rejected or transaction evicted after wasted work.
7. Attacker repeats at scale to exhaust tx-pool slots and miner CPU.

### Impact Explanation

- **Tx-pool pollution**: Non-enabled type script hash types bypass the intended early-rejection gate and occupy tx-pool memory.
- **Resource exhaustion**: Nodes and miners expend CPU on VM initialization and script execution for transactions that should have been rejected non-contextually.
- **Legitimate transaction displacement**: Tx-pool slot limits mean attacker-crafted transactions can crowd out valid ones.
- No consensus split or asset theft: all nodes ultimately reject such transactions during script execution.

### Likelihood Explanation

Any unprivileged actor with RPC access (the default for public nodes) can submit transactions. No keys, no stake, no special role required. The attack is cheap to mount at scale and requires only knowledge of the `ScriptHashType` encoding scheme, which is public.

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type for each output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        self.check_hash_type(output.lock().hash_type())?;
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            self.check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

This mirrors the completeness of `CellOutputReader::check_data`, which already validates both lock and type scripts for structural validity. [7](#0-6) 

### Proof of Concept

```rust
// Construct a transaction with a valid lock but non-enabled type script hash type
let tx = TransactionBuilder::default()
    .output(
        CellOutput::new_builder()
            .lock(Script::default().as_builder()
                .hash_type(ScriptHashType::Data)  // valid, enabled
                .build())
            .type_(Some(Script::default().as_builder()
                .hash_type(ScriptHashType::Data3)  // value=6, NOT in ENABLED_SCRIPT_HASH_TYPE
                .build()))
            .build(),
    )
    .build();

let verifier = ScriptHashTypeVerifier::new(&tx);
// This passes — the type script hash type is never checked
assert!(verifier.verify().is_ok());
// But select_version() would return Err(InvalidScriptHashType) during execution
```

The existing test `test_not_enabled_hash_type_output_lock` only covers the lock script case and has no counterpart for the type script, confirming the gap. [8](#0-7)

seq!(N in 3..=127 {
    /// Specifies how the script `code_hash` is used to match the script code and how to run the code.
    /// The hash type is split into the high 7 bits and the low 1 bit,
    /// when the low 1 bit is 1, it indicates the type,
    /// when the low 1 bit is 0, it indicates the data,
    /// and then it relies on the high 7 bits to indicate
    /// that the data actually corresponds to the version.
     #[derive(Default, Clone, Copy, PartialEq, Eq, Debug, Hash, FromRepr)]
     #[repr(u8)]
    pub enum ScriptHashType {
        /// Type "type" matches script code via cell type script hash.
        Type = 1,
        /// Type "data" matches script code via cell data hash, and run the script code in v0 CKB VM.
        #[default]
        Data = 0,
        /// Type "data1" matches script code via cell data hash, and run the script code in v1 CKB VM.
        Data1 = 2,
        /// Type "data2" matches script code via cell data hash, and run the script code in v2 CKB VM.
        Data2 = 4,
        #(
            #[doc = concat!("Type \"data", stringify!(N), "\" matches script code via cell data hash, and runs the script code in v", stringify!(N), " CKB VM.")]
            Data~N = N << 1,
        )*
    }
});

impl ScriptHashType {
    /// when the low 1 bit is 1, it indicates the type
    /// when the low 1 bit is 0, it indicates the data
    #[inline]
    pub fn verify_value(v: u8) -> bool {
        v.is_multiple_of(2) || v == 1
    }
