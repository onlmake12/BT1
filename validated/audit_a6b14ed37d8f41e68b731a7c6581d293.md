### Title
`MAX_PRICE_IDS_PER_SUBSCRIPTION = 255` Causes DOS via Gas Exhaustion on Low-Gas-Limit EVM Chains — (`File: target_chains/ethereum/pulse_sdk/solidity/SchedulerConstants.sol`)

---

### Summary

The `Scheduler` contract enforces a hardcoded `MAX_PRICE_IDS_PER_SUBSCRIPTION = 255`. When a subscription is created at this maximum, the `updatePriceFeeds` function must write 255 full `PriceFeed` structs (≈5 storage slots each) to cold storage, consuming approximately **25.5 million gas** — exceeding the block gas limits of EVM-compatible chains such as Scroll (10M) and Gnosis (17M). This permanently locks the subscription's funds.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint8 public constant MAX_PRICE_IDS_PER_SUBSCRIPTION = 255;
``` [1](#0-0) 

Any unprivileged user can call `createSubscription` with exactly 255 price IDs, which passes validation:

```solidity
if (params.priceIds.length > SchedulerConstants.MAX_PRICE_IDS_PER_SUBSCRIPTION) {
    revert SchedulerErrors.TooManyPriceIds(...);
}
``` [2](#0-1) 

When a keeper subsequently calls `updatePriceFeeds`, the function calls `_storePriceUpdates`, which writes every `PriceFeed` struct to a nested mapping in storage:

```solidity
function _storePriceUpdates(...) internal {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        _state.priceUpdates[subscriptionId][priceFeeds[i].id] = priceFeeds[i];
    }
}
``` [3](#0-2) 

`PythStructs.PriceFeed` contains `bytes32 id`, `Price price` (2 slots: packed int64/uint64/int32 + uint256 publishTime), and `Price emaPrice` (2 slots) — approximately **5 storage slots per feed**. Writing 255 feeds to cold storage costs:

> 255 feeds × 5 slots × 20,000 gas (cold SSTORE) = **25,500,000 gas**

This is before accounting for the gas consumed by `parsePriceFeedUpdatesWithConfig` (Merkle proof verification for 255 feeds), the slot-consistency loop, `_validateShouldUpdatePrices`, and `_processFeesAndPayKeeper`. [4](#0-3) 

---

### Impact Explanation

On EVM-compatible chains with block gas limits below ~26M (e.g., Scroll at 10M, Gnosis at 17M), `updatePriceFeeds` for a 255-feed subscription will **always revert** due to out-of-gas. The subscription's ETH balance is locked inside the contract because:

1. `withdrawFunds` requires the subscription to be inactive.
2. Deactivating via `updateSubscription` still calls `_validateSubscriptionParams`, which runs an O(n²) uniqueness loop over 255 price IDs (32,385 iterations) — itself a significant gas cost.
3. The subscription can never be successfully updated, so `priceLastUpdatedAt` never advances and the subscription is permanently stuck.

This matches the severity of the VotingEscrow analog: a user's funds are locked with no recovery path, caused solely by the hardcoded limit being too high for the target chain's block gas limit.

---

### Likelihood Explanation

The Scheduler (Pulse) is an EVM contract intended for deployment across multiple EVM-compatible chains. Any user who creates a subscription with 255 price IDs — a value explicitly permitted by the contract — will trigger this condition on low-gas chains. No special privileges, leaked keys, or external oracle manipulation are required. The attacker-controlled entry point is the public `createSubscription` function. [5](#0-4) 

---

### Recommendation

Reduce `MAX_PRICE_IDS_PER_SUBSCRIPTION` to a value whose worst-case `updatePriceFeeds` gas cost fits within the lowest target chain's block gas limit. Based on the gas benchmark test infrastructure already present in the repo, the team should run `PulseSchedulerGasBenchmark.t.sol` at the maximum feed count and validate against each deployment chain's block gas limit. [6](#0-5) 

A value of 10–20 price IDs per subscription would keep worst-case gas well within Scroll's 10M limit. Alternatively, enforce a per-chain configurable maximum set at deployment time rather than a single hardcoded constant.

---

    function createSubscription(
        SchedulerStructs.SubscriptionParams memory subscriptionParams
    ) external payable override returns (uint256 subscriptionId) {
        _validateSubscriptionParams(subscriptionParams);

        // Calculate minimum balance required for this subscription
        uint256 minimumBalance = this.getMinimumBalance(
            uint8(subscriptionParams.priceIds.length)
        );

        // Ensure enough funds were provided
        if (msg.value < minimumBalance) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Check deposit limit for permanent subscriptions
        if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        // Set subscription to active
        subscriptionParams.isActive = true;

        subscriptionId = _state.subscriptionNumber++;

        // Store the subscription parameters
        _state.subscriptionParams[subscriptionId] = subscriptionParams;

        // Initialize subscription status
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        status.priceLastUpdatedAt = 0;
        status.balanceInWei = msg.value;
        status.totalUpdates = 0;
        status.totalSpent = 0;

        // Map subscription ID to manager
        _state.subscriptionManager[subscriptionId] = msg.sender;

        _addToActiveSubscriptions(subscriptionId);

        emit SubscriptionCreated(subscriptionId, msg.sender);
        return subscriptionId;
    }
