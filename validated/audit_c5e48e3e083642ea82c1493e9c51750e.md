### Title
Excess Fee Paid by User Is Silently Credited to Provider Instead of Being Refunded — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.requestPriceUpdatesWithCallback`, when a caller sends `msg.value` greater than `requiredFee`, the entire surplus above `pythFeeInWei` is stored as `req.fee` and later credited to the provider's `accruedFeesInWei`. The user's excess ETH is permanently transferred to the provider rather than being refunded to the caller. This is the same receiver-mismatch class as the reference report: unspent funds flow to the wrong party.

### Finding Description
`Echo.requestPriceUpdatesWithCallback` computes the minimum required fee and enforces a floor check, but then stores the full `msg.value - pythFeeInWei` as the provider fee with no ceiling:

```solidity
// Echo.sol L75-84
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
...
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);   // <-- no upper bound
...
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [1](#0-0) 

Later, in `executeCallback`, the stored `req.fee` is credited directly to the provider:

```solidity
// Echo.sol L161-162
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The provider can then withdraw the excess via `withdrawAsFeeManager`. There is no refund path back to the original requester.

By contrast, `PythLazer.verifyUpdate` correctly refunds excess to `msg.sender`:

```solidity
// PythLazer.sol L75-77
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [3](#0-2) 

The `Entropy` contract explicitly documents that excess is not refunded ("Note that excess value is *not* refunded to the caller"), but `Echo` carries no such documentation and the excess does not even go to Pyth's treasury — it goes to the provider, which is a distinct and unintended recipient. [4](#0-3) 

### Impact Explanation
Any user who sends `msg.value > getFee(provider, callbackGasLimit, priceIds)` permanently loses the surplus ETH to the provider. The provider can call `withdrawAsFeeManager` to extract it. For integrators who add a small buffer to their fee estimate (a common defensive pattern), every single request leaks ETH to the provider. At scale this represents a material, unrecoverable financial loss for users.

### Likelihood Explanation
Callers routinely overpay by a small margin to guard against fee changes between the `getFee` call and the transaction landing on-chain. The `IEcho` interface provides no warning that excess is not refunded (unlike `IEntropy`). Any integrator following the standard pattern of `{value: estimatedFee + buffer}` will silently lose the buffer on every request.

### Recommendation
Cap the provider fee at the computed `requiredFee - pythFeeInWei` and refund any surplus to `msg.sender`:

```solidity
uint96 providerFee = requiredFee - _state.pythFeeInWei;
req.fee = providerFee;
_state.accruedFeesInWei += _state.pythFeeInWei;

uint256 excess = msg.value - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

### Proof of Concept
1. Deploy `Echo` with `pythFeeInWei = 100 wei` and a provider whose `getFee` returns `1000 wei` total.
2. Call `requestPriceUpdatesWithCallback{value: 2000 wei}(...)` — overpaying by `1000 wei`.
3. Observe `req.fee = 2000 - 100 = 1900 wei` stored (provider's required share is only `900 wei`).
4. Provider calls `executeCallback`; `accruedFeesInWei` for the provider increases by `1900 wei`.
5. Provider calls `withdrawAsFeeManager` and receives the extra `1000 wei` that belonged to the user.
6. The user has no mechanism to recover the overpayment. [5](#0-4)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
