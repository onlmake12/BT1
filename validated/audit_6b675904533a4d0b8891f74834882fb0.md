### Title
Unprotected `init_if_needed` on `parameters` PDA Allows Any Unprivileged Caller to Seize Authority Over OIS Staking Cap Parameters — (File: `pythnet/stake_caps_parameters/programs/stake_caps_parameters/src/lib.rs`)

---

### Summary

The `set_parameters` instruction in the `stake_caps_parameters` program uses `init_if_needed` to create the singleton `parameters` PDA. The authority guard inside the handler permits any signer to write when `current_authority == Pubkey::default()`. Because Anchor zero-initialises a freshly created account, the very first call to `set_parameters` — by **any** transaction sender — satisfies that condition, letting an unprivileged actor claim permanent authority over the staking-cap parameters `m` and `z` before the legitimate operator can.

---

### Finding Description

`SetParameters` declares the `parameters` account with `init_if_needed`:

```rust
#[account(
    init_if_needed,
    seeds = ["parameters".as_bytes()],
    bump,
    payer = signer,
    space = Parameters::LEN
)]
pub parameters: Account<'info, Parameters>,
``` [1](#0-0) 

The handler then checks authority as follows:

```rust
require!(
    ctx.accounts.signer.key() == stored_parameters.current_authority
        || stored_parameters.current_authority == Pubkey::default(),
    ErrorCode::WrongAuthority
);
**stored_parameters = parameters;
``` [2](#0-1) 

When `init_if_needed` allocates the account for the first time, Anchor zero-fills all fields, so `current_authority` is `Pubkey::default()`. The second branch of the `require!` is therefore unconditionally true on the very first call, regardless of who the signer is. The attacker can supply any `Parameters` struct — including setting `current_authority` to their own key — and the instruction succeeds. After that, the legitimate operator's key is locked out permanently unless the attacker cooperates.

The `init-if-needed` Anchor feature is explicitly opted into:

```toml
anchor-lang = {version ="0.30.1", features = ["init-if-needed"]}
``` [3](#0-2) 

---

### Impact Explanation

`m` (target stake per symbol) and `z` (low-publisher-count floor) are the two constants that determine every publisher's pool cap in the Oracle Integrity Staking formula:

$$C_p = M \cdot \sum_{s \in \text{Symbols}_p} \frac{1}{\max(n_s, Z)}$$

An attacker who wins the initialization race can:

- Set `m = 0` → all pool caps collapse to zero, making staking rewards impossible for every publisher.
- Set `z` to an extreme value → distort cap contributions across all symbols.
- Set `current_authority` to their own key → permanently lock out the legitimate Pyth operator from ever correcting the values.

This directly breaks the OIS reward and cap accounting for the entire protocol. [4](#0-3) 

---

### Likelihood Explanation

The window of vulnerability is the gap between program deployment and the first legitimate `set_parameters` call. Any actor who can submit a transaction to Pythnet during that window can exploit this. The CLI tool that the operator uses to call `set_parameters` is public, so the instruction interface is known. No special privilege, leaked key, or oracle collusion is required — only the ability to send a transaction first. [5](#0-4) 

---

### Recommendation

Replace `init_if_needed` with `init` so the account can only be created once and only by the intended deployer:

```diff
 #[account(
-    init_if_needed,
+    init,
     seeds = ["parameters".as_bytes()],
     bump,
     payer = signer,
     space = Parameters::LEN
 )]
 pub parameters: Account<'info, Parameters>,
```

If re-initialisation after a close is ever needed, add an explicit deployer/multisig constraint rather than relying on the zero-default of `current_authority`. [6](#0-5) 

---

### Proof of Concept

1. The `stake_caps_parameters` program is deployed; the `parameters` PDA does not yet exist.
2. An attacker submits `set_parameters` with:
   ```rust
   Parameters {
       current_authority: attacker_pubkey,
       m: 0,   // collapses all pool caps
       z: 0,
   }
   ```
3. `init_if_needed` creates the account; `current_authority` is `Pubkey::default()` at the moment of the authority check, so the `require!` passes.
4. The account is written with `current_authority = attacker_pubkey`, `m = 0`, `z = 0`.
5. Every subsequent call by the legitimate operator fails with `WrongAuthority`.
6. All OIS pool caps become zero; staking rewards are disrupted protocol-wide. [4](#0-3) [1](#0-0)
