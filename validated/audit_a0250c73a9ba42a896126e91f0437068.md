### Title
`get_twap_update_fee()` Always Returns Zero, Enabling Fee-Free TWAP Calls and Breaking TWAP Functionality — (File: `target_chains/stylus/contracts/pyth-receiver/src/lib.rs`)

---

### Summary

The Stylus (Arbitrum) Pyth receiver contract contains two stub functions — `get_twap_update_fee` and `parse_twap_price_feed_updates` — that always return hardcoded zero/empty values. `get_twap_update_fee` unconditionally returns `U256::from(0u8)` regardless of the supplied update data, and `parse_twap_price_feed_updates` returns `Vec::new()` without performing any validation, fee check, or data processing. Any unprivileged caller invoking these functions receives incorrect output: zero fee and empty TWAP data.

---

### Finding Description

In `target_chains/stylus/contracts/pyth-receiver/src/lib.rs`, two public functions are implemented as stubs:

**`get_twap_update_fee` (line 320–322):**
```rust
pub fn get_twap_update_fee(&self, _update_data: Vec<Vec<u8>>) -> U256 {
    U256::from(0u8)
}
```
The parameter is prefixed with `_`, signaling it is intentionally unused. The function always returns `0` regardless of how many price feeds are in the update data. The correct implementation — as seen in the EVM contract (`Pyth.sol`, line 123–152) — parses the accumulator header, counts the number of updates, and returns `getTotalFee(totalNumUpdates)`.

**`parse_twap_price_feed_updates` (line 513–519):**
```rust
pub fn parse_twap_price_feed_updates(
    &mut self,
    _update_data: Vec<Vec<u8>>,
    _price_ids: Vec<[u8; 32]>,
) -> Vec<PriceFeedReturn> {
    Vec::new()
}
```
Both parameters are unused. No fee is collected, no VAA is verified, no Merkle proof is checked, and no TWAP data is returned. The EVM counterpart (`Pyth.sol`, line 491–506) performs a full fee check (`if (msg.value < requiredFee) revert PythErrors.InsufficientFee()`), VAA parsing, and TWAP computation.

The root cause is identical to the reported bug class: a function that should compute and return a meaningful value instead returns a hardcoded zero/empty result, causing downstream callers to receive incorrect data.

---

### Impact Explanation

1. **Fee bypass:** Any caller invoking `get_twap_update_fee` is told the required fee is `0`. Since `parse_twap_price_feed_updates` performs no fee check, TWAP update calls can be submitted with zero ETH attached, bypassing the protocol's fee collection mechanism entirely for the Stylus deployment.

2. **TWAP data always empty:** `parse_twap_price_feed_updates` returns `Vec::new()` unconditionally. Any on-chain consumer or integration relying on TWAP price data from the Stylus Pyth receiver will always receive an empty result, silently breaking any downstream logic that depends on TWAP feeds (e.g., TWAP-based pricing, TWAP-based liquidation thresholds).

3. **No revert, silent failure:** Unlike the EVM contract which reverts on `InsufficientFee`, the Stylus contract silently accepts zero-fee calls and returns empty data, making the failure invisible to callers who do not explicitly check for empty results.

---

### Likelihood Explanation

Any unprivileged user or contract can call `parse_twap_price_feed_updates` on the Stylus Pyth receiver. There is no access control. The entry path is direct: submit a transaction calling `parse_twap_price_feed_updates` with any `update_data` and `price_ids`. The function will always succeed (no revert), always charge zero fee, and always return empty data. Likelihood is **High** — the broken behavior is triggered on every invocation.

---

### Recommendation

Implement `get_twap_update_fee` to mirror the EVM logic: parse the accumulator header of `update_data[0]`, extract the number of updates via `parseWormholeMerkleHeaderNumUpdates`, and return `total_num_updates * single_update_fee_in_wei + transaction_fee_in_wei`.

Implement `parse_twap_price_feed_updates` to:
1. Require `update_data.len() == 2` (start and end points).
2. Compute and enforce the required fee via `get_twap_update_fee`.
3. Parse and verify both VAAs.
4. Extract and compute TWAP values from the start/end `TwapPriceInfo` messages.

Until these are implemented, the functions should revert with an explicit `Unimplemented` error rather than silently returning zero/empty values.

    fn get_update_fee(&self, update_data: Vec<Vec<u8>>) -> Result<U256, PythReceiverError> {
        let mut total_num_updates: u64 = 0;
        for data in &update_data {
            let update_data_array: &[u8] = &data;
            let accumulator_update = AccumulatorUpdateData::try_from_slice(&update_data_array)
                .map_err(|_| PythReceiverError::InvalidAccumulatorMessage)?;
            match accumulator_update.proof {
                Proof::WormholeMerkle { vaa: _, updates } => {
                    let num_updates = u64::try_from(updates.len())
                        .map_err(|_| PythReceiverError::TooManyUpdates)?;
                    total_num_updates += num_updates;
                }
            }
        }
        Ok(
            U256::from(total_num_updates).saturating_mul(self.single_update_fee_in_wei.get())
                + self.transaction_fee_in_wei.get(),
        )
    }

    function getTwapUpdateFee(
        bytes[] calldata updateData
    ) public view override returns (uint feeAmount) {
        uint totalNumUpdates = 0;
        // For TWAP updates, updateData is always length 2 (start and end points),
        // but each VAA can contain multiple price feeds. We only need to count
        // the number of updates in the first VAA since both VAAs will have the
        // same number of price feeds.
        if (
            updateData[0].length > 4 &&
            UnsafeCalldataBytesLib.toUint32(updateData[0], 0) ==
            ACCUMULATOR_MAGIC
        ) {
            (
                uint offset,
                UpdateType updateType
            ) = extractUpdateTypeFromAccumulatorHeader(updateData[0]);
            if (updateType != UpdateType.WormholeMerkle) {
                revert PythErrors.InvalidUpdateData();
            }
            totalNumUpdates += parseWormholeMerkleHeaderNumUpdates(
                updateData[0],
                offset
            );
        } else {
            revert PythErrors.InvalidUpdateData();
        }

        return getTotalFee(totalNumUpdates);
    }
