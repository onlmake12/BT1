### Title
Keygen Scripts Write Private Key Files With Overly Broad Permissions - (`apps/hip-3-pusher/src/scripts/create_key.py`, `apps/pyth-lazer-pusher/bulk-trade-cli/src/main.rs`)

---

### Summary

Two Pyth keygen utilities write private key material to disk using OS-default file creation modes, resulting in world-readable key files (typically `0644`, or `0666` with a permissive umask). Any local user on the same machine can read the generated private key and impersonate the Pyth oracle pusher or Lazer updater.

---

### Finding Description

**Instance 1 — `apps/hip-3-pusher/src/scripts/create_key.py`**

The `main()` function generates an Ethereum private key and writes it to disk using Python's built-in `open()` in write mode:

```python
with open(args.output_file, "w") as f:
    f.write(private_key_hex)
```

Python's `open()` does not accept a `mode` (permission) argument; it delegates entirely to the OS `umask`. With a typical umask of `0022` the resulting file is `0644` (world-readable). With a permissive umask of `0000` it is `0666`. No `os.chmod` call follows to tighten the permissions.

Notably, the project's own README already acknowledges the risk with the comment `# SECURITY: chmod 600 this file` next to the `oracle_pusher_key_path` config entry, but the keygen script itself never enforces this. [1](#0-0) [2](#0-1) 

**Instance 2 — `apps/pyth-lazer-pusher/bulk-trade-cli/src/main.rs`**

Both `run_keygen()` and `run_key_grind()` write an Ed25519 private key (base58-encoded) using `std::fs::write()`:

```rust
std::fs::write(&output_path, &private_key_base58)
    .with_context(|| format!("failed to write key to {output_path}"))?;
```

On Linux, `std::fs::write` internally calls `open(path, O_WRONLY|O_CREAT|O_TRUNC, 0o666)`. After a typical umask of `0022` the file lands at `0644`. No `std::fs::set_permissions` call follows. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

- **hip-3-pusher key**: controls signing of Hyperliquid oracle price updates submitted by the Pyth HIP-3 pusher. An attacker who reads the file can submit arbitrary price data to Hyperliquid under the legitimate pusher's identity, directly manipulating on-chain oracle prices.
- **Lazer bulk-trade-cli key**: controls the Ed25519 signing key used by the Pyth Lazer updater. Compromise allows an attacker to forge signed Lazer price updates.

Both impacts are high-severity: unauthorized price manipulation by a party that appears to be a legitimate Pyth data provider.

---

### Likelihood Explanation

Likelihood is medium. The attack requires local filesystem access to the machine where key generation is performed (e.g., a shared CI runner, a developer workstation with multiple accounts, or a misconfigured server). This is a realistic scenario for infrastructure that is not single-user. The README comment `# SECURITY: chmod 600 this file` shows the project is aware of the risk but relies on the operator to manually fix permissions after the fact — a step that is easily forgotten.

---

### Recommendation

**Short term:**

In `create_key.py`, replace the bare `open()` call with an explicit `os.open()` that sets mode `0o600` atomically:

```python
import os
fd = os.open(args.output_file, os.O_WRONLY | os.O_CREAT | os.O_TRUNC, 0o600)
with os.fdopen(fd, "w") as f:
    f.write(private_key_hex)
```

In `main.rs`, use `OpenOptions` with explicit Unix permissions before writing:

```rust
use std::os::unix::fs::OpenOptionsExt;
std::fs::OpenOptions::new()
    .write(true).create(true).truncate(true)
    .mode(0o600)
    .open(&output_path)?
    .write_all(private_key_base58.as_bytes())?;
```

**Long term:** Investigate offloading private key storage to a secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager) to avoid writing raw key material to the filesystem entirely.

### Proof of Concept

```bash
# On a multi-user Linux system, operator generates a hip-3-pusher key:
python create_key.py --output-file /home/operator/oracle-key.txt

# Default umask 0022 → file created as 0644
ls -la /home/operator/oracle-key.txt
# -rw-r--r-- 1 operator operator 66 Jun 17 12:00 oracle-key.txt

# Attacker (different local user) reads the private key:
cat /home/operator/oracle-key.txt
# 0xdeadbeef...  ← full Ethereum private key

# Attacker now signs and submits arbitrary Hyperliquid oracle price updates
# as the legitimate Pyth HIP-3 pusher.
```
