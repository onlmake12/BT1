Audit Report

## Title
Stale Descendant-Weight Accounting in `remove_entry_and_descendants` Leaves Ancestor Entries with Inflated Eviction Keys - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::remove_entry_and_descendants` erases all link records for every entry in the removal set before calling `remove_entry` on each one. Because `update_ancestors_index_key` relies on `self.links.calc_ancestors` to locate surviving ancestors and decrement their descendant-weight fields, and those link records are already gone by the time it runs, the decrement is silently skipped. Ancestor transactions that remain in the pool are left with permanently inflated `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee`, corrupting their `evict_key` and therefore the pool's eviction ordering.

## Finding Description
`remove_entry_and_descendants` (L252–265) first calls `remove_entry_links` for every ID in the removal set, then calls `remove_entry` for each:

```rust
for id in &removed_ids {
    self.remove_entry_links(id);   // ALL links erased first
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry_links` (L418–430) calls `self.links.remove(id)`, which deletes the entry's record from `TxLinksMap::inner`. When `remove_entry` subsequently calls `update_ancestors_index_key` (L432–445), that function calls `self.links.calc_ancestors(&child.proposal_short_id())`. `calc_ancestors` delegates to `calc_relative_ids` (links.rs L37–50), which does `self.inner.get(short_id)` — returning `None` because the record was already deleted — and falls back to `unwrap_or_default()`, yielding an empty set. The loop over ancestors never executes; `sub_descendant_weight` is never called on any surviving ancestor outside the removed set.

The comment in the code ("update links state for remove, so that we won't update_descendants_index_key in remove_entry") confirms the intent was only to suppress redundant `update_descendants_index_key` calls for entries that are themselves being removed. The unintended side effect is that `update_ancestors_index_key` is equally suppressed for surviving ancestors.

The `EvictKey` (entry.rs L234–247) is computed directly from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`. With those fields inflated, the surviving ancestor's `evict_key` is permanently wrong for the lifetime of that entry in the pool. `next_evict_entry` (L380–385) iterates by `evict_key` to select the eviction candidate; `limit_size` (pool.rs L298–326) calls it in a loop until the pool is within size limits.

## Impact Explanation
The concrete impact is incorrect eviction ordering: a low-fee ancestor transaction whose descendant was removed via conflict resolution retains an inflated apparent descendant fee-rate, causing it to survive eviction rounds it should lose. Legitimate high-fee transactions may be evicted in its place. This maps to **Low (501–2000 points): Any other important performance improvements for CKB**, as it degrades the correctness and fairness of the mempool's fee-based transaction selection without crashing the node or affecting consensus.

## Likelihood Explanation
`remove_entry_and_descendants` is called on every conflict-resolution event via `resolve_conflict` (L305–332) and on every size-limit eviction via `limit_size` (pool.rs L307). Any unprivileged peer or RPC caller can trigger the ancestor-corruption path with three standard `send_transaction` calls: submit parent Tx A, child Tx B spending A's output, then conflicting Tx C spending the same input as B. This causes `resolve_conflict` → `remove_entry_and_descendants(B)`, leaving Tx A with stale descendant weights. The operation is low-cost, repeatable, and requires no special privilege.

## Recommendation
Move the link-removal step to after the ancestor-weight update. The simplest correct fix is to call `remove_entry` for each ID before erasing links, and handle the redundant `update_descendants_index_key` concern by checking membership in the removal set inside that function rather than by pre-erasing all links:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Call remove_entry first so update_ancestors_index_key can still
    // traverse links to find surviving ancestors and decrement their weights.
    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

To avoid redundant `update_descendants_index_key` calls for entries that are also being removed, pass the removal set into `update_descendants_index_key` and skip entries whose IDs are in that set.

## Proof of Concept
1. Submit Tx A (parent, low fee-rate) via `send_transaction`.
2. Submit Tx B (child of A, any fee-rate) via `send_transaction`.
3. Submit Tx C (double-spend of B's input) via `send_transaction` — triggers `resolve_conflict` → `remove_entry_and_descendants(B)`.
4. Inspect Tx A's pool entry: `descendants_count` is still 1, `descendants_size`/`descendants_fee`/`descendants_cycles` still include B's contribution.
5. Fill the pool to capacity with higher-fee transactions. Observe that `limit_size` → `next_evict_entry` selects a legitimate high-fee transaction for eviction instead of Tx A, because Tx A's `evict_key` is computed from its inflated `descendants_fee` and appears higher than it should.
6. A unit test can assert this invariant: after step 3, `pool_map.get(&a_id).unwrap().descendants_count == 1` and `descendants_size == a_size` (not `a_size + b_size`).
