### Title
Excess ETH Paid by Entropy Users Is Permanently Captured by Pyth Treasury Instead of Being Refunded â€” (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `requestHelper` internal function silently absorbs any ETH sent above the required fee into Pyth's own treasury (`accruedPythFeesInWei`) rather than refunding it to the caller. This is the direct analog of the OmniXMultisender "optimistic peer assumption" bug: instead of reverting or returning funds when the caller sends more than expected, the contract routes the surplus to an unintended destination (Pyth's fee pool), permanently removing it from the user's control.

---

### Finding Description

`requestHelper` is the shared internal entry point for all Entropy request variants: `request()`, `requestWithCallback()`, and `requestV2()`. [1](#0-0) 

The fee accounting at lines 237â€“239 is:

```solidity
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

The `requiredFee` (line 234) equals `providerFee + pythFeeInWei`. If a user sends `msg.value = requiredFee + X` (any overpayment `X > 0`), the arithmetic distributes funds as:

- `providerFee` â†’ provider's accrued balance âś“
- `pythFeeInWei + X` â†’ Pyth's `accruedPythFeesInWei` âś— (user loses `X`)

The excess `X` is permanently captured by Pyth's treasury. It can only be recovered by the Pyth admin via `withdrawFee()` in `EntropyGovernance.sol`. There is no mechanism for the original payer to reclaim it.

This is explicitly documented as a design choice in the NatSpec of every public entry point: [2](#0-1) [3](#0-2) 

The existing test suite confirms the behavior â€” an overpayment of 10,000 wei is silently absorbed into Pyth fees, not returned: [4](#0-3) 

---

### Impact Explanation

Any ETH sent above `getFeeV2(provider, gasLimit)` is **permanently lost to the caller**. The funds are not locked â€” they are immediately credited to Pyth's treasury and are withdrawable by the Pyth admin. The user has no recourse.

This matches the OmniXMultisender impact class: user funds routed to an unintended destination (Pyth treasury instead of back to the user) with no recovery path for the original sender.

**Impact: HIGH** â€” direct, irreversible loss of user ETH.

---

### Likelihood Explanation

Several realistic, unprivileged paths trigger this:

1. **Fee race condition**: A provider calls `setProviderFee()` or the admin calls `setPythFee()` between the user's `getFeeV2()` query and their `request()` submission. The user's cached fee is now stale; they send the old (higher) amount and overpay.
2. **Frontend/SDK safety buffer**: Consumer contracts or frontends commonly add a small buffer (e.g., `fee * 110 / 100`) to avoid `InsufficientFee` reverts. Every such call silently donates the buffer to Pyth.
3. **Direct user overpayment**: A user manually submitting a transaction with a round-number ETH value (e.g., `0.001 ETH`) when the fee is `0.00087 ETH` loses the difference permanently.

**Likelihood: MEDIUM** â€” the fee race condition is a standard MEV/timing scenario; the buffer pattern is a common integration practice.

---

### Recommendation

Refund excess `msg.value` to `msg.sender` after computing the required fee:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();

uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee); // only pythFeeInWei

// Refund excess to caller
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

Alternatively, enforce exact payment: `if (msg.value != requiredFee) revert`.
