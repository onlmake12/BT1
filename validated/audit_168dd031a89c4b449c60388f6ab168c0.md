### Title
Cross-Function Reentrancy in `executeCallback` Allows Malicious Consumer to Bloat Request Storage — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol`'s `executeCallback` makes an external call to a user-controlled contract (`req.requester._echoCallback`) without any reentrancy guard. Neither `executeCallback` nor `requestPriceUpdatesWithCallback` carries a `nonReentrant` modifier. A malicious consumer contract can exploit this to call `requestPriceUpdatesWithCallback` from inside its `echoCallback`, creating new requests mid-execution. This is the direct structural analog of the VUSD cross-function reentrancy: one function makes an external call to a user-controlled address, and a separate state-mutating entry point lacks protection against being called during that external call.

### Finding Description

In `Echo.sol`, `executeCallback` follows this sequence:

1. Credits the provider's accrued fees (`_state.providers[providerToCredit].accruedFeesInWei += ...`)
2. Clears the request (`clearRequest(sequenceNumber)`)
3. Advances `firstUnfulfilledSeq` past the cleared request
4. **Makes an external call** to `IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` [1](#0-0) 

The external call at step 4 transfers control to the requester's contract. `requestPriceUpdatesWithCallback` has no reentrancy guard: [2](#0-1) 

A malicious consumer's `echoCallback` can therefore call `requestPriceUpdatesWithCallback` one or more times during the callback. Each reentrant call:

- Increments `_state.currentSequenceNumber`
- Calls `allocRequest`, which — if the new sequence number's `shortKey` collides with an existing legitimate request — **silently moves that legitimate request into the overflow mapping** (`_state.requestsOverflow`) [3](#0-2) 

The fixed-size array has only 32 slots (`NUM_REQUESTS = 32`). Requests displaced to the overflow mapping cost more gas for providers to look up and fulfill. [4](#0-3) 

Neither `Echo.sol` nor `EchoUpgradeable.sol` imports or uses `ReentrancyGuard` or `nonReentrant`: [5](#0-4) [6](#0-5) 

### Impact Explanation

A malicious consumer can, during its `echoCallback`, repeatedly call `requestPriceUpdatesWithCallback` to:

1. Fill all 32 fixed-size slots with attacker-controlled requests, forcing every subsequent legitimate request into the overflow mapping.
2. Cause `allocRequest` to silently displace legitimate in-flight requests from the fast-path array into the overflow mapping, increasing gas costs for providers fulfilling those requests.
3. Inflate `currentSequenceNumber` and the pending request set, increasing the cost of `getFirstActiveRequests` and `firstUnfulfilledSeq` advancement for all participants.

The attacker must pay the required fee for each reentrant request (so it is not zero-cost), but the `callbackGasLimit` is set by the requester — a high gas limit allows many reentrant calls in a single transaction. The provider receives the fees but bears the increased fulfillment gas cost.

### Likelihood Explanation

Any registered consumer contract can set an arbitrarily high `callbackGasLimit` when calling `requestPriceUpdatesWithCallback`. The callback is invoked with exactly that gas limit. No privileged role is required. The attack path is fully reachable by an unprivileged Entropy/Echo user.

### Recommendation

Add OpenZeppelin's `ReentrancyGuard` to `Echo` (or `EchoUpgradeable`) and apply the `nonReentrant` modifier to both `executeCallback` and `requestPriceUpdatesWithCallback`, mirroring the fix recommended for VUSD's `processWithdrawals`, `mintWithReserve`, and `withdraw`.

```solidity
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";

abstract contract Echo is IEcho, EchoState, ReentrancyGuard {
    function requestPriceUpdatesWithCallback(...) external payable nonReentrant override returns (...) { ... }
    function executeCallback(...) external payable nonReentrant override { ... }
}
```

### Proof of Concept

```solidity
contract MaliciousConsumer is IEchoConsumer {
    IEcho echo;
    address provider;
    bytes32[] priceIds;

    constructor(address _echo, address _provider, bytes32[] memory _priceIds) {
        echo = IEcho(_echo);
        provider = _provider;
        priceIds = _priceIds;
    }

    function getEcho() internal view override returns (address) { return address(echo); }

    function echoCallback(uint64, PythStructs.PriceFeed[] memory) internal override {
        // Reenter requestPriceUpdatesWithCallback during executeCallback
        // Each call fills a slot in the 32-entry fixed array, potentially
        // displacing legitimate requests to the overflow mapping.
        uint96 fee = echo.getFee(provider, 50_000, priceIds);
        for (uint i = 0; i < 5; i++) {
            echo.requestPriceUpdatesWithCallback{value: fee}(
                provider,
                uint64(block.timestamp),
                priceIds,
                50_000
            );
        }
    }
}
```

1. Deploy `MaliciousConsumer` with a high `callbackGasLimit` (e.g. 2M).
2. Call `requestPriceUpdatesWithCallback` from `MaliciousConsumer`.
3. When the provider calls `executeCallback`, `echoCallback` fires and creates 5 new requests inside the same transaction.
4. Legitimate requests submitted by other users that hash to the same `shortKey` as the new attacker requests are silently moved to `requestsOverflow`, increasing provider fulfillment gas.
