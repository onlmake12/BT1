### Title
`mark_as_known_txs` Updates Relay Filter Before Confirming Successful Submission — (`sync/src/relayer/transactions_process.rs`)

---

### Summary

In `TransactionsProcess::execute()`, the relay deduplication filter (`tx_filter`) is updated to mark incoming transactions as "known" **before** those transactions are actually submitted to the tx pool. If the subsequent `submit_remote_tx` call fails (e.g., due to a full or closed channel), the transaction hash remains permanently recorded as "known" in the filter with no cleanup path, causing the node to silently discard all future relay messages for that transaction from any peer until the TTL expires.

---

### Finding Description

In `sync/src/relayer/transactions_process.rs`, `TransactionsProcess::execute()` processes incoming `RelayTransactions` P2P messages. After filtering for unknown and requested transactions, it calls `mark_as_known_txs` at line 76 **before** spawning the async task that actually submits the transactions:

```rust
// Line 76 — state updated unconditionally
shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));

// Lines 80–93 — submission happens later, in a spawned task
self.relayer.shared.shared().async_handle().spawn(async move {
    for (tx, declared_cycles) in txs {
        if let Err(e) = tx_pool
            .submit_remote_tx(tx.clone(), declared_cycles, peer)
            .await
        {
            error!("submit_tx error {}", e);  // no filter cleanup here
        }
    }
});
```

`mark_as_known_txs` (in `sync/src/types/mod.rs`, lines 1443–1451) does two things atomically:
1. Removes the hash from `unknown_tx_hashes` (preventing future re-requests to peers)
2. Inserts the hash into `tx_filter` (causing future relay messages to be silently dropped at line 50)

`TxPoolController::submit_remote_tx` (in `tx-pool/src/service.rs`, lines 278–285) uses the `send_message!` macro, which calls `try_send` on a bounded channel. Under high load, `try_send` returns `TrySendError::Full` or `TrySendError::Closed`, causing `submit_remote_tx` to return `Err(AnyError)`. This error is caught at line 90 and only logged — no rollback of the filter state occurs.

The only cleanup path, `remove_from_known_txs`, is called in `send_bulk_of_tx_hashes` (in `sync/src/relayer/mod.rs`, line 674) only when a `TxVerificationResult::Reject` message is received through the relay result channel. This path is never triggered when `submit_remote_tx` fails with a channel error, because the tx never reaches the verification stage.

---

### Impact Explanation

A valid transaction relayed to the node that fails to be submitted due to a full tx-pool service channel will be permanently marked as "known" (until TTL expiry) without ever being processed. The node will:
- Refuse to accept the same transaction from any other relay peer (filtered at line 50)
- Never re-request the transaction from any peer (removed from `unknown_tx_hashes`)

This causes silent transaction propagation failure: the node believes it has processed the transaction but has not. Under sustained channel pressure, multiple valid transactions can be lost from the node's perspective for the duration of the TTL window, degrading network-wide transaction propagation.

---

### Likelihood Explanation

The `TxPoolController` uses a bounded `mpsc` channel. Under high transaction relay load — which is a normal condition on mainnet during fee spikes — the channel can fill up, causing `try_send` to fail with `TrySendError::Full`. This is an ordinary operational condition, not a theoretical edge case. Any unprivileged relay peer sending a `RelayTransactions` message during such a period can trigger this path.

---

### Recommendation

Move `mark_as_known_txs` to after successful submission, or add a cleanup call (`remove_from_known_txs`) in the error branch of the spawned task:

```rust
self.relayer.shared.shared().async_handle().spawn(async move {
    for (tx, declared_cycles) in txs {
        if let Err(e) = tx_pool
            .submit_remote_tx(tx.clone(), declared_cycles, peer)
            .await
        {
            error!("submit_tx error {}", e);
            shared_state_clone.remove_from_known_txs(&tx.hash()); // rollback
        }
    }
});
```

Alternatively, defer `mark_as_known_txs` to inside the spawned task, after confirming `submit_remote_tx` returns `Ok`.

### Proof of Concept

1. A relay peer sends a `RelayTransactions` P2P message containing a valid transaction `T`.
2. The node's `TransactionsProcess::execute()` processes the message, passes all filters, and calls `mark_as_known_txs([T.hash()])` at line 76 — `T.hash()` is now in `tx_filter`.
3. The spawned async task calls `tx_pool.submit_remote_tx(T, ...)`. The tx-pool service channel is full (`TrySendError::Full`). The call returns `Err(...)`. The error is logged only.
4. A second relay peer sends the same transaction `T`. In `TransactionsProcess::execute()`, line 50 filters it out: `tx_filter.contains(&T.hash())` is `true`. The transaction is silently dropped.

    pub fn mark_as_known_txs(&self, hashes: impl Iterator<Item = Byte32> + std::clone::Clone) {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();
        let mut tx_filter = self.tx_filter.lock();

        for hash in hashes {
            unknown_tx_hashes.remove(&hash);
            tx_filter.insert(hash);
        }
    }
