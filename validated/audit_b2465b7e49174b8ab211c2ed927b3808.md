Audit Report

## Title
Inconsistent Fee-Rate Formulas Between Admission Check and Pool Entry Allow `min_fee_rate` Bypass for Cycle-Heavy Transactions - (File: tx-pool/src/util.rs, tx-pool/src/component/entry.rs)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` gates admission using only serialized transaction size, while all pool-internal accounting uses `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. Because cycles are unknown at `pre_check` time and no second fee check is performed after `verify_rtx`, an attacker can craft a small-serialized, high-cycle transaction that passes the size-based gate with a fee far below what the weight-based formula requires, inserting transactions whose true fee rate is orders of magnitude below `min_fee_rate`.

## Finding Description
The pipeline in `_process_tx` (`tx-pool/src/process.rs` L705–777):

1. **L715**: `pre_check` → internally calls `check_tx_fee(tx_pool, snapshot, rtx, tx_size)` — fee gate fires here, cycles unknown.
2. **L724–732**: `verify_rtx(...)` — scripts execute, actual cycles determined.
3. **L751**: `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles.
4. **L753**: `submit_entry(...)` — entry inserted into pool.

`check_tx_fee` (`tx-pool/src/util.rs` L42–52) computes:
```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```
The code comment at L42–44 explicitly acknowledges: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check."* No second weight-based check is performed after `verify_rtx` returns actual cycles.

`TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs` L115–117) computes:
```rust
let weight = get_transaction_weight(self.size, self.cycles);
FeeRate::calculate(self.fee, weight)
```
`get_transaction_weight` (`util/types/src/core/tx_pool.rs` L298–303) returns `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For any transaction where `cycles × DEFAULT_BYTES_PER_CYCLES > tx_size`, the weight exceeds the size and the true fee rate is strictly lower than the size-based rate used at admission. The same weight formula is used in `AncestorsScoreSortKey` (entry.rs L223–224), `EvictKey` (entry.rs L236–238), and fee-estimator algorithms — but not at the admission gate.

## Impact Explanation
An attacker submits transactions with small serialized size (e.g., 200 bytes) and near-maximum cycles (70,000,000). With `min_fee_rate = 1,000 shannons/KW`:

- Admission requires: `1,000 × 200 / 1,000 = 200 shannons` (passes).
- True weight: `max(200, 70,000,000 × 0.000_170_571_4) = 11,940`.
- True fee rate: `200 × 1,000 / 11,940 ≈ 16.7 shannons/KW` — ~60× below `min_fee_rate`.

These transactions enter the pool, consume pool memory (up to `max_tx_pool_size`), are sorted to the bottom (never mined), and displace legitimate transactions. The attacker's cost per transaction is 200 shannons. This matches **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation
The attack is reachable by any unprivileged user via the `send_transaction` JSON-RPC endpoint or P2P relay. No special privileges or keys are required. A standard RISC-V script that loops to consume ~70M cycles is trivial to write. The bypass magnitude grows linearly with cycles, so the maximum-cycle case is the easiest to target and produces the worst discrepancy. The attacker can repeat submissions continuously to maintain pool pressure.

## Recommendation
Add a second fee check after `verify_rtx` returns the actual cycles, using the same weight formula as the rest of the pool:

```rust
// After verify_rtx, before TxEntry::new (tx-pool/src/process.rs ~L751):
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee_weight.as_u64(), fee.as_u64())), snapshot));
}
```

Alternatively, refactor `check_tx_fee` to accept an optional `cycles` parameter and use `get_transaction_weight` when cycles are available, eliminating the size-only path entirely.

## Proof of Concept
1. Write a CKB lock script that loops until it consumes ~70,000,000 cycles. Deploy it on testnet.
2. Construct a transaction referencing that script via `code_hash`, keeping serialized size ≤ 200 bytes.
3. Set `inputs_capacity - outputs_capacity = 200 shannons` (fee = 200 shannons).
4. Submit via `send_transaction` RPC to a node with default `min_fee_rate = 1,000`.
5. Observe acceptance: size-based check `200 ≥ 1,000 × 200 / 1,000 = 200` passes exactly.
6. Call `get_pool_tx_detail_info`; observe the entry's actual fee rate ≈ 16.7 shannons/KW, far below `min_fee_rate`.
7. Repeat in a loop to fill the pool with sub-minimum-fee-rate transactions, displacing legitimate traffic.
