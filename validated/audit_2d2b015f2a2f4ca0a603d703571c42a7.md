### Title
Peer Store Permanent Saturation via Off-by-One in `check_purge` Group Eviction Threshold — (`network/src/peer_store/peer_store_impl.rs`)

### Summary

`check_purge`'s network-group eviction uses a strictly-greater-than-4 threshold (`addrs.len() > 4`). An attacker who fills all 16 384 peer-store slots with exactly 4 addresses per /24 subnet triggers a state where neither eviction path removes any entry, causing `Err(EvictionFailed)` to propagate through `add_addr`. The discovery protocol silently drops the error, so every subsequent honest address is permanently rejected and the store stays saturated with attacker-controlled entries.

### Finding Description

**`ADDR_COUNT_LIMIT`** is 16 384. [1](#0-0) 

**`check_purge`** runs two eviction passes when the store is full:

*Pass 1* — remove all addresses where `is_connectable` returns `false`. [2](#0-1) 

*Pass 2* — only if pass 1 removed nothing, group by network segment, sort descending by group size, take the top half of groups, and for each group evict 2 addresses **only if `addrs.len() > 4`** (strictly greater than 4): [3](#0-2) 

If both passes produce an empty candidate list, the function returns `Err(PeerStoreError::EvictionFailed)`: [4](#0-3) 

**`add_addr`** propagates that error with `?`: [5](#0-4) 

**`is_connectable`** returns `true` for any freshly-added address because `attempts_count` starts at 0, which is below `ADDR_MAX_RETRIES = 3`: [6](#0-5) [7](#0-6) 

**Discovery protocol** silently swallows the error at `debug` level: [8](#0-7) 

### Impact Explanation

With the store permanently saturated:

1. No new honest peer address can be stored.
2. `fetch_addrs_to_feeler` returns only attacker addresses (all have `last_connected_at_ms = 0`, satisfying the "never connected" filter).
3. When the victim feeler-connects to an attacker node and the connection succeeds, `add_outbound_addr` / `update_outbound_addr_last_connected_ms` sets `last_connected_at_ms` to now, promoting those entries into `fetch_addrs_to_attempt` candidates.
4. After a node restart the victim makes all outbound connections to attacker nodes — full eclipse.

An eclipsed node can be fed a fake chain, have its transactions censored, or be used to double-spend against it, all of which damage CKB economy.

### Likelihood Explanation

The attacker needs only **one TCP connection** to the victim. A single discovery `Nodes` message can carry up to 1 000 addresses; 17 such messages suffice to fill 16 384 slots. The addresses do not need to be reachable or controlled — they only need to be public IPs from 4 096 distinct /24 subnets (trivially synthesised). There is no rate-limit on discovery address ingestion visible in the code. The attack is therefore low-cost and repeatable.

### Recommendation

Change the eviction threshold from strictly greater than 4 to greater than or equal to 4 (i.e., `>= 4` or equivalently `> 3`) in `check_purge`:

Additionally, consider adding a per-session rate limit on the number of discovery addresses accepted, and logging `EvictionFailed` at `warn` or `error` level rather than `debug`.

```rust
fn test_eclipse_via_exact_group_saturation() {
    use crate::peer_store::{ADDR_COUNT_LIMIT, PeerStore};
    use crate::Flags;
    use p2p::{multiaddr::Multiaddr, secio::PeerId};

    let mut peer_store = PeerStore::default();

    // Fill store: 4096 groups × 4 addresses = 16384 (ADDR_COUNT_LIMIT)
    let mut count = 0usize;
    'outer: for subnet in 0u32..=255 {
        for host in 1u32..=255 {
            for port in 1u16..=4 {
                if count >= ADDR_COUNT_LIMIT { break 'outer; }
                let addr: Multiaddr = format!(
                    "/ip4/10.{}.{}.1/tcp/{}/p2p/{}",
                    subnet, host, port,
                    PeerId::random().to_base58()
                ).parse().unwrap();
                peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
                count += 1;
            }
        }
    }
    assert_eq!(peer_store.addr_manager().count(), ADDR_COUNT_LIMIT);

    // Now try to add an honest address — must succeed but currently fails
    let honest: Multiaddr = format!(
        "/ip4/8.8.8.8/tcp/8333/p2p/{}", PeerId::random().to_base58()
    ).parse().unwrap();
    let result = peer_store.add_addr(honest.clone(), Flags::COMPATIBILITY);
    // This assertion FAILS on the current code, proving the bug:
    assert!(result.is_ok(), "honest address must always be accepted");
    assert!(peer_store.addr_manager().get(&honest).is_some());
}
```

Running this test against the current code will fail at the `assert!(result.is_ok())` line, confirming that `check_purge` returns `Err(EvictionFailed)` and the honest address is dropped.

```rust
    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr

            return false;
```
