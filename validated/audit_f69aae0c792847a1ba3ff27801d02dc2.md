### Title
Division-by-Zero in TWAP Calculation Due to Missing Strict Slot Ordering Check — (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

`parseTwapPriceFeedUpdates` in the EVM Pyth contract does not enforce that the start and end TWAP messages come from **strictly different** Pythnet slots. When `publishSlot_start == publishSlot_end`, the divisor `slotDiff` in `calculateTwap` is zero, causing an EVM division-by-zero panic revert. Any unprivileged caller can trigger this with a single valid Wormhole-signed TwapMessage VAA submitted as both start and end.

---

### Finding Description

`validateTwapPriceInfo` uses a non-strict inequality for the slot ordering check:

```solidity
// Pyth.sol line 604
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This allows `publishSlot_start == publishSlot_end` to pass validation. [1](#0-0) 

`calculateTwap` then computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // = 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));  // panic: division by zero
uint128 twapConf  = confDiff  / uint128(slotDiff);         // panic: division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panic: division by zero
``` [2](#0-1) 

The Solana implementation correctly uses a **strict** greater-than check:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The EVM version is missing this strict enforcement. [4](#0-3) 

---

### Impact Explanation

Any call to `parseTwapPriceFeedUpdates` with equal-slot start/end data will revert with an EVM panic (division by zero). Protocols that rely on this function in critical paths — such as liquidation engines, margin borrowing, or settlement — will be unable to obtain a TWAP price, effectively freezing those operations for the duration of the attack. The function is `payable` and externally callable by any address. [5](#0-4) 

---

### Likelihood Explanation

The attack requires only a single valid Wormhole-signed TwapMessage VAA, which is freely available from the public Hermes API. The attacker submits the same VAA as both `updateData[0]` and `updateData[1]`. No privileged access, flash loan, or key compromise is needed. The cost is only the TWAP update fee. [6](#0-5) 

---

### Recommendation

Change the slot ordering check in `validateTwapPriceInfo` from non-strict to strict, matching the Solana implementation:

```solidity
// Before (allows equal slots → division by zero):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After (strict, consistent with Solana):
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [7](#0-6) 

---

### Proof of Concept

1. Fetch any valid Wormhole-signed TwapMessage accumulator update from the Hermes API (e.g., `GET /v2/updates/twap/{feed_id}/latest`).
2. Call `parseTwapPriceFeedUpdates` on the deployed EVM Pyth contract with `updateData = [vaa, vaa]` (same VAA for both start and end) and the corresponding `priceIds`.
3. `validateTwapPriceInfo` passes: `prevPublishTime < publishTime` holds for both (same message), and `publishSlot_start == publishSlot_end` is not rejected.
4. `calculateTwap` executes `slotDiff = 0`, triggering an EVM division-by-zero panic.
5. The transaction reverts. Any protocol calling this function in a liquidation or settlement path is now blocked. [8](#0-7)

    function validateTwapPriceInfo(
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure {
        // First validate each individual price's uniqueness
        if (
            twapPriceInfoStart.prevPublishTime >= twapPriceInfoStart.publishTime
        ) {
            revert PythErrors.InvalidTwapUpdateData();
        }
        if (twapPriceInfoEnd.prevPublishTime >= twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateData();
        }

        // Then validate the relationship between the two data points
        if (twapPriceInfoStart.expo != twapPriceInfoEnd.expo) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishTime > twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
    }

    function calculateTwap(
        bytes32 priceId,
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure returns (PythStructs.TwapPriceFeed memory twapPriceFeed) {
        twapPriceFeed.id = priceId;
        twapPriceFeed.startTime = twapPriceInfoStart.publishTime;
        twapPriceFeed.endTime = twapPriceInfoEnd.publishTime;

        // Calculate differences between start and end points for slots and cumulative values
        uint64 slotDiff = twapPriceInfoEnd.publishSlot -
            twapPriceInfoStart.publishSlot;
        int128 priceDiff = twapPriceInfoEnd.cumulativePrice -
            twapPriceInfoStart.cumulativePrice;
        uint128 confDiff = twapPriceInfoEnd.cumulativeConf -
            twapPriceInfoStart.cumulativeConf;

        // Calculate time-weighted average price (TWAP) and confidence by dividing
        // the difference in cumulative values by the number of slots between data points
        int128 twapPrice = priceDiff / int128(uint128(slotDiff));
        uint128 twapConf = confDiff / uint128(slotDiff);

        // The conversion from int128 to int64 is safe because:
        // 1. Individual prices fit within int64 by protocol design
        // 2. TWAP is essentially an average price over time (cumulativePrice₂-cumulativePrice₁)/slotDiff
        // 3. This average must be within the range of individual prices that went into the calculation
        // We use int128 only as an intermediate type to safely handle cumulative sums
        twapPriceFeed.twap.price = int64(twapPrice);
        twapPriceFeed.twap.conf = uint64(twapConf);
        twapPriceFeed.twap.expo = twapPriceInfoStart.expo;
        twapPriceFeed.twap.publishTime = twapPriceInfoEnd.publishTime;

        // Calculate downSlotsRatio as a value between 0 and 1,000,000
        // 0 means no slots were missed, 1,000,000 means all slots were missed
        uint64 totalDownSlots = twapPriceInfoEnd.numDownSlots -
            twapPriceInfoStart.numDownSlots;
        uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff;
