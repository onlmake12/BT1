Audit Report

## Title
Stale Aggregate Stat Overwrite After In-Flight Eviction Inflates `total_tx_size`/`total_tx_cycles` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::add_entry` captures a pre-eviction snapshot of `total_tx_size`/`total_tx_cycles` at lines 210–211, then calls `check_and_record_ancestors` at line 213, which may evict pool entries and correctly decrement those fields via `update_stat_for_remove_tx`. The stale snapshot is then unconditionally written back at lines 218–219, silently reversing every decrement that occurred during eviction. This permanently inflates `total_tx_size` and `total_tx_cycles`, causing `limit_size` to over-evict legitimate pending transactions from the pool.

## Finding Description
In `add_entry` (lines 200–221), the new aggregate totals are computed as a local snapshot before any evictions occur:

```rust
// lines 210-211
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
```

`updated_stat_for_add_tx` (lines 711–729) simply returns `(self.total_tx_size + tx_size, self.total_tx_cycles + cycles)` — a point-in-time snapshot of the pre-eviction state plus the new entry.

Immediately after, `check_and_record_ancestors` (lines 588–640) is called. When the new transaction has too many cell-dep ancestors exceeding `max_ancestors_count`, it evicts existing pool entries via `remove_entry_and_descendants` → `update_stat_for_remove_tx` (lines 733–758), which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` for each evicted entry.

But then, at the end of `add_entry`, the stale pre-eviction snapshot is unconditionally written back:

```rust
// lines 218-219
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

This overwrites the correctly-decremented running totals with the pre-eviction snapshot, effectively undoing every `update_stat_for_remove_tx` call that occurred inside `check_and_record_ancestors`. After `add_entry` returns, `total_tx_size` and `total_tx_cycles` are inflated by the combined size and cycles of all evicted transactions.

The eviction path in `check_and_record_ancestors` (lines 603–625) is reached when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count` — meaning the condition is reachable and the eviction loop at line 616–625 calls `remove_entry_and_descendants` for each candidate.

## Impact Explanation
`limit_size` (pool.rs lines 292–329) enforces the pool size cap by comparing `total_tx_size` against `max_tx_pool_size`:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee-rate entries
}
```

With an inflated `total_tx_size`, the pool believes it is fuller than it actually is. `limit_size` then evicts additional legitimate pending/proposed transactions to bring the phantom total back under the cap. Evicted transactions receive a `Reject::Full` callback and are permanently removed. An attacker can repeat this to continuously drain the pool of legitimate transactions, forcing honest users to resubmit indefinitely and causing sustained CKB network congestion at minimal cost.

**Impact class: High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs (10001–15000 points).**

## Likelihood Explanation
The eviction path requires: (1) a new transaction whose ancestor count exceeds `max_ancestors_count`, AND (2) removing cell-dep-only parents brings the count within the limit. An unprivileged attacker can deliberately construct this by first submitting a set of transactions forming a cell-dep chain in the pool, then submitting a new transaction referencing many of them as cell-dep ancestors to exceed `max_ancestors_count`. No special privilege is required — only the ability to submit transactions via RPC or P2P relay, available to any network participant. The attack is repeatable with fresh transactions to accumulate inflation across multiple submissions.

## Recommendation
Compute the new aggregate totals **after** `check_and_record_ancestors` returns, so eviction decrements are already reflected before the new entry's contribution is added:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Validate overflow only — do NOT capture a snapshot.
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Apply new entry's contribution AFTER evictions have decremented the running totals.
    self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

## Proof of Concept
**Setup:** Pool with `max_tx_pool_size = 1000` bytes, `max_ancestors_count = 3`. Pool holds transactions A (size=500), B (size=200), C (size=200); total = 900 bytes. B and C are cell-dep ancestors of a future transaction D.

1. Attacker submits D, which has cell-dep ancestors {A, B, C} → `ancestors_count = 4 > 3`. Since `4 - 2 (cell_ref_parents) = 2 <= 3`, the eviction path fires.
2. Inside `check_and_record_ancestors`, B (size=200) is evicted: `update_stat_for_remove_tx(200, ...)` → `self.total_tx_size = 700`.
3. Back in `add_entry`, the stale snapshot `total_tx_size = 900 + size(D) = 1050` is written back at line 218.
4. `limit_size` fires because `1050 > 1000`. It evicts A (size=500): phantom total becomes `550`. Real pool size is `700 - 500 + size(D) = 350` — well under the cap. A was evicted unnecessarily with `Reject::Full`.
5. Repeat with fresh transactions to continuously drain the pool of legitimate entries.
