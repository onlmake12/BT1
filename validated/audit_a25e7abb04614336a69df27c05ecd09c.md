### Title
Delegation Record Not Updated on Stake Account Split — (`governance/pyth_staking_sdk/src/idl/staking.json`, `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

When a user's stake account is split via `accept_split` in the staking program, a new stake account is created with delegation positions. However, the integrity pool's `delegation_record` — which tracks per-`(publisher, stake_account_positions)` reward accrual state — is never initialized or updated for the new account. When `advance_delegation_record` is later called on the new account, it initializes the record starting from the position's `activation_epoch`, which may predate the split. This allows the new account to claim rewards for epochs already claimed by the source account, resulting in double-claiming of staking rewards.

---

### Finding Description

The staking program's `accept_split` instruction creates a new stake account (`new_stake_account_positions`) and transfers `amount` tokens from the source account. The new account receives delegation positions (including their original `activation_epoch`).

The integrity pool program tracks reward accrual per `(publisher, stake_account_positions)` pair via a `DelegationRecord` PDA with seeds `["delegation_record", publisher, stake_account_positions]`. This record stores `last_epoch` (the last epoch rewards were claimed) and `next_slash_event_index`.

Critically, the `accept_split` instruction's account list does **not** include any `delegation_record` accounts: [1](#0-0) 

The instruction only touches staking-program accounts (`source_stake_account_positions`, `source_stake_account_metadata`, `new_stake_account_positions`, `new_stake_account_metadata`, custody accounts). No integrity pool `delegation_record` is initialized or updated.

The `DelegationRecord` struct has two fields: [2](#0-1) 

When `advance_delegation_record` is subsequently called for the new stake account, the SDK computes the starting epoch as `max(position.activationEpoch, delegationRecord?.lastEpoch)`. Since no `delegation_record` exists for the new account, `lastEpoch` is `undefined`, and the starting epoch falls back to `activation_epoch`: [3](#0-2) 

The `activation_epoch` of the copied positions predates the split. The source account already advanced its `delegation_record` up to some epoch `N > activation_epoch`. The new account's `delegation_record` starts from `activation_epoch`, so it claims rewards for epochs `[activation_epoch, N]` — the same epochs already claimed by the source account.

---

### Impact Explanation

**Double-claiming of OIS staking rewards.** The new stake account recipient can call `advance_delegation_record` and receive rewards for all epochs since the original position's `activation_epoch`, even though the source account already claimed those rewards. The integrity pool's `pool_reward_custody` is drained twice for the same staked tokens over the same epoch range. The magnitude scales with the staked amount and the number of epochs elapsed since the original position was created. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The `accept_split` flow requires `pda_authority` approval, which is a privileged role. However, splits are a documented, legitimate use case ("mostly useful to transfer unvested tokens"). Any user can call `request_split` permissionlessly; the `pda_authority` approves legitimate vesting transfers routinely. The vulnerability triggers automatically on any approved split where the source account had active integrity pool delegation positions — no additional attacker action is needed beyond calling `advance_delegation_record` on the new account, which is itself permissionless. [5](#0-4) 

---

### Recommendation

In `accept_split`, for each delegation position being transferred to the new stake account, initialize the new account's `delegation_record` with `last_epoch` set to the **current epoch** (not `activation_epoch`). This prevents the new account from claiming rewards for epochs already settled by the source account. Alternatively, require that `advance_delegation_record` is called on the source account (settling all pending rewards) before a split is accepted, and initialize the new account's `delegation_record` to the source's current `last_epoch`.

---

### Proof of Concept

1. User A has a stake account with a delegation position to publisher P, created at epoch 10. Current epoch is 100. User A has called `advance_delegation_record` up to epoch 90 (`delegation_record[P, A].last_epoch = 90`).
2. User A calls `request_split(amount, recipient=B)`.
3. `pda_authority` calls `accept_split`, creating new stake account B with the delegation position (`activation_epoch = 10`).
4. `delegation_record[P, B]` does not exist.
5. User B calls `advance_delegation_record` for their new account. The SDK computes starting epoch = `max(10, undefined)` = `10`.
6. User B receives rewards for epochs 10–100, even though User A already claimed epochs 10–90.
7. Epochs 10–90 are paid out twice from `pool_reward_custody`. [6](#0-5) [2](#0-1)
