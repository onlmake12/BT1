Audit Report

## Title
`prune()` Fails to Delete `Header`/`TxHash` Entries When No `ConsumedOutPoint` Entries Exist — (`util/indexer/src/indexer.rs`)

## Summary
In `prune()`, `min_block_number` is derived solely from `ConsumedOutPoint` entries and defaults to `BlockNumber::MAX` when none exist. When all pruneable blocks contain only cellbase transactions (which never produce `ConsumedOutPoint` entries), the `Header`/`TxHash` pruning iterator seeks to the end of the key space and immediately terminates, leaving all old `Header => Transactions` and `TxHash => TransactionInputs` entries in RocksDB indefinitely. This causes unbounded disk growth in the indexer store proportional to chain length.

## Finding Description
`min_block_number` is initialized to `BlockNumber::MAX` at line 765 and is only updated inside the `ConsumedOutPoint` loop (lines 766–781) when at least one entry with `block_number < prune_to_block` exists. When the loop body never executes, `min_block_number` remains `u64::MAX`.

At line 785, `key_prefix_header` is extended with `min_block_number.to_be_bytes()`, producing seek key `[224u8, 0xFF×8]` — the very end of the `Header` key space. The `take_while` guard at line 793 (`BlockNumber <= prune_to_block`) immediately terminates because no real block exists at block `u64::MAX`, so no `Header` or `TxHash` entries are ever deleted.

The root cause of absent `ConsumedOutPoint` entries for cellbase-only blocks is the `if tx_index > 0` gate at line 339, which skips all input processing for cellbase transactions. However, cellbase outputs ARE processed (line 424 onward): if any output matches the cell filter, `tx_matched = true` (line 438), causing a `TxHash` entry (lines 482–491) and a `Header` entry (lines 494–510) to be written. Additionally, `Header` entries are written unconditionally for every block passing the block filter (both branches of the if/else at lines 495–510 write a Header entry), so even blocks with zero matching transactions accumulate Header entries that are never pruned.

The existing `prune_bound` test (lines 1532–1600) exercises exactly this scenario — 21 cellbase-only blocks — but only asserts that `get_block_hash` works; it does not assert that old `Header`/`TxHash` entries are absent after pruning, leaving the invariant violation undetected.

## Impact Explanation
`TxHash => TransactionInputs` (prefix `192`) and `Header => Transactions` (prefix `224`) entries accumulate in the indexer's RocksDB store with no upper bound. The pruning mechanism that is supposed to cap storage to `keep_num` blocks is silently bypassed for these key families whenever the chain contains cellbase-only blocks. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**, as the indexer's storage grows unboundedly, degrading read/scan performance and consuming unbounded disk space.

## Likelihood Explanation
Triggered by any chain segment where non-cellbase transactions are absent — a routine condition during low-activity periods on mainnet. No attacker is required; any miner producing valid cellbase-only blocks (standard behavior) triggers this path. The condition is reachable without any special privilege and is a normal occurrence on the live network. The `prune_bound` test already sets up this exact scenario, confirming the code path is exercised in practice.

## Recommendation
Decouple the `Header`/`TxHash` pruning start key from `min_block_number`. When `min_block_number == BlockNumber::MAX` (no `ConsumedOutPoint` entries were pruned), the iterator should start from block `0` rather than `u64::MAX`:

```rust
let start_block = if min_block_number == BlockNumber::MAX {
    0u64
} else {
    min_block_number
};
let mut key_prefix_header = vec![KeyPrefix::Header as u8];
key_prefix_header.extend_from_slice(&start_block.to_be_bytes());
```

Additionally, the `prune_bound` test should be extended to assert that `Header` and `TxHash` entries for blocks `<= tip - keep_num - 1` are absent after pruning.

## Proof of Concept
1. Create an `Indexer` with `keep_num = 10`, `prune_interval = 1`.
2. Append `keep_num + 2 = 12` blocks, each containing only a cellbase transaction with one output matching the cell filter (no non-cellbase txs).
3. `prune()` is called automatically on each `append` since `prune_interval = 1`.
4. Scan all keys with prefix `KeyPrefix::TxHash` (`192`) and `KeyPrefix::Header` (`224`).
5. Assert that entries for blocks `<= tip - keep_num - 1` are absent — they will **not** be absent, demonstrating the bug.

The `prune_bound` test at line 1534 already sets up this exact scenario but omits the pruning invariant assertion, confirming the gap.
