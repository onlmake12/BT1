### Title
Any RPC Caller Can Remove Any Pending Transaction from the Tx Pool Without Authorization - (`rpc/src/module/pool.rs`)

### Summary

The `remove_transaction` and `clear_tx_pool` RPC methods in CKB's Pool module perform no authorization check on the caller. Any entity that can reach the RPC endpoint can remove any pending transaction (or all transactions) from the tx pool by supplying only the tx hash. This is a direct analog to the `removeFeeder()` missing-access-control pattern: a state-mutating function documented as an operator tool is callable by any unprivileged RPC client, enabling targeted transaction censorship or full pool wipe.

### Finding Description

`remove_transaction` is declared in the `PoolRpc` trait and implemented in `PoolRpcImpl` with no caller identity check:

```rust
// rpc/src/module/pool.rs  line 254-255
```

### Impact Explanation

An attacker who can reach the RPC endpoint can:

1. **Targeted censorship**: Supply any known `tx_hash` to `remove_transaction` and evict a victim's pending transaction (and all child transactions that depend on it) from the pool. The attacker can repeat this every time the victim resubmits, permanently preventing confirmation.
2. **Full pool wipe**: Call `clear_tx_pool` to atomically remove every pending and proposed transaction from the node, disrupting all in-flight activity on that node.
3. **Verify-queue drain**: Call `clear_tx_verify_queue` to drop all transactions currently being validated, stalling throughput.

For time-sensitive transactions (e.g., channel closures, DAO withdrawals at epoch boundaries, or any protocol built on CKB that relies on timely confirmation), repeated removal is equivalent to a targeted freeze of that operation on the affected node.

### Likelihood Explanation

The RPC binds to `127.0.0.1` by default, which limits the attack surface to local users. However:

- Many operators expose the RPC on a public or LAN interface for remote management; the code provides no fallback authentication layer for that case.
- Any process running on the same host (e.g., a compromised co-located service, a malicious script, or a web application with SSRF) can reach `127.0.0.1:8114` and call these methods.
- The attacker needs only the tx hash, which is public information visible in the mempool via `get_raw_tx_pool`.
- The `remove_transaction` RPC is exercised in integration tests without any credential, confirming it is reachable with a plain JSON-RPC call. [8](#0-7) 

### Recommendation

1. **Add caller authorization to `remove_transaction` and `clear_tx_pool`**: Restrict these methods to a configurable allowlist of IP addresses or require a bearer token/API key checked in the RPC server middleware, analogous to how Bitcoin Core gates `stop`, `setban`, and similar admin RPCs.
2. **Separate admin methods into a distinct, non-default module**: Move `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` into an `Admin` module that is disabled by default and requires explicit opt-in with authentication configured.
3. **Enforce binding-address checks at the code level**: If the RPC listen address is not loopback, require that an authentication token is configured; refuse to start otherwise.

### Proof of Concept

```bash
# 1. Submit a transaction and note its hash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"send_transaction","params":[<tx_json>,"passthrough"],"id":1}'
# -> {"result":"0xABCD..."}

# 2. Any unprivileged local caller removes it immediately — no credentials needed
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xABCD..."],"id":2}'
# -> {"result":true}

# 3. Repeat step 2 every time the victim resubmits to achieve permanent censorship.

# 4. Or wipe the entire pool in one call:
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":3}'
# -> {"result":null}
```

Expected outcome: the victim's transaction is evicted from the pool and will not be confirmed until the attacker stops interfering. On a node with a public RPC interface, any network peer can perform this attack.

```rust
const ALWAYS_SUCCESS_SCRIPT_CYCLE: u64 = 537;

pub struct RemoveTx;

impl Spec for RemoveTx {
    crate::setup!(num_nodes: 1);

    fn run(&self, nodes: &mut Vec<Node>) {
        let node0 = &mut nodes[0];
        node0.mine_until_out_bootstrap_period();

        let mut net = Net::new(
            self.name(),
            node0.consensus(),
            vec![SupportProtocols::RelayV3],
        );
        net.connect(node0);

        {
            // Remove a tx from orphan pool
            let parent_tx = node0.new_transaction_spend_tip_cellbase();
            let child_tx = node0.new_transaction(parent_tx.hash());

            node0.assert_tx_pool_statics(0, 0);

            relay_tx(&net, node0, child_tx.clone(), ALWAYS_SUCCESS_SCRIPT_CYCLE);
            let result = wait_until(5, || {
                let tx_pool_info = node0.get_tip_tx_pool_info();
                tx_pool_info.orphan.value() == 1 && tx_pool_info.pending.value() == 0
            });
            assert!(
                result,
                "Send child tx first, it will be added to orphan tx pool"
            );
            node0.assert_tx_pool_statics(0, 0);

            node0.remove_transaction(child_tx.hash());
            let result = wait_until(5, || {
                let tx_pool_info = node0.get_tip_tx_pool_info();
                tx_pool_info.orphan.value() == 0 && tx_pool_info.pending.value() == 0
            });
            assert!(result, "remove a tx from orphan tx pool");
```
