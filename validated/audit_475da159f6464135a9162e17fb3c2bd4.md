Audit Report

## Title
Stale-Snapshot Overwrite in `add_entry` Inflates `total_tx_size`/`total_tx_cycles`, Enabling Tx-Pool DoS — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::add_entry` captures projected pool totals into local variables before calling `check_and_record_ancestors`, which may evict existing entries and correctly decrement `self.total_tx_size`/`self.total_tx_cycles` via `update_stat_for_remove_tx`. After evictions complete, `add_entry` unconditionally overwrites those fields with the pre-eviction snapshot, re-inflating the totals by the aggregate size and cycles of every evicted transaction. An unprivileged RPC caller can craft a transaction graph that repeatedly triggers this path, permanently inflating pool accounting until `limit_size` evicts all legitimate transactions and the pool rejects every new submission with `Reject::Full`.

## Finding Description

**Root cause — `add_entry`, lines 200–221:**

`updated_stat_for_add_tx` (lines 210–211) is a read-only method that returns `(self.total_tx_size + tx_size, self.total_tx_cycles + cycles)` without mutating `self`. The result is stored in local variables `total_tx_size` / `total_tx_cycles`.

```rust
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213
// ...
self.total_tx_size  = total_tx_size;                            // line 218
self.total_tx_cycles = total_tx_cycles;                         // line 219
```

**Eviction path — `check_and_record_ancestors`, lines 603–625:**

When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, the function evicts `cell_ref_parents` one by one via `remove_entry_and_descendants` (line 618). That call chains to `remove_entry` (line 263) → `update_stat_for_remove_tx` (line 247), which correctly subtracts each evicted entry's `size` and `cycles` from `self.total_tx_size` / `self.total_tx_cycles`.

**The accounting error:**

Let `S` = pool size before the call, `E` = total size of evicted entries, `N` = new tx size.

| Step | `self.total_tx_size` |
|---|---|
| After `updated_stat_for_add_tx` snapshot | local var = `S + N` |
| After evictions inside `check_and_record_ancestors` | `self.total_tx_size = S − E` (correct) |
| After overwrite at line 218 | `self.total_tx_size = S + N` (stale, **E lost**) |

Correct post-add value: `S − E + N`. Actual value: `S + N`. Inflation per triggering call: `E`.

**Why existing checks do not prevent this:**

`updated_stat_for_add_tx` performs only an overflow/capacity check and returns computed values; it does not write to `self`. The subsequent overwrite at lines 218–219 is unconditional and has no awareness of intervening mutations to `self.total_tx_size` caused by evictions.

## Impact Explanation

`limit_size` (`pool.rs` lines 292–328) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` (default 180 MB) and evicts legitimate pending transactions. With `total_tx_size` artificially inflated:

- Legitimate transactions are evicted from the pool even when actual memory usage is well within the limit.
- New transactions are rejected with `Reject::Full` even when the pool has ample space.
- `total_tx_cycles` inflation distorts fee-rate estimation returned to miners and RPC callers.
- Repeated exploitation drives `total_tx_size` to `max_tx_pool_size`, after which the pool permanently rejects all incoming transactions — a complete mempool denial-of-service requiring a node restart to recover.

This matches the **High** impact: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs.*

## Likelihood Explanation

The trigger requires constructing a transaction graph where a new transaction has more than `max_ancestors_count` (default 25) ancestors and at least one ancestor is a `cell_ref_parent` (i.e., an unconfirmed transaction whose output is referenced as a cell dep). This is a valid, protocol-permitted CKB pattern. No privileged access, no key material, and no majority hashpower is required. The attack is executable by any RPC-accessible user. The default `max_ancestors_count` of 25 is low enough that the required chain depth is easily achievable. The inflation is permanent per round and accumulates monotonically.

## Recommendation

Move the stat update **after** all evictions have completed, so it reflects the post-eviction pool state. Replace the pre-eviction snapshot with a direct increment applied after `check_and_record_ancestors` returns:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Validate capacity (overflow check) without capturing stale values
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Recompute AFTER evictions so evicted sizes are not re-added
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, remove the local snapshot entirely and apply the increment directly to `self.total_tx_size` / `self.total_tx_cycles` after all mutations, mirroring the pattern used by `update_stat_for_remove_tx`.

## Proof of Concept

**Setup (default config: `max_ancestors_count = 25`, `max_tx_pool_size = 180 MB`):**

1. Submit a "dep-provider" transaction `D` whose output `D:0` is spendable and referenceable as a cell dep.
2. Submit 24 ancestor transactions `A1 → A2 → … → A24` (each spending the previous output), where `A1` also references `D:0` as a cell dep input — making `D` a `cell_ref_parent` of any transaction that descends from `A24`.
3. Submit a new transaction `T` spending `A24`'s output and referencing `D:0` as a cell dep.

**Execution trace:**

1. `add_entry(T)` is called.
2. `updated_stat_for_add_tx` snapshots `total_tx_size = S + size(T)` into a local variable.
3. `check_and_record_ancestors`: `ancestors_count = 25 + 1 = 26 > 25`; `cell_ref_parents = {D}`; `26 − 1 = 25 ≤ 25` → eviction branch fires.
4. `remove_entry_and_descendants(D)` → `update_stat_for_remove_tx(size(D), cycles(D))` → `self.total_tx_size = S − size(D)`.
5. Back in `add_entry`: `self.total_tx_size = total_tx_size` → `self.total_tx_size = S + size(T)`.
6. Correct value: `S − size(D) + size(T)`. Actual value: `S + size(T)`. **Inflation: `size(D)`.**

Repeating with fresh `D` transactions each round accumulates inflation. Once `total_tx_size ≥ 180 MB`, `limit_size` evicts all pending transactions and the pool rejects every subsequent submission with `Reject::Full`.
