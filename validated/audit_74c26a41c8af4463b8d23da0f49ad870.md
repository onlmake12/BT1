### Title
Unzeroized Heap Buffers Expose P2P Secret Key Material in Memory - (File: `util/app-config/src/configs/network.rs`, `util/crypto/src/secp/generator.rs`)

---

### Summary

The CKB node's secp256k1 P2P secret key is loaded from disk into a plain `Vec<u8>` heap buffer (`buf`) inside `read_secret_key`, and generated into a plain `Vec<u8>` heap buffer (`seed`) inside `gen_secret_key`. Neither buffer implements zeroization on drop. The raw 32-byte key material therefore persists in heap memory after the function returns, recoverable via core dumps, swap files, or `/proc/<pid>/mem` by a local attacker.

---

### Finding Description

**Location 1 — `read_secret_key` (`util/app-config/src/configs/network.rs`, lines 315–320):**

```rust
let mut buf = Vec::new();
file.read_to_end(&mut buf).and_then(|_read_size| {
    secio::SecioKeyPair::secp256k1_raw_key(&buf)
        .map(Some)
        .map_err(|_| Error::new(ErrorKind::InvalidData, "invalid secret key data"))
})
```

`buf` is a heap-allocated `Vec<u8>` that receives the full 32-byte raw secret key from disk. When `buf` goes out of scope at the end of the closure, Rust's standard `Vec::drop` deallocates the backing allocation **without zeroing it**. The key bytes remain readable in the freed heap region until the allocator reuses that page.

This function is called on every node startup via `fetch_private_key` → `read_secret_key`, and also directly by the `ckb peer-id from-secret` CLI subcommand. [1](#0-0) 

**Location 2 — `gen_secret_key` (`util/crypto/src/secp/generator.rs`, lines 34–42):**

```rust
fn gen_secret_key(&mut self) -> SecretKey {
    let mut seed = vec![0; 32];
    loop {
        self.rng.fill(seed.as_mut_slice());
        if let Ok(key) = SecretKey::from_slice(&seed) {
            return key;
        }
    }
}
```

`seed` is a heap-allocated `Vec<u8>` filled with the raw key entropy. It is dropped without zeroization when the function returns, leaving the key material in freed heap memory. [2](#0-1) 

**Contrast with `Privkey::drop`:** The `Privkey` wrapper struct correctly implements `Drop` by calling `zeroize()` using `ptr::write_volatile` and compiler fences. However, this only protects the final `Privkey` object — the intermediate heap buffers that held the raw bytes before being copied into `Privkey` are never cleared. [3](#0-2) 

**`generate_random_key` (`util/app-config/src/configs/network.rs`, lines 254–262)** also returns a plain `[u8; 32]` stack array with no zeroization, though stack memory is less persistent than heap. [4](#0-3) 

---

### Impact Explanation

The P2P secret key (`data/network/secret_key`) is the node's long-term secp256k1 identity used for secio-encrypted P2P sessions. Recovery of this key allows an attacker to:

1. **Impersonate the CKB node** on the P2P network, establishing authenticated connections as that peer identity.
2. **Decrypt recorded secio sessions** if the attacker captured past encrypted P2P traffic (forward secrecy depends on the session key derivation; the static identity key is used in the handshake).
3. **Disrupt peer reputation** — the node's peer ID is derived from this key; an attacker holding the key can forge the peer ID.

The key material is exposed in heap memory for the entire lifetime of the process after startup (since `fetch_private_key` is called once at startup and the `buf` is freed but not zeroed), and also transiently during every `ckb peer-id from-secret` CLI invocation. [5](#0-4) 

---

### Likelihood Explanation

Exploitation requires local access to the process's memory space, achievable via:

- **Core dumps**: If the `ckb` process crashes (e.g., due to an OOM kill, signal, or bug), the OS writes a core dump containing the full heap. On many Linux distributions, core dumps are enabled by default and written world-readable or to a central collector (`/proc/sys/kernel/core_pattern`).
- **Swap files**: If the system swaps memory to disk, the freed-but-unzeroed heap pages may be written to the swap partition, readable by root or by an attacker with disk access.
- **`/proc/<pid>/mem`**: A local process with `ptrace` capability (e.g., a debugger, or a process running as the same user) can read the CKB process's memory directly.
- **Memory forensics**: On cloud/VM environments, a hypervisor-level attacker or a co-tenant with access to memory snapshots can recover the key.

Likelihood is **Low-Medium**: requires local access or a crash scenario, but the attack surface is persistent (the freed heap pages are not zeroed and may not be reused for a long time).

---

### Recommendation

1. **Zeroize `buf` in `read_secret_key`** before it is dropped. Use a type like `zeroize::Zeroizing<Vec<u8>>` (from the `zeroize` crate) or manually zero the buffer after use:
   ```rust
   let mut buf = zeroize::Zeroizing::new(Vec::new());
   ```

2. **Zeroize `seed` in `gen_secret_key`** before it is dropped:
   ```rust
   let mut seed = zeroize::Zeroizing::new(vec![0u8; 32]);
   ```

3. **Zeroize `key` in `generate_random_key`** after writing to file:
   ```rust
   let mut key: [u8; 32] = [0; 32];
   // ... fill and write ...
   key.zeroize(); // before return
   ```

4. Consider using the `secrecy` or `zeroize` crate consistently for all secret-bearing buffers, rather than relying solely on the `Privkey::drop` zeroization, which only covers the final wrapper and not intermediate copies.

---

### Proof of Concept

1. Start a CKB node (`ckb run`). The node calls `fetch_private_key` → `read_secret_key`, which allocates `buf: Vec<u8>` on the heap with the 32-byte raw secret key, then drops `buf` without zeroing. [1](#0-0) 

2. From a shell running as the same user (or root), read the process heap:
   ```bash
   PID=$(pgrep ckb)
   # Read the heap region from /proc/<pid>/maps and /proc/<pid>/mem
   grep heap /proc/$PID/maps
   # dd the heap range and grep for the known key bytes
   ```

3. Alternatively, send `SIGABRT` to the CKB process to trigger a core dump, then use `strings` or a hex editor on the core file to locate the 32-byte key sequence.

4. The recovered bytes match the content of `data/network/secret_key`, confirming the key was left unzeroized in the freed heap allocation. [6](#0-5) [2](#0-1)
