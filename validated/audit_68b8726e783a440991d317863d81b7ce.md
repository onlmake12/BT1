Audit Report

## Title
Tx-Pool Promotes Transactions to `Proposed` Status Without Re-Verifying `since` Constraints After Reorg — (`tx-pool/src/pool.rs`, `tx-pool/src/process.rs`)

## Summary
During chain reorganization, `_update_tx_pool_for_reorg` promotes pending/gap transactions to `Proposed` status via `proposed_rtx` / `gap_rtx`, which perform only a status-label flip with no re-verification of time-relative (`since`) constraints against the new post-reorg snapshot. The block assembler's `calc_dao` also does not run `TimeRelativeTransactionVerifier` before including transactions in a block template. As a result, a miner's assembled block containing a `since`-violated transaction will be rejected by the network's full contextual block verifier, causing the miner to forfeit the block reward.

## Finding Description

**Root cause — `proposed_rtx` and `gap_rtx` are pure label flips:**

`proposed_rtx` (`tx-pool/src/pool.rs:405-422`) checks only for a duplicate status, then calls `set_entry_proposed`: [1](#0-0) 

`set_entry_proposed` delegates to `pool_map.set_entry` (`tx-pool/src/component/pool_map.rs:223-233`), which only modifies the `status` field — no fee check, no capacity check, no `TimeRelativeTransactionVerifier` call: [2](#0-1) 

`gap_rtx` (`tx-pool/src/pool.rs:386-403`) follows the identical pattern: [3](#0-2) 

**Call site — `_update_tx_pool_for_reorg` promotes without re-verification:**

The new post-reorg `snapshot` is available at this point but is never passed to any verifier before the status flip: [4](#0-3) 

**Contrast with `readd_detached_tx` — full pipeline including `verify_rtx`:**

Transactions from detached blocks go through `resolve_tx` → `check_tx_fee` → `verify_rtx`, which internally calls `TimeRelativeTransactionVerifier`. Transactions already in the pool that are merely promoted receive none of this: [5](#0-4) 

**Block assembler `calc_dao` does not run `TimeRelativeTransactionVerifier`:**

`calc_dao` only calls `entry.rtx.check()`, which verifies cell availability (double-spend, existence) but not `since` constraints. A transaction with a violated `since` passes this filter and enters the block template: [6](#0-5) 

**Full block verifier DOES enforce `since` at commit time:**

The contextual block verifier uses `TxVerifyEnv::new_commit` and runs `TimeRelativeTransactionVerifier` on every committed transaction. A `since` violation here causes the entire block to be rejected: [7](#0-6) 

**Concrete exploit scenario:**
1. Transaction T has `since = AbsoluteBlockNumber(100)`. Chain tip is at block 105; T is in `Pending`.
2. A block at height 106 proposes T. `_update_tx_pool_for_reorg` calls `proposed_rtx`, promoting T to `Proposed` (no `since` check performed, but tip ≥ 100 so it would pass anyway).
3. A reorg arrives (peer delivers a longer fork rooted at block 95). New tip is block 97.
4. T's proposal is detached; `remove_by_detached_proposal` moves T back to `Pending`.
5. The next block at height 98 re-proposes T. `_update_tx_pool_for_reorg` calls `proposed_rtx` again, promoting T to `Proposed` — **without checking that 98 < 100**.
6. `calc_dao` in the block assembler passes T (only checks cell availability).
7. T appears in the block template for block 99. Miner submits the block.
8. The network's contextual block verifier runs `TimeRelativeTransactionVerifier` with `TxVerifyEnv::new_commit` for block 99. `99 < 100` → `since` constraint fails → block rejected.
9. Miner forfeits the block reward.

## Impact Explanation

A miner operating in `mine_mode` will include a `since`-violated transaction in a block template after a reorg that reduces the tip height below the transaction's `since` threshold. The assembled block is rejected by the network's full contextual verifier, causing the miner to lose the block reward for that round. DAO withdrawal transactions, which are common on CKB and use epoch-based `since` fields, are a realistic trigger. This constitutes concrete economic damage to miners and maps to **Vulnerabilities which could easily damage CKB economy (Critical, 15001–25000 points)**.

## Likelihood Explanation

Reorgs are a normal, unprivileged network event — any peer can trigger one by relaying a longer valid chain. No special privileges, keys, or majority hashpower are required. Transactions with `since` time-locks (absolute block number, epoch, or timestamp) are common in CKB, particularly DAO withdrawal transactions. A reorg that reduces the tip height by even one block is sufficient if a transaction's `since` threshold falls in the gap. The condition is repeatable: an adversary who can reliably cause small reorgs can repeatedly invalidate a target miner's blocks.

## Recommendation

In `_update_tx_pool_for_reorg` (`tx-pool/src/process.rs`), before calling `proposed_rtx` or `gap_rtx`, run `time_relative_verify` (or the equivalent `TimeRelativeTransactionVerifier`) against the new snapshot for each candidate transaction. If verification fails, call `callbacks.call_reject` and remove the entry instead of promoting it. The `time_relative_verify` helper already exists in `tx-pool/src/util.rs:134-148` and accepts a snapshot, an `Arc<ResolvedTransaction>`, and a `TxVerifyEnv`. Use `TxStatus::Proposed.with_env(snapshot.tip_header())` to construct the appropriate environment. This mirrors the pattern already used in `readd_detached_tx` (`process.rs:888-912`).

## Proof of Concept

1. Configure a CKB node in mine mode (`block_assembler` enabled).
2. Submit a transaction T with `since = AbsoluteBlockNumber(N)` where N is slightly above the current tip (e.g., current tip = 95, N = 100).
3. Mine blocks until T is proposed and promoted to `Proposed` status (confirm via `get_raw_tx_pool` RPC showing T in the `proposed` map).
4. Inject a competing fork via a second node that reorgs the chain to a tip height below N (e.g., tip = 97), then re-proposes T in the new fork.
5. Observe via `get_raw_tx_pool` that T is in `Proposed` status even though `tip_height (97) < N (100)`.
6. Call `get_block_template` — T appears in the template's `transactions` list.
7. Submit the assembled block; the network rejects it with a `since` constraint violation (`TransactionError::Immature`).
8. Confirm the miner's block reward is forfeited.

    fn calc_dao(
        snapshot: &Snapshot,
        current_epoch: &EpochExt,
        cellbase: TransactionView,
        entries: Vec<TxEntry>,
    ) -> CalcDaoResult {
        let tip_header = snapshot.tip_header();
        let consensus = snapshot.consensus();
        let mut seen_inputs = HashSet::new();
        let mut transactions_checker = TransactionsChecker::new(iter::once(&cellbase));

        let mut checked_failed_txs = vec![];
        let checked_entries: Vec<_> = block_in_place(|| {
            entries
                .into_iter()
                .filter_map(|entry| {
                    let overlay_cell_checker =
                        OverlayCellChecker::new(&transactions_checker, snapshot);
                    if let Err(err) =
                        entry
                            .rtx
                            .check(&mut seen_inputs, &overlay_cell_checker, snapshot)
                    {
                        error!(
                            "Resolving transactions while building block template, \
                             tip_number: {}, tip_hash: {}, tx_hash: {}, error: {:?}",
                            tip_header.number(),
                            tip_header.hash(),
                            entry.transaction().hash(),
                            err
                        );
                        // Returning the out_point makes debugging easier and provides better logs.
                        checked_failed_txs
                            .push((entry.proposal_short_id(), err.out_point().cloned()));
                        None
                    } else {
                        transactions_checker.insert(entry.transaction());
                        Some(entry)
                    }
                })
                .collect()
        });

        let dummy_cellbase_entry = TxEntry::dummy_resolve(cellbase, 0, Capacity::zero(), 0);
        let entries_iter = iter::once(&dummy_cellbase_entry)
            .chain(checked_entries.iter())
            .map(|entry| entry.rtx.as_ref());

        // Generate DAO fields here
        let dao = DaoCalculator::new(consensus, &snapshot.borrow_as_data_loader())
            .dao_field_with_current_epoch(entries_iter, tip_header, current_epoch)?;

        Ok((dao, checked_entries, checked_failed_txs))
    }
