### Title
No Per-Peer QoS on Orphan Transaction Pool Allows DoS Eviction of Legitimate Orphans — (`File: tx-pool/src/component/orphan.rs`)

---

### Summary

The `OrphanPool` in CKB's tx-pool has a global capacity of 100 transactions with no per-peer quota. When the pool is full, eviction is performed by iterating a `HashMap` and removing the first key encountered — effectively random. A single unprivileged P2P peer can flood all 100 slots with crafted orphan transactions at zero fee cost, causing legitimate orphan transactions from honest peers to be randomly evicted. When the legitimate parent transaction later arrives, `process_orphan_tx` finds no orphan to resolve, silently breaking the transaction chain.

---

### Finding Description

`OrphanPool` is defined in `tx-pool/src/component/orphan.rs` with a hard global cap:

```rust
pub(crate) const DEFAULT_MAX_ORPHAN_TRANSACTIONS: usize = 100;
``` [1](#0-0) 

When the pool exceeds this limit, `limit_size()` evicts entries by iterating the `HashMap` and removing the first key it encounters — a non-deterministic, effectively random selection with no fairness guarantee:

```rust
while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
    // Evict a random orphan:
    let id = self.entries.keys().next().cloned().expect("bound checked");
    if let Some(entry) = self.remove_orphan_tx(&id) {
        evicted_txs.push(entry.tx.hash());
    }
}
``` [2](#0-1) 

`add_orphan_tx` inserts the new transaction first, then calls `limit_size()`, meaning the newly inserted legitimate transaction can itself be the one evicted: [3](#0-2) 

There is **no per-peer accounting** anywhere in `OrphanPool`. The `entries` field is a plain `HashMap<ProposalShortId, Entry>` with no per-`PeerIndex` quota: [4](#0-3) 

The entry point is the relay protocol. When a peer sends a `RelayTransactions` message containing a transaction whose inputs reference unknown outputs, `submit_remote_tx` is called, which calls `resumeble_process_tx`, which calls `add_orphan` when inputs are missing: [5](#0-4) 

The relay handler in `sync/src/relayer/transactions_process.rs` spawns an async task that calls `submit_remote_tx` for each transaction in the batch — up to `MAX_RELAY_TXS_NUM_PER_BATCH = 32767` hashes can be announced per message: [6](#0-5) 

Orphan transactions are added **before** script verification, so there is no cycle cost or fee barrier to filling the pool.

---

### Impact Explanation

A malicious peer can:

1. Connect to a CKB node as a normal P2P peer.
2. Relay 100 transactions that reference non-existent parent outputs (crafted orphans). These fill all 100 slots of the `OrphanPool` at zero fee cost.
3. Whenever a legitimate orphan transaction from an honest peer arrives and is inserted, `limit_size()` randomly evicts one entry. The attacker continuously sends new orphan transactions to maintain pool saturation.
4. When the legitimate parent transaction later arrives and `process_orphan_tx` is called, it finds no orphan to resolve — the transaction chain is silently broken. The honest user's child transaction is lost from the node's perspective. [7](#0-6) 

The downstream effect is that honest users' multi-transaction chains (parent → child) fail to propagate through the network, causing transaction confirmation delays or permanent loss of the child transaction from the mempool without any error surfaced to the user.

---

### Likelihood Explanation

- **Attacker-controlled entry path**: Any unprivileged P2P peer. No special role, key, or hashpower required.
- **Cost**: Zero fees. Orphan transactions are stored before verification. The attacker only pays network bandwidth.
- **Sustainability**: The attacker needs to send at most ~100 transactions per cycle to maintain pool saturation. With `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`, a single relay message can refill the pool many times over.
- **No existing per-peer limit**: Confirmed by inspection — `OrphanPool` has no per-`PeerIndex` quota. [8](#0-7) 

---

### Recommendation

1. **Add a per-peer orphan quota**: Track how many orphan transactions each `PeerIndex` has contributed. Reject or evict preferentially from peers that exceed their fair share (e.g., `DEFAULT_MAX_ORPHAN_TRANSACTIONS / connected_peers`).
2. **Evict from the most-contributing peer first**: Replace the random eviction in `limit_size()` with a policy that evicts from the peer holding the most slots, analogous to a fair-queue discipline.
3. **Evict expired entries before random eviction**: The current `limit_size()` already expires timed-out entries first, but the random fallback should be replaced with a peer-weighted policy.

---

### Proof of Concept

```
1. Attacker connects to a CKB node as a P2P peer via the Relay protocol.

2. Attacker constructs 100 transactions T_a1..T_a100, each spending a
   non-existent output (e.g., a random 32-byte tx hash, index 0).
   These are valid in structure but have unknown inputs → they become orphans.

3. Attacker sends RelayTransactionHashes for T_a1..T_a100, then responds
   to GetRelayTransactions with the full transactions.
   → submit_remote_tx → resumeble_process_tx → add_orphan (missing input)
   → OrphanPool now has 100 entries, all from the attacker.

4. Honest peer sends orphan transaction T_h (child of a not-yet-arrived parent P_h).
   → add_orphan_tx inserts T_h (pool size = 101)
   → limit_size() evicts one entry at random from the HashMap
   → T_h may be evicted (probability ≈ 1/101 per cycle, but attacker
     immediately refills, so T_h is evicted with high probability over
     repeated attempts).

5. Honest peer sends parent P_h.
   → process_orphan_tx(P_h) searches orphan pool for T_h → not found.
   → T_h is silently lost. The honest user's transaction chain is broken.

6. Attacker repeats step 3 continuously to maintain pool saturation.
   Cost: ~100 crafted transactions per cycle, zero fees.
``` [3](#0-2) [8](#0-7)

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
