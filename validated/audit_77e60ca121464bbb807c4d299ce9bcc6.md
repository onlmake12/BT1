### Title
`PythLazer.verifyUpdate()` Does Not Validate Payload Timestamp Freshness, Enabling Stale Price Replay - (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

The `verifyUpdate` function in `PythLazer.sol` validates the ECDSA signature and the signer's certificate expiration, but performs **no staleness check on the price payload's embedded timestamp**. Any unprivileged caller can submit an arbitrarily old but validly-signed Lazer price update, and the contract will accept it as valid, returning the stale payload to the consumer.

---

### Finding Description

`PythLazer.verifyUpdate()` performs exactly two security checks:

1. ECDSA signature recovery and validity
2. Signer certificate expiration via `isValidSigner(signer)` — which checks `block.timestamp < trustedSignerToExpiresAtMapping[signer]` [1](#0-0) 

It does **not** inspect the `timestamp` field embedded in the payload itself. The Lazer payload format encodes a `uint64 timestamp` (in microseconds) as the first field after the magic number, as parsed by `parsePayloadHeader` in `PythLazerLib.sol`: [2](#0-1) 

The official integration documentation confirms the timestamp is present and parseable by consumers: [3](#0-2) 

However, `verifyUpdate` returns the raw `payload` and `signer` without any freshness guarantee. The signer certificate expiration (`expiresAt`) governs how long the **signing key** is trusted, not how fresh the **price data** is. These are orthogonal concerns: a signer certificate valid for months means old payloads signed by that key remain verifiable indefinitely. [4](#0-3) 

By contrast, Pyth Core's price feeds enforce data freshness at the protocol level via `getPriceNoOlderThan`, which checks the price's `publishTime` against `block.timestamp`: [5](#0-4) 

No equivalent protection exists in `PythLazer.verifyUpdate()`.

---

### Impact Explanation

Consumer contracts that call `pythLazer.verifyUpdate(update)` and use the returned payload for financial decisions (lending, trading, liquidations) receive **no on-chain guarantee that the price data is recent**. An attacker who has saved old Lazer payloads can submit them to consumer contracts at a strategically chosen time when the stale price is favorable. For example:

- Stale price is lower than current → attacker borrows more than allowed against collateral
- Stale price is higher than current → attacker avoids liquidation or opens a position at a favorable price

The `verifyUpdate` function is the necessary vulnerable step: it is the sole on-chain security boundary for Lazer price updates, and it does not enforce data freshness.

---

### Likelihood Explanation

**Low.** The attacker must have previously subscribed to the Lazer WebSocket stream and saved signed payloads. The Lazer stream is publicly accessible to subscribers. Old payloads remain cryptographically valid as long as the signer's certificate has not expired. Signer certificates are set with long expiration windows (e.g., `3000000000000000` in tests), making saved payloads exploitable for extended periods. [6](#0-5) 

---

### Recommendation

Add a staleness check inside `verifyUpdate` (or a new `verifyUpdateNoOlderThan` variant) that:

1. Parses the `uint64 timestamp` from the first bytes of the payload (after the magic number)
2. Converts it from microseconds to seconds
3. Asserts `block.timestamp - (payloadTimestamp / 1_000_000) <= maxAge`

This mirrors the `getPriceNoOlderThan` pattern used in Pyth Core and the `check_price_is_fresh` pattern used in Move-based receivers. [7](#0-6) 

---

### Proof of Concept

1. Attacker subscribes to the Lazer WebSocket stream and records a signed payload `oldPayload` at time `T` when BTC price is $50,000.
2. Time passes; BTC price rises to $100,000.
3. Attacker calls `consumer.updatePrice(oldPayload)`.
4. Consumer calls `pythLazer.verifyUpdate{value: fee}(oldPayload)`.
5. `verifyUpdate` recovers the signer, checks `isValidSigner` — the signer certificate is still valid — and returns `(payload, signer)` with no timestamp check.
6. Consumer stores the stale BTC price of $50,000.
7. Attacker exploits the stale price (e.g., opens a leveraged position or borrows against collateral at the stale valuation).

The root cause is at `lazer/contracts/evm/src/PythLazer.sol` lines 70–106: the function never reads or validates the `timestamp` field embedded in the payload bytes. [1](#0-0)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
