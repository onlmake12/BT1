Audit Report

## Title
Duplicate tx_hash Amplification in GetTransactionsProof — No Deduplication Before DB Access - ([File: util/light-client-protocol-server/src/components/get_transactions_proof.rs])

## Summary
A remote peer can send a `GetTransactionsProof` message containing up to 1000 copies of the same `tx_hash`. Because no deduplication occurs before DB access, the server performs up to 2000 redundant DB reads (1000× `get_transaction_info` + 1000× `get_transaction_with_info`) for what is logically a single-item request. Additionally, `CBMT::build_merkle_proof` is invoked with 1000 duplicate indices and the result is unwrapped with `.expect()`, risking a panic.

## Finding Description
`GET_TRANSACTIONS_PROOF_LIMIT` is set to 1000. [1](#0-0) 

The only guard before processing is a raw-length upper-bound check. A message with 1000 identical hashes passes this check without any deduplication. [2](#0-1) 

The partition loop at lines 54–64 iterates over every element of `tx_hashes` without deduplication, calling `snapshot.get_transaction_info(tx_hash)` and `snapshot.is_main_chain()` for each one — 1000 calls for 1000 identical hashes. [3](#0-2) 

The `found` accumulation loop at lines 67–75 then calls `snapshot.get_transaction_with_info(&tx_hash)` for every element in `found` without deduplication, pushing 1000 identical `(tx, index)` pairs into the same `HashMap` entry. [4](#0-3) 

Finally, `CBMT::build_merkle_proof` is called with a `Vec` of 1000 duplicate indices and the result is unwrapped with `.expect("build proof with verified inputs should be OK")`. If the CBMT implementation rejects duplicate indices, this panics the handler. [5](#0-4) 

## Impact Explanation
Per malicious message: **2000 DB reads** (1000× `get_transaction_info` + 1000× `get_transaction_with_info`) instead of 2. This is a **1000× amplification** of DB I/O from a single P2P message. Sustained at even modest message rates this saturates the RocksDB read path. This matches the allowed CKB bounty impact: **Low (501–2000 points) — Any other important performance improvements for CKB**.

## Likelihood Explanation
The attacker needs only a valid on-chain tx hash (publicly observable via any block explorer) and the ability to connect as a light-client peer (no authentication required). The exploit is trivially constructable, requires no special privilege, and is indefinitely repeatable.

## Recommendation
Deduplicate `tx_hashes` immediately after the length check, before any DB access:

```rust
let tx_hashes: Vec<_> = {
    let mut seen = std::collections::HashSet::new();
    self.message.tx_hashes().to_entity()
        .into_iter()
        .filter(|h| seen.insert(h.clone()))
        .collect()
};
if tx_hashes.is_empty() { ... }
if tx_hashes.len() > constant::GET_TRANSACTIONS_PROOF_LIMIT { ... }
```

This reduces DB calls to at most `GET_TRANSACTIONS_PROOF_LIMIT` unique hashes and eliminates the duplicate-index risk in `build_merkle_proof`.

## Proof of Concept
1. Obtain any valid on-chain tx hash `H` (e.g., from a block explorer).
2. Construct a `GetTransactionsProof` message with `tx_hashes = [H; 1000]` and a valid `last_hash` pointing to a main-chain block.
3. Send to a light-client protocol server peer.
4. Instrument `snapshot.get_transaction_info`: assert it is called **1000 times** rather than 1.
5. Instrument `snapshot.get_transaction_with_info`: assert it is called **1000 times** rather than 1.
6. Observe 2000 total DB reads for a logically single-item request, confirming the amplification.
