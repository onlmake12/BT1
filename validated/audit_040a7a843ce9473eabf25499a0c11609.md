### Title
Network Alert Multisig Uses 2-of-4 (50%) Threshold with All Internal Signers, Enabling Fraudulent Network-Wide Alert Broadcast on Key Compromise — (`util/app-config/src/configs/alert_signature.toml`, `util/network-alert/src/verifier.rs`)

---

### Summary

CKB's network alert system — which broadcasts critical messages to every node on the P2P network — is protected by a hardcoded 2-of-4 (50%) multisig threshold, with all four keys held exclusively by Nervos Foundation developers. This is a weaker authorization posture than the 3/5 (60%) configuration flagged in the external report, and well below the recommended 70% for high-impact operations. Compromise of any two of the four keys is sufficient to broadcast arbitrary alert messages to all CKB nodes.

---

### Finding Description

**Hardcoded threshold configuration:**

The default alert signing configuration is compiled directly into the binary via `include_bytes!`:

```toml
# need 2 signatures to send alert
signatures_threshold = 2
public_keys = [
  "0x03933a9b116c5017561742c37ae69acb0dca3329a52c479c85df5bb387c8ac8715",
  "0x038da23240e5a4234601902cf3db3cdfc1b3fdb2db2a54ba6204f6ca1d6ef6129a",
  "0x02adc94b64a9809019139fe70bd26aa0d787772a1ad645a4bcb1456fb3e1105f09",
  "0x0369eca725513fc94685cd0b8ccebc7be874afea38d23ccb090566aa1c50d696b1",
]
``` [1](#0-0) 

This is loaded as the `Default` implementation for `NetworkAlertConfig`: [2](#0-1) 

At node startup, if `alert_signature` is absent from `ckb.toml`, the launcher falls back to this default: [3](#0-2) 

**Verification path:**

`Verifier::verify_signatures` calls `verify_m_of_n` with `self.config.signatures_threshold` (= 2) against the hardcoded 4-key set: [4](#0-3) 

`verify_m_of_n` itself is correct — it properly deduplicates public keys and counts verified signatures — but the *threshold value* it receives is the weak 2-of-4: [5](#0-4) 

**Broadcast path on success:**

Once `verify_signatures` returns `Ok(())`, the RPC handler immediately broadcasts the alert to all connected peers with no further authorization check: [6](#0-5) 

The P2P relayer (`AlertRelayer::received`) also verifies and then re-broadcasts to all connected peers: [7](#0-6) 

**All signers are internal:**

The RPC module documentation confirms all four keys belong to "early CKB developers": [8](#0-7) 

---

### Impact Explanation

An attacker who obtains any 2 of the 4 Nervos Foundation private keys can:

1. Craft an arbitrary `packed::Alert` with any `message`, `id`, `cancel`, `priority`, and `notice_until`.
2. Sign it with the 2 compromised keys.
3. Submit it via the `send_alert` RPC (reachable by any local or remote RPC caller with access to the RPC port) or inject it directly into the P2P network.
4. The alert propagates to **every connected CKB node** on the network, where it is stored, displayed to users, and re-relayed.

The `Notifier::add` method stores the alert and calls `notify_controller.notify_network_alert`, which triggers all registered subscribers (including external notification scripts configured by node operators): [9](#0-8) 

A fraudulent alert with a convincing message (e.g., "Critical vulnerability — stop all transactions immediately") could cause widespread user panic, halt economic activity, or serve as a social engineering vector for a follow-on attack.

---

### Likelihood Explanation

**Low** — mirrors the external report's assessment. Exploitation requires compromise of 2 of 4 private keys held by Nervos Foundation developers, presumably on hardware wallets. However, the 2-of-4 (50%) threshold means the attacker's task is materially easier than a 3-of-4 (75%) or 3-of-5 (60%) configuration. All four signers are internal to a single organization, eliminating the diversity benefit of external signers. A single organizational incident (insider threat, targeted attack on developer infrastructure, or supply-chain compromise) affecting two individuals is sufficient.

---

### Recommendation

1. **Increase the threshold** from 2-of-4 (50%) to at least 3-of-4 (75%) in `alert_signature.toml`. This directly mirrors the external report's recommendation to exceed 70% for high-impact operations.
2. **Add external signers**: Expand to a 4-of-6 or 5-of-7 configuration with at least one or two signers from outside the Nervos Foundation (e.g., trusted ecosystem partners or security firms), eliminating single-organization concentration risk.
3. **Validate `signatures_threshold` at startup**: The `NetworkAlertConfig` struct does not validate that `signatures_threshold > 0` or that `signatures_threshold <= public_keys.len()`. A zero threshold would cause `verify_m_of_n` to return `Ok(())` for any alert with zero signatures, since `0 > sigs.len()` is never true and `verified_sig_count (0) < 0` is never true.
4. **Annual review**: Implement a documented procedure for rotating the signer set and testing recovery scenarios.

---

### Proof of Concept

**Step 1 — Obtain 2 of the 4 hardcoded private keys** (e.g., via insider threat or targeted key extraction).

**Step 2 — Craft and sign a fraudulent alert:**

```rust
let raw_alert = packed::RawAlert::new_builder()
    .id(99999u32)
    .priority(100u32)
    .notice_until(far_future_timestamp)
    .message("CRITICAL: CKB network compromised. Stop all transactions.")
    .build();

let hash = raw_alert.calc_alert_hash();
let sig1 = compromised_key_1.sign_recoverable(&hash.into()).unwrap();
let sig2 = compromised_key_2.sign_recoverable(&hash.into()).unwrap();

let alert = packed::Alert::new_builder()
    .raw(raw_alert)
    .signatures(vec![sig1.serialize().into(), sig2.serialize().into()])
    .build();
```

**Step 3 — Submit via RPC:**

```json
{
  "jsonrpc": "2.0",
  "method": "send_alert",
  "params": [{ "id": "0x1869f", "cancel": "0x0", "priority": "0x64",
               "message": "CRITICAL: ...", "notice_until": "0xffffffffffff",
               "signatures": ["0x<sig1>", "0x<sig2>"] }],
  "id": 1
}
```

**Step 4 — Observe:** `verify_m_of_n` passes (2 valid sigs ≥ threshold of 2), the alert is stored locally and broadcast to all peers via `broadcast_with_handle`, and every node on the network displays the fraudulent message.

The root cause is the hardcoded `signatures_threshold = 2` against a 4-key set in `util/app-config/src/configs/alert_signature.toml`, consumed without additional validation by `util/network-alert/src/verifier.rs`.

    pub fn add(&mut self, alert: &Alert) {
        let alert_id = alert.raw().id().into();
        let alert_cancel = alert.raw().cancel().into();
        if self.has_received(alert_id) {
            return;
        }
        // checkout cancel_id
        if alert_cancel > 0 {
            self.cancel(alert_cancel);
        }
        // add to received alerts
        self.received_alerts.insert(alert_id, alert.clone());

        // check conditions, figure out do we need to notice this alert
        if !self.is_version_effective(alert) {
            debug!("Received a version ineffective alert {:?}", alert);
            return;
        }

        if self.noticed_alerts.contains(alert) {
            return;
        }
        self.notify_controller.notify_network_alert(alert.clone());
        self.noticed_alerts.push(alert.clone());
        // sort by priority
        self.noticed_alerts.sort_by_key(|a| {
            let priority: u32 = a.raw().priority().into();
            u32::MAX - priority
        });
    }
