### Title
Provider Re-Registration Abandons Pending Requests and Steals User Fees Without Refund — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy `register()` function is fully permissionless and allows any address to re-register as a provider at any time, resetting the commitment chain. Because user fees are credited to the provider immediately upon request — with no refund path — a malicious provider can register, collect fees from users requesting randomness, then re-register to rotate the commitment chain, making all pending requests permanently unfulfillable while retaining the accrued fees.

---

### Finding Description

The `register()` function in `Entropy.sol` is `public` with no access control and explicitly supports re-registration:

```
// Re-registering the same provider rotates the provider's commitment (and updates the feeInWei).
```

On re-registration, the following fields are overwritten:
- `provider.currentCommitment` → new commitment
- `provider.currentCommitmentSequenceNumber` → current `sequenceNumber`
- `provider.originalCommitment` / `originalCommitmentSequenceNumber` → reset
- `provider.endSequenceNumber` → `sequenceNumber + chainLength`
- `provider.sequenceNumber += 1`

Critically, `accruedFeesInWei` is **intentionally not reset** (noted in the code comment at line 124–127).

In `requestHelper()`, when a user requests randomness, the provider fee is credited immediately:

```solidity
providerInfo.accruedFeesInWei += providerFee;
```

The stored request captures `req.commitment = keccak256(userCommitment || providerInfo.currentCommitment)` at request time. After re-registration, `currentCommitment` is replaced with a new value. The provider can no longer produce a valid hash-chain preimage for the old commitment, making all pre-registration requests permanently unfulfillable.

There is no refund mechanism. The `IEntropy` interface explicitly states: *"Note that excess value is not refunded to the caller."* No `requestCancel` or `claimRefund` function exists.

---

### Impact Explanation

A malicious provider can:
1. Register with a valid commitment chain.
2. Attract users to request randomness (fees are immediately credited to the provider's `accruedFeesInWei`).
3. Call `register()` again with a new commitment, rotating the chain and making all pending requests unfulfillable.
4. Call `withdraw()` to extract all accrued fees.

Users lose their paid fees and receive no randomness. Callback-based consumers (`requestWithCallback`) are additionally left with stuck contract state waiting for a callback that will never arrive. The attack is repeatable across multiple cycles.

---

### Likelihood Explanation

Any address can register as a provider — there is no vetting, staking requirement, or governance approval. The attack requires only two transactions (`register` then `withdraw`) after collecting fees. The default provider is governance-controlled, but users can freely choose any registered provider, and integrators may select non-default providers. The attack is low-cost and fully on-chain.

---

### Recommendation

1. **Track pending request count per provider.** Disallow re-registration (or commitment rotation) while `pendingRequests > 0`.
2. **Alternatively, implement a time-locked re-registration.** Require a delay between re-registration calls, giving users time to receive fulfillment or claim refunds.
3. **Add a user refund path.** Allow users to reclaim their fee if a request has not been fulfilled within N blocks, debiting the provider's `accruedFeesInWei`.
4. **Require a minimum stake or bond** for provider registration to create a financial disincentive for abandonment.

---

### Proof of Concept

1. Attacker calls `register(feeInWei, commitment, metadata, chainLength, uri)` — becomes a valid provider.
2. Victim calls `requestWithCallback{value: fee}(attackerAddress, userRandomNumber)` — `attackerAddress.accruedFeesInWei += providerFee` is executed immediately.
3. Attacker calls `register(feeInWei, newCommitment, newMetadata, chainLength, uri)` — `currentCommitment` is replaced; the victim's stored `req.commitment` (computed against the old commitment) can never be satisfied.
4. Attacker calls `withdraw(accruedFeesInWei)` — transfers all collected fees to themselves.
5. Victim's callback contract is permanently stuck; no refund is available.

```text
    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }
```

```text
    function withdraw(uint128 amount) public override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            msg.sender
        ];

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer
        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "withdrawal to msg.sender failed");

        emit EntropyEvents.Withdrawal(msg.sender, msg.sender, amount);
        emit EntropyEventsV2.Withdrawal(
            msg.sender,
            msg.sender,
            amount,
            bytes("")
        );
    }
```
