### Title
Missing Strict Slot Ordering Validation Causes Division-by-Zero DoS in `parseTwapPriceFeedUpdates` — (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

`validateTwapPriceInfo` in `Pyth.sol` uses a non-strict inequality (`>`) when comparing `publishSlot` values of the start and end TWAP messages. This allows `startSlot == endSlot`, producing `slotDiff = 0`. The downstream `calculateTwap` function then performs three unchecked divisions by `slotDiff`, causing a Solidity 0.8 panic revert and rendering `parseTwapPriceFeedUpdates` completely unusable for that input.

---

### Finding Description

`validateTwapPriceInfo` checks slot ordering with:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This permits `startSlot == endSlot`. When that happens, `calculateTwap` computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // Panic: division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // Panic: division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // Panic: division by zero
```

All three divisions revert with `Panic(0x12)` in Solidity ≥ 0.8.

The Solana counterpart (`validate_twap_messages`) correctly uses a strict inequality:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
```

This is a direct inconsistency between the two implementations of the same protocol logic.

---

### Impact Explanation

Any caller of `parseTwapPriceFeedUpdates` who submits two Wormhole-verified TWAP messages sharing the same `publishSlot` will cause the entire transaction to revert. Because the fee is charged before the revert path is reached (the fee check at line 506 precedes `validateTwapPriceInfo`), the caller loses gas but the TWAP feature is completely unavailable for that input. Downstream protocols relying on `parseTwapPriceFeedUpdates` for TWAP-gated logic (e.g., lending protocols using TWAP for liquidation guards) would be unable to function for the affected price feed window.

---

### Likelihood Explanation

A user can submit the same Wormhole-verified VAA for both the `updateData[0]` (start) and `updateData[1]` (end) slots. The existing validation does not reject this: `prevPublishTime < publishTime` is satisfied for a single message, `expo` matches trivially, and the non-strict slot check passes. This requires no privileged access, no key compromise, and no external oracle manipulation — only a valid Wormhole-attested message, which any user can obtain from Hermes.

---

### Recommendation

Change the slot comparison in `validateTwapPriceInfo` from non-strict to strict, mirroring the Solana implementation:

```solidity
// Before (allows slotDiff == 0):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After (matches Solana's validate_twap_messages):
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

---

### Proof of Concept

1. Obtain any valid Wormhole-attested TWAP message from Hermes for a price feed.
2. Call `parseTwapPriceFeedUpdates` with `updateData[0] == updateData[1]` (same VAA for start and end).
3. `validateTwapPriceInfo` passes: `prevPublishTime < publishTime` ✓, `expo` matches ✓, `startSlot > endSlot` is false ✓.
4. `calculateTwap` executes `priceDiff / int128(uint128(0))` → Solidity panic revert.
5. The transaction reverts; the TWAP result is never returned.
