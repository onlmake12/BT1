Audit Report

## Title
DAO Withdrawal `header_dep_index` Parsed as `u64` in Rust vs. Lowest Byte in On-Chain C Script — Consensus Split via Crafted Witness - (File: `util/dao/src/lib.rs`)

## Summary
`DaoCalculator::transaction_maximum_withdraw` in `util/dao/src/lib.rs` reads the DAO withdrawal witness `header_dep_index` as a full 8-byte little-endian `u64`, while the on-chain C DAO script reads only the lowest byte (treating it as `u8`). A crafted withdrawal transaction with `header_dep_index = 257` (LE bytes `[0x01, 0x01, 0x00, ...]`) causes the C VM to resolve `header_deps[1]` (correct deposit block) while Rust resolves `header_deps[257]` (wrong block), producing a block number mismatch that causes Rust to return `DaoError::InvalidOutPoint` and reject the block — splitting consensus.

## Finding Description
In `util/dao/src/lib.rs` at line 91, the witness `input_type` field is decoded as a full `u64`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

This value is then used directly as a `usize` index into `header_deps` at lines 94–98:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
    .and_then(|hash| header_deps.get(&hash))
    .ok_or(DaoError::InvalidOutPoint)
```

The on-chain C DAO script reads only the lowest byte of the same 8-byte field. With `input_type = 257` (0x0101000000000000 LE):
- **C VM**: lowest byte = `0x01` → index 1 → correct deposit block hash → block number check passes → **accepts**
- **Rust**: full u64 = `257` → index 257 → withdraw block hash (number 200) → `deposit_header.number() != deposited_block_number` (200 ≠ 100) at line 105 → `DaoError::InvalidOutPoint` → **rejects**

The test `check_dao_withdraw_header_dep_index_exceeds_u8` at `util/dao/src/tests.rs:475–537` was written explicitly to document this discrepancy. It constructs a 258-entry `header_deps` list with the deposit block at index 1 and the withdraw block at index 257, sets `input_type = 257u64`, and asserts `result.is_err()` — confirming Rust's divergent behavior. The `withdrawing_header_hash` guard at lines 67–72 does not prevent this: the withdraw block hash is present in the `header_deps` `HashSet` (it is at position 257), so that check passes before the wrong deposit block is resolved.

## Impact Explanation
This is a **Critical** impact: **consensus deviation**. A miner including a crafted DAO withdrawal transaction in a block will have that block accepted by C-VM-based nodes (on-chain DAO script satisfied) and rejected by Rust nodes whose `FeeCalculator` calls `DaoCalculator::transaction_fee`, which returns `DaoError::InvalidOutPoint`. The network forks. This directly matches the allowed impact class: "Vulnerabilities which could easily cause consensus deviation."

## Likelihood Explanation
Any DAO depositor — no privilege required — can construct this transaction. The only additional requirement is miner inclusion, which can be incentivized with a high fee. The transaction is structurally valid (correct witness length, valid `header_deps` list, correct lock script), passing all non-contextual checks. The discrepancy is triggered purely by choosing `header_dep_index > 255` with a specific `header_deps` layout. The attack is repeatable and deterministic.

## Recommendation
In `util/dao/src/lib.rs`, after parsing the `u64` index, validate it fits within `u8` to match the C VM's behavior:

```rust
let index = LittleEndian::read_u64(&header_deps_index_data.unwrap());
if index > u8::MAX as u64 {
    return Err(DaoError::InvalidDaoFormat);
}
```

Alternatively, update the on-chain C DAO script to read the full `u64`. Either fix must be applied consistently to both the Rust node and the on-chain script.

## Proof of Concept
The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` at `util/dao/src/tests.rs:475–537` directly demonstrates the Rust side of the split:

1. Build deposit block (number 100) and withdraw block (number 200), insert both into store.
2. Construct 258-entry `header_deps`: `header_deps[1]` = deposit block hash, `header_deps[257]` = withdraw block hash.
3. Set `input_type = 257u64` (LE: `[0x01, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`).
4. Call `DaoCalculator::transaction_fee(&rtx)`.
5. Rust reads index 257 → withdraw block (number 200) → `deposit_header.number()` (200) ≠ `deposited_block_number` (100) → `DaoError::InvalidOutPoint`.
6. The test asserts `result.is_err()` — confirmed.
7. The C VM reads lowest byte = 1 → deposit block (number 100) → check passes → accepts.
8. A miner broadcasting a block with this transaction causes a chain split. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
```
