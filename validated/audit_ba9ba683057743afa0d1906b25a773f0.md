### Title
Decompressed-size not bounded by per-protocol `max_frame_length` allows remote memory exhaustion â€” (`network/src/compress.rs`)

### Summary

`LengthDelimitedCodecWithCompress::decode` enforces `max_frame_length` only on the **wire (compressed) frame size**. After that check passes, the decompressed allocation is bounded solely by the global `MAX_UNCOMPRESSED_LEN` (8 MB). An unprivileged remote peer can craft a snappy frame whose wire size fits within any protocol's `max_frame_length` but whose snappy varint header claims a decompressed size just below 8 MB, causing an 8 MB heap allocation per message, multiplied across all open sessions.

---

### Finding Description

The decode path in `LengthDelimitedCodecWithCompress::decode`:

1. **Wire-size check** â€” `self.length_delimited.decode(src)?` rejects frames whose compressed wire size exceeds `max_frame_length`. [1](#0-0) 

2. **Decompressed-size check** â€” only rejects if `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` (8 MB). There is **no comparison against `max_frame_length`**. [2](#0-1) 

3. **Allocation** â€” `BytesMut::zeroed(decompressed_bytes_len)` allocates up to 8 MB **before** decompression is attempted or validated. [3](#0-2) 

The per-protocol limits are:

| Protocol | `max_frame_length` | Max allocation per message |
|---|---|---|
| Ping | 1 024 B (1 KB) | 8 MB |
| Feeler | 1 024 B | 8 MB |
| DisconnectMessage | 1 024 B | 8 MB |
| Time | 1 024 B | 8 MB |
| Identify | 2 048 B | 8 MB |
| Sync | 2 MB | 8 MB |
| RelayV3 | 4 MB | 8 MB | [4](#0-3) 

The snappy format stores the uncompressed length as a varint at the very beginning of the compressed stream. An attacker can craft a byte sequence where:
- The varint header encodes `MAX_UNCOMPRESSED_LEN - 1` (â‰ˆ 8 MB)
- The remaining bytes are minimal (a few bytes of valid or invalid snappy literal data)
- Total wire size stays below `max_frame_length`

`decompress_len` reads only the varint header and returns the claimed size. The code then allocates that many bytes. If decompression subsequently fails, the error is returned â€” but the allocation already occurred and must be freed by the allocator. [5](#0-4) 

---

### Impact Explanation

With `N` concurrent peer sessions each sending crafted frames at high rate:

- **Peak memory per session** â‰ˆ 8 MB (for Ping, 8 000Ã— the intended 1 KB limit)
- **Peak memory for 100 sessions** â‰ˆ 800 MB
- **Peak memory for 1 000 sessions** â‰ˆ 8 GB

CKB nodes accept inbound connections from any peer. The default max peer count is in the hundreds. Even at modest session counts this can OOM the node process, causing a crash or severe degradation. The attacker does not need any stake, key, or PoW â€” only the ability to open TCP connections to the node's P2P port.

---

### Likelihood Explanation

- The P2P port is publicly reachable on all full nodes.
- No authentication or rate-limiting on the compressed-frame path prevents this.
- Crafting a snappy varint header with a large claimed size is trivial (a few bytes).
- The `CKBProtocol::build` path wires `LengthDelimitedCodecWithCompress` for **all** protocols that set `compress: true` (the default). [6](#0-5) 
- The attack is amplified by protocols with small `max_frame_length` (Ping, Feeler, Time, DisconnectMessage) where the compression ratio required is enormous but the varint trick bypasses any ratio check entirely.

---

### Recommendation

After the wire-size check passes and the compress flag is detected, add a second guard before allocating:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {
    return Err(io::ErrorKind::InvalidData.into());
}
// ADD: also enforce the per-protocol limit
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
    return Err(io::ErrorKind::InvalidData.into());
}
let mut buf = BytesMut::zeroed(decompressed_bytes_len);
```

This ensures the decompressed allocation is bounded by the same limit that governs the wire frame, making the invariant hold end-to-end. [7](#0-6)

### Proof of Concept

```rust
// Craft a snappy frame for the Ping protocol (max_frame_length = 1024)
// Wire size: ~10 bytes; claimed decompressed size: 8MB - 1

fn craft_malicious_frame() -> Vec<u8> {
    let mut frame = vec![0b1000_0000u8]; // compress flag byte
    // snappy varint: encode (8MB - 1) = 8388607
    // varint encoding of 8388607: 0xFF, 0xFF, 0xFF, 0x03
    frame.extend_from_slice(&[0xFF, 0xFF, 0xFF, 0x03]);
    // minimal snappy literal tag: 1 byte literal "A"
    frame.extend_from_slice(&[0x00, 0x41]); // literal len=1, data=0x41
    frame // total: 7 bytes, well within Ping's 1024-byte limit
}

// decompress_len(&frame[1..]) returns 8_388_607
// BytesMut::zeroed(8_388_607) allocates ~8MB
// decompress() fails â†’ Err returned, but 8MB was allocated
```

Send this frame repeatedly across hundreds of sessions to exhaust node memory.
