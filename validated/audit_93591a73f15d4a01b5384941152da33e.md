## Vulnerability Analysis

### The `take(len / 2)` Integer Division Bug

The core logic in `check_purge` is:

```rust
let len = peers_by_network_group.len();
// ...
peers
    .into_iter()
    .take(len / 2)   // integer division: if len=1, take(0) → empty
    .flat_map(move |addrs| {
        if addrs.len() > 4 { Some(...) } else { None }
    })
    .flatten()
    .collect()
// ...
if candidate_peers.is_empty() {
    return Err(PeerStoreError::EvictionFailed.into());
}
``` [1](#0-0) 

When `len = 1` (all addresses in one `/16` group), `1 / 2 = 0` in integer arithmetic, so `take(0)` produces an empty iterator, `candidate_peers` is empty, and `Err(EvictionFailed)` is returned unconditionally.

### Attack Path Is Concrete

**Step 1 — Entry point:** The discovery protocol's `add_new_addrs` calls `peer_store.add_addr()` for every address received from any connected peer, with no per-session rate limit on the total number of addresses injected. [2](#0-1) 

**Step 2 — Filling the store:** `ADDR_COUNT_LIMIT = 16384`. A single connected attacker peer sends `Nodes` messages (up to 1000 addresses per non-announce message, plus unlimited announce messages of 10 each) with addresses all in the same `/16` subnet (e.g., `1.2.x.x`). Each address needs a distinct multiaddr key — the attacker uses different peer IDs or ports within `1.2.x.x`. [3](#0-2) 

**Step 3 — Freshly added addresses are always connectable:** `AddrInfo::new(addr, 0, score, flags)` sets `last_connected_at_ms=0`, `attempts_count=0`. With those values, `is_connectable` returns `true` (neither the `ADDR_MAX_RETRIES` nor `ADDR_MAX_FAILURES` threshold is reached). [4](#0-3) 

**Step 4 — `check_purge` fails permanently:** Once the store hits 16384 entries all in `Group::IP4([1, 2])`:
- Phase 1 (evict non-connectable): finds nothing, `candidate_peers` is empty.
- Phase 2 (group eviction): `len = 1`, `take(1/2)` = `take(0)` → empty → `Err(EvictionFailed)`. [5](#0-4) 

**Step 5 — `add_addr` propagates the error:** The `?` operator in `add_addr` returns the error to all callers — discovery, identify protocol, and feeler connections all fail to add new addresses. [6](#0-5) 

### What the Existing Test Misses

The existing `test_eviction` test always seeds the store with **4 distinct network groups** (`225.0.x.x`, `192.163.x.x`, `255.255.x.x`, IPv6), so `len = 4`, `take(2)` works. The single-group case is never tested. [7](#0-6) 

### Impact Correction

The vulnerability is real and reachable, but the impact claim of **"consensus deviation"** is overstated. The actual impact is:

- **Peer store permanently poisoned**: no new addresses can be added via discovery or identify protocol.
- **Node cannot discover honest peers**: `fetch_addrs_to_feeler` and `fetch_addrs_to_attempt` will only return attacker-controlled addresses from `1.2.x.x`.
- **Existing connections are unaffected**: `connected_peers` is a separate structure; the node keeps its current sessions.
- **Inbound connections still accepted**: `PeerRegistry::accept_peer` is not blocked.

This is a **peer discovery DoS / peer store poisoning**, not full network isolation or consensus deviation. The node degrades to only connecting to attacker-controlled addresses for new outbound connections, which can lead to eclipse-like conditions over time as existing connections cycle out.

### Title
Peer Store Permanently Poisoned via Single-Subnet Discovery Flood — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
Integer division `take(len / 2)` in `check_purge` produces `take(0)` when all 16384 peer store entries belong to a single `/16` network group, causing `Err(EvictionFailed)` on every subsequent `add_addr` call. An unprivileged remote peer can trigger this by flooding the discovery protocol with addresses from one `/16` subnet.

### Finding Description
`check_purge` (lines 327–404, `peer_store_impl.rs`) has two eviction phases. Phase 2 groups addresses by `/16` network segment and calls `.take(len / 2)` where `len` is the number of distinct groups. When `len = 1`, integer division yields 0, the iterator is empty, `candidate_peers` is empty, and `Err(PeerStoreError::EvictionFailed)` is returned. This error propagates through `add_addr` (line 75) to all callers including `DiscoveryAddressManager::add_new_addrs` and `IdentifyCallback::add_remote_listen_addrs`.

### Impact Explanation
After the attack, the node's peer store is full of attacker-controlled addresses. All calls to `add_addr` return `Err(EvictionFailed)`. The node cannot learn about honest peers via discovery or identify. As existing connections cycle out, the node's outbound connection pool will be drawn exclusively from attacker-controlled `1.2.x.x` addresses, enabling a slow eclipse attack.

### Likelihood Explanation
Requires one successful inbound or outbound connection to the victim. The attacker sends ~17 non-announce `Nodes` messages (1000 addresses each) with distinct peer IDs all in `1.2.x.x`. No PoW, no privileged access, no Sybil majority required. The `is_valid_addr` filter in `DiscoveryAddressManager` only checks `is_reachable`, which passes for public IPs.

### Recommendation
Replace `take(len / 2)` with `take((len + 1) / 2)` or `take(len.max(1))` to ensure at least one group is considered when `len >= 1`. Additionally, add a per-group cap (e.g., max 256 addresses per `/16`) at insertion time in `add_addr` to prevent a single subnet from monopolizing the store.

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {
                            Some(
                                addrs
                                    .iter()
                                    .choose_multiple(&mut rand::thread_rng(), 2)
                                    .into_iter()
                                    .map(|addr| addr.addr.clone())
                                    .collect::<Vec<Multiaddr>>(),
                            )
                        } else {
                            None
                        }
                    })
                    .flatten()
                    .collect()
            };

            for key in candidate_peers.iter() {
                self.addr_manager.remove(key);
            }

            if candidate_peers.is_empty() {
                return Err(PeerStoreError::EvictionFailed.into());
            }
        }
        Ok(())
    }
