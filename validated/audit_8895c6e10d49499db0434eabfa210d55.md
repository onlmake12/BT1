### Title
Undocumented and Unenforced Fee Accounting Invariant Between `txs_fee` and `miner_reward` Components in `get_block_economic_state` — (`rpc/src/module/chain.rs`, `util/reward-calculator/src/lib.rs`)

---

### Summary

The `get_block_economic_state` RPC exposes `txs_fee` (total fees committed in a block) alongside `miner_reward.committed` and `miner_reward.proposal`. These three fields originate from **different transaction sets** and have no documented or enforced invariant relationship. Specifically, `miner_reward.committed + miner_reward.proposal` can significantly exceed `txs_fee` — a counterintuitive property that is never stated in code or documentation. Additionally, the `proposal_reward` calculation in `RewardCalculator` uses `zip` to pair transaction IDs with stored fees without any length-equality assertion, creating a silent truncation path that produces incorrect (zero) proposal rewards if the two vectors diverge.

---

### Finding Description

**Component 1 — `get_block_economic_state` invariant gap**

`get_block_economic_state` in `rpc/src/module/chain.rs` constructs a `BlockEconomicState` from two independent computations:

```
txs_fee  = Σ block_ext.txs_fees          (100 % of fees committed in target block)
miner_reward.committed = RewardCalculator::txs_fees()   (≈ 60 % of same fees)
miner_reward.proposal  = RewardCalculator::proposal_reward()  (40 % of fees from OTHER blocks)
``` [1](#0-0) 

`txs_fee` and `miner_reward` are computed independently and then placed side-by-side in the response with no cross-validation. Because `miner_reward.proposal` draws from fees of transactions committed in **other** blocks (those that were merely *proposed* by the target block), the sum `miner_reward.committed + miner_reward.proposal` is not bounded by `txs_fee`. No code comment, no RPC documentation field, and no runtime assertion states this. [2](#0-1) [3](#0-2) [4](#0-3) 

**Component 2 — silent `zip` truncation in `proposal_reward`**

Inside `proposal_reward`, two closures are defined and then zipped:

```rust
let committed_idx_proc = |hash| -> Vec<ProposalShortId> {
    store.get_block_txs_hashes(hash).into_iter().skip(1)…
};
let txs_fees_proc = |hash| -> Vec<Capacity> {
    store.get_block_ext(hash).expect("block ext stored").txs_fees
};
…
for (id, tx_fee) in committed_idx.into_iter().zip(txs_fees_proc(&index.hash()).iter()) {
``` [5](#0-4) 

The comment at line 202 explicitly states the two iterators **must** be the same length and order, but no assertion enforces this. `txs_fees` is populated in `insert_ok_ext` only when `cache_entries` is `Some`:

```rust
if let Some(entries) = cache_entries {
    let (txs_fees, cycles) = entries.iter().map(|e| (e.fee, e.cycles)).unzip();
    ext.txs_fees = txs_fees;
}
``` [6](#0-5) 

When `cache_entries` is `None` (the `switch.disable_all()` branch at line 723), `txs_fees` remains `vec![]` while `committed_idx` has `N` entries. The `zip` silently produces zero iterations, and the proposal reward for every transaction committed in that block is computed as zero. [7](#0-6) 

The `BlockExt` struct documents the required length relationship but provides no runtime enforcement: [8](#0-7)

### Impact Explanation

**RPC-level accounting inconsistency (Component 1):** Any RPC caller — exchange, wallet, block explorer, or analytics tool — that reads `get_block_economic_state` and assumes `miner_reward.committed + miner_reward.proposal ≤ txs_fee` will silently compute incorrect fee balances. A target block that proposes many high-fee transactions (committed in later blocks) can have `miner_reward.proposal` orders of magnitude larger than `txs_fee`. For example: target block commits 1 tx with fee 100 shannons and proposes 10 txs each with fee 1 000 shannons committed elsewhere → `txs_fee = 100`, `miner_reward.committed ≈ 60`, `miner_reward.proposal = 4 000`. The total miner fee reward (4 060) is 40× the reported `txs_fee`.

**Silent zero proposal reward (Component 2):** If any block in the chain was stored with an empty `txs_fees` (e.g., via the `disable_all` switch or a future code path), every subsequent `proposal_reward` calculation that references that block silently computes zero reward for all transactions committed in it. This produces an incorrect (lower) total block reward, which the `RewardVerifier` would then enforce as the canonical reward, permanently under-paying the miner whose block proposed those transactions. [9](#0-8) 

---

### Likelihood Explanation

**Component 1** is reachable by any unprivileged RPC caller invoking `get_block_economic_state` on any finalized block where the target block proposed transactions committed elsewhere. This is the normal operation of the two-phase commit protocol and occurs on every active chain. The invariant violation is structural and permanent.

**Component 2** is reachable when `switch.disable_all()` is true (e.g., `--skip-verify` import mode or testing). While not the default production path, the absence of a length assertion means any future code change that introduces a `None` cache-entries path would silently corrupt proposal reward accounting with no error or warning.

---

### Recommendation

1. **Document the invariant explicitly** in `BlockEconomicState` and the `get_block_economic_state` RPC documentation: state that `miner_reward.committed + miner_reward.proposal` is **not** bounded by `txs_fee` because the proposal reward originates from fees of transactions committed in other blocks.

2. **Add a runtime assertion** in `proposal_reward` that `committed_idx.len() == txs_fees.len()` before the `zip`, analogous to the existing comment at line 202:
   ```rust
   assert_eq!(committed_idx.len(), txs_fees.len(),
       "committed_idx and txs_fees must have the same length");
   ```

3. **Enforce the `BlockExt` length invariant** in `insert_ok_ext`: assert that when `cache_entries` is `Some`, `entries.len() == block.transactions().len() - 1`.

4. **Consider adding a `total_miner_fee_reward` field** to `BlockEconomicState` that equals `miner_reward.committed + miner_reward.proposal`, making the actual miner payout from fees unambiguous.

---

### Proof of Concept

**Invariant violation (Component 1):**

```
Block T (target):
  - Commits tx_A with fee = 100 shannons
  - Proposes tx_B, tx_C, …, tx_K (10 txs), each with fee = 1 000 shannons
    (these are committed in blocks T+2 … T+10, within the proposal window)

get_block_economic_state(T):
  txs_fee              = 100          (fees committed IN block T)
  miner_reward.committed = 60         (60% of 100)
  miner_reward.proposal  = 4 000      (40% of 10 × 1 000, from other blocks)

  committed + proposal = 4 060  >>  txs_fee = 100
```

No code path checks or documents this relationship. An RPC consumer that computes `assert(committed + proposal <= txs_fee)` will fail or silently produce wrong balances.

**Silent truncation (Component 2):**

```
Block B processed with switch.disable_all() = true:
  insert_ok_ext(…, cache_entries = None, …)
  → ext.txs_fees = vec![]   (empty, not populated)
  → stored in DB

Later, proposal_reward() for a block that proposed txs committed in B:
  committed_idx = [tx1_id, tx2_id, tx3_id]   (len = 3)
  txs_fees      = []                           (len = 0, from block_ext)
  zip produces 0 iterations → reward += 0 for all 3 txs
  → proposal_reward = 0 (should be 40% of tx1_fee + tx2_fee + tx3_fee)
``` [5](#0-4) [6](#0-5) [1](#0-0) [10](#0-9)

            .checked_add(proposal_window.closest())

        // Transaction can be committed at height H(c): H(c) > H(w_close)

            block_number.saturating_sub(proposal_window.length()),
            closest_start,

        // same order, the order of transactions in block.
        let committed_idx_proc = |hash: &Byte32| -> Vec<ProposalShortId> {
            store
                .get_block_txs_hashes(hash)
                .into_iter()
                .skip(1)

                .zip(txs_fees_proc(&index.hash()).iter())

        let cellbase = &self.resolved[0];
        let no_finalization_target =
            (self.parent.number() + 1) <= self.context.consensus.finalization_delay_length();

        let (target_lock, block_reward) = self.context.finalize_block_reward(self.parent)?;
        let output = CellOutput::new_builder()

            .lock(target_lock.clone())
            .build();

        if no_finalization_target || insufficient_reward_to_create_cell {
            let ret = if cellbase.transaction.outputs().is_empty() {
                Ok(())
            } else {
                Err((CellbaseError::InvalidRewardTarget).into())
            };
            return ret;
        }

        if !insufficient_reward_to_create_cell {
            if cellbase.transaction.outputs_capacity()? != block_reward.total {
                return Err((CellbaseError::InvalidRewardAmount).into());
            }
            if cellbase
                .transaction
                .outputs()
                .get(0)
                .expect("cellbase should have output")
                .lock()
                != target_lock
            {
                return Err((CellbaseError::InvalidRewardTarget).into());
            }
        }

        Ok(())
    }
