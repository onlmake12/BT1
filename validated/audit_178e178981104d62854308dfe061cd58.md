### Title
Hardcoded `signature_set` keypair blocklist in `PostVaa::constraints` is trivially bypassable via a fresh keypair — (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_vaa.rs`)

### Summary
`INVALID_SIGNATURE_SET_KEYS` blocks 16 specific `signature_set` keypairs from being used in `PostVaa`. Because `signature_set` is a free keypair account (not a PDA), an attacker can create a new keypair, populate it with the same valid guardian signatures for a blocked message hash via `VerifySignatures`, and call `PostVaa` — bypassing the blocklist entirely.

### Finding Description
`PostVaa::constraints` checks whether the `signature_set` account's pubkey is in a 16-entry hardcoded string array:

```rust
require!(
    !INVALID_SIGNATURE_SET_KEYS.contains(&signature_set.key().to_string().as_str()),
    CoreBridgeError::InvalidSignatureSet
);
```

The `signature_set` account is declared in `VerifySignatures` as `init_if_needed` with no PDA seeds — it is a plain keypair account. There is no on-chain binding between a keypair and a message hash. The `posted_vaa` PDA is seeded by `message_hash` (not by the `signature_set` pubkey), so any `signature_set` carrying the same `message_hash` will target the same PDA.

An attacker generates a fresh keypair K, calls `VerifySignatures` with K and the real guardian signatures for the blocked VAA (which are public), then calls `PostVaa(signature_set=K)`. The blocklist check passes because K is not one of the 16 entries.

### Impact Explanation
If the 16 blocked signature_sets correspond to VAAs whose `PostedVaaV1` accounts do not yet exist (i.e., `PostVaa` was never successfully called for those message hashes — which is precisely the scenario the blocklist was designed to prevent), an attacker can force-post those VAAs. Depending on the VAA content, this could mean:
- Re-executing blocked governance actions (guardian set updates, contract upgrades, parameter changes).
- Posting stale or otherwise-rejected price update VAAs that downstream consumers accept.

### Likelihood Explanation
The attack requires only: (1) knowledge of the original VAA bytes for a blocked hash (public, available from the Wormhole guardian API), and (2) the ability to submit two unprivileged transactions (`VerifySignatures` + `PostVaa`). No privileged access, no key compromise, no brute force. Likelihood is high given the information is public.

### Recommendation
The blocklist approach is architecturally unsound for this use case. The correct fix is to block at the **message hash** level, not the keypair level. Options:
1. Maintain a denylist of `MessageHash` values and check `signature_set.message_hash` instead of `signature_set.key()`.
2. Pre-initialize the `PostedVaaV1` PDA for each blocked message hash with a sentinel value, causing the `init` constraint to fail for any future `PostVaa` call targeting that hash.
3. Derive `signature_set` as a PDA from the message hash, making it impossible to create a second signature_set for the same hash.

### Proof of Concept
```
1. Obtain VAA bytes for any of the 16 blocked message hashes from the Wormhole guardian API.
2. Generate fresh Solana keypair K.
3. Submit tx: [secp256k1_verify(guardian_sigs, blocked_hash), VerifySignatures(signature_set=K, guardian_set=<correct index>)]
   → K.sig_verify_successes now has quorum-many `true` entries; K.message_hash = blocked_hash.
4. Submit tx: PostVaa(signature_set=K, args=<VAA body matching blocked_hash>)
   → INVALID_SIGNATURE_SET_KEYS.contains(K) == false → passes
   → num_verified() >= quorum → passes
   → PostedVaaV1 PDA for blocked_hash is created.
5. Assert the PostedVaaV1 account exists on-chain.
```
