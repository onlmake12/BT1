### Title
Stale Pool-Size Write-Back in `add_entry` Inflates `total_tx_size`/`total_tx_cycles` After Ancestor Evictions — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new running totals (`total_tx_size`, `total_tx_cycles`) are computed from the **pre-eviction** state and then unconditionally written back **after** `check_and_record_ancestors` has already decremented those same counters through ancestor evictions. The result is that every eviction triggered during ancestor-limit enforcement is silently un-done, permanently inflating the pool's size and cycle counters. This is the direct CKB analog of the ERC1155Supply TOCTOU: a running total is read before a mutating operation updates it, and the stale value is written back afterward.

---

### Finding Description

`PoolMap::add_entry` (`tx-pool/src/component/pool_map.rs`, lines 200–221) follows this sequence:

```
1. (line 210-211) total_tx_size  = self.total_tx_size  + entry.size    // snapshot BEFORE evictions
                  total_tx_cycles = self.total_tx_cycles + entry.cycles

2. (line 213)     evicts = self.check_and_record_ancestors(&mut entry)
                  // may call remove_entry_and_descendants → remove_entry
                  // → update_stat_for_remove_tx, which DECREMENTS
                  //   self.total_tx_size  and self.total_tx_cycles

3. (line 218-219) self.total_tx_size  = total_tx_size   // OVERWRITES the decremented value
                  self.total_tx_cycles = total_tx_cycles // with the pre-eviction snapshot
``` [1](#0-0) 

`check_and_record_ancestors` evicts "cell-ref parent" transactions when the incoming transaction would exceed `max_ancestors_count`: [2](#0-1) 

Each eviction calls `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles`: [3](#0-2) 

But the write-back at lines 218–219 uses the snapshot taken **before** those decrements, so every byte and cycle belonging to evicted transactions is added back into the counters. The correct post-eviction value should be `(original_total − evicted_sizes + entry.size)`, but the actual written value is `(original_total + entry.size)`.

The `updated_stat_for_add_tx` helper that produces the stale snapshot: [4](#0-3) 

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict further transactions:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size { … evict … }
``` [5](#0-4) 

`limit_size` is called immediately after `_submit_entry` in `submit_entry`: [6](#0-5) 

With inflated counters:

1. **Cascading over-eviction**: `limit_size` sees a pool that appears larger than it is and evicts additional legitimate transactions that should have been kept.
2. **Persistent `PoolIsFull` rejections**: Every subsequent `add_entry` call reads the inflated `total_tx_size` in `updated_stat_for_add_tx`, causing honest transactions to be rejected with `Reject::Full` even when the pool has real capacity.
3. **Inflated `tx_pool_info` RPC output**: `total_tx_size` and `total_tx_cycles` are surfaced directly to callers, producing incorrect observability data. [7](#0-6) 

---

### Likelihood Explanation

The trigger condition — a transaction whose ancestor count exceeds `max_ancestors_count` only because of cell-dep parents — is reachable by any unprivileged `send_transaction` RPC caller or P2P relay peer:

1. Submit a "pivot" transaction T0 that creates a live cell.
2. Submit N transactions T1…TN that each list T0's output as a `cell_dep` (N > `max_ancestors_count`, default 25).
3. Submit T_consume that spends T0's output as an input.

When T_consume enters `add_entry`, `check_and_record_ancestors` finds `ancestors_count > max_ancestors_count` but `ancestors_count − cell_ref_parents.len() ≤ max_ancestors_count`, so it evicts the lowest-fee cell-ref parents. Each eviction decrements the counters, but the write-back at lines 218–219 restores the inflated value. The inflation per attack round equals the total size/cycles of the evicted transactions.

No privileged access, no majority hash power, and no social engineering is required.

---

### Recommendation

Compute the new totals **after** `check_and_record_ancestors` completes, so that any eviction-driven decrements are already reflected:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, Default::default()));
    }
    // Perform evictions first …
    let evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // … then update totals from the already-correct self.total_tx_* values
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size  = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, remove the local snapshot entirely and call `update_stat_for_add_tx` (a mutating variant) only after all evictions are complete.

### Proof of Concept

**Setup** (default `max_ancestors_count = 25`):

1. Mine a block to get a spendable cell C.
2. Submit T0 spending C, producing output O0.
3. Submit 30 transactions T1…T30, each with `cell_dep = O0` (they become cell-ref parents of any future tx that spends O0).
4. Record `pool_info_before = tx_pool_info()` → `total_tx_size = S0`.
5. Submit T_consume spending O0 as an input.
   - `check_and_record_ancestors` finds 31 ancestors (30 cell-ref + T0), exceeds limit 25.
   - It evicts the 6 lowest-fee cell-ref parents (sizes s1…s6), calling `update_stat_for_remove_tx` for each → `self.total_tx_size -= (s1+…+s6)`.
   - Lines 218–219 write back `total_tx_size = S0 + size(T_consume)`, ignoring the decrements.
6. Record `pool_info_after = tx_pool_info()` → `total_tx_size = S0 + size(T_consume)`.
7. Verify: the pool actually contains `S0 − (s1+…+s6) + size(T_consume)` bytes of transactions, but `total_tx_size` reports `S0 + size(T_consume)` — inflated by `s1+…+s6`.
8. Repeat steps 2–7 until `total_tx_size > max_tx_pool_size`; observe that `send_transaction` now returns `PoolIsFull` for honest transactions even though the pool has real capacity.

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
