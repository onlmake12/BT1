### Title
`SinceVerifier::verify_absolute_lock` Uses Current Tip Epoch Instead of Projected Commit Epoch for Epoch-Based Since Locks, Causing Inconsistency with Block-Number Locks - (`verification/src/transaction_verifier.rs`)

---

### Summary

`SinceVerifier::verify_absolute_lock` uses `self.tx_env.epoch()` (the current tip epoch) when checking absolute epoch-based `since` locks, but uses `self.tx_env.block_number(proposal_window)` (the **projected commit block number**, accounting for the proposal window) when checking absolute block-number-based `since` locks. This is the same structural defect as the reference report: the wrong time reference is used in a time-based access control check. The result is that transactions with epoch-based since locks are rejected from the tx-pool until the current epoch reaches the required epoch, and are then committed only after an additional `proposal_window.closest()` blocks — later than intended and inconsistent with block-number-based since locks.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, `SinceVerifier::verify_absolute_lock` handles three metric types. For `SinceMetric::BlockNumber` it correctly uses the **projected commit block number**:

```rust
// Lines 635–639
Some(SinceMetric::BlockNumber(block_number)) => {
    let proposal_window = self.consensus.tx_proposal_window();
    if self.tx_env.block_number(proposal_window) < block_number {
        return Err((TransactionError::Immature { index }).into());
    }
}
```

`block_number(proposal_window)` returns `tip + 1 + proposal_window.closest()` for the `Submitted` phase — the earliest block the tx can actually be committed in.

For `SinceMetric::EpochNumberWithFraction`, however, the code uses `self.tx_env.epoch()`:

```rust
// Lines 641–649
Some(SinceMetric::EpochNumberWithFraction(epoch_number_with_fraction)) => {
    if !epoch_number_with_fraction.is_well_formed_increment() {
        return Err((TransactionError::InvalidSince { index }).into());
    }
    let a = self.tx_env.epoch().to_rational();   // ← current tip epoch, NOT projected commit epoch
    let b = epoch_number_with_fraction.normalize().to_rational();
    if a < b {
        return Err((TransactionError::Immature { index }).into());
    }
}
```

`TxVerifyEnv::epoch()` unconditionally returns `self.epoch` — the raw tip epoch stored at construction time — regardless of the `TxVerifyPhase`:

```rust
// script/src/verify_env.rs lines 116–119
/// The earliest epoch which the transaction will committed in.
pub fn epoch(&self) -> EpochNumberWithFraction {
    self.epoch   // ← always the tip epoch; doc comment is wrong for Submitted/Proposed phases
}
```

The doc comment is incorrect for `Submitted` and `Proposed` phases. The `TxVerifyEnv` struct already provides `epoch_number(proposal_window)` which correctly computes the minimum epoch number after `1 + proposal_window.closest()` blocks (the projected commit epoch), but it is never called in `verify_absolute_lock`.

The same wrong reference appears in `verify_relative_lock` for epoch-based relative locks at lines 692–696:

```rust
let a = self.tx_env.epoch().to_rational();   // ← same issue
let b = info.block_epoch.to_rational()
    + epoch_number_with_fraction.normalize().to_rational();
if a < b {
    return Err((TransactionError::Immature { index }).into());
}
```

---

### Impact Explanation

For **block-number-based** absolute since locks: a transaction with lock `N` can be submitted to the pool when `tip + 1 + proposal_window.closest() >= N`. The tx enters the pool `proposal_window.closest()` blocks before the lock expires and is committed exactly at block `N`.

For **epoch-based** absolute since locks: a transaction with lock epoch `E` can only be submitted when `current_tip_epoch >= E`. The tx then waits another `proposal_window.closest()` blocks before it is committed. The tx is committed at approximately `epoch E + proposal_window_epochs` — **later than intended**. A user who wants their tx committed at the start of epoch `E` cannot submit it early; they must wait until epoch `E` is already the tip epoch, then wait the proposal window on top.

This is the same structural defect as the reference report: the check uses the current time reference (`epoch()`) instead of the future time reference (projected commit epoch), causing the operation to complete later than intended.

---

### Likelihood Explanation

Any unprivileged RPC caller who submits a transaction via `send_transaction` with an absolute epoch-based `since` lock (flag bits `0x2000...`) will encounter this behavior. The `SinceVerifier` is invoked on every tx-pool submission through `time_relative_verify` in `tx-pool/src/process.rs`. No special privileges or unusual configuration are required.

---

### Recommendation

In `verify_absolute_lock`, replace `self.tx_env.epoch()` with a comparison against the projected commit epoch, consistent with how `block_number(proposal_window)` is used for block-number locks. The `TxVerifyEnv::epoch_number(proposal_window)` method already computes the correct projected epoch number and should be used here. The misleading doc comment on `TxVerifyEnv::epoch()` should also be corrected to reflect that it returns the raw tip epoch, not the projected commit epoch.

---

### Proof of Concept

**Setup:** Consensus with `proposal_window.closest() = 2`, current tip at epoch `5/10` (epoch 5, index 5, length 10). A transaction has an absolute epoch since lock of `epoch 5, index 7/10` (rational value 5.7).

**Current behavior (wrong reference):**
- `self.tx_env.epoch().to_rational()` = 5.5 (current tip epoch)
- Required = 5.7
- 5.5 < 5.7 → `Immature` error; tx rejected

**Correct behavior (projected commit epoch):**
- Projected commit epoch ≈ 5.5 + (2 blocks / 10 blocks per epoch) = 5.7
- 5.7 >= 5.7 → tx accepted; committed exactly at epoch 5.7

**Block-number equivalent (for comparison):**
- A tx with absolute block-number lock `N = tip_block + 2` is accepted immediately because `block_number(proposal_window) = tip + 1 + 2 = tip + 3 >= N`. The epoch check does not offer the same early-submission window.

The inconsistency is directly observable by submitting two otherwise-identical transactions — one with an absolute block-number lock and one with an equivalent absolute epoch lock — and noting that the epoch-locked tx is rejected while the block-number-locked tx is accepted.         if since.is_absolute() {
            match since.extract_metric() {
                Some(SinceMetric::BlockNumber(block_number)) => {
                    let proposal_window = self.consensus.tx_proposal_window();
                    if self.tx_env.block_number(proposal_window) < block_number {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                Some(SinceMetric::EpochNumberWithFraction(epoch_number_with_fraction)) => {
                    if !epoch_number_with_fraction.is_well_formed_increment() {
                        return Err((TransactionError::InvalidSince { index }).into());
                    }
                    let a = self.tx_env.epoch().to_rational();
                    let b = epoch_number_with_fraction.normalize().to_rational();
                    if a < b {
                        return Err((TransactionError::Immature { index }).into());
                    }

                    }
                }

                }

        }

    /// The block number of the earliest block which the transaction will committed in.
    pub fn block_number(&self, proposal_window: ProposalWindow) -> BlockNumber {
        match self.phase {
            TxVerifyPhase::Submitted => self.number + 1 + proposal_window.closest(),
            TxVerifyPhase::Proposed(already_proposed) => {
                self.number.saturating_sub(already_proposed) + proposal_window.closest()
            }
            TxVerifyPhase::Committed => self.number,
        }
    }

    /// The epoch number of the earliest epoch which the transaction will committed in.
    pub fn epoch_number(&self, proposal_window: ProposalWindow) -> EpochNumber {
        let n_blocks = match self.phase {
            TxVerifyPhase::Submitted => 1 + proposal_window.closest(),
            TxVerifyPhase::Proposed(already_proposed) => {
                proposal_window.closest().saturating_sub(already_proposed)
            }
            TxVerifyPhase::Committed => 0,
        };
        self.epoch.minimum_epoch_number_after_n_blocks(n_blocks)
    }

    /// The parent block hash of the earliest block which the transaction will committed in.
    pub fn parent_hash(&self) -> Byte32 {
        match self.phase {
            TxVerifyPhase::Submitted => &self.hash,
            TxVerifyPhase::Proposed(_) => &self.hash,
            TxVerifyPhase::Committed => &self.parent_hash,
        }
        .to_owned()
    }

    /// The earliest epoch which the transaction will committed in.
    pub fn epoch(&self) -> EpochNumberWithFraction {
        self.epoch
    }
