### Title
Unconditional Zero-Value ETH Transfer in `_processFeesAndPayKeeper` Can Permanently Block `updatePriceFeeds` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
`Scheduler._processFeesAndPayKeeper` unconditionally calls `msg.sender.call{value: totalKeeperFee}("")` without first checking whether `totalKeeperFee > 0`. When `tx.gasprice == 0` (possible on certain L2s/testnets) and `singleUpdateKeeperFeeInWei == 0` (admin-configurable), `totalKeeperFee` evaluates to exactly zero. A keeper implemented as a contract whose `receive()` function reverts on zero-value ETH transfers will cause the entire `updatePriceFeeds` call to revert, permanently blocking that keeper from updating any subscription's price feeds.

### Finding Description

In `Scheduler.sol`, `_processFeesAndPayKeeper` computes the keeper reward as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [1](#0-0) 

When both `tx.gasprice == 0` and `singleUpdateKeeperFeeInWei == 0`, `totalKeeperFee` is exactly `0`. The balance check `status.balanceInWei < totalKeeperFee` trivially passes (0 ≤ any balance), and the unconditional transfer is then executed:

```solidity
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
if (!sent) {
    revert SchedulerErrors.KeeperPaymentFailed();
}
``` [2](#0-1) 

If `msg.sender` is a keeper contract whose `receive()` reverts on zero-value ETH (a known pattern used by contracts that enforce non-trivial deposits), `sent` is `false` and `KeeperPaymentFailed` is thrown. Because `_processFeesAndPayKeeper` is called at the end of `updatePriceFeeds`, the entire transaction reverts, including all price-feed state writes that preceded it. [3](#0-2) 

### Impact Explanation

Any keeper contract that guards its `receive()` against zero-value ETH is permanently unable to call `updatePriceFeeds` under the described conditions. All price-feed state updates for every subscription are rolled back on each attempt, effectively freezing on-chain price data for consumers of those subscriptions. The subscription balance is also never deducted, so the funds are not lost, but the service is completely non-functional for the affected keeper.

### Likelihood Explanation

Two conditions must coincide:

1. **`singleUpdateKeeperFeeInWei == 0`** — This is an admin-settable parameter (`_state.singleUpdateKeeperFeeInWei`). An admin may legitimately set it to zero to reduce keeper overhead or during initial deployment.
2. **`tx.gasprice == 0`** — Possible on certain EVM-compatible L2s, testnets, or networks where the base fee is zero. The Scheduler is a multi-chain contract deployed across Pyth's supported networks.

When both hold, any keeper contract with a non-trivial `receive()` is permanently blocked. The likelihood is low-to-medium given the multi-chain deployment surface and the admin-configurable fee.

### Recommendation

Add a zero-value guard before the ETH transfer, mirroring the fix pattern from the referenced Cozy report:

```solidity
if (totalKeeperFee > 0) {
    (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
    if (!sent) {
        revert SchedulerErrors.KeeperPaymentFailed();
    }
}
``` [4](#0-3) 

### Proof of Concept

1. Admin deploys `SchedulerUpgradeable` with `singleUpdateKeeperFeeInWei = 0`.
2. Deploy a keeper contract `MaliciousKeeper` with:
   ```solidity
   receive() external payable { require(msg.value > 0, "no zero ETH"); }
   ```
3. On a network where `tx.gasprice == 0`, `MaliciousKeeper` calls `updatePriceFeeds(subscriptionId, updateData)`.
4. All price-feed writes succeed, then `_processFeesAndPayKeeper` computes `totalKeeperFee = 0`.
5. `call{value: 0}("")` to `MaliciousKeeper` triggers its `receive()`, which reverts.
6. `sent == false` → `KeeperPaymentFailed` is thrown → entire `updatePriceFeeds` reverts.
7. No price feed is ever updated for this keeper, regardless of how many times it retries. [4](#0-3)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
