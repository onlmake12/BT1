Audit Report

## Title
Remote peer forces full `max_block_cycles` VM execution per verify worker via declared-cycles off-by-one — (`tx-pool/src/process.rs`, `tx-pool/src/util.rs`)

## Summary
The relay guard in `TransactionsProcess::execute` rejects only `declared_cycles > max_block_cycles`, admitting the exact maximum value. Inside `_process_tx`, `max_cycles` is set directly from `declared_cycles` with no secondary cap from `max_tx_verify_cycles`, so a remote peer can force each verify worker to run CKB-VM script execution for the full `max_block_cycles` budget before the node detects the mismatch and bans the peer. With N workers, N such transactions saturate the entire verification pipeline at near-zero recurring cost.

## Finding Description

**Guard 1 — relay off-by-one:** The check at `sync/src/relayer/transactions_process.rs` L64–66 uses strict `>`: [1](#0-0) 
`declared_cycles == max_block_cycles` is admitted.

**Guard 2 — no secondary cap in `_process_tx`:** At `tx-pool/src/process.rs` L720, `max_cycles` is set directly from `declared_cycles` with no clamp to `max_tx_verify_cycles`: [2](#0-1) 

**Guard 3 — full VM execution up to `max_cycles`:** `verify_rtx` is called with this uncapped value; on the worker path (`command_rx = Some`), it calls `verify_with_pause(max_tx_verify_cycles, command_rx)` where the parameter named `max_tx_verify_cycles` is actually the `max_block_cycles`-valued `max_cycles` from above: [3](#0-2) 

**Guard 4 — mismatch detected only after execution:** The `DeclaredWrongCycles` rejection is produced only after `verify_rtx` returns: [4](#0-3) 

**Guard 5 — ban fires after CPU cost is paid:** `ban_malformed` is called in `after_process` only after `_process_tx` has already returned: [5](#0-4) 

**`max_tx_verify_cycles` is routing-only:** The config is consumed solely as `large_cycle_threshold` in the verify queue to select which worker class handles the tx — it never caps `max_cycles` for execution: [6](#0-5) 

## Impact Explanation

Each malicious transaction monopolizes one verify worker for the maximum possible verification duration (`max_block_cycles` = 3.5 billion cycles on mainnet). With the default worker count of `3/4 * cpu_cores`, a number of simultaneous transactions equal to the worker count saturates the entire verification pipeline, halting admission of legitimate transactions. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation

The attacker needs one valid UTXO locked by a high-cycle script (a one-time on-chain cost). The `unknown_tx_hashes` precondition is trivially satisfied: announce the tx hash via `RelayTransactionHashes`, wait for the node to issue `GetRelayTransactions`, then deliver the tx with `declared_cycles = max_block_cycles`. Because rejected transactions are never confirmed on-chain, the UTXO is never consumed. New transactions with different witnesses or outputs produce different tx hashes, bypassing the `tx_filter` deduplication check at L50. The per-peer ban is circumvented by opening new peer connections or using multiple IPs.

## Recommendation

In `_process_tx` (`tx-pool/src/process.rs` L720), cap `max_cycles` for remote transactions:

```rust
let max_cycles = declared_cycles
    .map(|c| c.min(self.tx_pool_config.max_tx_verify_cycles))
    .unwrap_or_else(|| self.consensus.max_block_cycles());
```

This ensures no single remote transaction can consume more than the configured per-tx cycle budget regardless of what the peer declares. The `max_tx_verify_cycles` config already exists for this purpose but is currently only used for worker routing. [7](#0-6) 

## Proof of Concept

1. Deploy a lock script on-chain that loops for exactly `max_block_cycles - 1` cycles.
2. Create a UTXO locked by this script.
3. Connect to the target node as a peer.
4. Announce the tx hash via `RelayTransactionHashes`; wait for the node to send `GetRelayTransactions`.
5. Send `RelayTransactions` with `declared_cycles = max_block_cycles`.
6. Observe the verify worker blocked for the full `max_block_cycles` duration before emitting `DeclaredWrongCycles`.
7. Repeat with a new tx (different witness/output, same UTXO) from a new peer connection to bypass the ban.
8. With N simultaneous such transactions (N = worker count), measure that legitimate transaction throughput drops to zero.
