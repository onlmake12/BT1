### Title
Unsafe `.transfer()` for ETH Refund in `verifyUpdate` Causes Permanent DoS on zkSync-Compatible Chains — (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` uses `payable(msg.sender).transfer(...)` to refund excess ETH to callers. On zkSync and zkSync-compatible chains (Abstract, Cronos zkEVM) — all of which are explicitly configured in the Pyth deployment infrastructure — the 2300 gas stipend hardcoded by `.transfer()` is insufficient, causing every call that overpays to revert. This makes the Lazer price-verification contract permanently non-functional for any caller that sends more than exactly `verification_fee` wei on those chains.

---

### Finding Description

`PythLazer.verifyUpdate()` is a `payable` function that accepts a fee and refunds any excess:

```solidity
// lazer/contracts/evm/src/PythLazer.sol, lines 70-77
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
        payable(msg.sender).transfer(msg.value - verification_fee);  // ← vulnerable
    }
    ...
}
``` [1](#0-0) 

`.transfer()` forwards exactly 2300 gas. On zkSync Era and its derivatives (Abstract, Cronos zkEVM), gas accounting is dynamic and even a simple ETH transfer to an EOA can consume more than 2300 gas, as documented by zkSync and confirmed by the [921 ETH stuck incident](https://medium.com/coinmonks/gemstoneido-contract-stuck-with-921-eth-an-analysis-of-why-transfer-does-not-work-on-zksync-era-d5a01807227d). The [zkSync best-practices docs](https://docs.zksync.io/build/developer-reference/best-practices#use-call-over-send-or-transfer) explicitly state: *"Avoid using `payable(addr).send(x)/payable(addr).transfer(x)` because the 2300 gas stipend may not be enough."*

The Pyth deployment infrastructure explicitly targets these chains. The hardhat config registers `zkSyncMainnet` (chainId 324), `abstract`, and `cronosZkEvmMainnet` as zkSync-type networks: [2](#0-1) 

The `PythLazerDeploy.s.sol` script uses `CreateX` for deterministic deployment to *any EVM chain*, and the deployment manager script accepts any chain from the store: [3](#0-2) 

Additionally, on any EVM chain, if `msg.sender` is a contract with any logic in its `receive` or `fallback` function (a common pattern for DeFi protocols integrating Lazer), the 2300 gas stipend will be insufficient and the refund will revert.

---

### Impact Explanation

Any caller of `verifyUpdate` who sends `msg.value > verification_fee` on a zkSync-compatible chain will have their transaction unconditionally revert. Since callers cannot reliably predict the exact `verification_fee` at call time (it is owner-settable via storage), overpayment is a normal and expected pattern. This renders `verifyUpdate` — the sole price-verification entry point of the entire Lazer EVM contract — permanently broken on zkSync chains. Downstream consumer contracts that integrate `PythLazer` for price feeds will be unable to verify any Lazer update on these chains.

---

### Likelihood Explanation

- PythLazer is explicitly deployed to any EVM chain via a chain-agnostic deployment script, and zkSync-family chains are already registered in the Pyth deployment infrastructure.
- Any caller (EOA or contract) that sends even 1 wei above `verification_fee` triggers the vulnerable path.
- Consumer contracts integrating Lazer (the primary intended users) routinely use `msg.value` forwarding patterns that may not match the fee exactly.
- The `verification_fee` is mutable by the owner, making exact-fee matching unreliable.

---

### Recommendation

Replace `.transfer()` with a low-level `.call{}` and check the return value:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "ETH refund failed");
}
```

This forwards all available gas and is compatible with zkSync, contract receivers, and standard EVM chains.

---

### Proof of Concept

1. Deploy `PythLazer` to zkSync Era (chainId 324) using the existing `PythLazerDeploy.s.sol` script.
2. Add a trusted signer via `updateTrustedSigner`.
3. Call `verifyUpdate{value: verification_fee + 1}(validUpdate)` from any address.
4. The transaction reverts at `payable(msg.sender).transfer(1)` because the 2300 gas stipend is insufficient on zkSync's dynamic gas model.
5. The caller cannot verify any Lazer price update unless they send exactly `verification_fee` wei — an exact match that is fragile and breaks whenever the owner changes the fee. [4](#0-3) [5](#0-4)
