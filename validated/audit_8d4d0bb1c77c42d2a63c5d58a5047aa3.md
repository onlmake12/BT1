### Title
Panic in `BlockTransactionsVerifier::verify` via Mismatched Compact Block `txs_len` — (`sync/src/relayer/block_transactions_verifier.rs`)

---

### Summary

A malicious P2P peer can crash a CKB full node by exploiting a mismatch between the compact block stored in `pending_compact_blocks` and the `expected_transaction_indexes` computed from a different compact block with the same header hash but a larger `txs_len`. This triggers an unconditional `expect("should never outbound")` panic in `BlockTransactionsVerifier::verify`.

---

### Finding Description

The vulnerability class from the external report — data passed to a decoder that doesn't match the expected structure, causing a hard failure — maps directly to CKB's compact block relay pipeline.

**Root cause: `block_transactions_verifier.rs` line 18**

In `BlockTransactionsVerifier::verify`, the node indexes into `block_short_ids` (derived from the stored compact block) using `expected_transaction_indexes` (derived from a potentially different compact block): [1](#0-0) 

The `expect("should never outbound")` assumes the indexes are always valid for the stored compact block. This assumption is violated when two compact blocks with the same header hash but different `txs_len` values are processed.

**How the mismatch arises:**

In `compact_block_process.rs`, when a second peer sends a compact block for an already-pending block hash, `or_insert_with` keeps the **first** compact block but inserts the **second** peer's missing indexes (computed from the second compact block): [2](#0-1) 

The check that guards against duplicate processing only rejects the **same peer** re-sending the same block hash — a different peer is allowed through: [3](#0-2) 

When the second peer's `BlockTransactions` arrives, `block_transactions_process.rs` calls the verifier with the **first** compact block but the **second** peer's indexes: [4](#0-3) 

**`CompactBlockVerifier::verify` does not prevent this attack.** It only checks internal consistency of a single compact block (cellbase prefilled, no duplicate short IDs, ordered prefilled indexes), not consistency with any previously stored compact block for the same hash: [5](#0-4) 

The `block_short_ids()` helper always returns a `Vec` of exactly `txs_len` elements for the stored compact block: [6](#0-5) 

If the stored compact block has `txs_len = N` and the attacker's compact block has `txs_len = M > N`, the missing indexes computed from the attacker's block include values `>= N`, which are out of bounds for the stored block's `block_short_ids`, triggering the panic.

---

### Impact Explanation

The `expect("should never outbound")` panic at `block_transactions_verifier.rs:18` is an unconditional process abort. Any CKB full node running the relay protocol can be crashed remotely by a single unprivileged P2P peer. This is a **remote denial-of-service** against any node that participates in compact block relay (i.e., all non-IBD full nodes).

---

### Likelihood Explanation

The attack is straightforward:
1. Monitor the P2P network for a compact block with hash `H` and `txs_len = N` being propagated.
2. Connect to the target node.
3. Send a crafted compact block with the same header (same hash `H`) but with `N + K` short IDs (any `K > 0` fake `ProposalShortId` values that pass `CompactBlockVerifier::verify`).
4. The node stores the honest compact block but records the attacker's inflated missing indexes `[1, 2, ..., N+K-1]` for the attacker's peer entry.
5. Send a `BlockTransactions` message for hash `H`.
6. `BlockTransactionsVerifier::verify` accesses index `N` on a `Vec` of length `N` → panic → node crash.

No special privileges, no key material, and no majority hashpower are required. The attacker only needs a standard P2P connection.

---

### Recommendation

Replace the `expect("should never outbound")` with a bounds-checked path that returns a `Status` error instead of panicking:

```rust
let entry = block_short_ids.get(*index as usize);
if entry.is_none() {
    return StatusCode::ProtocolMessageIsMalformed
        .with_context(format!("index {} out of bounds (len {})", index, block_short_ids.len()));
}
```

Additionally, when a second peer sends a compact block for an already-pending hash, the node should verify that the new compact block's `txs_len` matches the stored one, or recompute the missing indexes against the **stored** compact block rather than the newly received one.

---

### Proof of Concept

```
1. Honest peer relays CompactBlock{hash=H, prefilled=[cellbase@0], short_ids=[s1,s2,s3]}
   → txs_len = 4; node stores compact_block_honest; missing=[1,2,3] for honest peer

2. Attacker connects and sends CompactBlock{hash=H, prefilled=[cellbase@0], short_ids=[s1,s2,s3,fake4,fake5,fake6]}
   → txs_len = 7; CompactBlockVerifier::verify passes (valid structure)
   → reconstruct_block fails (fake short IDs not in tx pool)
   → missing=[1,2,3,4,5,6] stored for attacker peer
   → pending_compact_blocks[H] still holds compact_block_honest (txs_len=4)

3. Attacker sends BlockTransactions{block_hash=H, transactions=[]}
   → BlockTransactionsVerifier::verify(compact_block_honest, [1,2,3,4,5,6], [])
   → block_short_ids = compact_block_honest.block_short_ids()  // len = 4
   → block_short_ids.get(4).expect("should never outbound")   // PANIC → node crash
```

    fn block_short_ids(&self) -> Vec<Option<packed::ProposalShortId>> {
        let txs_len = self.txs_len();
        let mut block_short_ids: Vec<Option<packed::ProposalShortId>> = Vec::with_capacity(txs_len);
        let prefilled_indexes = self
            .prefilled_transactions()
            .into_iter()
            .map(|tx_index| tx_index.index().into())
            .collect::<HashSet<usize>>();

        let mut index = 0;
        for i in 0..txs_len {
            if prefilled_indexes.contains(&i) {
                block_short_ids.push(None);
            } else {
                block_short_ids.push(self.short_ids().get(index));
                index += 1;
            }
        }
        block_short_ids
    }
