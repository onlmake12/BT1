Audit Report

## Title
RPC Batch Request Limit Disabled by Default Allows Unbounded CPU Exhaustion via Expensive Calls - (File: rpc/src/server.rs)

## Summary
The CKB JSON-RPC server's batch request size limit is opt-in and disabled by default. `JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is never initialized unless the operator explicitly sets `rpc_batch_limit` in the config, causing the batch size guard to be entirely skipped for default deployments. A local attacker can submit a single HTTP POST containing thousands of expensive calls, causing sustained CPU exhaustion and RPC unresponsiveness within the 30-second timeout window.

## Finding Description
In `rpc/src/server.rs` lines 53–55, `JSONRPC_BATCH_LIMIT` is only initialized when `config.rpc_batch_limit` is `Some(...)`:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

In `handle_jsonrpc` lines 274–282, the guard is conditional on `JSONRPC_BATCH_LIMIT.get()` returning `Some`. When the `OnceLock` is uninitialized (the default), the check is skipped entirely:

```rust
Request::Batch(calls) => {
    if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
        && calls.len() > *batch_size
    { ... }
    // falls through with no limit when OnceLock is None
```

The config field `rpc_batch_limit: Option<usize>` (confirmed at `util/app-config/src/configs/rpc.rs` line 44) has no safe default, and the production template explicitly leaves it commented out (`resource/ckb.toml` lines 205–208) with the comment "By default, there is no limitation on the size of batch request size." Calls are processed sequentially via `stream::iter(calls).then(...)` (lines 284–289), and the `TimeoutLayer` of 30 seconds (lines 125–128) limits request duration.

The `get_raw_tx_pool(verbose=true)` method (`rpc/src/module/pool.rs` lines 703–718) calls `tx_pool.get_all_entry_info()`, which in the tx-pool service (`tx-pool/src/service.rs` lines 1000–1005) acquires a tokio `RwLock` read guard on the entire pool and collects all entry metadata before releasing it. With `max_tx_pool_size = 180_000_000` (180 MB), each call performs a full pool scan under the read lock.

**Correction to the submitted OOM claim**: The report's assertion that 500 calls produce "90 GB of simultaneous memory usage" is incorrect. The batch stream is lazy (`stream::iter(calls).then(...)`) and the response is streamed via `StreamBodyAs::json_array(stream)`, meaning each call's result is produced and streamed before the next call begins. Peak memory is bounded by approximately one pool state at a time (~180 MB), not 500 × 180 MB. The actual impact is CPU exhaustion and repeated read-lock acquisition on the tx pool, not process-level OOM.

## Impact Explanation
**Note (0–500 points). Any local RPC API crash.** An unlimited batch of expensive `get_raw_tx_pool(verbose=true)` calls causes sustained CPU exhaustion and repeated tx-pool read-lock contention within the 30-second timeout window, rendering the local RPC unresponsive for the duration of the request. The sequential streaming architecture prevents the OOM crash claimed in the report; the node process does not crash. The impact is a local RPC denial-of-service, not a node crash.

## Likelihood Explanation
The RPC is bound to `127.0.0.1:8114` by default, restricting the attacker to local access. The bounty scope includes "RPC caller" and "supported local CLI/RPC user" as valid attacker roles. No authentication, no special privileges, and no prior chain knowledge are required. The attack is a single HTTP POST with a crafted JSON array and is trivially repeatable. The production config template explicitly documents that no batch limit is set by default.

## Recommendation
- Change `rpc_batch_limit` from `Option<usize>` to a field with a hardcoded safe default (e.g., 200), enforced unconditionally by initializing `JSONRPC_BATCH_LIMIT` at startup regardless of config.
- Alternatively, initialize `JSONRPC_BATCH_LIMIT` with a safe fallback when `config.rpc_batch_limit` is `None`, rather than skipping the check entirely.
- Emit a startup warning when no batch limit is configured.

## Proof of Concept
```bash
# Build a batch of 500 get_raw_tx_pool(verbose=true) calls
python3 -c "
import json
calls = [{'id': i, 'jsonrpc': '2.0', 'method': 'get_raw_tx_pool', 'params': [True]} for i in range(500)]
print(json.dumps(calls))
" > batch.json

# Send to a node with no rpc_batch_limit configured (default)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  --data @batch.json
# Each call acquires the tx-pool read lock and scans all entries.
# 500 sequential calls within the 30-second window cause sustained CPU load
# and repeated lock contention, making the RPC unresponsive for the duration.
```
