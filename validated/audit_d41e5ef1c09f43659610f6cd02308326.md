### Title
`PythPriceOracleGetter` Returns Raw Price Without Confidence Interval Check, Enabling Bad Debt in Aave Lending Deployments — (`target_chains/ethereum/contracts/contracts/aave/PythPriceOracleGetter.sol`)

---

### Summary

Pyth ships a first-party Aave oracle adapter, `PythPriceOracleGetter`, that fetches a price via `getPriceNoOlderThan()` but **never inspects `price.conf`**. Pyth's own best-practices documentation explicitly states that lending protocols must use `price - confidence` for collateral valuation and must halt activity when `conf/price` exceeds a threshold. The adapter ignores this entirely. When a wrapped or bridged asset (e.g., WBTC, stETH, cbBTC) depegs from its underlying, Pyth publishers diverge and the confidence interval widens dramatically — but the adapter returns the raw mid-price unchanged, giving Aave a falsely high collateral value and allowing unlimited borrowing against a worthless asset.

---

### Finding Description

`PythPriceOracleGetter.getAssetPrice()` is Pyth's production Aave oracle adapter. Its full price-reading logic is:

```solidity
PythStructs.Price memory price = pyth().getPriceNoOlderThan(
    priceId,
    PythAssetRegistry.validTimePeriodSeconds()
);

if (price.price <= 0) {
    revert InvalidNonPositivePrice();
}
uint256 normalizedPrice = uint64(price.price);
```

The returned `PythStructs.Price` struct contains both `price` (the aggregate mid-price) and `conf` (the aggregate confidence interval). The function reads `price.price` and `price.expo` but **never reads `price.conf`**. There is no check of the form `require(price.conf * MAX_CONF_RATIO <= uint64(price.price))` and no circuit-breaker that reverts or pauses when the confidence-to-price ratio is too wide.

Pyth's own documentation states:

> *"For lending protocols, use conservative pricing: Collateral valuation: price - confidence (lower bound) … Pause activity if the confidence-to-price ratio exceeds your threshold (e.g., conf/price > 2%)."*

The adapter violates this guidance in its own implementation.

**Depeg scenario root cause chain:**

1. An Aave deployment uses `PythPriceOracleGetter` and registers WBTC (or any bridged asset) against a price feed ID.
2. The bridge backing WBTC is compromised; WBTC begins trading at a significant discount to BTC on-chain.
3. Pyth publishers observe the divergence across venues and report widely differing prices. The aggregation algorithm produces a large `conf` value reflecting the spread.
4. `getAssetPrice()` is called by Aave's `PriceOracle` during a borrow or liquidation check.
5. The function returns `price.price` — the mid-price, still close to BTC/USD — ignoring the wide `conf` that signals the depeg.
6. Aave values WBTC collateral at near-BTC price; borrowers drain the protocol against worthless collateral.

---

### Impact Explanation

An Aave deployment using `PythPriceOracleGetter` with any wrapped or bridged asset will:

- **Over-value depegged collateral** at the underlying asset's price rather than the actual market price of the wrapped token.
- **Allow unbounded borrowing** against collateral that is effectively worthless, accumulating bad debt.
- **Fail to trigger liquidations** because the oracle still reports a high collateral value, so health factors remain above 1.

This is a direct financial loss to Aave liquidity providers and is the same impact class as M-18.

---

### Likelihood Explanation

- Wrapped/bridged assets (WBTC, cbBTC, stETH, rETH, etc.) are standard Aave collateral types.
- Bridge exploits and depeg events have occurred historically (e.g., Multichain, Ronin, Wormhole).
- Any borrower can exploit this permissionlessly once a depeg begins — no privileged access required.
- The adapter is Pyth's own production code intended for deployment; any Aave fork or new Aave market using it is exposed.

---

### Recommendation

Add a confidence-ratio guard inside `getAssetPrice()`:

```solidity
// Example: revert if conf/price > 2% (configurable)
uint256 maxConfBps = 200; // 2%
require(
    uint256(price.conf) * 10_000 <= uint256(uint64(price.price)) * maxConfBps,
    "PythOracle: confidence too wide"
);
```

Alternatively, return the conservative lower bound for collateral valuation:

```solidity
// Use price - conf as the collateral value
uint256 conservativePrice = uint64(price.price) - price.conf;
```

The `maxConfBps` threshold should be a constructor parameter so deployers can tune it per asset class.

---

### Proof of Concept

1. Deploy `PythPriceOracleGetter` with WBTC mapped to a BTC/USD Pyth price feed ID.
2. Simulate a depeg: call `pyth.updatePriceFeeds()` with a `PriceFeedMessage` where `price = 60_000e8` (BTC price) but `conf = 30_000e8` (50% confidence interval, reflecting a 50% depeg).
3. Call `pythOracleGetter.getAssetPrice(wbtcAddress)`.
4. Observe that the function returns `60_000e8` (normalized) — the full BTC price — with no revert, despite `conf/price = 50%` signaling that WBTC is worth at most ~$30,000.
5. Aave accepts this price for collateral valuation; a borrower can borrow up to the LTV-adjusted value of $60,000 against WBTC that is actually worth $30,000.
